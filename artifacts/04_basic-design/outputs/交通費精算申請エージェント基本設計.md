## 2. 目的・役割定義

### 2.1 エージェントの目的
- 社員から移動情報（移動日、出発地、目的地、交通手段、業務目的）を一括確認で対話収集し、交通費精算申請書（Excel）を自動生成する
- 運賃データから交通費を自動計算し、手動入力による計算ミスを防止する
- 申請期限チェック・上長承認チェックを自動実施し、申請ルールへの準拠を支援する
- Human-in-the-Loop による申請書生成前確認を設け、誤申請を防止する

### 2.2 主な責務
- **移動情報の対話収集**: 移動日、出発地、目的地、交通手段、業務目的を一括確認で収集する。費用は自動計算するため確認しない（計算できない場合に確認する）
- **駅名正規化**: 入力された駅名の表記ゆれを正規化する（「渋谷駅」→「渋谷」）。プロンプト指示によりLLMが実行する
- **交通費自動計算**: 運賃データ（経路テーブル・固定運賃）を検索し、交通費を自動計算する（TOOL-001）
- **申請期限チェック**: 経費発生日（移動日）から3ヶ月以内かを判定し、超過時はエラーを通知する
- **上長承認チェック**: 交通費合計が10,000円を超える場合、上長承認が必要な旨を通知する
- **申請書自動生成**: 収集した情報をテンプレートに入力し、交通費精算申請書（Excel）を生成する（TOOL-002）

### 2.3 非責務（やらないこと）
- **申請種別の判断**: AG-001（申請受付窓口エージェント）が担当
- **申請者名の受付**: main.pyがアプリケーション起動時に取得し、invocation_state経由で受け取る
- **申請書の提出（申請システムへの登録）**: 本システムのスコープ外

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 運賃データ・申請ルールの参照 | 運賃マスタデータ（D-011）、申請ルール（knowledge/） |
| 実行 | 移動情報の対話収集、駅名正規化、交通費自動計算、申請期限チェック、上長承認チェック | 情報収集・計算・判定処理全般 |
| 実行（承認必須） | 申請書ファイルの生成・保存 | TOOL-002（申請書生成ツール）の実行前にHumanApprovalHookで承認を得る |

### 3.2 判断権限の範囲
- **自律判断可能**: 交通費の自動計算（BRL-12）、駅名正規化（BRL-15）、申請期限判定（BRL-08）、上長承認要否判定（BRL-10）
- **人間確認必要**: 申請書生成前の内容確認（OK/修正/キャンセル）、運賃データに該当経路が存在しない場合の手動金額入力
- **実行不可**: 申請書の提出、承認権限の行使

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始 → 移動情報の入力依頼 → 移動情報の受付 → 駅名正規化・交通費計算 → 申請期限チェック → 上長承認チェック → 申請書ドラフト提示（テキスト整理、ツール呼び出しなし） → HumanApprovalHook承認（BeforeToolCallEvent） → 申請書生成・提示 → 終了
```

**ドラフト提示とツール実行の分離**:
- 「申請書ドラフト提示」ステップでは、収集済み申請情報をテキストとして整理・提示する（ツール呼び出しなし）
- 「申請書生成・提示」ステップでは、HumanApprovalHookがBeforeToolCallEventでトリガーし、申請者の承認（OK）後にはじめて申請書生成ツール（TOOL-002）を実行する
- Human-in-the-Loop承認は1回のみ発生し、承認前にツールを実行してはならない

**フローの詳細**: 詳細設計書 2.3.3章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理（CF-003 Step 1〜Step 8の順序で実行）
- **処理をスキップする考え方**: 全ステップ必須。スキップは行わない
- **同一処理を複数回実行**: 修正選択時はStep 1（移動情報の入力依頼）へ戻り、全区間の情報を再収集する

### 4.3 状態管理
- **移動情報リスト（D-007）**: 会話履歴内で保持。各区間の移動日・出発地・目的地・交通手段・費用・業務目的を含む
- **計算済み運賃**: 会話履歴内で保持。TOOL-001の計算結果
- **申請者名・申請日**: invocation_state経由で受け取り、セッション内で参照

## 6. ツール設計

### 6.1 利用ツール

本エージェントが使用するツールは、業務機能（計算・生成）に特化したツールのみとする。汎用ツール（ファイル操作・Web検索等）は使用しない。

- **TOOL-001（交通費計算ツール - calculate_transport_fare）**: 運賃データを検索し、出発地・目的地・交通手段に基づいて交通費を計算する。電車は経路テーブル検索、バス/タクシー/飛行機は固定運賃
- **TOOL-002（申請書生成ツール - generate_transport_report）**: 申請書テンプレート（Excel）に収集した情報を入力し、交通費精算申請書ファイルを生成・保存する

TOOL-001は移動情報の入力受付後に各区間ごとに呼び出す。TOOL-002はユーザーがOKを選択した後（HumanApprovalHook承認後）に1回のみ呼び出す。

**ドラフト提示とツール実行の分離**:
- 申請書生成前に、エージェントは収集済み申請情報をテキストとして整理・提示する（この段階ではツール呼び出しを行わない）
- ドラフト提示後、HumanApprovalHookがBeforeToolCallEventでトリガーし、申請者の承認（OK）後にはじめてTOOL-002を実行する
- 承認前にTOOL-002を実行してはならない。Human-in-the-Loop承認は1回のみ発生する

**ツールの詳細**: 詳細設計書 2.4.2章および各ツールの設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 交通費精算申請エージェントとしての役割・責務を定義する
- 移動情報の一括確認収集フロー（CF-003）を指示する
- 駅名正規化ルール（BRL-15）をLLMに指示する
- 申請期限チェック（BRL-08）の判断基準（現在日付と3ヶ月前の日付）を提供する
- 上長承認チェック（BRL-10）の閾値を提供する

### 7.2 プロンプト構成要素
- **役割定義**: 交通費精算申請書の作成を支援する専門エージェント
- **処理フロー**: CF-003に基づく移動情報収集→計算→チェック→承認→生成の流れ
- **判断プロトコル**: 申請期限チェック（経費発生日から3ヶ月以内）、上長承認チェック（10,000円超）
- **主な責任**: 移動情報の一括確認収集、駅名正規化、交通費自動計算、申請書生成
- **動的情報**: 現在日付（application_date）、3ヶ月前の日付（申請期限の基準日）
- **エラー発生時の振る舞い**: システム系エラー（ツールエラー・システム障害等）が発生した場合、エラー内容を要約して呼び出し元エージェント（AG-001）に返却する

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-003 Step 4で申請期限チェック（経費発生日から3ヶ月以内）をLLMに判断させるため、現在日付と3ヶ月前の日付をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 現在日付（YYYY-MM-DD）、3ヶ月前の日付（YYYY-MM-DD）、申請者名、申請期限（YYYY-MM-DD） |

### 7.4 主要な設計意図
- **駅名正規化のプロンプト指示**: BRL-15をツールではなくプロンプト指示で実現し、LLMが「渋谷駅」→「渋谷」等の正規化を行ってからTOOL-001に渡す
- **申請期限の動的埋め込み**: 現在日付と3ヶ月前の日付をプロンプトに埋め込むことで、LLMが移動日と比較して期限超過を判断できる
- **費用確認の省略**: 費用は自動計算するため確認しない方針をプロンプトで明示し、不要な質問を防止する

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
※初期化時の入力はなし。AG-001からAgent as Toolsとして呼び出される際にinvocation_state経由で情報を受け取る。

### 10.2 対話ループでの入力

- **承認選択**: OK/修正/キャンセルの3択（HumanApprovalHook経由）

### 10.3 invocation_stateの入力
AG-001（申請受付窓口エージェント）から渡される情報:
```python

    "applicant_name": str,      # 申請者名
    "application_date": str,    # 申請日（YYYY-MM-DD）
    "session_id": str           # セッションID
}
```

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
- **ツール名**: transport_agent
- **デコレータ**: @tool(context=True)
- **入力**: query (str), tool_context (ToolContext)

### 13.2 ToolContextの使用
- invocation_stateからapplicant_name、application_date、session_idを取得

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 移動情報の一括確認収集が正しく動作すること、駅名正規化が正しく行われること、交通費自動計算が運賃データに基づき正しい結果を返すこと、申請書が正しく生成されること
- **ドラフト提示テスト**: 申請書生成前に収集済み情報がテキストとして整理・提示されること（ツール呼び出しなし）、HumanApprovalHook承認（OK）後にはじめてTOOL-002が実行されること、承認前にツールが実行されないこと
- **異常系テスト**: 経路不存在時の手動入力案内、申請期限超過時のエラー通知、テンプレート不存在時のエラー処理、LoopLimitError発生時の安全な終了
- **性能テスト**: 複数区間（5区間以上）の移動情報を処理した場合のレスポンス時間

**詳細なテストケース**: 詳細設計書 7章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- CLIインターフェースのみ対応（Web/Slack等のUIは対象外）
- 運賃データはローカルJSONファイルに依存（リアルタイム運賃検索API不使用）
- 申請書フォーマットは固定テンプレート（Excel）に依存

### 16.2 業務的制約
- 対応交通手段は電車・バス・タクシー・飛行機の4種類のみ
- 申請期限は経費発生日（移動日）から3ヶ月以内（BRL-08）
- 上長承認閾値は交通費合計10,000円超（BRL-10）
- 申請書の提出（申請システムへの登録）は本システムのスコープ外

### 16.3 前提条件
- AG-001（申請受付窓口エージェント）から申請種別「交通費精算申請」として委譲されること
- invocation_stateにapplicant_name、application_date、session_idが含まれること
- 運賃データファイル（data/train_routes.json、data/fixed_fares.json）が利用可能であること
- 交通費精算申請書テンプレート（template/交通費精算申請書_template.xlsx）が存在すること
