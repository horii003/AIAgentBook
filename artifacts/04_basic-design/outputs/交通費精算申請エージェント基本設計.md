## 2. 目的・役割定義

### 2.1 エージェントの目的
- 交通費精算申請の移動情報を一括収集し、運賃を自動計算する
- 申請期限チェック・業務目的確認・上長承認要否判断を実施する
- 収集済み情報から交通費精算申請書（Excel）を自動生成する
- 申請書の確認・修正を支援する（提出は人が実行）

### 2.2 主な責務
- **移動情報収集**: 移動区間ごとに出発地・目的地・交通手段・移動日・業務目的を収集する（FR-004, BRL-07）
- **交通費自動計算**: TOOL-001（交通費計算ツール）を呼び出して運賃を自動計算する（FR-005, BRL-08）
- **申請期限チェック**: 経費発生日から 3 ヶ月以内であることを確認する（FR-008, BRL-09）
- **業務目的確認**: 業務目的が記載されていることを確認する（FR-009, BRL-12）
- **不足情報対話収集**: 申請書テンプレートの必須項目と収集済み情報を照合して不足情報を対話で収集する（FR-010）
- **上長承認要否判断**: 交通費合計が 10,000 円を超える場合に上長承認必要と判断する（FR-011, BRL-10, COND-002）
- **申請書自動生成**: TOOL-002（generate_transport_expense_form）を呼び出して交通費精算申請書を生成する（FR-012, BRL-05）
- **申請書確認・修正**: 修正指示に基づいて申請書を再生成する（FR-013）

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001（申請受付窓口エージェント）が担当する
- **経費精算申請書生成**: AG-003（経費精算申請エージェント）が担当する
- **申請書の提出**: BRL-06 に従い、提出は人が実行する

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 運賃データ参照・申請書テンプレート参照 | ACT-READ-03（運賃データ参照）、ACT-READ-04（申請書テンプレート参照） |
| 実行 | 申請期限チェック・上長承認要否判断・交通費計算・申請書生成 | ACT-DEC-02, ACT-DEC-03, ACT-DEC-05, ACT-GEN-01, ACT-GEN-02 |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請期限の適否判断（3 ヶ月以内）、上長承認要否判断（10,000 円超）、運賃計算、不足情報の特定
- **人間確認必要**: 申請書の最終確認・承認（HumanApprovalHook による承認）、申請書の提出
- **実行不可**: 申請書の提出（ACT-EXEC-01）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001 から委譲）
  → 移動情報収集（一区間ずつ）
    → 駅名正規化
    → 交通費計算（TOOL-001）
    → 申請期限チェック（3ヶ月以内）
    → 業務目的確認
  → 不足情報対話収集
  → 上長承認要否判断（10,000円超）
  → 申請書ドラフト提示（ツール呼び出しなし）
    → 収集済み申請情報をテキストとして整理・提示する
  → 申請書生成確認（HumanApprovalHook: BeforeToolCallEvent でトリガー）
    → OK: 申請書生成（TOOL-002: generate_transport_expense_form）
    → 修正: 修正指示受付 → 申請書再生成
    → キャンセル: 生成中止
  → 申請書ファイルパス提示
→ 終了
```

**フローの詳細**: 詳細設計書 エージェント詳細設計を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 移動情報収集 → 交通費計算 → 申請期限チェック → 不足情報収集 → 申請書ドラフト提示 → HumanApprovalHook 承認 → 申請書生成の逐次処理
- **ドラフト提示とツール呼び出しの分離**: 申請書ドラフト提示はテキストとして行い（ツール呼び出しなし）、申請者の承認（OK）後にはじめて TOOL-002 を呼び出す。承認前にツールを実行してはならない
- **HumanApprovalHook の発火タイミング**: TOOL-002 呼び出し直前の BeforeToolCallEvent でトリガーされる。Human-in-the-Loop 承認は 1 回のみ発生する
- **処理をスキップする考え方**: 申請期限超過時は申請不可を案内して処理を終了する（BRL-09）
- **同一処理を複数回実行**: 移動区間が複数ある場合は区間ごとに交通費計算を繰り返す（BRL-07）

### 4.3 状態管理
- **収集済み移動情報**: 会話履歴（SlidingWindowConversationManager, window_size=20）で管理する
- **申請者名・申請日**: invocation_state から取得する（tool_context.invocation_state.get("user_name") 等）
- **申請書ファイルパス**: TOOL-002 の戻り値から取得し、ユーザーへの提示に使用する

## 6. ツール設計

### 6.1 利用ツール

- **calculate_transport_fare（TOOL-001）**: 移動区間ごとに交通費を計算する。電車は電車経路テーブル検索、バス・タクシー・飛行機は固定運賃参照
- **generate_transport_expense_form（TOOL-002 の交通費用関数）**: 収集済み移動情報から交通費精算申請書（Excel）を生成する

各エージェントが使用するツールは業務機能（計算・生成など）に特化したツールのみとする。TOOL-001 は情報収集完了後に各移動区間に対して呼び出す。TOOL-002 は HumanApprovalHook による承認（BeforeToolCallEvent）後に呼び出す。承認前にツールを実行してはならない。

**ツールの詳細**: artifacts/04_basic-design/outputs/交通費計算ツール基本設計.md・申請書生成ツール基本設計.md を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 交通費精算申請の情報収集手順を定義する
- 申請期限チェック・業務目的確認・上長承認要否判断の基準を定義する
- 申請書生成前の確認手順を定義する

### 7.2 プロンプト構成要素
- **役割定義**: 交通費精算申請エージェントとしての役割（情報収集・申請書生成）
- **処理フロー**: 移動情報収集 → 交通費計算 → 申請期限チェック → 申請書ドラフト提示 → HumanApprovalHook 承認 → 申請書生成の流れ
- **判断プロトコル**: 申請期限チェック基準（BRL-09）・上長承認要否判断基準（BRL-10）・業務目的確認（BRL-12）
- **主な責任**: 交通費精算申請書の生成まで。提出は人が実行
- **動的情報**: 申請日（request_date）・申請期限（request_date から 3 ヶ月前の日付）
- **エラー発生時の振る舞い**: ツールエラー・システム障害等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返す

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（BRL-09）で「経費発生日から 3 ヶ月以内」を判断するため、実行時の申請日（request_date）と申請期限（3 ヶ月前の日付）をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 申請者名（applicant_name）、申請日（application_date）、申請期限（deadline: application_date から 3 ヶ月前の日付） |

### 7.4 主要な設計意図
- **動的プロンプト生成**: 申請期限チェックに実行時の日付を使用するため、動的生成関数で申請日・期限日をプロンプトに埋め込む
- **業務ルールの埋め込み**: 申請期限・上長承認閾値等の業務ルールをプロンプトに埋め込み、LLM が正しく判断できるようにする
- **ツール呼び出し指示**: 交通費計算・申請書生成のタイミングをプロンプトで明示する

**プロンプト全文**: 詳細設計書 システムプロンプト詳細設計を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
AG-002 はファクトリ関数（_build_transportation_expense_agent）で生成されるため、初期化時の直接入力はない。

### 10.2 対話ループでの入力
- **移動情報**: 出発地・目的地・交通手段・移動日・業務目的（ユーザーが自然文で入力）
- **修正指示**: 申請書確認後の修正内容（任意）

### 10.3 invocation_stateの入力
AG-001 から渡される情報:
```python
{
    "user_name": str,       # 申請者名
    "request_date": str     # 申請日（YYYY-MM-DD形式）
}
```
（session_id はファクトリ関数で消費済みのため除外）

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: AG-001 への応答文字列（str）
- **出力内容**: 申請書生成完了メッセージ・申請書ファイルパス

- **ファイル名**: `交通費精算申請書_{YYYYMMDD_HHMMSS}.xlsx`
- **保存先**: `output/`
- **内容**: 収集済み移動情報・申請者名・申請日・上長承認欄（必要時）を反映した Excel ファイル

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 移動情報が正しく収集されること。交通費が正しく計算されること。申請書が正しく生成されること
- **ドラフト生成テスト**: 申請書ドラフトがツール呼び出しなしでテキストとして提示されること。ドラフト提示ステップとツール呼び出しステップが明確に分離されていること
- **承認フローテスト**: HumanApprovalHook が BeforeToolCallEvent でトリガーされること。承認（OK）後にはじめて TOOL-002 が実行されること。承認前にツールが実行されないこと。Human-in-the-Loop 承認が 1 回のみ発生すること
- **異常系テスト**: 申請期限超過時に申請不可が案内されること。TOOL-001 のデータ未発見時にエラーが返ること。システム系エラー発生時にエラー内容が要約されて AG-001 に返ること
- **性能テスト**: LoopControlHook の 10 回制限が正しく機能すること

**詳細なテストケース**: 詳細設計書 テスト観点章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- AWS Strands SDK（strands-agents >= 0.1.0）を使用すること
- Amazon Bedrock（jp.anthropic.claude-sonnet-4-5-20250929-v1:0）を LLM バックエンドとして使用すること
- openpyxl >= 3.1.0 が必要（申請書生成）

### 16.2 業務的制約
- 申請書の提出（申請の実行）は AI が行わず、必ず人が実行する（BRL-06）
- 申請期限は経費発生日から 3 ヶ月以内（BRL-09）
- 交通費 10,000 円超の場合は上長承認が必要（BRL-10）

### 16.3 前提条件
- AG-001 から申請種別が交通費精算申請として確定した状態で委譲されること
- invocation_state に user_name・request_date が設定されていること
- data/train_fares.json・data/fixed_fares.json が配置されていること
- template/交通費精算申請書_template.xlsx が配置されていること
