## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- Strands SDK の FileSessionManager を統一的に生成・管理するファクトリクラスを提供する
- セッションIDの生成ロジックを一元化し、全エージェントで一貫したセッション管理を実現する
- セッションの再開（Resume）に対応し、中断後の会話継続を可能にする

### 2.2 主な責務
- **セッションID生成**: `session_{YYYYMMDD_HHMMSS}_{8桁ランダム文字列}` 形式のセッションIDを生成する
- **FileSessionManager生成**: セッションIDと保存先パスを指定してFileSessionManagerインスタンスを生成する
- **保存先ディレクトリ管理**: storage/sessions/ ディレクトリの存在確認・自動作成を行う
- **セッション存在確認**: 指定されたセッションIDのセッションデータが存在するかを確認する

### 2.3 非責務（やらないこと）
- **会話履歴の読み書き**: FileSessionManager（Strands SDK）が担当
- **セッション状態の遷移管理**: FileSessionManagerが内部的に管理
- **invocation_stateの管理**: オーケストレーターエージェント（AG-001）が担当
- **ログ出力設定**: main.pyが担当

### 2.4 必要性
- 全エージェント（AG-001, AG-002, AG-003）がFileSessionManagerを利用するため、生成ロジックの一元化が必要
- セッションID形式の統一により、セッション単位でのトレース・監査を実現する
- セッション再開時の存在確認ロジックを共通化し、重複実装を防止する

### 2.5 主要な設計意図
- **薄いラッパー**: FileSessionManagerの機能を拡張せず、生成ロジックのみを集約する
- **ステートレス**: SessionManagerFactory自体は状態を持たず、都度FileSessionManagerを生成する
- **設定の外部化**: 保存先パスはsettings.pyから取得し、ハードコードしない

## 3. SessionManagerFactoryクラス

### 3.1 クラス概要
- **クラス名**: SessionManagerFactory
- **説明**: Strands SDK の FileSessionManager インスタンスを生成するファクトリクラス。セッションIDの生成と保存先ディレクトリの管理を担当する

### 3.2 クラス変数
- **DEFAULT_STORAGE_DIR**: セッションデータの保存先ディレクトリパス（初期値: `"storage/sessions"`）

**詳細**: 詳細設計書で定義

---

## 4. 主要メソッド

### 4.1 generate_session_id
- **目的**: 一意なセッションIDを生成する
- **引数**: なし
- **戻り値**: `str`（セッションID文字列）
- **フォーマット**: `session_{YYYYMMDD_HHMMSS}_{8桁ランダム文字列}`（例: `session_20260517_162400_a1b2c3d4`）

### 4.2 create_session_manager
- **目的**: 指定されたセッションIDでFileSessionManagerインスタンスを生成する
- **引数**: `session_id: str`（セッションID）
- **戻り値**: `FileSessionManager`（Strands SDK提供のセッションマネージャインスタンス）
- **デフォルトパス**: `storage/sessions`（settings.pyから取得）

### 4.3 session_exists
- **目的**: 指定されたセッションIDのセッションデータが存在するかを確認する
- **引数**: `session_id: str`（セッションID）
- **戻り値**: `bool`（存在する場合True、存在しない場合False）

### 4.4 get_session_dir
- **目的**: 指定されたセッションIDに対応するセッションデータの保存ディレクトリパスを返す
- **引数**: `session_id: str`（セッションID）
- **戻り値**: `str`（セッションデータの保存ディレクトリパス）
- **フォーマット**: `storage/sessions/{session_id}/`

**詳細**: 詳細設計書で定義

## 5. セッション管理の仕組み

**システム設計書の方針**: artifacts/03_system-design/outputs/セッション管理方針.md を参照してください。

### 5.1 セッション生成処理
1. main.py起動時に `SessionManagerFactory.generate_session_id()` でセッションIDを生成する
2. `SessionManagerFactory.create_session_manager(session_id)` でFileSessionManagerインスタンスを生成する
3. 保存先ディレクトリ（storage/sessions/）が存在しない場合は自動作成する

### 5.2 セッション再開処理
1. main.py起動時にユーザーに既存セッションIDの入力を促す（または新規セッション開始を選択）
2. `SessionManagerFactory.session_exists(session_id)` でセッションデータの存在を確認する
3. 存在する場合は `SessionManagerFactory.create_session_manager(session_id)` で既存セッションを復元する

### 5.3 オーケストレーターでのセッション利用
1. main.pyで生成したFileSessionManagerインスタンスをオーケストレーター（AG-001）のインスタンス変数として保持する
2. AG-001のAgent()コンストラクタの`session_manager`パラメータに渡す
3. セッションIDはinvocation_state経由で専門エージェントに伝播する

### 5.4 専門エージェントでのセッション利用
1. 専門エージェントのツール関数内で`tool_context.invocation_state.get("session_id")`からセッションIDを取得する
2. ファクトリ関数（`_build_specialist_x_agent`）内で`SessionManagerFactory.create_session_manager(session_id)`を呼び出す
3. 生成したFileSessionManagerインスタンスを専門エージェントのAgent()コンストラクタに渡す
4. 専門エージェントは呼び出しごとに新しいFileSessionManagerインスタンスを使用する（都度作成）

**詳細**: 詳細設計書で定義

## 6. マルチエージェント連携

### 6.1 セッション共有の仕組み

**オーケストレーター（AG-001）**:
1. main.pyでセッションIDを生成し、FileSessionManagerを作成する
2. FileSessionManagerをAG-001のインスタンス変数（`self._session_manager`）として保持する
3. AG-001のAgent()コンストラクタの`session_manager`パラメータに渡す
4. invocation_stateにセッションIDを含めて専門エージェントに伝播する

**専門エージェント（AG-002, AG-003）**:
1. ツール関数内でinvocation_stateからセッションIDを取得する
2. ファクトリ関数内でSessionManagerFactory.create_session_manager(session_id)を呼び出す
3. 生成したFileSessionManagerを専門エージェントのAgent()コンストラクタに渡す

### 6.2 会話履歴の共有範囲
- **共有される情報**: セッションID（全エージェントで同一のセッションIDを使用し、セッション単位でのトレースを可能にする）
- **共有されない情報**: 各エージェントの会話履歴（エージェントごとに独立したFileSessionManagerインスタンスで管理。AG-001の会話履歴はAG-002/AG-003からは参照されない）

## 9. 設定値

### 9.1 保存先ディレクトリ
- **パス**: `storage/sessions`（settings.pyで管理）
- **自動作成**: あり（create_session_manager呼び出し時にディレクトリが存在しない場合は自動作成）

### 9.2 セッションID形式
- **フォーマット**: `session_{YYYYMMDD_HHMMSS}_{8桁ランダム文字列}`
- **例**: `session_20260517_162400_a1b2c3d4`
- **ランダム部分**: 英小文字+数字の8桁（`random.choices(string.ascii_lowercase + string.digits, k=8)`）

### 9.3 セッション状態
- **フォーマット**: CREATED / ACTIVE / WAITING / CLOSED / TERMINATED
- **例**: セッション生成直後はCREATED、申請者名取得後はACTIVE

## 10. テスト観点

| 観点 | テスト内容 | 期待結果 |
|-----|----------|---------|
| 正常系 | generate_session_idで一意なセッションIDが生成される | `session_{YYYYMMDD_HHMMSS}_{8桁ランダム}` 形式の文字列が返される |
| 正常系 | create_session_managerでFileSessionManagerが生成される | FileSessionManagerインスタンスが返される |
| 正常系 | 保存先ディレクトリが存在しない場合に自動作成される | storage/sessions/ ディレクトリが作成される |
| 正常系 | session_existsで既存セッションの存在確認ができる | セッションデータが存在する場合Trueが返される |
| 正常系 | session_existsで存在しないセッションの確認ができる | セッションデータが存在しない場合Falseが返される |
| 正常系 | 同一セッションIDで再度create_session_managerを呼び出すとセッションが復元される | 既存の会話履歴が復元されたFileSessionManagerが返される |
| 性能 | 複数回のgenerate_session_id呼び出しで重複が発生しない | 100回連続生成しても全て一意のIDが生成される |

## 11. 詳細設計への引き渡し事項

### 11.1 実装時の識別子
- **クラス名**: SessionManagerFactory

### 11.2 SDK実装時の注意事項
- FileSessionManagerのコンストラクタには`session_id`と`storage_dir`を渡す
- FileSessionManagerはStrands SDKの`strands.session`モジュールからインポートする
- セッションIDのランダム部分は`random`モジュールではなく`secrets`モジュールの使用を検討する（セキュリティ要件に応じて詳細設計で決定）

### 11.3 モック／スタブ要否
- 単体テスト時はFileSessionManagerをモック化し、ファイルI/Oを発生させない
- 結合テスト時は一時ディレクトリを使用して実際のファイルI/Oを検証する

## 13. 制約事項・前提条件

### 13.1 技術的制約
- FileSessionManagerはStrands SDK（strands-agents >= 0.1.0）が提供するクラスに依存する
- セッションデータはローカルファイルシステムに保存する（リモートストレージ非対応）
- セッションデータのファイル形式はFileSessionManagerの内部実装に依存する（JSON形式）

### 13.2 業務的制約
- 1回の申請処理（申請受付から申請書生成完了まで）を1セッションとする
- セッションデータの保持期間は要件上未定義（運用設計フェーズで決定）
- TERMINATED状態のセッションは再開不可（新規セッションを開始する）

### 13.3 前提条件
- Python 3.10以上の実行環境が利用可能であること
- storage/sessions/ ディレクトリへの書き込み権限があること
