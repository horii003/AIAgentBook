## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- ツール関数の入出力データに対して型安全性を保証する
- 外部データ（運賃データ JSON）の読み込み時にデータ品質を保証する
- エージェント間で共有する invocation_state の構造を定義する
- 共通バリデーター関数により表記ゆれの正規化を一元管理する

### 2.2 主な責務
- **ツール入力バリデーション**: ツール関数に渡されるパラメータの型・制約を検証する
- **ツール出力構造定義**: ツール関数の戻り値の構造を定義する
- **マスタデータバリデーション**: 運賃データ JSON の読み込み時にデータ構造を検証する
- **共通バリデーター提供**: 日付正規化・駅名正規化・交通手段正規化等の共通バリデーター関数を提供する

### 2.3 非責務（やらないこと）
- **業務ロジック実行**: ツール関数（tools/）が担当
- **エラーメッセージ生成**: ErrorHandler（handlers/error_handler.py）が担当
- **設定値管理**: settings.py（config/）が担当

### 2.4 必要性
- LLM がツールパラメータを構築する際、型不正・値不正が発生する可能性がある
- 運賃データ JSON の形式が不正な場合、実行時エラーを防止する必要がある
- 複数ツール関数で共通するバリデーションロジックを一元管理する必要がある

### 2.5 主要な設計意図
- **Pydantic v2 採用**: Field 定義による宣言的バリデーションと field_validator による正規化を活用する
- **invocation_state は辞書リテラル**: マルチエージェント連携設計に準拠し、専用の Pydantic モデルは定義しない
- **モデルカテゴリ分離**: ツール入力・ツール出力・マスタデータの3カテゴリに分離し、責務を明確化する

## 4. 共通バリデーター

### 4.1 validate_date_format
- **目的**: 日付文字列を YYYY-MM-DD 形式に正規化する
- **引数**: v: str（日付文字列）
- **戻り値**: str（YYYY-MM-DD 形式）
- **例外**: ValueError（解析不能な日付文字列）

### 4.2 validate_station_name
- **目的**: 駅名の表記ゆれを正規化する（「渋谷駅」→「渋谷」）
- **引数**: v: str（駅名文字列）
- **戻り値**: str（正規化済み駅名）
- **例外**: ValueError（空文字列）

### 4.3 normalize_transport_type
- **目的**: 交通手段の表記ゆれを正規化する（"train"→"電車"）
- **引数**: v: str（交通手段文字列）
- **戻り値**: str（正規化済み交通手段）
- **マッピング**: train→電車, bus→バス, taxi→タクシー, airplane/plane→飛行機

**詳細**: 詳細設計書 4章を参照

## 5. 主要モデルの概要

### 5.1 TransportCalculatorInput
- **目的**: 交通費計算ツール（calculate_transport_fare）の入力パラメータを検証する
- **フィールド**:
  - `departure: str` — 出発地・正規化済み駅名（DATA-001 出発駅）
  - `destination: str` — 目的地・正規化済み駅名（DATA-001 到着駅）
  - `transport_type: Literal["電車", "バス", "タクシー", "飛行機"]` — 交通手段
  - `travel_date: str` — 移動日（YYYY-MM-DD形式）。申請期限チェック（経費発生日から3ヶ月以内、BRL-08）に使用
- **バリデーション**: departure/destination に validate_station_name 適用、transport_type に normalize_transport_type 適用、travel_date に validate_date_format 適用、min_length=1 で空文字禁止

### 5.2 TransportCalculatorOutput
- **目的**: 交通費計算ツールの出力構造を定義する
- **フィールド**:
  - `success: bool` — 計算成否
  - `fare: Optional[int]` — 交通費金額（円）、成功時のみ
  - `error_message: Optional[str]` — エラーメッセージ、失敗時のみ
- **バリデーション**: fare は ge=0（0以上の整数）

### 5.3 ExpenseReportInput
- **目的**: 経費精算申請書生成ツール（generate_expense_report）の入力パラメータを検証する
- **フィールド**:
  - `items: list[ExpenseItem]` — 経費明細リスト
- **バリデーション**: items は min_length=1（空リスト禁止）

### 5.4 ExpenseItem
- **目的**: 経費明細の1件を定義する
- **フィールド**:
  - `purchase_date: str` — 購入日（YYYY-MM-DD）
  - `store_name: str` — 店舗名
  - `item_name: str` — 品目
  - `expense_category: Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]` — 経費区分
  - `amount: int` — 金額（円）
  - `business_purpose: str` — 業務目的
- **バリデーション**: purchase_date に validate_date_format 適用、amount は ge=0、各文字列フィールドは min_length=1

### 5.5 TransportReportInput
- **目的**: 交通費精算申請書生成ツール（generate_transport_report）の入力パラメータを検証する
- **フィールド**:
  - `items: list[TransportItem]` — 移動明細リスト
- **バリデーション**: items は min_length=1（空リスト禁止）

### 5.6 TransportItem
- **目的**: 移動明細の1件を定義する
- **フィールド**:
  - `travel_date: str` — 移動日（YYYY-MM-DD）
  - `departure: str` — 出発地
  - `destination: str` — 目的地
  - `transport_type: Literal["電車", "バス", "タクシー", "飛行機"]` — 交通手段
  - `fare: int` — 費用（円）
  - `business_purpose: str` — 業務目的
- **バリデーション**: travel_date に validate_date_format 適用、fare は ge=0、各文字列フィールドは min_length=1

### 5.7 ReportOutput
- **目的**: 申請書生成ツールの出力構造を定義する
- **フィールド**:
  - `success: bool` — 生成成否
  - `file_path: Optional[str]` — 生成ファイルパス、成功時のみ
  - `error_message: Optional[str]` — エラーメッセージ、失敗時のみ
- **バリデーション**: なし（出力モデルのため）

### 5.8 TrainRouteRecord
- **目的**: 電車経路・運賃データ（train_routes.json）の1レコードを検証する
- **フィールド**:
  - `departure: str` — 出発駅（DATA-001 出発駅）
  - `destination: str` — 到着駅（DATA-001 到着駅）
  - `fare: int` — 運賃（円）（DATA-001 運賃）
- **バリデーション**: fare は gt=0（正の整数）、各文字列フィールドは min_length=1

### 5.9 FixedFareRecord
- **目的**: 固定運賃データ（fixed_fares.json）の1レコードを検証する
- **フィールド**:
  - `transport_type: Literal["バス", "タクシー", "飛行機"]` — 交通手段（DATA-001-2 交通手段）
  - `fare: int` — 固定運賃（円）（DATA-001-2 運賃）
- **バリデーション**: fare は gt=0（正の整数）

**詳細**: 詳細設計書 5章を参照

## 6. バリデーションルール

**システム設計書の方針**: artifacts/03_system-design/outputs/バリデーション方針.md を参照

### 6.1 日付
- **フォーマット**: YYYY-MM-DD（ISO 8601）
- **例**: OK: "2026-05-17", NG: "2026/5/17"（正規化で対応可能）

### 6.2 交通手段
- **許可される値**: "電車", "バス", "タクシー", "飛行機"
- **正規化**: "train"→"電車", "bus"→"バス", "taxi"→"タクシー", "airplane"/"plane"→"飛行機"

### 6.3 駅名
- **制約**: 空文字禁止、「〇〇駅」→「〇〇」に正規化
- **範囲**: 任意の文字列（運賃データとの照合はツール関数内で実施）

### 6.4 金額
- **制約**: 0以上の整数（int, ge=0）
- **最小値/最大値**: 最小0、最大値は要件上未定義

**詳細**: 詳細設計書 6章を参照

## 7. マルチエージェント連携時の扱い

### 7.1 データモデルの共有
- models/data_models.py に全モデルを定義し、ツール関数から import して使用する
- invocation_state は辞書リテラルで渡す（専用の Pydantic モデルは定義しない）
- ツール関数内で .get() による安全なアクセスを行う

### 7.2 データの受け渡し
- **エージェント間（invocation_state）**: AG-001 → AG-002/AG-003 へ辞書リテラル `{"applicant_name": str, "application_date": str, "session_id": str}` を渡す
- **エージェント→ツール**: LLM がツールパラメータを構築し、ツール関数内で Pydantic モデルによりバリデーションする
- **ツール→エージェント**: ツール関数が辞書形式で結果を返し、LLM が解釈する

## 11. 制約事項・前提条件

### 11.1 技術的制約
- Pydantic v2（pydantic >= 2.0.0）に依存
- python-dateutil（>= 2.8.2）に依存（日付パース）
- invocation_state は辞書リテラルで渡す（Pydantic モデル化しない）

### 11.2 業務的制約
- 交通手段は4種類のみ（電車・バス・タクシー・飛行機）
- 経費区分は4種類のみ（事務用品費・宿泊費・資格精算費・その他経費）
- 金額は円単位の整数（小数点以下なし）

### 11.3 前提条件
- バリデーション方針.md に定義された方針に準拠すること
- マルチエージェント連携設計.md に定義された invocation_state フィールド構成に準拠すること
