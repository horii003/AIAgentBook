## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- 全ツール・全エージェントで使用するデータ構造を Pydantic v2 モデルとして一元定義し、型安全性を保証する
- バリデーションルールを一箇所に集約し、全コンポーネントで一貫したデータ検証を実施する
- invocation_state のフィールド構成をマルチエージェント連携設計に準拠して定義する

### 2.2 主な責務
- **ツール入力モデル定義**: 各ツール関数の入力パラメータを Pydantic モデルとして定義する
- **ツール出力モデル定義**: 各ツール関数の戻り値を Pydantic モデルとして定義する
- **マスタデータモデル定義**: data/ 配下の JSON ファイルの型保証モデルを定義する
- **共通バリデーター定義**: 複数モデルで再利用するバリデーター関数を定義する

### 2.3 非責務（やらないこと）
- **バリデーション実行**: ツール関数の入口で実施する（tools/{domain}_tools.py の責務）
- **エラーメッセージ生成**: ErrorHandler が担当する（handlers/error_handler.py の責務）
- **ビジネスロジック実装**: 各ツール関数・エージェントが担当する

### 2.4 必要性
- LLM が渡す値の型・形式が不定であるため、Pydantic による型強制が必要
- 複数エージェント・ツールで同一データ構造を使用するため、一元定義が必要
- バリデーションルールの重複実装を防ぐため、共通バリデーターの集約が必要

### 2.5 主要な設計意図
- **定義場所の分離**: 共通モデル（InvocationState 等）は models/data_models.py に定義し、ツール固有モデルは tools/{domain}_tools.py に定義する（バリデーション方針.md 3.2 節準拠）
- **invocation_state は辞書リテラル**: マルチエージェント連携設計.md に準拠し、invocation_state は辞書リテラルで渡す。専用の Pydantic モデルは定義しない
- **未使用モデルの禁止**: 定義したモデルは対応するツール関数内で実際に使用する（デッドコード禁止）

## 4. 共通バリデーター

### 4.1 validate_date_format
- **目的**: 日付文字列が YYYY-MM-DD 形式の有効な日付であることを検証する
- **引数**: v: str（日付文字列）
- **戻り値**: str（検証済み日付文字列）
- **例外**: ValueError（形式不正・存在しない日付）
- **適用モデル**: TransportRecord（移動日）、ExpenseRecord（経費発生日）

### 4.2 normalize_transport_type
- **目的**: 交通手段の表記を正規化する（英語表記・略称を日本語正規表記に変換）
- **引数**: v: str（交通手段文字列）
- **戻り値**: str（正規化済み交通手段文字列）
- **マッピング**: "train" → "電車"、"bus" → "バス"、"taxi" → "タクシー"、"airplane" / "plane" → "飛行機"

### 4.3 normalize_expense_category
- **目的**: 経費区分の表記を正規化する（英語表記・略称を日本語正規表記に変換）
- **引数**: v: str（経費区分文字列）
- **戻り値**: str（正規化済み経費区分文字列）
- **マッピング**: "office_supplies" → "事務用品費"、"accommodation" → "宿泊費"、"qualification" → "資格精算費"、"other" / "その他" → "その他経費"

## 5. 主要モデルの概要

### 5.1 TransportCalculatorInput
- **目的**: calculate_transport_fare ツールの入力バリデーション
- **フィールド**:
  - `departure: str` — 出発地（正規化済み駅名・地点名）。min_length=1（データ一覧.md DATA-003 出発地）
  - `destination: str` — 目的地（正規化済み駅名・地点名）。min_length=1（データ一覧.md DATA-003 目的地）
  - `transport_type: Literal["電車", "バス", "タクシー", "飛行機"]` — 交通手段。normalize_transport_type バリデーター適用
- **バリデーション**: departure・destination は空文字禁止。transport_type は許容値以外で ValidationError

### 5.2 TransportRecord
- **目的**: 移動区間1件の情報（TransportExpenseFormInput の要素）
- **フィールド**:
  - `travel_date: str` — 移動日（YYYY-MM-DD形式）。validate_date_format バリデーター適用
  - `departure: str` — 出発地。min_length=1
  - `destination: str` — 目的地。min_length=1
  - `transport_type: Literal["電車", "バス", "タクシー", "飛行機"]` — 交通手段。normalize_transport_type バリデーター適用
  - `fare: int` — 運賃（円）。ge=0
  - `business_purpose: str` — 業務目的。min_length=1（BRL-12）
- **バリデーション**: 全フィールド必須。fare は 0 以上の整数

### 5.3 TransportExpenseFormInput
- **目的**: generate_transport_expense_form ツールの入力バリデーション
- **フィールド**:
  - `transport_records: list[TransportRecord]` — 移動区間リスト。min_length=1
  - `needs_approval: bool` — 上長承認要否フラグ（BRL-10）
- **バリデーション**: transport_records は1件以上必須。申請者名・申請日は invocation_state から取得するためフィールドに含めない

### 5.4 ExpenseRecord
- **目的**: 経費1件の情報（GeneralExpenseFormInput の要素）
- **フィールド**:
  - `expense_date: str` — 経費発生日（YYYY-MM-DD形式）。validate_date_format バリデーター適用
  - `store_name: str` — 店舗名。min_length=1
  - `item_name: str` — 品目。min_length=1
  - `amount: int` — 金額（円）。ge=0
  - `category: Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]` — 経費区分（BRL-14）。normalize_expense_category バリデーター適用
  - `business_purpose: str` — 業務目的。min_length=1（BRL-12）
- **バリデーション**: 全フィールド必須。amount は 0 以上の整数

### 5.5 GeneralExpenseFormInput
- **目的**: generate_general_expense_form ツールの入力バリデーション
- **フィールド**:
  - `expense_records: list[ExpenseRecord]` — 経費情報リスト。min_length=1
  - `needs_approval: bool` — 上長承認要否フラグ（BRL-16）
- **バリデーション**: expense_records は1件以上必須。申請者名・申請日は invocation_state から取得するためフィールドに含めない

### 5.6 TransportCalculatorOutput
- **目的**: calculate_transport_fare ツールの出力スキーマ
- **フィールド**:
  - `success: bool` — 計算成否フラグ
  - `fare: Optional[int]` — 計算済み運賃（円）。正常時のみ設定（ge=0）
  - `error_message: Optional[str]` — エラーメッセージ。エラー時のみ設定
- **バリデーション**: success=True の場合は fare が設定される。success=False の場合は error_message が設定される

### 5.7 FormGeneratorOutput
- **目的**: 申請書生成ツール（generate_transport_expense_form / generate_general_expense_form）の出力スキーマ
- **フィールド**:
  - `success: bool` — 生成成否フラグ
  - `file_path: Optional[str]` — 生成済み申請書ファイルパス。正常時のみ設定
  - `error_message: Optional[str]` — エラーメッセージ。エラー時のみ設定
- **バリデーション**: success=True の場合は file_path が設定される。success=False の場合は error_message が設定される

### 5.8 TrainFareData
- **目的**: data/train_fares.json の1レコードの型保証
- **フィールド**:
  - `departure: str` — 出発地（正規化済み駅名）。min_length=1
  - `destination: str` — 目的地（正規化済み駅名）。min_length=1
  - `fare: int` — 運賃（円）。ge=0
- **バリデーション**: 全フィールド必須

### 5.9 FixedFareData
- **目的**: data/fixed_fares.json の1レコードの型保証
- **フィールド**:
  - `transport_type: Literal["バス", "タクシー", "飛行機"]` — 交通手段
  - `fare: int` — 固定運賃（円）。ge=0
- **バリデーション**: 全フィールド必須

## 6. バリデーションルール

**システム設計書の方針**: artifacts/03_system-design/outputs/バリデーション方針.md を参照

### 6.1 日付フォーマット
- **フォーマット**: YYYY-MM-DD（例: 2026-05-21）
- **検証方法**: python-dateutil の parser.parse() で有効な日付であることを確認する
- **例**: OK: "2026-05-21"、NG: "2026/05/21"、NG: "2026-13-01"（存在しない日付）

### 6.2 交通手段
- **許可される値**: "電車"、"バス"、"タクシー"、"飛行機"
- **正規化**: normalize_transport_type バリデーターで英語表記・略称を日本語正規表記に変換する
- **エラー**: 許容値以外は ValidationError

### 6.3 経費区分
- **許可される値**: "事務用品費"、"宿泊費"、"資格精算費"、"その他経費"
- **正規化**: normalize_expense_category バリデーターで英語表記・略称を日本語正規表記に変換する
- **エラー**: 許容値以外は ValidationError

### 6.4 金額・運賃
- **制約**: 0 以上の整数（ge=0）
- **最小値**: 0（無料の場合を許容）
- **エラー**: 負の値は ValidationError

### 6.5 必須文字列
- **制約**: 空文字禁止（min_length=1）
- **対象**: departure、destination、store_name、item_name、business_purpose 等の必須フィールド
- **エラー**: 空文字は ValidationError

**詳細**: 詳細設計書 バリデーション実装章を参照

## 7. マルチエージェント連携時の扱い

### 7.1 データモデルの共有
- ツール入力モデル・出力モデルは tools/{domain}_tools.py に定義し、対応するツール関数内でのみ使用する
- マスタデータモデル（TrainFareData・FixedFareData）は tools/transport_tools.py に定義する
- 共通バリデーター関数は models/data_models.py に定義し、各ツールファイルからインポートして使用する

### 7.2 データの受け渡し
- **エージェント間（invocation_state）**: 辞書リテラルで渡す。専用の Pydantic モデルは定義しない（マルチエージェント連携設計.md 6.1 節準拠）
  - AG-001 → AG-002/AG-003: `{"user_name": str, "request_date": str, "session_id": str}`
  - AG-002/AG-003 → ツール関数内: `{"user_name": str, "request_date": str}`（session_id はファクトリ関数で消費済みのため除外）
- **エージェント → ツール（ツールパラメータ）**: LLM が推論して渡す。Pydantic モデルでバリデーションする
- **ツール → エージェント（戻り値）**: 辞書形式で返す。キー名は出力 Pydantic モデルのフィールド名と一致させる

## 8. テスト観点

| 観点 | テスト内容 | 期待結果 |
|-----|----------|---------|
| 正常系 | 有効な交通手段（"電車"）で TransportCalculatorInput を生成 | バリデーション通過 |
| 正常系 | 英語表記の交通手段（"train"）で TransportCalculatorInput を生成 | normalize_transport_type により "電車" に正規化されてバリデーション通過 |
| 異常系 | 無効な交通手段（"自転車"）で TransportCalculatorInput を生成 | ValidationError 発生 |
| 正常系 | 有効な日付（"2026-05-21"）で TransportRecord を生成 | バリデーション通過 |

| 異常系 | 負の運賃（-1）で TransportRecord を生成 | ValidationError 発生 |
| 異常系 | 空文字の出発地（""）で TransportCalculatorInput を生成 | ValidationError 発生 |
| 正常系 | 有効な経費区分（"事務用品費"）で ExpenseRecord を生成 | バリデーション通過 |
| 異常系 | 無効な経費区分（"交通費"）で ExpenseRecord を生成 | ValidationError 発生 |

## 9. 詳細設計への引き渡し事項

### 9.1 実装時の識別子
- **モジュール名**: models/data_models.py（共通バリデーター）、tools/transport_tools.py（交通費計算ツール関連モデル）、tools/output_generator.py（申請書生成ツール関連モデル）

### 9.2 SDK実装時の注意事項
- Pydantic v2 の field_validator を使用する（v1 の validator は使用しない）
- 共通バリデーター関数は `classmethod(validate_xxx)` として直接渡す（ラムダラッパー禁止）
- モデルの docstring は日本語で記載する（R8 言語ポリシー準拠）
- フィールド名・バリデーター名は英語で記載する（R7 命名規則準拠）

### 9.3 モック／スタブ要否
- 単体テスト時は Pydantic モデルのインスタンスを直接生成してテストする（モック不要）

## 11. 制約事項・前提条件

### 11.1 技術的制約
- Pydantic v2（pydantic >= 2.0.0）を使用すること
- python-dateutil（python-dateutil >= 2.8.2）を日付バリデーションに使用すること
- Python 3.10 以上が必要（型ヒント・match 文対応）

### 11.2 業務的制約
- 交通手段は電車・バス・タクシー・飛行機の4種類のみ（BRL-08）
- 経費区分は事務用品費・宿泊費・資格精算費・その他経費の4種類のみ（BRL-14）
- 業務目的は必須（BRL-12）

### 11.3 前提条件
- invocation_state は辞書リテラルで渡す（専用 Pydantic モデルは定義しない）
- 定義したモデルは対応するツール関数内で実際に使用すること（未使用モデル禁止）
