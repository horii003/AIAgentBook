## 2. ErrorHandler 基本設計

**システム設計書の方針**: システム設計書 例外処理方針.md（`artifacts/03_system-design/outputs/例外処理方針.md`）を参照してください。

### 2.1 ErrorHandlerの位置づけ

#### システム全体における役割
- 全エージェント（AG-001, AG-002, AG-003）および全ツール（TOOL-001, TOOL-002）で発生する例外に対して、ユーザー向けメッセージを生成する共通コンポーネント
- 例外分類（EX-01〜EX-08）ごとに対応するメソッドを提供し、一貫したエラーメッセージ生成を実現する

#### 主な責務（概要）
- 例外分類ごとのユーザー向け日本語メッセージの生成
- 技術的な詳細の隠蔽（スタックトレース・例外クラス名等をユーザーに見せない）
- 次の行動の提示（再入力案内・担当者連絡案内・再試行案内）

#### 非責務（やらないこと）
- **ログ出力**: 呼び出し元モジュールが `_logger.error(exc_info=True)` で行う
- **例外の検知**: エージェントまたはツールが検知し、ErrorHandler に委譲する
- **セッション状態の更新**: セッション管理コンポーネント（FileSessionManager）が行う
- **リトライ制御**: ModelRetryStrategy（Strands SDK）が行う

#### 詳細
詳細設計書 1章を参照

### 2.2 利用コンポーネント

| コンポーネントID | コンポーネント名 | 利用目的 |
|----------------|----------------|---------|
| AG-001 | 申請受付窓口エージェント | 入力例外・判断不能・業務ルール違反・想定外エラーのメッセージ生成 |
| AG-002 | 経費精算申請エージェント | バリデーションエラー・ファイルエラー・期限超過・想定外エラーのメッセージ生成 |
| AG-003 | 交通費精算申請エージェント | バリデーションエラー・ファイルエラー・期限超過・想定外エラーのメッセージ生成 |
| TOOL-001 | 交通費計算ツール | 運賃データ不存在時のエラーメッセージ生成 |
| TOOL-002 | 申請書生成ツール | テンプレート不存在・書き込みエラー・バリデーションエラーのメッセージ生成 |

### 2.3 主要メソッド一覧

| メソッド名 | 目的 | 対応する例外分類 | 利用タイミング |
|-----------|-----|----------------|--------------|
| handle_validation_error | Pydanticバリデーションエラーのメッセージ生成 | EX-01, EX-06 | ツール関数の入力バリデーション失敗時、invocation_state のバリデーション失敗時 |
| handle_loop_limit_error | ループ制限到達時のメッセージ生成 | EX-08 | LoopControlHook が LoopLimitError を発生させた時 |
| handle_unexpected_error | 予期しないエラーのメッセージ生成 | EX-08 | 未分類の Exception が発生した時 |
| handle_deadline_error | 申請期限超過時のメッセージ生成 | EX-03 | 経費発生日から3ヶ月超過を検知した時 |
| handle_file_error | ファイル操作エラーのメッセージ生成 | EX-06, EX-07 | テンプレート不存在・ファイル書き込み失敗・運賃データ読み込み失敗時 |

### 2.4 メソッドの詳細
詳細設計書 2.3章を参照

### 2.5 ErrorHandlerの設計方針と設計意図

**全メソッド@staticmethod（インスタンス化不要）**:
- ErrorHandler は状態を持たないユーティリティクラスとして設計する。`ErrorHandler.handle_xxx()` として直接呼び出す

**ログ出力の責務分離**:
- ErrorHandler はメッセージ生成のみを担当し、ログ出力は呼び出し元が行う。これにより、ログレベルの判断を呼び出し元に委ねることができる

**主要な設計意図**:
- **一貫したUX**: 全エージェント・全ツールで統一されたエラーメッセージフォーマットを提供する
- **技術詳細の隠蔽**: ユーザーに内部実装の詳細を見せず、業務的な原因と次の行動を提示する
- **拡張性**: 新しい例外分類が追加された場合、対応する handle_* メソッドを追加するだけで対応可能

**詳細**: 詳細設計書 2.5章を参照

## 3. LoopControlHook 基本設計

**システム設計書の方針**: システム設計書 実行制御方針.md（`artifacts/03_system-design/outputs/実行制御方針.md`）10章を参照してください。

### 3.1 LoopControlHookの位置づけ

#### システム全体における役割
- 全エージェント（AG-001, AG-002, AG-003）の ReAct ループを監視し、無限ループを防止するフックコンポーネント
- HookProvider を継承し、Strands SDK のフックメカニズムに統合される

#### 主な責務（概要）
- ReAct ループ回数のカウント（LLM呼び出し1回を1カウント）
- 最大ループ回数（10回）到達時の LoopLimitError 発生
- エージェント呼び出し開始・完了時のカウンタリセット

#### 非責務（やらないこと）
- **エラーメッセージの生成**: ErrorHandler（HD-001）が担当する
- **ログ出力**: 呼び出し元モジュールが担当する
- **セッション状態の更新**: セッション管理コンポーネントが担当する
- **承認制御**: HumanApprovalHook（HD-003）が担当する

#### 詳細
詳細設計書 1章を参照

### 3.2 利用エージェント

| エージェントID | エージェント名 | 利用目的 |
|---------------|---------------|---------|
| AG-001 | 申請受付窓口エージェント | 申請種別判断ループの無限ループ防止 |
| AG-002 | 経費精算申請エージェント | 情報収集・申請書生成ループの無限ループ防止 |
| AG-003 | 交通費精算申請エージェント | 情報収集・交通費計算・申請書生成ループの無限ループ防止 |

### 3.3 フック登録の概要

**フックの目的**: エージェントの ReAct ループが最大回数（10回）を超えて実行されることを防止し、システムリソースの枯渇とユーザー体験の劣化を防ぐ

**登録するイベント**:
- BeforeInvocationEvent: エージェント呼び出し開始前にループカウンタを0にリセットする
- AfterModelCallEvent: LLM呼び出し完了後にカウンタをインクリメントし、上限到達を監視する
- AfterInvocationEvent: エージェント呼び出し完了後にループカウンタを0にリセットする

### 3.4 フック登録の詳細
詳細設計書 2.4.1章を参照

### 3.5 LoopControlHookの処理概要

**処理の流れ**:
1. BeforeInvocationEvent 発火時: ループカウンタを0にリセットする
2. AfterModelCallEvent 発火時: ループカウンタをインクリメントし、max_iterations（10回）と比較する
3. 上限到達時: LoopLimitError を raise する（エージェントの処理が強制終了される）
4. AfterInvocationEvent 発火時: ループカウンタを0にリセットする（正常完了時のクリーンアップ）

**処理の目的**: ReAct ループの無限ループを防止し、エージェントが制御不能な自律動作を行わないようにする

### 3.6 処理フローの詳細
詳細設計書 3.1章を参照

## 4. HumanApprovalHook 基本設計

**システム設計書の方針**: システム設計書 実行制御方針.md（`artifacts/03_system-design/outputs/実行制御方針.md`）11章を参照してください。

### 4.1 HumanApprovalHookの位置づけ

#### システム全体における役割
- 専門エージェント（AG-002, AG-003）の申請書生成ツール実行前に、ユーザーの承認を得るフックコンポーネント
- HookProvider を継承し、Strands SDK のフックメカニズムに統合される
- 自律度 Lv4（承認後実行）を実現する Human-in-the-Loop 機構

#### 主な責務（概要）
- 対象ツール（generate_expense_report, generate_transport_report）の実行前に承認ゲートを設ける
- ユーザーに OK/修正/キャンセル の3択を提示する
- 選択結果に応じてツール実行の継続・キャンセルを制御する

#### 非責務（やらないこと）
- **申請内容の表示**: エージェント（AG-002, AG-003）のシステムプロンプトで制御する
- **ループ制御**: LoopControlHook（HD-002）が担当する
- **エラーメッセージの生成**: ErrorHandler（HD-001）が担当する
- **対象ツール以外のフィルタリング以外の処理**: 対象外ツールはスキップする

#### 詳細
詳細設計書 1章を参照

### 4.2 利用エージェント

| エージェントID | エージェント名 | 利用目的 |
|---------------|---------------|---------|
| AG-002 | 経費精算申請エージェント | 経費精算申請書生成前の承認ゲート（APR-001） |
| AG-003 | 交通費精算申請エージェント | 交通費精算申請書生成前の承認ゲート（APR-001） |

### 4.3 フック登録の概要

**フックの目的**: 申請書ファイルの誤生成を防止し、ユーザーが申請内容を最終確認する機会を提供する

**登録するイベント**:
- BeforeToolCallEvent: ツール実行前に発火し、対象ツールの場合に承認ゲートを実行する

### 4.4 フック登録の詳細
詳細設計書 2.4.1章を参照

### 4.5 HumanApprovalHookの処理概要

**処理の流れ**:
1. エージェントが収集済み申請情報をテキストとして整理・提示する（ドラフト提示ステップ、ツール呼び出しなし）
2. エージェントが申請書生成ツールを呼び出そうとした時点で BeforeToolCallEvent が発火する
3. `event.tool_use["name"]` で対象ツールかを判定する
4. 対象外ツールの場合: 何もせずリターンする（全ツール呼び出しで発火するため必須）
5. 対象ツールの場合: ユーザーに OK/修正/キャンセル の選択肢を提示する
6. OK 選択時: そのままリターンする（ツール実行が継続される）
7. 修正選択時: `event.cancel_tool = "修正を選択しました。情報を修正してください。"` を設定する
8. キャンセル選択時: `event.cancel_tool = "キャンセルを選択しました。申請処理を終了します。"` を設定する

**ドラフト提示とツール実行の分離**:
- ドラフト提示はエージェントのプロンプト指示により実行される（ツール呼び出しなし）
- HumanApprovalHookはBeforeToolCallEventでのみトリガーし、承認（OK）後にはじめてツールが実行される
- Human-in-the-Loop承認は1回のみ発生する。承認前にツールを実行してはならない

**処理の目的**: 自律度 Lv4（承認後実行）を実現し、ユーザーが最終判断を行った上で申請書を生成する

### 4.6 処理フローの詳細
詳細設計書 3.1章を参照

## 5. マルチエージェント連携時の扱い

### 5.1 ErrorHandler

**共有方法**:
- 全メソッドが @staticmethod であり、インスタンス化不要。`ErrorHandler.handle_xxx()` として直接呼び出す
- 全エージェント・全ツールから同一のクラスメソッドを参照する

**スレッドセーフ性**:
- 状態を持たない（インスタンス変数なし）ため、スレッドセーフ性の問題は発生しない

### 5.2 LoopControlHook

**共有方法**:
- 各エージェントの Agent() コンストラクタの hooks パラメータに個別のインスタンスを登録する
- AG-001, AG-002, AG-003 それぞれが独立した LoopControlHook インスタンスを持つ

**競合防止**:
- 各エージェントが独立したインスタンスを持つため、カウンタの競合は発生しない
- オーケストレーター（AG-001）から専門エージェント（AG-002/AG-003）への委譲時、各エージェントのカウンタは独立して動作する

### 5.3 HumanApprovalHook

**共有方法**:
- AG-002, AG-003 の Agent() コンストラクタの hooks パラメータに個別のインスタンスを登録する
- AG-001（オーケストレーター）には登録しない（申請書生成を行わないため）

**競合防止**:
- 各エージェントが独立したインスタンスを持つため、承認状態の競合は発生しない
- 1セッション内で AG-002 または AG-003 のいずれか一方のみが実行されるため、並列実行による競合は発生しない

## 6. テスト観点

### 6.1 ErrorHandler
- **正常系**: 各例外分類（EX-01〜EX-08）に対応するメソッドが適切な日本語メッセージを返すこと
- **異常系**: None や空文字列が引数に渡された場合でも例外を発生させず、安全なメッセージを返すこと

### 6.2 LoopControlHook
- **正常系**: 最大ループ回数（10回）未満の場合、LoopLimitError が発生しないこと
- **正常系**: 最大ループ回数（10回）到達時に LoopLimitError が発生すること
- **正常系**: BeforeInvocationEvent / AfterInvocationEvent でカウンタが正しくリセットされること

### 6.3 HumanApprovalHook
- **正常系**: 対象ツール（generate_expense_report, generate_transport_report）の実行前に承認ゲートが発火すること
- **正常系**: OK 選択時にツール実行が継続されること（event.cancel_tool が False のまま）
- **正常系**: 修正選択時に event.cancel_tool に修正メッセージが設定されること
- **正常系**: キャンセル選択時に event.cancel_tool にキャンセルメッセージが設定されること
- **正常系**: ドラフト提示ステップ（テキスト整理）ではツール呼び出しが発生せず、HumanApprovalHookがトリガーされないこと
- **正常系**: 承認（OK）後にはじめてツールが実行され、Human-in-the-Loop承認が1回のみ発生すること
- **異常系**: 対象外ツールの場合に承認ゲートが発火しないこと

## 7. 詳細設計への引き渡し事項

### 7.1 実装時の識別子
- **クラス名**: ErrorHandler, LoopControlHook, HumanApprovalHook
- **ファイル名**: handlers/error_handler.py, handlers/loop_control_hook.py, handlers/human_approval_hook.py

### 7.2 SDK実装時の注意事項
- LoopControlHook・HumanApprovalHook は `HookProvider` を継承し、`register_hooks` メソッドで `registry.add_callback()` を使用してイベントを登録する（`add_hook()` は存在しない）
- HumanApprovalHook の BeforeToolCallEvent ハンドラーでは、全ツール呼び出しで発火するため対象ツールのフィルタリングが必須
- ツール呼び出しのキャンセルは `event.cancel_tool = "メッセージ"` で行う（`event.cancel()` は存在しない）
- ErrorHandler の全メソッドは `@staticmethod` とし、インスタンス化不要で呼び出し可能にする

### 7.3 モック／スタブ要否
- **ErrorHandler**: 単体テストではモック不要（状態を持たない純粋関数的なメソッド群）
- **LoopControlHook**: テスト時は max_iterations を小さい値（例: 3）に設定してテストする
- **HumanApprovalHook**: テスト時はユーザー入力（input()）をモックし、OK/修正/キャンセルの各パターンを検証する

## 9. 制約事項・前提条件

### 9.1 技術的制約
- ErrorHandler は状態を持たない @staticmethod のみで構成する（インスタンス変数禁止）
- LoopControlHook・HumanApprovalHook は HookProvider を継承する必要がある（Strands SDK の制約）
- HumanApprovalHook はCLI の標準入力（input()）でユーザー入力を受け付ける（GUI/Web 対応は対象外）

### 9.2 業務的制約
- 承認対象ツールは generate_expense_report と generate_transport_report の2つのみ（TOOL-002 の分割後の2関数）
- 最大ループ回数（10回）は全エージェント共通の固定値とする
- 承認選択肢は OK/修正/キャンセル の3択固定とする

### 9.3 前提条件
- Strands SDK（strands-agents >= 0.1.0）が利用可能であること
- HookProvider, BeforeInvocationEvent, AfterModelCallEvent, AfterInvocationEvent, BeforeToolCallEvent が Strands SDK から提供されること
- LoopLimitError 例外クラスが定義されていること（handlers/loop_control_hook.py 内で定義）
- 環境変数（LOG_LEVEL 等）が .env ファイルで設定されていること
