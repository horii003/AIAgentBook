## 2. 目的・役割定義

### 2.1 エージェントの目的
- 社員の申請内容テキストから申請種別を自動判断し、対応する専門エージェントへ処理を委譲する
- 申請者名を収集し、invocation_stateとして専門エージェントへ伝播する
- 判断不能時はユーザーへ選択肢を提示し、申請種別を確定させる

### 2.2 主な責務
- **申請者名収集**: アプリケーション起動時（対話ループ開始前）に申請者名を収集する
- **申請種別判断**: 申請内容テキストからJD-01〜JD-03に基づき申請種別を自動判断する
- **専門エージェントへの委譲**: 申請種別確定後にAG-002またはAG-003へAgent as Toolsで委譲する
- **invocation_state生成**: applicant_name・application_date・session_idを生成してinvocation_stateとして渡す
- **委譲結果の提示**: 専門エージェントからの処理結果をユーザーへ提示する

### 2.3 非責務（やらないこと）
- **申請書生成**: AG-002・AG-003が担当（AG-001はTOOL-002を呼び出さない）
- **移動情報・経費情報の収集**: AG-002・AG-003が担当
- **交通費計算**: AG-002・TOOL-001が担当

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 申請ルール参照（ACT-READ-01） | 申請種別判断の根拠として申請ルールを参照する |
| 提案 | 申請種別判断（ACT-DEC-01） | 申請内容テキストから申請種別を自動判断し委譲する |
| 実行 | 専門エージェントへの委譲 | 申請種別確定後に専門エージェントへ委譲する |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請種別の自動判断（JD-01〜JD-02）・専門エージェントへの委譲
- **人間確認必要**: 申請種別が判断不能な場合（JD-03）のユーザー選択
- **実行不可**: 申請書ファイル生成（TOOL-002の呼び出し禁止）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始 → 申請者名収集 → 申請内容テキスト受付 → 申請種別判断（JD-01〜JD-03）
  → [交通費精算] AG-002へ委譲 → 委譲結果提示 → 終了
  → [経費精算] AG-003へ委譲 → 委譲結果提示 → 終了
  → [判断不能] 選択肢提示 → ユーザー選択 → 再ルーティング
```

**フローの詳細**: 詳細設計書 agents/orchestrator_agent.py を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 申請者名収集 → 申請内容受付 → 申請種別判断 → 委譲の逐次処理
- **処理をスキップする考え方**: 申請種別が明確な場合（JD-01/JD-02）はユーザー確認をスキップして自動委譲
- **同一処理を複数回実行**: 申請種別判断不能時（JD-03）はユーザー選択後に再ルーティング

### 4.3 状態管理
- **applicant_name**: アプリ起動時に取得。セッション中保持（invocation_stateで専門エージェントへ伝播）
- **session_id**: アプリ起動時に生成（`{YYYYMMDD_HHMMSS}_{8桁ランダム英数字}`）。セッション中保持
- **会話履歴**: FileSessionManagerがstorage/sessions/に永続化

## 5. 専門エージェントとの連携設計

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 交通費精算申請エージェント | 交通費精算申請書の作成 | 申請内容が交通費・移動・電車・バス・タクシー・飛行機に関連する（JD-01） |
| AG-003 | 経費精算申請エージェント | 経費精算申請書の作成 | 申請内容が経費・購入・備品・宿泊・資格に関連する（JD-02） |

### 5.2 振り分け判断基準

- **AG-002**: 交通費・移動・電車・バス・タクシー・飛行機に関連するキーワードが含まれる場合（JD-01）
- **AG-003**: 経費・購入・備品・宿泊・資格に関連するキーワードが含まれる場合（JD-02）
- **判断不能時**: 「交通費精算申請」「経費精算申請」の選択肢をユーザーへ提示（JD-03）

**詳細な判断基準**: 詳細設計書 agents/orchestrator_agent.py を参照

### 5.3 invocation_stateの受け渡し

専門エージェントにapplicant_name・application_date・session_idを渡す。

```python
{
    "applicant_name": str,      # 申請者名（アプリ起動時に取得）
    "application_date": str,    # 申請日（YYYY-MM-DD、システム日付を自動取得）
    "session_id": str           # セッションID（アプリ起動時に生成）
}
```

**データ構造の詳細**: 詳細設計書 agents/orchestrator_agent.py を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 申請者名の収集方法を指示する
- 申請種別の判断基準（JD-01〜JD-03）を定義する
- 専門エージェントへの委譲タイミングと方法を指示する

### 7.2 プロンプト構成要素
- **役割定義**: 申請受付窓口エージェントとしての役割
- **処理フロー**: 申請者名収集 → 申請内容受付 → 申請種別判断 → 委譲の流れ
- **判断プロトコル**: JD-01〜JD-03の判断基準
- **主な責任**: 申請種別判断・委譲のみ（申請書生成は担当外）

**プロンプト全文**: 詳細設計書 prompt/prompt_orchestrator.py を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001は申請種別判断・委譲のみ担当。日付ベースの業務ルール判断はAG-002/AG-003が担当） |
| 動的に埋め込む情報 | なし |

### 7.4 主要な設計意図
- **静的定数採用**: AG-001は申請種別判断・委譲のみ担当するため、日付依存の動的生成は不要
- **申請書生成禁止の明示**: システムプロンプトでTOOL-002の呼び出しを明示的に禁止する
- **フォールバック指示**: JD-03（判断不能）時の選択肢提示フローを明示する

**プロンプト全文**: 詳細設計書 prompt/prompt_orchestrator.py を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
- **申請者名**: 必須。アプリ起動時（対話ループ開始前）にユーザーへ入力を求める
- 入力方法: CLIの標準入力（`input()`）
- バリデーション: 空文字禁止（min_length=1）

### 10.2 対話ループでの入力
- **申請内容テキスト**: ユーザーが申請したい内容を自由テキストで入力する
- **特殊コマンド**:
  - `exit` / `quit`: 対話ループを終了する
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットし、申請者名の収集から再開する

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: CLIの標準出力（`print()`）

- 委譲結果: 専門エージェントからの応答テキストをそのまま提示

## 16. 制約事項・前提条件

### 16.1 技術的制約
- AG-001はTOOL-002（申請書生成ツール）を呼び出してはならない
- invocation_stateは辞書リテラルで渡す（専用Pydanticモデルは定義しない）

### 16.2 業務的制約
- 申請書生成はAG-002・AG-003が担当する（AG-001は委譲のみ）
- 申請種別判断不能時（JD-03）はユーザーへ選択肢を提示する

### 16.3 前提条件
- アプリ起動時に申請者名を取得済みであること
- session_idが生成済みであること
