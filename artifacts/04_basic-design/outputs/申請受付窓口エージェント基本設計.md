## 2. 目的・役割定義

### 2.1 エージェントの目的
- 社員の申請内容を受け付け、申請種別（交通費精算申請・経費精算申請）を判断して申請先を提示する
- 申請種別が判断できない場合はユーザーに選択肢を提示して確認する
- 申請種別確定後に専門エージェント（AG-002 または AG-003）へ処理を委譲する
- 申請ルール・方法が不明な利用者に社内申請ルールを案内する

### 2.2 主な責務
- **申請内容受付**: ユーザーの自然文入力を受け付ける（FR-001）
- **申請種別判断・提示**: 申請内容から交通費精算申請・経費精算申請のいずれかを判断して提示する（FR-002）
- **申請先提示**: 申請種別に応じた申請書名・申請先を提示する（FR-003）
- **専門エージェントへの委譲**: 申請種別確定後に AG-002 または AG-003 へ処理を委譲する
- **申請ルール案内**: 申請ルール・方法が不明な利用者に案内する（FR-014）

### 2.3 非責務（やらないこと）
- **交通費情報収集・申請書生成**: AG-002（交通費精算申請エージェント）が担当する
- **経費情報収集・申請書生成**: AG-003（経費精算申請エージェント）が担当する
- **申請書の提出**: BRL-06 に従い、提出は人が実行する

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルールの参照 | ACT-READ-01（申請内容受付）、ACT-READ-02（社内申請ルール参照） |
| 提案 | 申請種別の候補提示（最終選択は人） | ACT-DEC-01（申請種別判断） |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請内容から申請種別（交通費精算申請・経費精算申請）を判断して提示する
- **人間確認必要**: 申請種別が判断不能の場合、ユーザーに選択肢を提示して確認する（COND-001）
- **実行不可**: 申請書の生成・提出（ACT-GEN-01, ACT-EXEC-01）は実行しない

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始 → 申請者名取得 → 対話ループ開始
  → ユーザー入力受付（500文字チェック）
  → 申請種別判断
    → 判断可能: 申請先提示 → 専門エージェントへ委譲
    → 判断不能: 選択肢提示 → ユーザー選択 → 専門エージェントへ委譲
    → 申請ルール質問: 社内申請ルール案内
  → 対話ループ継続（30回上限）
→ 終了
```

**フローの詳細**: 詳細設計書 エージェント詳細設計を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 申請種別判断 → 申請先提示 → 専門エージェント委譲の逐次処理
- **処理をスキップする考え方**: 申請ルール案内のみの場合は専門エージェントへの委譲をスキップする
- **同一処理を複数回実行**: 申請種別判断不能時は選択肢提示を繰り返す（ユーザーが選択するまで）

### 4.3 状態管理
- **申請者名（user_name）**: アプリケーション起動時に取得し、インスタンス変数として保持する
- **申請日（request_date）**: 対話ループ開始時にシステム日付（YYYY-MM-DD形式）を自動取得する
- **セッションID（session_id）**: セッション管理モジュールが生成し、インスタンス変数として保持する
- **会話履歴**: SlidingWindowConversationManager（window_size=30）で管理する

## 5. 専門エージェントとの連携設計

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 交通費精算申請エージェント | 交通費精算申請の情報収集・申請書生成を委譲する | 申請種別が交通費精算申請として確定した場合 |
| AG-003 | 経費精算申請エージェント | 経費精算申請の情報収集・申請書生成を委譲する | 申請種別が経費精算申請として確定した場合 |

### 5.2 振り分け判断基準

- **AG-002（交通費精算申請）**: 申請内容に交通費・移動・電車・バス・タクシー・飛行機等のキーワードが含まれる場合
- **AG-003（経費精算申請）**: 申請内容に経費・領収書・購入・宿泊・資格等のキーワードが含まれる場合
- **判断不能時**: ユーザーに「交通費精算申請」「経費精算申請」の選択肢を提示して確認する（COND-001）

**詳細な判断基準**: 詳細設計書 エージェント詳細設計を参照

### 5.3 invocation_stateの受け渡し

専門エージェントに申請者名・申請日・セッションIDを渡す。

```python
invocation_state = {
    "user_name": self._user_name,       # str: 申請者名
    "request_date": request_date,        # str: 申請日（YYYY-MM-DD形式）
    "session_id": self._session_id       # str: セッションID
}
```

**データ構造の詳細**: artifacts/03_system-design/outputs/マルチエージェント連携設計.md 7.2 節を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 申請内容から申請種別を判断して提示する役割を定義する
- 申請種別判断不能時の選択肢提示手順を定義する
- 社内申請ルール（申請種別・申請先・申請期限等）を埋め込む

### 7.2 プロンプト構成要素
- **役割定義**: 申請受付窓口エージェントとしての役割（申請種別判断・申請先提示・申請ルール案内）
- **処理フロー**: 申請内容受付 → 申請種別判断 → 申請先提示 → 専門エージェント委譲の流れ
- **判断プロトコル**: 申請種別判断基準（BRL-01）・判断不能時の選択肢提示（BRL-02）・申請先特定（BRL-03）
- **主な責任**: 申請種別判断（提案）のみ。申請書生成・提出は行わない

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001 は申請種別判断・申請先提示のみ担当。日付ベースの業務ルール判断は AG-002/AG-003 が担当） |
| 動的に埋め込む情報 | なし |

### 7.4 主要な設計意図
- **社内申請ルールの埋め込み**: RAG を使用せず、社内申請ルールをシステムプロンプトに直接埋め込む（現フェーズの方針）
- **申請種別判断の提案**: 最終選択はユーザーが行う（自律度 Lv2）
- **専門エージェントへの委譲**: 申請種別確定後は AG-002/AG-003 に処理を委譲し、AG-001 は関与しない

**プロンプト全文**: 詳細設計書 システムプロンプト詳細設計を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
- **申請者名（user_name）**: 必須。アプリケーション起動時に main.py がユーザーに入力を求める
- 入力方法: main.py の対話ループ開始前に input() で取得する
- バリデーション: 空文字禁止（min_length=1）

### 10.2 対話ループでの入力
- **ユーザー自然文**: 申請内容・申請ルール質問・申請種別選択等
- **特殊コマンド**:
  - `exit` / `quit`: セッション終了
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして最初から開始する

### 10.3 invocation_stateの入力
AG-001 はオーケストレーターのため、invocation_state を受け取らない（生成して専門エージェントへ渡す）。

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: CLI 標準出力（print）
- **出力内容**: 申請種別・申請先の提示、申請ルール案内、専門エージェントからの応答の中継

### 11.2 出力メッセージ
- 申請種別提示: `"申請種別は「{申請種別}」です。申請先は「{申請先}」です。"`

- 申請ルール案内: 社内申請ルールに基づく案内文

## 16. 制約事項・前提条件

### 16.1 技術的制約
- AWS Strands SDK（strands-agents >= 0.1.0）を使用すること
- Amazon Bedrock（jp.anthropic.claude-sonnet-4-5-20250929-v1:0）を LLM バックエンドとして使用すること
- Python 3.10 以上が必要

### 16.2 業務的制約
- 申請書の提出（申請の実行）は AI が行わず、必ず人が実行する（BRL-06）
- 申請種別の最終選択はユーザーが行う（自律度 Lv2）
- 1 セッションの対話回数上限は 30 回（FR-016）

### 16.3 前提条件
- 社内申請ルールがシステムプロンプトに設定されていること
- AWS 認証情報（AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY）が環境変数に設定されていること
- AG-002・AG-003 が @tool(context=True) でラップされてツールとして登録されていること
