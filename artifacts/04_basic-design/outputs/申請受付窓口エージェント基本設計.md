## 2. 目的・役割定義

### 2.1 エージェントの目的
- 社員が自然言語で入力した申請内容から、申請種別（交通費精算申請・経費精算申請）を自動判断する
- 判断した申請種別に対応する申請書名・申請先を社員に案内する
- 適切な専門エージェント（AG-002またはAG-003）へ処理を委譲し、申請書作成を開始する
- 判断不能時はユーザーに選択肢を提示し、対象外の場合は担当者へのエスカレーションを案内する

### 2.2 主な責務
- **申請者名の受付**: アプリ起動時（main.py）に申請者名を取得し、初期化パラメータとして保持する
- **申請内容の受付**: 社員が入力した自然言語テキストを受け付ける（FR-001）
- **申請種別の判断**: 申請ルール（ナレッジ）を参照し、申請種別を自動判定する（FR-002）
- **申請種別・申請書・申請先の案内**: 判断結果に基づき申請書名と申請先を提示する（FR-003）
- **選択肢提示**: 判断不能時に「交通費精算申請」「経費精算申請」の2択を提示する（FR-004）
- **エスカレーション案内**: 対象外の申請に対して担当者への連絡を案内する（FR-005）
- **専門エージェントへの振り分け**: 確定した申請種別に応じてAG-002またはAG-003をAgent as Toolsとして呼び出す

### 2.3 非責務（やらないこと）
- **申請書の作成処理**: AG-002（経費精算申請エージェント）またはAG-003（交通費精算申請エージェント）が担当
- **領収書画像の読み取り・経費情報抽出**: AG-002が担当
- **交通費の計算・運賃データ検索**: AG-003が担当
- **申請書生成前の承認確認（Human-in-the-Loop）**: AG-002・AG-003が担当

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 申請内容テキストの受付、申請ルール（ナレッジ）の参照 | FR-001（申請内容受付）、ACT-READ-01、ACT-READ-02 |
| 提案 | 申請種別・申請書・申請先の提示、判断不能時の選択肢提示 | FR-003（案内提示）、FR-004（選択肢提示） |
| 実行 | 申請種別の自動判定、専門エージェントへの振り分け、エスカレーション案内 | FR-002（申請種別判断）、FR-005（エスカレーション案内）、ACT-DEC-01、ACT-EXEC-03 |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請種別の自動判定（BRL-01〜BRL-03、JD-01〜JD-04）、対象外判定時のエスカレーション案内（BRL-05）
- **人間確認必要**: 判断不能時の申請種別選択（最終決定はユーザー）
- **実行不可**: 申請書の作成・生成（P-EXEC-001、P-EXEC-002は禁止権限）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始 → 申請者名受付（main.py） → 申請内容入力促進 → 申請内容受付 → 申請種別判断 → [分岐] → 終了
                                                                              ├→ 経費精算申請 → AG-002へ委譲
                                                                              ├→ 交通費精算申請 → AG-003へ委譲
                                                                              ├→ 判断不能 → 選択肢提示 → ユーザー選択 → 委譲
                                                                              └→ 対象外 → エスカレーション案内
```

**フローの詳細**: 詳細設計書 2.3.1章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理（申請者名受付→申請内容受付→種別判断→案内/委譲の順序で実行）
- **処理をスキップする考え方**: 申請者名はアプリ起動時に取得済みのため、対話ループ内ではスキップする
- **同一処理を複数回実行**: 入力が空の場合は再入力を促す（CF-001 Step 1→2、Step 3→4のループ）
- **リセットコマンド**: `reset` / `リセット` / `最初から` が入力された場合、会話履歴と申請者情報をクリアし、申請者名の入力からやり直す

### 4.3 状態管理
- **申請者名（applicant_name）**: main.pyで取得し、インスタンス変数として保持。セッション内で不変
- **申請日（application_date）**: システム日付を自動取得（YYYY-MM-DD形式）。セッション内で不変
- **セッションID（session_id）**: main.py起動時に生成。FileSessionManagerによる永続化に使用
- **申請種別判断結果（D-003）**: LLM推論結果として会話履歴内に保持

## 5. 専門エージェントとの連携設計

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 経費精算申請エージェント | 経費精算申請書の作成を委譲する | 申請内容が経費精算申請に該当（BRL-02）、またはユーザーが「経費精算申請」を選択 |
| AG-003 | 交通費精算申請エージェント | 交通費精算申請書の作成を委譲する | 申請内容が交通費精算申請に該当（BRL-01）、またはユーザーが「交通費精算申請」を選択 |

### 5.2 振り分け判断基準

- **AG-003（交通費精算申請エージェント）**: 申請内容に「業務上の交通費精算（電車・バス・タクシー・飛行機）」に関する内容が含まれる場合（BRL-01、JD-01）
- **AG-002（経費精算申請エージェント）**: 申請内容に「業務上の経費精算（物品購入・宿泊・資格取得等）」に関する内容が含まれる場合（BRL-02、JD-02）

**詳細な判断基準**: 詳細設計書 2.3.1章を参照

### 5.3 invocation_stateの受け渡し

専門エージェントに申請者名・申請日・セッションIDを辞書リテラルで渡します。

```python
invocation_state = {
    "applicant_name": self._applicant_name,       # str: 申請者名（main.pyで取得済み）
    "application_date": datetime.now().strftime("%Y-%m-%d"),  # str: 申請日（システム日付）
    "session_id": self._session_id                # str: セッションID
}
```

**データ構造の詳細**: 詳細設計書 2.5.1章を参照

## 6. ツール設計

### 6.1 利用ツール

AG-001は機能ツール一覧（TOOL-XXX）に登録されたツールを直接使用しない。専門エージェント（AG-002、AG-003）をAgent as Toolsパターンで呼び出すことで、申請書作成処理を委譲する。

- **expense_agent（AG-002）**: 経費精算申請書作成を委譲するためのAgent as Tool
- **transport_agent（AG-003）**: 交通費精算申請書作成を委譲するためのAgent as Tool

AG-001のtools引数には、上記の専門エージェントツール関数（@tool(context=True)でデコレートされた関数）を登録する。

**ツールの詳細**: 詳細設計書 2.4.2章および各エージェント基本設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 申請受付窓口としての役割と振る舞いを定義する
- 申請種別判断のためのルール（BRL-01〜BRL-03、BRL-05〜BRL-07）を埋め込む
- 判断不能時・対象外時の対応フローを指示する

### 7.2 プロンプト構成要素
- **役割定義**: 申請受付窓口エージェントとしての役割（申請種別判断・案内・振り分け）
- **処理フロー**: 申請者名確認→申請内容受付→種別判断→案内/委譲の流れ
- **判断プロトコル**: BRL-01（交通費精算該当判断）、BRL-02（経費精算該当判断）、BRL-03（判断不能時選択肢提示）、BRL-05（対象外エスカレーション）の適用順序
- **案内ルール**: BRL-06（交通費精算申請の申請先）、BRL-07（経費精算申請の申請先）に基づく申請書名・申請先の提示
- **主な責任**: 正確な申請種別判断、ユーザーフレンドリーな案内、適切な専門エージェントへの委譲

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（日付ベースの業務ルール判断はこのエージェントのステップに含まれない） |
| 動的に埋め込む情報 | なし |

### 7.4 主要な設計意図
- **申請ルールのプロンプト埋め込み**: RAGを使用せず、knowledge/application_policies.pyのテキスト定数をシステムプロンプトに直接埋め込む方式を採用。レイテンシ削減と判断精度の安定化を図る
- **判断優先順序の明示**: BRL-01（交通費）→BRL-02（経費）→BRL-03（判断不能）→BRL-05（対象外）の順で評価するよう明示し、LLMの判断ブレを抑制する
- **専門エージェントへの委譲指示**: 申請種別確定後は速やかに対応する専門エージェントツールを呼び出すよう指示し、不要な対話を防止する

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 8. 共通設定

**システム設計書の方針**: artifacts/03_system-design/outputs/共通設定方針.md を参照してください。

### 8.1 設定方針

| 設定項目 | 設定値 | 選択理由 |
|---------|--------|---------|
| 会話管理 | SlidingWindow(30) | オーケストレーターとして複数の専門エージェントとのやり取りを保持する必要があるため、最大サイズを採用 |
| リトライ戦略 | 標準設定（max_attempts=6, initial_delay=4, max_delay=240） | Amazon Bedrock APIの一時的なエラーに対する自動リトライ。全エージェント共通設定 |
| ループ制御 | 最大10回 | ReActループの無限ループ防止。業務処理に必要な最大ステップ数を考慮した上限値 |
| 承認フック | なし | AG-001は申請書生成を行わないため、HumanApprovalHookは不要 |
| セッション管理 | FileSessionManager | 会話履歴の永続化・中断後の再開（Resume）に対応 |

**設定の詳細**: 詳細設計書 2.4.1章および共通設定パターンを参照

## 9. エラー・例外処理設計

**システム設計書の方針**: artifacts/03_system-design/outputs/例外処理方針.md を参照してください。

### 9.1 エラー対応方針

| エラー種別 | 対応方針 |
|-----------|----------|
| 入力例外（EX-01）: 空入力・500文字超過 | 再入力を促すメッセージを表示し、ユーザーに再入力を求める |

| 外部I/F例外（EX-04）: Bedrock APIエラー | ModelRetryStrategy（最大6回）で自動リトライ。リトライ失敗時はエラーメッセージを表示して業務終了 |
| 想定外（EX-08）: LoopLimitError・予期しない例外 | LoopLimitError → Exceptionの順でキャッチし、ErrorHandlerに委譲。エラーメッセージを表示して業務終了 |

## 10. 入力データ設計

### 10.1 初期化時の入力
- **申請者名（applicant_name）**: 必須、空文字禁止
  - 入力方法: アプリ起動時（main.py）にCLIから取得し、AG-001の初期化パラメータとして渡される
  - バリデーション: 空文字・空白のみの入力は再入力を促す

### 10.2 対話ループでの入力
- **申請内容テキスト（D-001）**: 社員が自然言語で入力する申請内容
- **特殊コマンド**:
  - `終了` / `quit`: アプリケーションを終了する
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットし、申請者名の入力からやり直す
- **入力制約**:
  - 500文字以内（FR-016）。超過時は入力が長すぎる旨を通知し再入力を促す
  - 空入力は再入力を促す（BRL-04）

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: CLI（標準出力）
- **出力内容**: 申請種別判断結果、申請書名・申請先の案内、選択肢提示、エスカレーション案内、専門エージェントの実行結果

### 11.2 出力メッセージ
- 申請者名入力促進: `"申請者名を入力してください。"`
- 申請内容入力促進: `"申請したい内容を入力してください。"`
- 申請種別案内（交通費精算）: `"交通費精算申請ですね。申請書は「交通費精算申請書」、申請先は所属部門の経理担当者です。"`
- 申請種別案内（経費精算）: `"経費精算申請ですね。申請書は「経費精算申請書」、申請先は所属部門の経理担当者です。"`
- 選択肢提示: `"申請種別を判断できませんでした。以下から選択してください：1. 交通費精算申請 2. 経費精算申請"`
- エスカレーション案内: `"申し訳ございませんが、この申請は本システムの対象外です。担当者にお問い合わせください。"`

### 11.3 生成ファイル

AG-001はファイルを生成しない。ファイル生成は専門エージェント（AG-002、AG-003）が担当する。

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 申請種別判断の正確性（交通費精算・経費精算・判断不能・対象外の各パターン）、専門エージェントへの正しい振り分け、申請書名・申請先の正しい案内

## 16. 制約事項・前提条件

### 16.1 技術的制約
- CLIインターフェースのみ対応（Web/Slack等のUIは対象外）
- 申請ルールはRAGを使用せず、システムプロンプトに直接埋め込む方式
- Agent as Toolsパターンによる同期呼び出し（非同期委譲は行わない）
- 最大委譲深度は1（AG-001→AG-002またはAG-003の1段階のみ）

### 16.2 業務的制約
- 対応申請種別は交通費精算申請・経費精算申請の2種類のみ
- 申請書の提出（申請システムへの登録）は本システムのスコープ外
- 1セッション内で委譲先は1エージェントのみ（AG-002またはAG-003の排他選択）
- 申請者名はアプリ起動時に1回のみ取得し、セッション内で変更不可

### 16.3 前提条件
- Amazon Bedrockが利用可能であること（AWS認証情報が設定済み）
- knowledge/application_policies.pyに申請ルール（ナレッジ）が定義されていること
- storage/sessions/ディレクトリが存在すること（main.py起動時に自動作成）
- 専門エージェント（AG-002、AG-003）が正常に動作すること
