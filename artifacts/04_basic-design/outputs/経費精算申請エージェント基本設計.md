## 2. 目的・役割定義

### 2.1 エージェントの目的
- 領収書画像から経費情報（購入日・店舗名・品目・金額）を自動抽出する
- 品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断する
- 申請期限チェック・業務目的確認・上長承認要否判断を実施する
- 収集済み情報から経費精算申請書（Excel）を自動生成する

### 2.2 主な責務
- **領収書情報抽出**: 領収書画像から購入日・店舗名・品目・金額を自動抽出する（FR-006, BRL-13）
- **経費区分判断**: 品目から経費区分を自動判断する（FR-007, BRL-14）
- **申請期限チェック**: 経費発生日から 3 ヶ月以内であることを確認する（FR-008, BRL-15）
- **業務目的確認**: 業務目的が記載されていることを確認する（FR-009, BRL-12）
- **不足情報対話収集**: 申請書テンプレートの必須項目と収集済み情報を照合して不足情報を対話で収集する（FR-010）
- **上長承認要否判断**: 経費合計が 5,000 円を超える場合に上長承認必要と判断する（FR-011, BRL-16, COND-003）
- **申請書自動生成**: TOOL-002（generate_general_expense_form）を呼び出して経費精算申請書を生成する（FR-012, BRL-05）
- **申請書確認・修正**: 修正指示に基づいて申請書を再生成する（FR-013）

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001（申請受付窓口エージェント）が担当する
- **交通費精算申請書生成**: AG-002（交通費精算申請エージェント）が担当する
- **申請書の提出**: BRL-06 に従い、提出は人が実行する

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 申請書テンプレート参照 | ACT-READ-04（申請書テンプレート参照） |
| 実行 | 領収書情報抽出・経費区分判断・申請期限チェック・上長承認要否判断・申請書生成 | ACT-DEC-02, ACT-DEC-03, ACT-DEC-04, ACT-GEN-01, ACT-GEN-02 |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請期限の適否判断（3 ヶ月以内）、上長承認要否判断（5,000 円超）、経費区分判断、不足情報の特定
- **人間確認必要**: 申請書の最終確認・承認（HumanApprovalHook による承認）、申請書の提出。領収書画像読み取り失敗時の手動入力
- **実行不可**: 申請書の提出（ACT-EXEC-01）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001 から委譲）
  → 領収書画像受付
    → 画像解析（LLM の画像解析機能）
    → 経費情報抽出（購入日・店舗名・品目・金額）
    → 読み取り失敗時: 手動入力を促す
  → 経費区分判断（LLM プロンプトベース）
  → 申請期限チェック（3ヶ月以内）
  → 業務目的確認
  → 不足情報対話収集
  → 上長承認要否判断（5,000円超）
  → 申請書ドラフト提示（ツール呼び出しなし）
    → 収集済み申請情報をテキストとして整理・提示する
  → 申請書生成確認（HumanApprovalHook: BeforeToolCallEvent でトリガー）
    → OK: 申請書生成（TOOL-002: generate_general_expense_form）
    → 修正: 修正指示受付 → 申請書再生成
    → キャンセル: 生成中止
  → 申請書ファイルパス提示
→ 終了
```

**フローの詳細**: 詳細設計書 エージェント詳細設計を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 領収書情報抽出 → 経費区分判断 → 申請期限チェック → 不足情報収集 → 申請書ドラフト提示 → HumanApprovalHook 承認 → 申請書生成の逐次処理
- **ドラフト提示とツール呼び出しの分離**: 申請書ドラフト提示はテキストとして行い（ツール呼び出しなし）、申請者の承認（OK）後にはじめて TOOL-002 を呼び出す。承認前にツールを実行してはならない
- **HumanApprovalHook の発火タイミング**: TOOL-002 呼び出し直前の BeforeToolCallEvent でトリガーされる。Human-in-the-Loop 承認は 1 回のみ発生する
- **処理をスキップする考え方**: 申請期限超過時は申請不可を案内して処理を終了する（BRL-15）
- **同一処理を複数回実行**: 複数の領収書がある場合は領収書ごとに情報抽出を繰り返す

### 4.3 状態管理
- **収集済み経費情報**: 会話履歴（SlidingWindowConversationManager, window_size=15）で管理する
- **申請者名・申請日**: invocation_state から取得する（tool_context.invocation_state.get("user_name") 等）
- **申請書ファイルパス**: TOOL-002 の戻り値から取得し、ユーザーへの提示に使用する

## 6. ツール設計

### 6.1 利用ツール

- **generate_general_expense_form（TOOL-002 の経費用関数）**: 収集済み経費情報から経費精算申請書（Excel）を生成する

各エージェントが使用するツールは業務機能（計算・生成など）に特化したツールのみとする。領収書画像の解析は LLM の画像解析機能（strands_tools の image_reader）を使用する（フレームワーク組み込み機能のため独立ツール不要）。経費区分判断は LLM プロンプトベースで実施する（独立ツール不要）。TOOL-002 は HumanApprovalHook による承認（BeforeToolCallEvent）後に呼び出す。承認前にツールを実行してはならない。

**ツールの詳細**: artifacts/04_basic-design/outputs/申請書生成ツール基本設計.md を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 経費精算申請の情報収集手順を定義する
- 領収書画像からの情報抽出手順を定義する
- 経費区分判断基準・申請期限チェック・上長承認要否判断の基準を定義する

### 7.2 プロンプト構成要素
- **役割定義**: 経費精算申請エージェントとしての役割（領収書情報抽出・経費区分判断・申請書生成）
- **処理フロー**: 領収書情報抽出 → 経費区分判断 → 申請期限チェック → 申請書ドラフト提示 → HumanApprovalHook 承認 → 申請書生成の流れ
- **判断プロトコル**: 申請期限チェック基準（BRL-15）・上長承認要否判断基準（BRL-16）・経費区分判断基準（BRL-14）・業務目的確認（BRL-12）
- **主な責任**: 経費精算申請書の生成まで。提出は人が実行
- **動的情報**: 申請日（request_date）・申請期限（request_date から 3 ヶ月前の日付）
- **エラー発生時の振る舞い**: ツールエラー・システム障害等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返す

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（BRL-15）で「経費発生日から 3 ヶ月以内」を判断するため、実行時の申請日（request_date）と申請期限（3 ヶ月前の日付）をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 申請者名（applicant_name）、申請日（application_date）、申請期限（deadline: application_date から 3 ヶ月前の日付） |

### 7.4 主要な設計意図
- **動的プロンプト生成**: 申請期限チェックに実行時の日付を使用するため、動的生成関数で申請日・期限日をプロンプトに埋め込む
- **LLM 画像解析の活用**: 領収書画像の解析は LLM の画像解析機能を使用し、独立ツールを不要にする
- **LLM プロンプトベースの経費区分判断**: 経費区分判断は LLM プロンプトで実施し、独立ツールを不要にする

**プロンプト全文**: 詳細設計書 システムプロンプト詳細設計を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
AG-003 はファクトリ関数（_build_general_expense_agent）で生成されるため、初期化時の直接入力はない。

### 10.2 対話ループでの入力
- **領収書画像**: ファイルパスまたは画像データ（ユーザーが提供）
- **経費情報**: 購入日・店舗名・品目・金額・業務目的（手動入力または画像から抽出）
- **修正指示**: 申請書確認後の修正内容（任意）

### 10.3 invocation_stateの入力
AG-001 から渡される情報:
```python
{
    "user_name": str,       # 申請者名
    "request_date": str     # 申請日（YYYY-MM-DD形式）
}
```
（session_id はファクトリ関数で消費済みのため除外）

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: AG-001 への応答文字列（str）
- **出力内容**: 申請書生成完了メッセージ・申請書ファイルパス

### 11.2 出力メッセージ
- 申請書生成完了: `"経費精算申請書を生成しました。ファイルパス: {file_path}"`
- 申請期限超過: `"申請期限（経費発生日から 3 ヶ月以内）を超過しているため、申請を受け付けることができません。"`
- エラー: ErrorHandler が生成するユーザー向けメッセージ

### 11.3 生成ファイル
- **ファイル名**: `経費精算申請書_{YYYYMMDD_HHMMSS}.xlsx`
- **保存先**: `output/`
- **内容**: 収集済み経費情報・申請者名・申請日・上長承認欄（必要時）を反映した Excel ファイル

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 領収書画像から経費情報が正しく抽出されること。経費区分が正しく判断されること。申請書が正しく生成されること
- **ドラフト生成テスト**: 申請書ドラフトがツール呼び出しなしでテキストとして提示されること。ドラフト提示ステップとツール呼び出しステップが明確に分離されていること
- **承認フローテスト**: HumanApprovalHook が BeforeToolCallEvent でトリガーされること。承認（OK）後にはじめて TOOL-002 が実行されること。承認前にツールが実行されないこと。Human-in-the-Loop 承認が 1 回のみ発生すること
- **異常系テスト**: 申請期限超過時に申請不可が案内されること。領収書画像読み取り失敗時に手動入力が促されること。システム系エラー発生時にエラー内容が要約されて AG-001 に返ること
- **性能テスト**: LoopControlHook の 10 回制限が正しく機能すること

**詳細なテストケース**: 詳細設計書 テスト観点章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- AWS Strands SDK（strands-agents >= 0.1.0）を使用すること
- Amazon Bedrock（jp.anthropic.claude-sonnet-4-5-20250929-v1:0）を LLM バックエンドとして使用すること
- openpyxl >= 3.1.0 が必要（申請書生成）
- Pillow が必要（strands_tools image_reader の間接依存）

### 16.2 業務的制約
- 申請書の提出（申請の実行）は AI が行わず、必ず人が実行する（BRL-06）
- 申請期限は経費発生日から 3 ヶ月以内（BRL-15）
- 経費 5,000 円超の場合は上長承認が必要（BRL-16）

### 16.3 前提条件
- AG-001 から申請種別が経費精算申請として確定した状態で委譲されること
- invocation_state に user_name・request_date が設定されていること
- template/経費精算申請書_template.xlsx が配置されていること
