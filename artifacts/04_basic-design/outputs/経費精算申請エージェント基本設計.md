## 2. 目的・役割定義

### 2.1 エージェントの目的
- 領収書画像から経費情報（購入日、店舗名、品目、金額）を自動抽出し、社員の手入力を最小化する
- 品目から経費区分を自動判断し、社員が経費区分を判断する手間を省く
- 不足情報を一括確認で対話収集し、効率的に申請書作成に必要な情報を揃える
- 申請期限チェック・上長承認チェックを自動実施し、申請ルールへの準拠を支援する
- 収集した情報をテンプレートに入力し、経費精算申請書（Excel）を自動生成する

### 2.2 主な責務
- **領収書情報自動抽出**: 領収書画像からLLMの画像解析機能（strands_toolsのimage_reader）を使用して経費情報（購入日、店舗名、品目、金額）を品目ごとに抽出する
- **経費区分自動判断**: 抽出した品目から経費区分（事務用品費、宿泊費、資格精算費、その他経費）を自動判断する（BRL-14）
- **不足情報の一括確認**: 申請に必要な不足情報（業務目的等）をまとめて質問し、計算可能な情報は確認しない（BRL-04, BRL-16）
- **申請期限チェック**: 経費発生日から3ヶ月以内かを判定し、超過時はエラーを通知する（BRL-09）
- **上長承認チェック**: 経費が5,000円を超えるか判定し、超過時は上長承認が必要な旨を通知する（BRL-11）
- **申請書自動生成**: 収集した全情報をユーザーに提示し、承認後に経費精算申請書（Excel）を生成する（TOOL-002）

### 2.3 非責務（やらないこと）
- **申請種別の判断**: AG-001（申請受付窓口エージェント）が担当
- **交通費の計算・交通費精算申請書の作成**: AG-003（交通費精算申請エージェント）が担当
- **申請書の提出（申請システムへの登録）**: 本システムのスコープ外

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 申請ルール（ナレッジ）の参照、領収書画像の読取 | knowledge/配下のポリシー参照、LLM画像解析による情報抽出 |
| 実行 | 経費区分の自動判断、申請期限判定、上長承認要否判定、不足情報の対話収集 | BRL-14による経費区分判断、BRL-09による期限チェック、BRL-11による承認チェック |
| 実行（承認必須） | 経費精算申請書の生成・保存 | HumanApprovalHookによるユーザー承認後にTOOL-002を実行 |

### 3.2 判断権限の範囲
- **自律判断可能**: 経費区分の自動判断（BRL-14）、申請期限の判定（BRL-09）、上長承認要否の判定（BRL-11）、情報充足確認（JD-05）
- **人間確認必要**: 申請書生成前の内容確認（OK/修正/キャンセルの選択）（APR-001）
- **実行不可**: 申請書の提出、承認権限の行使、申請種別の判断

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始 → 領収書画像依頼 → 経費情報自動抽出 → 経費区分自動判断 → 不足情報一括確認 → 情報充足確認 → 申請期限チェック → 上長承認チェック → 申請書ドラフト提示（テキスト整理、ツール呼び出しなし） → HumanApprovalHook承認（BeforeToolCallEvent） → 申請書生成・提示 → 終了
```

**ドラフト提示とツール実行の分離**:
- 「申請書ドラフト提示」ステップでは、収集済み申請情報をテキストとして整理・提示する（ツール呼び出しなし）
- 「申請書生成・提示」ステップでは、HumanApprovalHookがBeforeToolCallEventでトリガーし、申請者の承認（OK）後にはじめて申請書生成ツール（TOOL-002）を実行する
- Human-in-the-Loop承認は1回のみ発生し、承認前にツールを実行してはならない

**フローの詳細**: 詳細設計書 2.3.2章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: CF-002のステップ定義に従い逐次処理を行う（Step 1〜Step 12）
- **処理をスキップする考え方**: 計算可能な情報（合計金額等）はユーザーに確認せずエージェントが自動計算する
- **同一処理を複数回実行**: 情報充足確認（Step 7）で不足がある場合はStep 5（不足情報の一括確認）へ戻る。承認時に修正が選択された場合もStep 5へ戻る

### 4.3 状態管理
- **会話履歴**: SlidingWindowConversationManager（window_size=15）で直近15件のメッセージを保持
- **業務コンテキスト**: invocation_state経由で申請者名・申請日・セッションIDを保持（リクエスト単位）
- **経費明細データ**: LLMのコンテキストウィンドウ内で保持（領収書から抽出した情報＋対話収集した情報）

## 6. ツール設計

### 6.1 利用ツール

本エージェントが使用するツールは、業務機能（生成）に特化したツールのみとする。汎用ツール（ファイル操作・Web検索等）は使用しない。

- **TOOL-002（申請書生成ツール - generate_expense_report）**: 収集した経費情報をテンプレートに入力し、経費精算申請書（Excel）を生成・保存する

申請書生成ツールは、HumanApprovalHookによるユーザー承認（OK選択）後にのみ実行される。承認前にツールが呼び出された場合はHumanApprovalHookがBeforeToolCallEventで発火し、ユーザーの選択を待機する。

**ドラフト提示とツール実行の分離**:
- 申請書生成前に、エージェントは収集済み申請情報をテキストとして整理・提示する（この段階ではツール呼び出しを行わない）
- ドラフト提示後、HumanApprovalHookがBeforeToolCallEventでトリガーし、申請者の承認（OK）後にはじめてTOOL-002を実行する
- 承認前にTOOL-002を実行してはならない。Human-in-the-Loop承認は1回のみ発生する

**ツールの詳細**: 詳細設計書 2.4.2章および申請書生成ツール基本設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 経費精算申請エージェントとしての役割・責務を定義する
- CF-002の会話フローに従った処理手順をLLMに指示する
- 適用すべき業務ルール（BRL-04, BRL-09, BRL-11, BRL-13, BRL-14, BRL-16）をLLMに提示する
- 申請期限チェックのための日付情報（現在日付・3ヶ月前の日付）を動的に埋め込む

### 7.2 プロンプト構成要素
- **役割定義**: 経費精算申請書の作成を支援する専門エージェントとしての役割
- **処理フロー**: CF-002に基づく領収書画像依頼→情報抽出→不足情報確認→チェック→承認→生成の流れ
- **判断プロトコル**: 経費区分判断基準（BRL-14）、申請期限チェック基準（BRL-09）、上長承認チェック基準（BRL-11）
- **主な責任**: 領収書からの正確な情報抽出、不足情報の一括確認、申請ルールの適用
- **動的情報**: 現在日付（application_date）、3ヶ月前の日付（申請期限判定の基準日）
- **エラー発生時の振る舞い**: システム系エラー（ツールエラー・システム障害等）が発生した場合、エラー内容を要約して呼び出し元エージェント（AG-001）に返却する

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-002 Step 8で申請期限チェック（経費発生日から3ヶ月以内）をLLMに判断させるため、現在日付と3ヶ月前の日付をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 現在日付（YYYY-MM-DD）、3ヶ月前の日付（YYYY-MM-DD）、申請者名、申請日、申請期限 |

### 7.4 主要な設計意図
- **LLMによる申請期限判断**: 申請期限チェックをツール化せずLLMの推論で判断させるため、プロンプトに現在日付と3ヶ月前の日付を埋め込む
- **一括確認パターン**: 不足情報を1つずつ聞くのではなく、まとめて質問することで対話回数を削減する
- **計算可能情報の非確認**: 合計金額等の計算可能な情報はユーザーに確認せず、エージェントが自動計算する

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 9. エラー・例外処理設計

**システム設計書の方針**: artifacts/03_system-design/outputs/例外処理方針.md を参照してください。

### 9.1 エラー対応方針

| エラー種別 | 対応方針 |
|-----------|----------|
| 入力例外（EX-01） | 再入力を促すメッセージを表示し、ユーザーに再入力を求める（空入力、500文字超過等） |
| 業務ルール違反（EX-03） | 申請期限超過時はエラーメッセージを表示して業務終了。担当者への連絡を案内する |
| 外部I/F例外（EX-04） | ModelRetryStrategy（最大6回）で自動リトライ。リトライ失敗時はエラーメッセージを表示して業務終了 |
| 実行不可（EX-05） | キャンセル選択時は業務終了。修正選択時は情報収集ステップへ戻る |
| データ不整合（EX-06） | テンプレートファイル不存在・バリデーションエラー時はエラーメッセージを表示し担当者への連絡を案内 |
| 想定外（EX-08） | LoopLimitError → Exceptionの順でキャッチ。エラーメッセージを表示し担当者への連絡を案内して業務終了 |
| システム系エラー（ツールエラー・システム障害等） | エラー内容を要約し、呼び出し元エージェント（AG-001）に要約メッセージとして返却する |

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
- **ツール名**: expense_agent
- **デコレータ**: @tool(context=True)
- **入力**: query (str), tool_context (ToolContext)
- **出力**: str（エージェントからの応答）

### 13.2 ToolContextの使用
- invocation_stateからapplicant_name、application_date、session_idを取得
- session_idを使用してFileSessionManagerを初期化し、エージェントのセッション管理を行う

- 子エージェントへのinvocation_stateにはsession_idを除いたapplicant_nameとapplication_dateのみを渡す

**詳細**: 詳細設計書 2.5.2章を参照

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 領収書画像からの情報抽出精度、経費区分の自動判断正確性、不足情報の一括確認動作、申請期限チェック・上長承認チェックの正確性、申請書ファイルの正常生成
- **ドラフト提示テスト**: 申請書生成前に収集済み情報がテキストとして整理・提示されること（ツール呼び出しなし）、HumanApprovalHook承認（OK）後にはじめてTOOL-002が実行されること、承認前にツールが実行されないこと
- **異常系テスト**: 領収書画像読取失敗、申請期限超過時のエラー通知、テンプレートファイル不存在時のエラー処理、ユーザーキャンセル時の業務終了、LoopLimitError発生時の安全な停止
- **性能テスト**: 会話ウィンドウ（15件）超過時の切り詰め動作、ReActループ上限（10回）到達時の動作

**詳細なテストケース**: 詳細設計書 7章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- 領収書画像の解析はLLMの画像解析機能（strands_toolsのimage_reader）に依存する
- 申請書フォーマットは固定テンプレート（Excel: template/経費精算申請書_template.xlsx）に依存する
- CLIインターフェースのみ対応（画像ファイルパスの入力による提供）

### 16.2 業務的制約
- 対応する経費区分は4種類のみ（事務用品費、宿泊費、資格精算費、その他経費）
- 申請期限は経費発生日から3ヶ月以内（BRL-09）
- 上長承認閾値は経費5,000円超（BRL-11）
- 申請書の提出（申請システムへの登録）は本システムのスコープ外

### 16.3 前提条件
- AG-001（申請受付窓口エージェント）から経費精算申請として振り分けられた後に実行される
- invocation_stateにapplicant_name、application_date、session_idが設定されていること
- 経費精算申請書テンプレート（template/経費精算申請書_template.xlsx）が存在すること
- Amazon Bedrock（Claude Sonnet 4.5）が利用可能であること
