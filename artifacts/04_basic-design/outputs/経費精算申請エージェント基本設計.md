## 2. 目的・役割定義

### 2.1 エージェントの目的
- AG-001から委譲された経費精算申請を完結させる
- 経費情報を対話で収集し、経費区分を自動判断してユーザーへ確認を求める
- ユーザーのOK選択後に経費精算申請書（Excel）を生成する

### 2.2 主な責務
- **経費情報収集**: 購入日・品目・金額・業務目的を対話で収集する
- **経費区分判断**: 品目から経費区分を自動判断し、必ずユーザーへ確認を求める（BRL-14、自動確定禁止）
- **申請期限チェック**: 購入日から申請日までが90日以内かを確認し、超過時は警告を提示する（BRL-15）
- **上長承認要否通知**: 経費合計が5,000円超の場合に上長承認が必要である旨を通知する（BRL-16）
- **申請書ドラフト提示・承認確認**: 収集情報をテキストで提示し、OK/修正/キャンセルを求める
- **申請書生成**: ユーザーのOK選択後にTOOL-002（generate_expense_form）を呼び出す

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001が担当
- **交通費精算申請書生成**: AG-002が担当
- **承認ゲート制御**: HumanApprovalHook（HD-003）が担当
- **交通費計算**: AG-002・TOOL-001が担当

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 申請ルール参照（ACT-READ-01） | 申請ルール・経費区分ルールの参照 |
| 提案 | 経費区分判断（ACT-DEC-02） | 品目から経費区分を自動判断し候補を提示する（確認必須、自動確定禁止） |
| 実行 | 申請書ドラフト提示（ACT-GEN-01） | 収集情報をテキストで提示する |
| 実行（承認必須） | 申請書ファイル生成（ACT-EXEC-01） | ユーザーのOK選択後のみTOOL-002を呼び出す（COND-002） |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請期限チェック（BRL-15）・上長承認要否判断（BRL-16）
- **人間確認必要**: 経費区分の確定（BRL-14、COND-003）・申請書ドラフト確認（OK/修正/キャンセル）
- **実行不可**: ユーザー未承認での申請書ファイル生成・経費区分の自動確定

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001からの委譲受付）
  → 経費情報収集（購入日・品目・金額・業務目的）
  → 経費区分自動判断 → ユーザーへ確認（BRL-14、自動確定禁止）
  → 申請期限チェック（BRL-15）・上長承認要否通知（BRL-16）
  → 申請書ドラフト提示（収集情報をテキストで整理・提示。ツール呼び出しなし）
  → ユーザーへOK/修正/キャンセルを確認
  → [OK] HumanApprovalHookがBeforeToolCallEventでトリガー → ユーザー承認後にTOOL-002呼び出し（generate_expense_form）→ ファイルパス通知 → 終了
  → [修正] 経費情報収集へ戻る
  → [キャンセル] 処理中断 → 終了
```

**フローの詳細**: 詳細設計書 agents/expense_agent.py を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 経費情報収集 → 経費区分判断・確認 → 申請期限チェック → ドラフト提示 → 申請書生成の逐次処理
- **処理をスキップする考え方**: 経費区分が明確な場合でも必ずユーザー確認を実施（スキップ禁止）
- **同一処理を複数回実行**: 修正要望時は経費情報収集から再実行
- **ドラフト提示とツール呼び出しの分離**: 申請書ドラフトはテキストで提示（ツール呼び出しなし）。ユーザーのOK選択後にはじめてgenerate_expense_formを呼び出す。Human-in-the-Loop承認（HumanApprovalHook）はBeforeToolCallEventでトリガーし、承認前にツールを実行してはならない

### 4.3 状態管理
- **収集済み経費情報**: 会話履歴（SlidingWindowConversationManager）で保持
- **経費区分確認結果**: 会話履歴で保持
- **invocation_state**: tool_context.invocation_state.get()で参照（リクエスト単位）

## 6. ツール設計

### 6.1 利用ツール

- **generate_expense_form（TOOL-002の一部）**: 収集済み経費情報から経費精算申請書（Excel）を生成する。ユーザーのOK選択後のみ呼び出す

申請書ドラフトの提示はツール呼び出しなしでエージェントがテキストとして整理・提示する。ドラフト提示ステップとツール呼び出しステップは明確に分離し、Human-in-the-Loop承認（HumanApprovalHook）が1回のみ発生するようにする。

**ツールの詳細**: 詳細設計書 tools/output_generator.py を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 経費情報の収集方法を指示する
- 経費区分の自動判断・ユーザー確認フローを指示する（BRL-14、自動確定禁止）
- 申請期限チェック・上長承認要否判断の実施方法を指示する

### 7.2 プロンプト構成要素
- **役割定義**: 経費精算申請エージェントとしての役割
- **処理フロー**: 経費情報収集 → 経費区分判断・確認 → 申請期限チェック → ドラフト提示 → 申請書生成の流れ
- **判断プロトコル**: BRL-10〜BRL-17の申請ルール
- **主な責任**: 経費情報収集・経費区分判断・申請書生成
- **動的情報**: 申請日（application_date）・申請期限（application_date - 90日）
- **エラー発生時の振る舞い**: ツールエラー・システム障害等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返す

**プロンプト全文**: 詳細設計書 prompt/prompt_expense.py を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（BRL-15）でLLMが購入日と申請日を比較するため、実行時の申請日・申請期限日をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 申請者名（applicant_name）、申請日（application_date）、申請期限（application_dateから90日前の日付） |

### 7.4 主要な設計意図
- **動的生成採用**: BRL-15（申請期限チェック）でLLMが日付比較を行うため、実行時の申請日・期限日をプロンプトに埋め込む
- **経費区分自動確定禁止の明示**: BRL-14に従い、経費区分の自動確定を明示的に禁止する
- **申請ルール埋め込み**: BRL-10〜BRL-17をシステムプロンプトに直接埋め込む（RAG不使用）

**プロンプト全文**: 詳細設計書 prompt/prompt_expense.py を参照

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
- **ツール名**: `expense_agent`
- **デコレータ**: `@tool(context=True)`
- **入力**: `query: str`（AG-001からの指示テキスト）, `tool_context: ToolContext`
- **出力**: `str`（処理結果テキスト）

### 13.2 ToolContextの使用
- invocation_stateから`applicant_name`・`application_date`・`session_id`を取得
- `session_id`を使用してFileSessionManagerを生成（専門エージェントのセッション管理）
- 子エージェントへはsession_idを除いた`applicant_name`・`application_date`のみを伝播

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 経費情報収集 → 経費区分判断・確認 → ドラフト提示 → 申請書生成の一連フローが正常に動作すること
- **ドラフト提示テスト**: 申請書ドラフトがツール呼び出しなしでテキストとして提示されること
- **承認フローテスト**: HumanApprovalHookがBeforeToolCallEventでトリガーされ、ユーザー承認後にのみgenerate_expense_formが呼び出されること。承認前にツールが実行されないこと
- **異常系テスト**: 対応外経費区分でバリデーションエラーが発生すること。経費区分が自動確定されないこと（BRL-14）
- **システム系エラーテスト**: ツールエラー・LoopLimitError発生時にエラー内容が要約されてAG-001に返されること
- **性能テスト**: 申請書生成が正常に完了すること

**詳細なテストケース**: 詳細設計書 tests/ を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- invocation_stateはtool_context.invocation_state.get()で安全に参照すること
- HumanApprovalHookのtarget_toolsに`generate_expense_form`を登録すること

### 16.2 業務的制約
- 申請書生成はユーザーのOK選択後のみ実行可能（APR-001）
- 経費区分の自動確定禁止（BRL-14）。必ずユーザーへ確認を求めること

### 16.3 前提条件
- invocation_stateにapplicant_name・application_date・session_idが含まれていること
- TOOL-002が正常に動作すること
