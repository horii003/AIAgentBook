## 5. 主要設計決定事項のサマリー

| 設計項目 | 決定内容 |
|---------|---------|
| マルチエージェント方式 | Agent as Tools パターン（階層型）、委譲深度1 |
| エージェント間通信 | invocation_state（applicant_name, application_date, session_id）、辞書リテラルで渡す |
| ウィンドウサイズ | AG-001:30、AG-003:20、AG-002:15 |
| セッション管理 | FileSessionManager（storage/sessions/）、SlidingWindowConversationManager |
| 例外処理 | LoopLimitError → Exception の2層構造、ツール関数は辞書形式でエラー返却 |
| エラー伝播 | モジュール（ログ出力）→ ErrorHandler（メッセージ生成）→ 呼び出し元（伝播） |
| 実行制御 | LoopControlHook（最大10回、BeforeInvocation/AfterModelCall/AfterInvocation）、HumanApprovalHook（OK/修正/キャンセル） |
| リトライ戦略 | ModelRetryStrategy（max_attempts=6, initial_delay=4, max_delay=240） |
| バリデーション | Pydantic v2+、field_validator、ValidationError→ErrorHandler委譲。invocation_stateは辞書リテラル（Pydanticモデル不使用） |
| ログ出力 | Python標準logging、RotatingFileHandler（10MB×5世代）、デフォルトWARNING（LOG_LEVEL環境変数で制御）、推論トレース記録なし |
| ガードレール | Amazon Bedrock Guardrails + HumanApprovalHook + LoopControlHook の3層構成 |
| テスト方式 | pytest + strands-agents-evals、単体はモック、統合/E2Eは実呼び出し |
| 実行環境 | ローカルPC上でCLI使用、データはPCの所定フォルダにファイル格納 |
| 申請者情報取得 | 申請者名はアプリ起動時に取得、申請日はシステム日付自動取得（YYYY-MM-DD） |
