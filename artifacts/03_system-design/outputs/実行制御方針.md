## 1. 目的・適用範囲

### 1.1 目的

システム全体として **処理の開始・継続・中断・停止・再開** を一貫したルールで制御するための方針を定義する

### 1.2 適用範囲

- マルチエージェント処理（AG-001, AG-002, AG-003）
- 外部I/F実行（Amazon Bedrock、ファイルシステム）
- 人間承認を含む業務処理（HumanApprovalHook）

### 1.3 本方針で定義しないこと

- 個々のエージェントの内部制御ロジック
- SDK固有のキュー／スレッド制御
- 実装コードレベルの設定値

---

## 2. 実行制御の基本原則

- **制御はシステムが行う（エージェント任せにしない）**: LoopControlHook・HumanApprovalHook によりシステムが制御する
- **処理は必ず状態を持つ**: セッション状態（CREATED/ACTIVE/WAITING/CLOSED/TERMINATED）で管理する
- **再実行可能性を前提にする**: FileSessionManager によるセッション永続化で再開に対応する
- **制御不能な自律動作をさせない**: 最大ループ回数・対話回数上限・入力文字数制限で制御する
- **人が介入できる余地を残す**: 申請書生成前に HumanApprovalHook で承認ゲートを設ける


---

## 3. 実行単位の定義

### 3.1 実行単位の種類

| 単位 | 説明 | 識別子 |
|-----|------|-------|
| セッション | 1回の申請処理全体（申請受付から申請書生成完了または業務終了まで） | session_id |
| エージェント呼び出し | AG-001 の1回の対話処理、または AG-001 から AG-002/AG-003 への1回の委譲 | invocation_id（Strands SDK 内部管理） |
| ループ | エージェントの ReAct ループ（思考→ツール実行→観察の1サイクル） | iteration（LoopControlHook でカウント） |
| ツール実行 | TOOL-001（交通費計算）または TOOL-002（申請書生成）の1回の実行 | tool_name + execution_id |


### 3.2 実行単位の識別子

**session_id**:
- 形式: `session_{YYYYMMDD_HHMMSS}_{8桁ランダム文字列}`
- 例: `session_20260517_162400_a1b2c3d4`
- 生成: main.py 起動時に生成し、FileSessionManager に渡す

## 4. 実行状態管理

### 4.1 実行状態定義

| 状態 | 説明 | 適用対象 |
|-----|------|---------|
| READY | 実行待ち（セッション生成済み、処理未開始） | セッション |
| RUNNING | 実行中（エージェントが処理を実行中） | セッション、エージェント呼び出し |
| WAITING | 外部／人待ち（HumanApprovalHook 発火中） | セッション、エージェント呼び出し |
| COMPLETED | 正常終了（申請書生成完了またはキャンセル） | セッション、エージェント呼び出し、ツール実行 |
| FAILED | 失敗（ツール実行失敗、バリデーションエラー等） | ツール実行 |
| TERMINATED | 強制終了（LoopLimitError、予期しないエラー） | セッション、エージェント呼び出し |


### 4.2 状態遷移の責務

**状態管理主体**: FileSessionManager（セッション状態）、LoopControlHook（ループ状態）、HumanApprovalHook（承認状態）

**エージェントの役割**: エージェントは状態を直接変更せず、ハンドラー経由で状態を更新する

---

## 5. 実行開始制御

### 5.1 実行開始条件

**前提条件充足**:
- セッションIDが生成されていること
- FileSessionManager が初期化されていること
- 環境変数（AWS認証情報・ガードレールID等）が設定されていること

**権限確認完了**:
- 申請書生成（TOOL-002）は HumanApprovalHook による承認（OK 選択）が完了していること

**ガードレールチェック通過**:
- Amazon Bedrock Guardrails による入力コンテンツフィルタリングが通過していること


### 5.2 実行抑止条件

**同一セッションでの重複実行**:
- 同一 session_id での並列実行は禁止。FileSessionManager がセッションロックを管理する

**排他対象リソース使用中**:
- TOOL-002（申請書生成）は承認前の実行を禁止。HumanApprovalHook が event.cancel_tool で実行をキャンセルする

**システム負荷超過**:
- 対応なし（本システムはシングルユーザー・シングルセッション前提）


---

## 6. 並列・逐次制御方針

### 6.1 並列実行可否の判断軸

**データ競合有無**:
- 申請書ファイル（output/）への書き込みは1セッション内で1回のみ。並列書き込みは禁止

**外部I/F制約**:
- Amazon Bedrock API はリクエスト単位で逐次呼び出し。並列呼び出しは行わない

**業務整合性**:
- 1申請処理で委譲先エージェントは1つのみ（AG-002 または AG-003 の排他選択）

### 6.2 排他制御対象

| 対象 | 排他理由 | 実装方法 |
|-----|---------|---------|
| TOOL-002（申請書生成） | 承認前の実行禁止・重複生成防止 | HumanApprovalHook による event.cancel_tool |
| output/ ディレクトリ | 同一ファイル名での上書き防止 | ファイル名にタイムスタンプを付与（{申請種別}_{YYYYMMDD_HHMMSS}.xlsx） |

## 7. 再試行（Retry）方針

### 7.1 再試行対象

| 対象 | 再試行可否 | 最大回数 | 理由 |
|-----|----------|---------|------|
| Amazon Bedrock API 呼び出し | ○ | 6回 | ModelRetryStrategy による自動リトライ（指数バックオフ） |
| TOOL-001（交通費計算） | × | — | 運賃データ不存在は再試行しても解決しない。手動入力を案内する |
| TOOL-002（申請書生成） | × | — | 承認後1回のみ実行。失敗時は担当者エスカレーション |
| ユーザー入力（EX-01） | ○ | 制限なし（対話回数上限30回まで） | 再入力を促すことで解決可能 |

### 7.2 再試行制約

**最大回数**: 6回（Amazon Bedrock API、ModelRetryStrategy で設定）

**クールダウン**: 指数バックオフ（初期遅延4秒、最大遅延240秒）

**冪等性前提**: Amazon Bedrock API 呼び出しは冪等性を前提とする。TOOL-002 は冪等性なし（承認後1回のみ実行）

---

## 8. 中断・停止制御

### 8.1 中断（Pause）

**中断可能条件**:
- HumanApprovalHook 発火時（WAITING 状態）: ユーザーの OK/修正/キャンセル選択待ち

**中断時の状態保持**:
- FileSessionManager により会話履歴をファイルに永続化する。中断後の再開時に復元可能

### 8.2 停止（Terminate）

**停止条件**:
- LoopLimitError 発生（最大ループ回数10回到達）
- 予期しないエラー（Exception）発生
- ユーザーがキャンセルを選択（CLOSED 状態）

**停止後の復旧可否**:
- LoopLimitError: 復旧不可（新規セッションで再試行）
- 予期しないエラー: 復旧不可（新規セッションで再試行）
- ユーザーキャンセル: 復旧不可（新規セッションで再申請）

---

## 9. タイムアウト制御

### 9.1 タイムアウト対象

| 対象 | タイムアウト時間 | 実装状況 |
|-----|---------------|---------|
| Amazon Bedrock API 呼び出し | 要件上未定義 | ModelRetryStrategy の max_delay（240秒）が実質的な上限 |
| ユーザー入力待ち | 要件上未定義 | CLI の標準入力待ちのため、タイムアウトなし |
| HumanApprovalHook 承認待ち | 要件上未定義 | CLI の標準入力待ちのため、タイムアウトなし |

### 9.2 タイムアウト時の扱い

**例外分類**: EX-04（外部I/F例外）

**再試行 or エスカレーション**: ModelRetryStrategy による自動リトライ（最大6回）。リトライ失敗時はエラーメッセージを表示して業務終了

---

## 10. ループ制御

### 10.1 ループ制御の目的

- ReAct ループの無限ループを防止する
- エージェントの暴走を防止する
- システムリソースの枯渇を防止する

### 10.2 ループ制御の仕組み

**最大ループ回数**: 10回（デフォルト）

**実装クラス**: `LoopControlHook`（HookProvider として実装）

**制御方式**: LoopControlHook が登録イベントをフックし、ループ回数をカウントする

**登録イベント**:

| イベント | タイミング | 処理内容 |
|---------|----------|---------|
| BeforeInvocationEvent | エージェント呼び出し開始時 | ループカウンタを0にリセット |
| AfterModelCallEvent | LLM呼び出し後 | ループカウンタをインクリメントし、上限到達を監視。上限到達時に LoopLimitError を発生させる |
| AfterInvocationEvent | エージェント呼び出し完了後 | ループカウンタを0にリセット |

**カウント対象**: エージェントの ReAct ループ（LLM呼び出し1回を1カウント）

### 10.3 ループ制限到達時の処理

**処理**: LoopControlHook が LoopLimitError を発生させ、エージェントの処理を強制終了する

**ユーザーへの通知**: ErrorHandler が「処理の上限回数に達したため、申請処理を終了します。最初からやり直してください。」を表示する

**セッション状態**: TERMINATED に更新する

> ※ 具体的な実装はハンドラー基本設計.md を参照

## 11. 人間承認制御

### 11.1 承認制御の目的

- 申請書の誤生成を防止する
- ユーザーが申請内容を最終確認する機会を提供する
- 自律度 Lv4（承認後実行）を実現する

### 11.2 承認制御の仕組み

**発火タイミング**: TOOL-002（申請書生成ツール）の実行前（BeforeToolCallEvent）

**承認選択肢**:
1. OK（承認）: 申請書生成を実行する
2. 修正（修正要望）: 情報収集ステップへ戻り、再収集を行う
3. キャンセル（拒否）: 業務を終了する

### 11.3 承認結果の処理

| 選択肢 | 処理内容 |
|-------|---------|
| OK | event.cancel_tool = False（デフォルト）のまま TOOL-002 を実行する |
| 修正 | event.cancel_tool = "修正を選択しました。情報を修正してください。" でツール実行をキャンセルし、情報収集ステップへ戻る |
| キャンセル | event.cancel_tool = "キャンセルを選択しました。申請処理を終了します。" でツール実行をキャンセルし、業務を終了する |

> ※ 具体的な実装はハンドラー基本設計.md を参照

## 12. エスカレーション連携

**実行不能状態の扱い**:
- 例外処理方針のエスカレーション対象例外（EX-03, EX-04, EX-06, EX-07, EX-08）が発生した場合、ErrorHandler がユーザーへの担当者連絡案内メッセージを生成する

**実行中断 → エスカレーションの流れ**:
1. 例外発生（エージェント/ツール）
2. ErrorHandler が例外分類・継続可否判定
3. エスカレーション対象と判定
4. ユーザーへの担当者連絡案内メッセージを表示
5. セッション状態を TERMINATED に更新して業務終了

**実行再開条件**:
- エスカレーション後の再開は不可。担当者対応後に新規セッションで再申請する

---

## 13. セッション管理との連携

**実行状態とセッション状態の関係**:
- 実行状態（RUNNING/WAITING/COMPLETED/TERMINATED）はセッション状態（ACTIVE/WAITING/CLOSED/TERMINATED）と対応する
- 実行状態の変化は FileSessionManager を通じてセッション状態に反映される

**再開時の制御ポイント**:
- セッション再開時は FileSessionManager から会話履歴を復元し、ACTIVE 状態から処理を継続する
- 再開時に申請者名・申請日の再入力を促す（invocation_state は揮発性のため）

---

## 14. ガードレール連携

**実行前チェック**:
- Amazon Bedrock Guardrails による入力コンテンツフィルタリング（プロンプトインジェクション・有害コンテンツ検出）
- ガードレールトレース（guardrail_trace="enabled"）で検知結果を記録する

**実行中監視**:
- Amazon Bedrock Guardrails による出力コンテンツフィルタリング（業務スコープ外トピックの拒否）
- LoopControlHook によるループ回数監視

**実行後検証**:
- TOOL-002 の実行後に生成ファイルの存在確認を行う
- 申請書生成完了を INFO ログに記録する

## 15. ログ・監査方針

### 15.1 実行制御ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| event_type | イベントタイプ（処理開始・完了・ループ制限到達等） | logs/app.log（INFO/WARNING） |
| agent_name | エージェント名（AG-001〜AG-003） | logs/app.log（INFO） |
| iteration | ループ回数（LoopControlHook がカウント） | logs/app.log（WARNING：制限到達時） |
| tool_name | ツール名（TOOL-001, TOOL-002） | logs/app.log（INFO） |
| timestamp | タイムスタンプ | logs/app.log（自動付与） |

### 15.2 監査観点

**実行判断の説明可能性**:
- ループ制限到達・承認結果・エスカレーション判定をログに記録し、実行制御の判断経路を説明可能にする

**人間介入の記録**:
- HumanApprovalHook による承認結果（OK/修正/キャンセル）を INFO ログに記録する

## 16. 設計上の留意点・リスク

**過剰制御による性能劣化**:
- LoopControlHook が全ツール呼び出しで発火するため、対象ツール以外はスキップ処理が必要
- 対策: BeforeToolCallEvent のハンドラー内で対象ツール（TOOL-002）をフィルタリングする

**制御不足による暴走**:
- 最大ループ回数（10回）を超えた場合に LoopLimitError が発生しない可能性がある
- 対策: LoopControlHook の実装で確実にカウントし、上限到達時に LoopLimitError を発生させる

**再試行ループ**:
- ModelRetryStrategy の再試行が最大6回を超えて継続する可能性がある
- 対策: max_attempts=6 を厳守し、超過時は例外を上位に伝播させる
