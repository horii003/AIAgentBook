## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通するバリデーション方針を定義し、入力データの品質保証と業務ルールの適用を一貫して実現する。

### 1.2 適用範囲

- 全エージェント（AG-001, AG-002, AG-003）のツール関数入力
- invocation_state（エージェント間共有コンテキスト）
- 外部データ（運賃データ JSON ファイル）の読み込み時
- 申請書生成ツール（TOOL-002）の入力データ

### 1.3 本方針で定義すること

- バリデーションの基本原則とタイミング
- Pydantic v2+ を使用したデータモデルの使用方針
- データ型別バリデーション（日付・金額・文字列・列挙型）
- 業務ルールに基づくカスタムバリデーション
- バリデーションエラーメッセージの方針

### 1.4 本方針で定義しないこと

- 具体的な Pydantic モデルのフィールド定義（データモデル基本設計書に委ねる）
- バリデーション実装の詳細コード（詳細設計書に委ねる）

## 2. バリデーションの基本原則

### 2.1 基本方針

- **早期検証**: ツール関数の入口・invocation_state 受取時に即座にバリデーションを実施する
- **型安全性**: Pydantic v2+ により全入力データの型を保証する
- **ユーザーフレンドリー**: バリデーションエラーは日本語で分かりやすく通知する
- **自動正規化**: 駅名・経費区分等の表記ゆれは field_validator で自動正規化する
- **一貫性**: 全ツール関数で同一のバリデーションパターンを使用する

### 2.2 バリデーションのタイミング

| タイミング | 対象 | 実装場所 |
|----------|------|---------|
| ツール関数の入口 | ツールパラメータ（TOOL-001, TOOL-002 の入力） | tools/{domain}_tools.py |
| invocation_state 受取時 | エージェント間共有コンテキスト（applicant_name, application_date, session_id） | tools/{domain}_tools.py（.get() による安全なアクセス） |
| invocation_state の再バリデーション | 専門エージェントツール関数内での invocation_state 参照時 | tools/output_generator.py |
| 外部データ読み込み時 | 運賃データ JSON ファイルの読み込み後 | tools/transport_tools.py |

### 2.3 バリデーション失敗時の扱い

**ValidationError発生時**:
- ValidationError をキャッチし、ErrorHandler.handle_validation_error() に委譲する
- ErrorHandler がユーザー向けの日本語エラーメッセージを生成する
- 呼び出し元モジュールが ERROR ログを出力する（exc_info=True でスタックトレースを付与）
- ツール関数は辞書形式でエラーを返す（例外を再送出しない）

## 3. Pydanticモデルの使用方針

### 3.1 基本原則

- 外部から受け取るデータは全て Pydantic モデルでバリデーションを行う
- バリデーション済みデータはモデルのインスタンス属性または model_dump() を通じてアクセスする
- 未使用モデルの定義は禁止する（デッドコード防止）

### 3.2 Pydanticモデルの定義場所

**定義場所**: `models/data_models.py`（共有モデル）、`tools/{domain}_tools.py`（ツール固有モデル）

**命名規則**:
- ツール入力モデル: `{ToolName}Input`（例: `TransportCalculatorInput`）
- ツール出力モデル: `{ToolName}Output`（例: `TransportCalculatorOutput`）
- マスタデータモデル: `{DataName}Record`（例: `TrainRouteRecord`）

> ※ invocation_state は辞書リテラルで渡すため、専用の Pydantic モデル（InvocationState クラス）は定義しない

### 3.3 Pydanticモデルの基本構造

```python
from pydantic import BaseModel, field_validator, Field
from typing import Optional, Literal

class TransportCalculatorInput(BaseModel):
    """交通費計算ツールの入力モデル"""
    departure: str = Field(..., min_length=1, description="出発地（正規化済み駅名）")
    destination: str = Field(..., min_length=1, description="目的地（正規化済み駅名）")
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"] = Field(
        ..., description="交通手段"
    )

    # 共通バリデーター関数の適用
    _validate_departure = field_validator("departure", mode="before")(
        classmethod(validate_station_name)
    )
    _validate_destination = field_validator("destination", mode="before")(
        classmethod(validate_station_name)
    )
```

## 4. 日付のバリデーション

### 4.1 基本原則

- 日付は YYYY-MM-DD 形式（ISO 8601）を標準とする
- python-dateutil を使用して柔軟な日付文字列を解析し、標準形式に変換する
- 未来日付・過去日付の制約は業務ルールに基づいて適用する

### 4.2 日付の定義

**標準形式**: `YYYY-MM-DD`（例: `2026-05-17`）

**許容値**: 有効な日付文字列（python-dateutil で解析可能なもの）

### 4.3 日付バリデーションの実装

```python
from datetime import date
from dateutil import parser as dateutil_parser

def validate_date_format(v: str) -> str:
    """日付文字列を YYYY-MM-DD 形式に正規化する共通バリデーター"""
    try:
        parsed = dateutil_parser.parse(v)
        return parsed.strftime("%Y-%m-%d")
    except (ValueError, TypeError):
        raise ValueError(f"日付の形式が正しくありません: {v}（YYYY-MM-DD 形式で入力してください）")
```

### 4.4 日付の特殊ルール

**申請期限チェック（BRL-08, BRL-09）**:
- 経費発生日から申請日までが3ヶ月以内であること
- 超過時は ValidationError ではなく業務エラーとして処理する（ErrorHandler.handle_deadline_error()）

## 5. 金額のバリデーション

### 5.1 基本原則

- 金額は整数（円）で管理する
- 0以上の値のみ許容する
- 上長承認閾値チェックはバリデーションではなく業務ロジックとして実装する

### 5.2 金額の定義

**標準形式**: `int`（円単位）

**許容値**: 0以上の整数

### 5.3 金額バリデーションの実装

```python
from pydantic import Field

amount: int = Field(..., ge=0, description="金額（円）")
```

### 5.4 金額の特殊ルール

**上長承認チェック（BRL-10, BRL-11）**:
- 交通費合計 > 10,000円: 上長承認が必要な旨を通知（バリデーションエラーではない）
- 経費合計 > 5,000円: 上長承認が必要な旨を通知（バリデーションエラーではない）
- 上長承認チェックはツール関数内の業務ロジックとして実装する

## 6. 文字列のバリデーション

### 6.1 基本原則

- 必須文字列フィールドは min_length=1 で空文字を禁止する
- ユーザー入力は500文字以内に制限する（FR-016）
- 申請者名・業務目的等の必須項目は空文字・空白のみを禁止する

### 6.2 文字列の定義

**標準形式**: `str`

**許容値**: 空文字以外の文字列（必須フィールドの場合）

### 6.3 文字列バリデーションの実装

```python
from pydantic import Field

# 必須文字列（空文字禁止）
applicant_name: str = Field(..., min_length=1, description="申請者名")

# 任意文字列
notes: Optional[str] = Field(None, description="備考")
```

### 6.4 文字列の特殊ルール

**ユーザー入力文字数制限（FR-016）**:
- main.py の入力受付ループで500文字超過を検知し、再入力を促す
- Pydantic モデルのバリデーションではなく、main.py の入力処理で実装する

## 7. 列挙型のバリデーション

### 7.1 基本原則

- 列挙型フィールドは `Literal` 型で定義し、受け入れ可能な値を明示する
- LLM が渡す可能性のある表記バリエーション（別表記・略称等）を field_validator で正規化する
- 正規化後の値を後続処理で使用する（入力値をそのまま使用しない）

### 7.2 列挙型の定義

**交通手段**:
- 許容値: `"電車"`, `"バス"`, `"タクシー"`, `"飛行機"`
- 別表記: `"train"`, `"bus"`, `"taxi"`, `"airplane"`, `"plane"` 等

**経費区分**:
- 許容値: `"事務用品費"`, `"宿泊費"`, `"資格精算費"`, `"その他経費"`
- 別表記: `"事務用品"`, `"宿泊"`, `"資格"`, `"その他"` 等

### 7.3 列挙型バリデーションの実装

```python
from typing import Literal
from pydantic import field_validator

class TransportCalculatorInput(BaseModel):
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"] = Field(
        ..., description="交通手段"
    )

    @field_validator("transport_type", mode="before")
    @classmethod
    def normalize_transport_type(cls, v: str) -> str:
        """交通手段の表記ゆれを正規化する"""
        mapping = {
            "train": "電車", "電車": "電車",
            "bus": "バス", "バス": "バス",
            "taxi": "タクシー", "タクシー": "タクシー",
            "airplane": "飛行機", "plane": "飛行機", "飛行機": "飛行機",
        }
        normalized = mapping.get(v.lower() if isinstance(v, str) else v)
        if normalized is None:
            raise ValueError(f"交通手段の値が正しくありません: {v}（電車・バス・タクシー・飛行機のいずれかを指定してください）")
        return normalized
```

## 8. カスタムバリデーションルール

### 8.1 駅名正規化（BRL-15）

**目的**: 入力された駅名の表記ゆれを正規化する（「渋谷駅」→「渋谷」）

**実装パターン**:
```python
def validate_station_name(v: str) -> str:
    """駅名の表記ゆれを正規化する共通バリデーター"""
    if not v:
        raise ValueError("駅名を入力してください")
    # 「〇〇駅」→「〇〇」に正規化
    return v.removesuffix("駅").strip()
```

### 8.2 invocation_state バリデーション

**目的**: エージェント間で渡す共有コンテキストの値を検証する

**実装方針**:
- invocation_state は辞書リテラルで渡す。専用の Pydantic モデル（InvocationState クラス）は定義しない
- バリデーションはツール関数内で `.get()` による安全なアクセスと、必要に応じた値チェックで実施する

**フィールド構成**:
| フィールド名 | 型 | 説明 |
|------------|---|------|
| applicant_name | str | 申請者名 |
| application_date | str | 申請日（YYYY-MM-DD） |
| session_id | str | セッションID |

**参照パターン**:
```python
# ツール関数内での invocation_state 参照
applicant_name = tool_context.invocation_state.get("applicant_name")
application_date = tool_context.invocation_state.get("application_date")
```

## 9. バリデーションエラーメッセージ

### 9.1 基本原則

- **わかりやすさ**: 技術的な詳細を隠蔽し、ユーザーが理解できる言葉で説明する
- **具体性**: どのフィールドで何が問題かを明示する
- **一貫性**: 全バリデーションエラーで同一の形式を使用する

### 9.2 エラーメッセージの形式

**基本形式**:
```
{フィールド名}のエラー: {エラー内容}
対処方法: {対処方法}
```

### 9.3 エラーメッセージの実装

```python
from pydantic import ValidationError

try:
    validated = TransportCalculatorInput(**data)
except ValidationError as e:
    error_message = ErrorHandler.handle_validation_error(e)
    _logger.error("バリデーションエラー: %s", e, exc_info=True)
    return {"success": False, "error_message": error_message}
```

**具体的なエラーメッセージ例**:
```
# 交通手段が不正
「交通手段のエラー: 指定された交通手段が正しくありません。
対処方法: 電車・バス・タクシー・飛行機のいずれかを指定してください。」

# 駅名が空
「出発地のエラー: 駅名を入力してください。
対処方法: 出発地の駅名を入力してください。」

# 申請日の形式が不正
「申請日のエラー: 日付の形式が正しくありません。
対処方法: YYYY-MM-DD 形式（例: 2026-05-17）で入力してください。」
```

## 10. バリデーションのテスト

### 10.1 テスト観点

- 正常系: 有効な入力値でバリデーションが通過すること
- 異常系: 無効な入力値でバリデーションエラーが発生すること
- 境界値: 最小値・最大値・境界値でのバリデーション動作
- 正規化: 表記ゆれが正規化されること
- エラーメッセージ: 日本語で適切なメッセージが生成されること

### 10.2 テストケースの例

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 正常系: 交通手段（電車） | `transport_type="電車"` | バリデーション通過 |
| 正常系: 交通手段（英語表記） | `transport_type="train"` | 正規化後 `"電車"` |
| 異常系: 交通手段（不正値） | `transport_type="自転車"` | ValidationError |
| 正常系: 駅名（「渋谷駅」） | `departure="渋谷駅"` | 正規化後 `"渋谷"` |
| 異常系: 駅名（空文字） | `departure=""` | ValidationError |
| 正常系: 金額（0円） | `amount=0` | バリデーション通過 |
| 異常系: 金額（負の値） | `amount=-1` | ValidationError |
| 正常系: 日付（YYYY-MM-DD） | `application_date="2026-05-17"` | バリデーション通過 |
| 異常系: 日付（不正形式） | `application_date="2026/05/17"` | 正規化後 `"2026-05-17"` または ValidationError |

> テスト設計の詳細は評価テスト方式設計.md を参照

## 11. 設計上の留意点・リスク

### 11.1 LLM の出力バリデーション

**リスク**: LLM が Pydantic モデルのフィールド名と異なるキー名でツールパラメータを渡す可能性がある

**対策**: @tool デコレータの docstring に各フィールドの説明を明記し、LLM が正しいパラメータを構築できるようにする（R9.9.2 参照）

### 11.2 ツール出力モデルのキー名不一致

**リスク**: ツール関数のエラー返却辞書のキー名が出力 Pydantic モデルのフィールド名と一致しない場合、エラーが正しく処理されない

**対策**: ツール関数のエラー返却辞書のキー名は出力 Pydantic モデルのフィールド名と必ず一致させる（例: `{"error_message": ...}` ← `success`, `error_message` フィールドを持つモデルに対応）
