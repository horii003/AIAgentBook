## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通するバリデーション方針を定義し、各エージェント・ツール・データモデルで一貫したデータ検証を実施する。

### 1.2 適用範囲

- 全エージェント（AG-001, AG-002, AG-003）のツールパラメータ
- 全ツール（TOOL-001, TOOL-002）の入力データ
- invocation_state（エージェント間で共有するコンテキスト情報）
- 外部データファイル（電車経路テーブル・固定運賃データ）の読み込み時

### 1.3 本方針で定義すること

- バリデーションの基本原則とタイミング
- Pydanticモデルの使用方針（定義場所・命名規則・基本構造）
- データ型別バリデーション（日付・金額・文字列・列挙型）
- 業務ルールに基づくカスタムバリデーション
- バリデーションエラーメッセージの方針

### 1.4 本方針で定義しないこと

- 具体的なPydanticモデルのフィールド定義（データモデル基本設計書に委ねる）
- ガードレールによる入力フィルタリング（ガードレール処理方式設計に委ねる）
- テストコードの実装（テスト設計書に委ねる）

## 2. バリデーションの基本原則

### 2.1 基本方針

- **早期検証**: ツール関数の入口・invocation_state受取時に即座にバリデーションを実施する
- **型安全性**: Pydantic v2+を使用してすべての入力と状態モデルの型を保証する
- **ユーザーフレンドリー**: バリデーションエラーメッセージは日本語で記述し、対処方法を提示する
- **自動正規化**: 列挙型フィールドは受け入れ後に統一表記へ変換する（field_validatorで実装）
- **一貫性**: 全ツール・全エージェントで同一のバリデーションパターンを使用する

### 2.2 バリデーションのタイミング

| タイミング | 対象 | 実装場所 |
|----------|------|---------|
| ツール関数の入口 | ツールパラメータ（TOOL-001, TOOL-002の入力） | tools/{domain}_tools.py |
| invocation_state受取時 | エージェント間で共有するコンテキスト情報 | agents/orchestrator_agent.py |
| invocation_stateの再バリデーション | 専門エージェントが受け取るコンテキスト情報 | tools/output_generator.py |
| マスタデータ読み込み時 | 電車経路テーブル・固定運賃データのJSONデータ | tools/transport_tools.py |

### 2.3 バリデーション失敗時の扱い

**ValidationError発生時**:
- ValidationErrorをキャッチし、ErrorHandler.handle_validation_error()に委譲する
- ユーザー向けの日本語エラーメッセージを生成し、技術的な詳細を隠蔽する
- ツール関数はエラー情報を辞書形式で返す（例外を再送出しない）
- エラー返却辞書のキー名は出力Pydanticモデルのフィールド名と一致させる

## 3. Pydanticモデルの使用方針

### 3.1 基本原則

- 外部から受け取るデータはすべてPydanticモデルでバリデーションを行う
- バリデーション済みデータはPydanticモデルのインスタンス属性またはmodel_dump()を通じてアクセスする
- 定義したモデルは対応するツール関数内で実際に使用する（未使用モデルはデッドコードとして削除する）

### 3.2 Pydanticモデルの定義場所

**定義場所**: `models/data_models.py`（共通モデル）、`tools/{domain}_tools.py`（ツール固有のモデル）

**命名規則**:
- ツール入力モデル: `{ツール名}Input`（例: `TransportCalculatorInput`）
- ツール出力モデル: `{ツール名}Output`（例: `TransportCalculatorOutput`）
- マスタデータモデル: `{データ名}Data`（例: `TrainFareData`）

> **invocation_stateについて**: invocation_stateは辞書リテラルで渡す。専用のPydanticモデルは定義しない（8.1節参照）。

### 3.3 Pydanticモデルの基本構造

```python
from pydantic import BaseModel, field_validator, Field
from typing import Optional, Literal

class TransportCalculatorInput(BaseModel):
    """交通費計算ツールの入力モデル"""
    departure: str = Field(..., min_length=1, description="出発地（正規化済み駅名）")
    destination: str = Field(..., min_length=1, description="目的地（正規化済み駅名）")
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"] = Field(
        ..., description="交通手段"
    )

    @field_validator("transport_type", mode="before")
    @classmethod
    def normalize_transport_type(cls, v: str) -> str:
        """交通手段の表記を正規化する"""
        mapping = {
            "train": "電車",
            "bus": "バス",
            "taxi": "タクシー",
            "airplane": "飛行機",
            "飛行機": "飛行機",
        }
        return mapping.get(v, v)
```

## 4. 日付のバリデーション

### 4.1 基本原則

- 日付はYYYY-MM-DD形式の文字列として管理する
- python-dateutilを使用して日付文字列をパースし、形式の正当性を検証する
- 申請期限チェック（経費発生日から3ヶ月以内）はバリデーション後に実施する

### 4.2 日付の定義

**標準形式**: `YYYY-MM-DD`（例: `2026-05-21`）

**許容値**: 有効な日付（存在する日付のみ）

### 4.3 日付バリデーションの実装

```python
from pydantic import field_validator
from dateutil import parser as dateutil_parser

@field_validator("expense_date", mode="before")
@classmethod
def validate_expense_date(cls, v: str) -> str:
    """経費発生日の形式を検証する（YYYY-MM-DD形式）"""
    try:
        dateutil_parser.parse(v)
        return v
    except (ValueError, TypeError):
        raise ValueError(f"日付の形式が正しくありません: {v}（YYYY-MM-DD形式で入力してください）")
```

### 4.4 日付の特殊ルール

**申請期限チェック（BRL-09, BRL-15）**:
- 経費発生日から申請日（request_date）までの期間が3ヶ月以内であることを確認する
- 期限超過時はバリデーションエラーではなく業務ルール違反（EX-03）として処理する
- 日付計算にはpython-dateutilのrelativedeltaを使用する

## 5. 金額のバリデーション

### 5.1 基本原則

- 金額は整数型（int）または浮動小数点型（float）として管理する
- 金額は0以上の値のみ許容する（負の金額は禁止）
- 上長承認要否判断（BRL-10, BRL-16）はバリデーション後に実施する

### 5.2 金額の定義

**標準形式**: 整数（円単位）

**許容値**: 0以上の整数

### 5.3 金額バリデーションの実装

```python
from pydantic import Field

class ExpenseInput(BaseModel):
    """経費情報の入力モデル"""
    amount: int = Field(..., ge=0, description="金額（円）")
```

### 5.4 金額の特殊ルール

**上長承認要否判断（BRL-10）**:
- 交通費合計が10,000円を超える場合は上長承認欄を申請書に追加する
- 閾値比較はツール関数内で実施する（バリデーションではなく業務ロジック）

**上長承認要否判断（BRL-16）**:
- 経費合計が5,000円を超える場合は上長承認欄を申請書に追加する
- 閾値比較はツール関数内で実施する（バリデーションではなく業務ロジック）

## 6. 文字列のバリデーション

### 6.1 基本原則

- 必須文字列フィールドは空文字を禁止する（min_length=1）
- ユーザー入力の文字数制限（500文字）はmain.pyの入力ループで実施する（Pydanticモデルでは実施しない）
- 申請者名・業務目的等の必須フィールドは空文字を禁止する

### 6.2 文字列の定義

**標準形式**: UTF-8文字列

**許容値**: 空文字以外の文字列（必須フィールドの場合）

### 6.3 文字列バリデーションの実装

```python
from pydantic import Field

class ApplicationInput(BaseModel):
    """申請情報の入力モデル"""
    applicant_name: str = Field(..., min_length=1, description="申請者名")
    business_purpose: str = Field(..., min_length=1, description="業務目的")
```

### 6.4 文字列の特殊ルール

**業務目的確認（BRL-12）**:
- 業務目的が記載されていることを確認する（min_length=1で空文字を禁止）
- 未記載時はバリデーションエラーとして処理し、入力を促す

## 7. 列挙型のバリデーション

### 7.1 基本原則

- 列挙型フィールドはLiteral型またはEnumで定義し、受け入れ可能な値を明示する
- LLMが渡す可能性のある表記バリエーション（別表記・略称等）を洗い出し、field_validatorで正規化する
- 正規化後の値を後続処理で使用する（入力値をそのまま使用しない）

### 7.2 列挙型の定義

**交通手段**: `Literal["電車", "バス", "タクシー", "飛行機"]`

**経費区分**: `Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]`

**申請種別**: `Literal["交通費精算申請", "経費精算申請"]`

### 7.3 列挙型バリデーションの実装

```python
from typing import Literal
from pydantic import field_validator

class ExpenseCategoryInput(BaseModel):
    """経費区分の入力モデル"""
    category: Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"] = Field(
        ..., description="経費区分"
    )

    @field_validator("category", mode="before")
    @classmethod
    def normalize_category(cls, v: str) -> str:
        """経費区分の表記を正規化する"""
        mapping = {
            "office_supplies": "事務用品費",
            "accommodation": "宿泊費",
            "qualification": "資格精算費",
            "other": "その他経費",
            "その他": "その他経費",
        }
        return mapping.get(v, v)
```

### 7.4 列挙型の特殊ルール

**経費区分判断（BRL-14）**:
- 品目から経費区分を自動判断する処理はLLMプロンプトベースで実施する
- 判断結果をfield_validatorで正規化し、Literal型で型安全性を保証する

## 8. カスタムバリデーションルール

### 8.1 invocation_stateのバリデーション

invocation_stateは辞書リテラルで渡す。専用のPydanticモデルは定義しない。ツール関数内では`tool_context.invocation_state.get("key")`で安全にアクセスする。

```python
# invocation_stateの参照例（ツール関数内）
user_name = tool_context.invocation_state.get("user_name")
request_date = tool_context.invocation_state.get("request_date")
```

**invocation_stateのフィールド**:
| フィールド | 型 | 説明 |
|----------|---|------|
| user_name | str | 申請者名（必須） |
| request_date | str | 申請日（YYYY-MM-DD形式、必須） |
| session_id | str | セッションID（オーケストレーター→専門エージェント間のみ。専門エージェント内部では除外） |

### 8.2 マスタデータのバリデーション

電車経路テーブル・固定運賃データはJSONファイルからの静的読み込みで取得し、Pydanticモデルでバリデーションを実施する。

```python
from pydantic import BaseModel, Field

class TrainFareData(BaseModel):
    """電車経路テーブルの1レコード"""
    departure: str = Field(..., min_length=1, description="出発地")
    destination: str = Field(..., min_length=1, description="目的地")
    fare: int = Field(..., ge=0, description="運賃（円）")
```

## 9. バリデーションエラーメッセージ

### 9.1 基本原則

- **わかりやすさ**: 技術的な詳細を隠蔽し、ユーザーが理解できる日本語で記述する
- **具体性**: エラーの原因と対処方法を具体的に提示する
- **一貫性**: 全ツール・全エージェントで同一のエラーメッセージ形式を使用する

### 9.2 エラーメッセージの形式

**基本形式**:
```
{フィールド名}のエラー: {エラー内容}
対処方法: {対処方法}
```

### 9.3 エラーメッセージの実装

```python
from pydantic import ValidationError
from handlers.error_handler import ErrorHandler

try:
    validated = TransportCalculatorInput(**data)
except ValidationError as e:
    error_message = ErrorHandler.handle_validation_error(e)
    return {"success": False, "error_message": error_message}
```

**具体的なエラーメッセージ例**:
```
出発地のエラー: 空文字は入力できません
対処方法: 出発地を入力してください

経費発生日のエラー: 日付の形式が正しくありません（2026/05/21）
対処方法: YYYY-MM-DD形式で入力してください（例: 2026-05-21）

交通手段のエラー: 許可されていない値です（新幹線）
対処方法: 電車・バス・タクシー・飛行機のいずれかを入力してください
```

## 10. バリデーションのテスト

### 10.1 テスト観点

- 正常系: 有効な入力値でバリデーションが通過すること
- 異常系: 無効な入力値でValidationErrorが発生すること
- 境界値: 境界値（0円・10,000円・5,000円等）でバリデーションが正しく動作すること
- 正規化: 別表記の入力値が正規化されること
- エラーメッセージ: バリデーションエラー時に日本語メッセージが生成されること

### 10.2 テストケースの例

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 正常系: 有効な交通手段 | transport_type="電車" | バリデーション通過 |
| 正常系: 別表記の交通手段 | transport_type="train" | 正規化後"電車"でバリデーション通過 |
| 異常系: 無効な交通手段 | transport_type="自転車" | ValidationError発生 |
| 正常系: 有効な金額 | amount=10000 | バリデーション通過 |
| 異常系: 負の金額 | amount=-1 | ValidationError発生 |
| 境界値: 0円 | amount=0 | バリデーション通過 |
| 正常系: 有効な日付 | expense_date="2026-05-21" | バリデーション通過 |
| 異常系: 無効な日付形式 | expense_date="2026/05/21" | ValidationError発生 |
| 異常系: 空文字の申請者名 | user_name="" | ValidationError発生 |

> ※ テストコードの実装はtests/unit/test_data_models.py等のテスト設計書を参照

## 11. 設計上の留意点・リスク

### 11.1 LLMの出力バリデーション

**リスク**: LLMが予期しない形式の値を出力し、バリデーションエラーが頻発する可能性がある

**対策**: field_validatorで表記バリエーションを正規化し、Literal型で許容値を明示する。LLMへのシステムプロンプトに出力形式を明記する

### 11.2 バリデーションエラーの握り潰し

**リスク**: ValidationErrorをキャッチせずに処理を継続することで、不正なデータが後続処理に渡る可能性がある

**対策**: ツール関数の入口で必ずtry/exceptでValidationErrorをキャッチし、ErrorHandlerに委譲する
