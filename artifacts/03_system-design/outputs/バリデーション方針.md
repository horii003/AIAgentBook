## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通するバリデーション方針を定義し、各エージェント・ツール・データモデルで一貫したデータ検証を実施する。

### 1.2 適用範囲

- 全ツール（TOOL-001〜TOOL-002）のツールパラメータ
- invocation_state（AG-001からAG-002/AG-003へ渡す業務コンテキスト）
- マスタデータ（train_fares.json・fixed_fares.json）の読み込み時
- 申請書生成時の出力データ

### 1.3 本方針で定義すること

- バリデーションの基本原則とタイミング
- Pydanticモデルの使用方針
- データ型別バリデーション（日付・金額・文字列・列挙型）
- 業務ルールに基づくカスタムバリデーション
- バリデーションエラーメッセージの方針

### 1.4 本方針で定義しないこと

- 個々のPydanticモデルの具体的なフィールド定義（データモデル基本設計書で定義）
- LLMの出力バリデーション（ガードレール要件定義で定義）
- ファイルシステムのアクセス制御

## 2. バリデーションの基本原則

### 2.1 基本方針

- **早期検証**: ツール関数の入口・invocation_state受取時に即座にバリデーションを実施する
- **型安全性**: Pydantic v2+を使用し、型ヒントによる静的型チェックと実行時バリデーションを両立する
- **ユーザーフレンドリー**: ValidationError発生時はErrorHandlerが日本語のユーザー向けメッセージを生成する
- **自動正規化**: 駅名・交通手段・経費区分等の表記ゆれをfield_validatorで自動正規化する
- **一貫性**: 全ツール・全モデルで同一のバリデーションパターンを使用する

### 2.2 バリデーションのタイミング

| タイミング | 対象 | 実装場所 |
|----------|------|---------|
| ツール関数の入口 | ツールパラメータ（TOOL-001/TOOL-002の入力） | tools/{domain}_tools.py, tools/output_generator.py |
| エージェント呼び出し前 | invocation_state（applicant_name/application_date/session_id） | agents/orchestrator_agent.py |
| invocation_stateの再バリデーション | invocation_state（専門エージェントツール関数内） | tools/output_generator.py |
| マスタデータ読み込み時 | train_fares.json・fixed_fares.json | tools/transport_tools.py |

### 2.3 バリデーション失敗時の扱い

**ValidationError発生時**:
- ValidationErrorをキャッチし、ErrorHandler.handle_validation_error()に委譲する
- ErrorHandlerがユーザー向けの日本語メッセージを生成する
- ツール関数はエラー時に辞書形式でエラーメッセージを返す（例外を再送出しない）
- エラーログ（logs/error.log）にスタックトレースを記録する（exc_info=True）

## 3. Pydanticモデルの使用方針

### 3.1 基本原則

- Pydantic v2+（pydantic >= 2.0.0）を使用する
- 外部から受け取るデータはすべてPydanticモデルでバリデーションを行う
- field_validatorでカスタムバリデーションを実装する
- 共通バリデーター関数は複数モデルで再利用する

### 3.2 Pydanticモデルの定義場所

**定義場所**: `models/data_models.py`（共有データモデル）、`tools/{domain}_tools.py`（ツール固有モデル）

**命名規則**:
- ツール入力モデル: `{ツール名}Input`（例: `TransportCalculatorInput`）
- ツール出力モデル: `{ツール名}Output`（例: `TransportCalculatorOutput`）
- マスタデータモデル: `{データ名}Record`（例: `TrainFareRecord`）
- ※ invocation_stateは辞書リテラルで渡すため、専用のPydanticモデル（InvocationState等）は定義しない

### 3.3 Pydanticモデルの基本構造

```python
from pydantic import BaseModel, field_validator, Field
from typing import Optional, Literal

class TransportCalculatorInput(BaseModel):
    """交通費計算ツールの入力モデル"""
    departure: str = Field(..., min_length=1, description="出発地（正規化済み）")
    destination: str = Field(..., min_length=1, description="目的地（正規化済み）")
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"] = Field(
        ..., description="交通手段"
    )

    # 共通バリデーター関数の適用
    _validate_departure = field_validator("departure", mode="before")(
        classmethod(validate_station_name)
    )
    _validate_destination = field_validator("destination", mode="before")(
        classmethod(validate_station_name)
    )
```

## 4. 日付のバリデーション

### 4.1 基本原則

- 日付はYYYY-MM-DD形式の文字列として受け取る
- python-dateutilを使用して日付文字列をパースし、形式の正当性を検証する
- 申請期限チェック（BRL-06/BRL-15）は日付バリデーション後に実施する

### 4.2 日付の定義

**標準形式**: `YYYY-MM-DD`（例: `2026-05-10`）

**許容値**: 有効な日付（存在する日付のみ）

### 4.3 日付バリデーションの実装

```python
from dateutil import parser as date_parser
from pydantic import field_validator

def validate_date_format(v: str) -> str:
    """日付形式（YYYY-MM-DD）のバリデーション共通関数"""
    try:
        date_parser.parse(v)
        return v
    except (ValueError, TypeError):
        raise ValueError(f"日付の形式が正しくありません。YYYY-MM-DD形式で入力してください: {v}")
```

### 4.4 日付の特殊ルール

**申請期限チェック（BRL-06/BRL-15）**:
- 移動日/購入日から申請日（application_date）までの日数が90日を超える場合は警告を提示する
- バリデーションエラーではなく警告として処理を継続する（EX-03）
- 申請期限チェックはツール関数内ではなく、エージェントのシステムプロンプトで実施する

## 5. 金額のバリデーション

### 5.1 基本原則

- 金額は数値型（float）として受け取る
- 0以上の値のみ許容する（負の金額は不可）
- 上長承認要否チェック（BRL-07/BRL-16）は金額バリデーション後に実施する

### 5.2 金額の定義

**標準形式**: 数値（float）、単位：円

**許容値**: 0以上の数値（ge=0）

### 5.3 金額バリデーションの実装

```python
from pydantic import Field

class ExpenseInput(BaseModel):
    """経費情報の入力モデル"""
    amount: float = Field(..., ge=0, description="金額（円）")
```

### 5.4 金額の特殊ルール

**上長承認要否チェック（BRL-07）**:
- 交通費合計が10,000円を超える場合は上長承認が必要である旨を通知する
- バリデーションエラーではなく通知として処理を継続する（EX-03）

**上長承認要否チェック（BRL-16）**:
- 経費合計が5,000円を超える場合は上長承認が必要である旨を通知する
- バリデーションエラーではなく通知として処理を継続する（EX-03）

## 6. 文字列のバリデーション

### 6.1 基本原則

- 必須文字列フィールドはmin_length=1で空文字を禁止する
- ユーザー入力文字列は500文字以内に制限する（FR-018）
- 駅名は正規化（接尾語除去）を行う（BRL-08）

### 6.2 文字列の定義

**標準形式**: str型

**許容値**: 空文字以外の文字列（必須フィールド）

### 6.3 文字列バリデーションの実装

```python
from pydantic import Field, field_validator

class TransportInput(BaseModel):
    """移動情報の入力モデル"""
    departure: str = Field(..., min_length=1, description="出発地")
    destination: str = Field(..., min_length=1, description="目的地")
    purpose: str = Field(..., min_length=1, description="業務目的（BRL-09）")

def validate_station_name(v: str) -> str:
    """駅名の正規化（BRL-08）：接尾語を除去する"""
    suffixes = ["駅", "バス停", "空港"]
    for suffix in suffixes:
        if v.endswith(suffix):
            return v[:-len(suffix)]
    return v
```

### 6.4 文字列の特殊ルール

**業務目的の必須記載（BRL-09/BRL-17）**:
- 業務目的（purpose）は必須フィールド（min_length=1）として定義する
- 未入力の場合はValidationErrorを発生させ、再入力を促す

## 7. 列挙型のバリデーション

### 7.1 基本原則

- 列挙型フィールドはLiteral型で定義し、受け入れ可能な値を明示する
- LLMが渡す可能性のある表記バリエーション（別表記・略称）を許容する場合はfield_validatorで正規化する
- 正規化後の値を後続処理で使用する（入力値をそのまま使用しない）

### 7.2 交通手段のバリデーション（BRL-03）

**許容値**: `["電車", "バス", "タクシー", "飛行機"]`

**実装パターン**:
```python
from typing import Literal
from pydantic import field_validator

class TransportCalculatorInput(BaseModel):
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"] = Field(
        ..., description="交通手段（BRL-03）"
    )

    @field_validator("transport_type", mode="before")
    @classmethod
    def normalize_transport_type(cls, v: str) -> str:
        """交通手段の表記ゆれを正規化する"""
        mapping = {
            "train": "電車", "電鉄": "電車", "鉄道": "電車",
            "bus": "バス",
            "taxi": "タクシー", "cab": "タクシー",
            "airplane": "飛行機", "plane": "飛行機", "flight": "飛行機", "航空": "飛行機",
        }
        return mapping.get(v.lower(), v)
```

### 7.3 経費区分のバリデーション（BRL-12）

**許容値**: `["事務用品費", "宿泊費", "資格精算費", "その他経費"]`

**実装パターン**:
```python
from typing import Literal
from pydantic import field_validator

class ExpenseInput(BaseModel):
    expense_category: Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"] = Field(
        ..., description="経費区分（BRL-12）"
    )

    @field_validator("expense_category", mode="before")
    @classmethod
    def normalize_expense_category(cls, v: str) -> str:
        """経費区分の表記ゆれを正規化する"""
        mapping = {
            "事務用品": "事務用品費", "文具": "事務用品費",
            "宿泊": "宿泊費", "ホテル": "宿泊費",
            "資格": "資格精算費", "資格取得": "資格精算費",
            "その他": "その他経費",
        }
        return mapping.get(v, v)
```

## 8. カスタムバリデーションルール

### 8.1 invocation_stateのバリデーション

invocation_stateは辞書リテラルで渡す。専用のPydanticモデルは定義しない。ツール関数内でtool_context.invocation_state.get()を使用して安全に参照する。

invocation_stateのフィールド構成（オーケストレーター → 専門エージェント）：
- `applicant_name`（申請者名）
- `application_date`（申請日、YYYY-MM-DD形式）
- `session_id`（セッションID）

専門エージェント内部でエージェントに渡すinvocation_stateのフィールド構成：
- `applicant_name`（申請者名）
- `application_date`（申請日）
- ※ `session_id` はセッションマネージャーの初期化に直接使用し、エージェントへのinvocation_stateには含めない

### 8.2 マスタデータのバリデーション

train_fares.json・fixed_fares.jsonの読み込み時にPydanticモデルでバリデーションを行う。

```python
class TrainFareRecord(BaseModel):
    """経路テーブルの1レコード"""
    departure: str = Field(..., min_length=1, description="出発地")
    destination: str = Field(..., min_length=1, description="目的地")
    fare: float = Field(..., ge=0, description="運賃（円）")

class FixedFareRecord(BaseModel):
    """固定運賃データの1レコード"""
    transport_type: Literal["バス", "タクシー", "飛行機"] = Field(
        ..., description="交通手段"
    )
    fare: float = Field(..., ge=0, description="固定運賃（円）")
```

## 9. バリデーションエラーメッセージ

### 9.1 基本原則

- **わかりやすさ**: 技術的な詳細を隠蔽し、ユーザーが理解できる日本語で記述する
- **具体性**: エラーの原因と対処方法を具体的に提示する
- **一貫性**: ErrorHandler.handle_validation_error()を通じて一貫したメッセージ形式を使用する

### 9.2 エラーメッセージの形式

**基本形式**:
```
{フィールド名}のエラー: {エラー内容}
対処方法: {対処方法}
```

### 9.3 エラーメッセージの実装

```python
from pydantic import ValidationError
from handlers.error_handler import ErrorHandler

try:
    validated = TransportCalculatorInput(**data)
except ValidationError as e:
    error_message = ErrorHandler.handle_validation_error(e)
    return {"success": False, "error_message": error_message}
```

**具体的なエラーメッセージ例**:
```
【交通手段が対応外の場合（BRL-03）】
交通手段のエラー: 対応していない交通手段が入力されました。
対処方法: 電車・バス・タクシー・飛行機のいずれかを入力してください。

【経費区分が対応外の場合（BRL-12）】
経費区分のエラー: 対応していない経費区分が入力されました。
対処方法: 事務用品費・宿泊費・資格精算費・その他経費のいずれかを選択してください。

【日付形式が不正の場合】
申請日のエラー: 日付の形式が正しくありません。
対処方法: YYYY-MM-DD形式（例: 2026-05-10）で入力してください。

【必須項目が未入力の場合（BRL-09/BRL-17）】
業務目的のエラー: 業務目的が入力されていません。
対処方法: 業務目的を入力してください。
```

## 10. バリデーションのテスト

### 10.1 テスト観点

- 正常系：有効な入力値でバリデーションが通過すること
- 異常系：無効な入力値でValidationErrorが発生すること
- 境界値：境界値（最小値・最大値）でのバリデーション動作
- 正規化：表記ゆれが正規化されること
- エラーメッセージ：エラーメッセージが日本語で適切に生成されること

### 10.2 テストケースの例

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 正常系：有効な交通手段 | transport_type="電車" | バリデーション通過 |
| 正常系：表記ゆれの正規化 | transport_type="train" | "電車"に正規化されてバリデーション通過 |
| 異常系：対応外交通手段 | transport_type="自転車" | ValidationError（BRL-03） |
| 正常系：有効な日付 | application_date="2026-05-10" | バリデーション通過 |
| 異常系：不正な日付形式 | application_date="2026/05/10" | ValidationError |
| 正常系：金額0円 | amount=0 | バリデーション通過（ge=0） |
| 異常系：負の金額 | amount=-100 | ValidationError |
| 境界値：金額0円 | amount=0 | バリデーション通過 |
| 正常系：有効な経費区分 | expense_category="事務用品費" | バリデーション通過 |
| 正常系：経費区分の正規化 | expense_category="事務用品" | "事務用品費"に正規化されてバリデーション通過 |
| 異常系：対応外経費区分 | expense_category="交際費" | ValidationError（BRL-12） |
| 異常系：業務目的が空文字 | purpose="" | ValidationError（BRL-09/BRL-17） |

> ※ テスト実装はtests/unit/test_data_models.py・tests/unit/test_transport_tools.py等を参照

## 11. 設計上の留意点・リスク

### 11.1 LLMの出力バリデーション

**リスク**: LLMが対応外の交通手段・経費区分を出力した場合、バリデーションエラーが発生する

**対策**: field_validatorで表記ゆれを正規化し、正規化後もLiteral型に一致しない場合のみValidationErrorを発生させる

### 11.2 マスタデータの整合性

**リスク**: train_fares.json・fixed_fares.jsonのスキーマが変更された場合、バリデーションエラーが発生する

**対策**: マスタデータ読み込み時にPydanticモデルでバリデーションを行い、スキーマ不整合を早期に検出する
