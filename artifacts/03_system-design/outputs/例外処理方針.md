## 1. 目的・適用範囲

### 1.1 目的

システム全体として **「例外とは何か」「どう分類し」「どう収束させるか」** の共通方針を定義する

個々のエージェント設計に **一貫した判断軸** を提供する

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001, AG-002, AG-003）
- 業務処理、ツール実行、UI/UX、運用を含む
- 外部システム連携は本プロジェクトでは発生しない（Amazon Bedrock はフレームワーク基盤として利用）

### 1.3 本方針で定義しないこと

- エージェント内部の例外検知ロジック
- SDK／ライブラリ依存の例外クラス設計
- 具体的な再試行アルゴリズム

---

## 2. 例外処理の基本原則（システム共通）

- **例外は必ず収束させる（無限リトライ禁止）**: LoopControlHook により最大10回のループ制限を設ける
- **失敗は状態として管理する**: 例外発生時はセッション状態を TERMINATED に更新する
- **例外＝即エラー表示ではない**: 再試行・代替処理を試みてから最終的にエラーを表示する
- **ユーザーに内部失敗を見せない**: 技術的な詳細を隠蔽し、ユーザーフレンドリーな日本語メッセージを表示する
- **後続処理に影響を残さない**: 例外発生時は業務を安全に終了し、次回の申請処理に影響を与えない


---

## 3. 例外の定義と分類（システム観点）

### 3.1 例外の定義

「通常の業務フローから逸脱し、継続判断が必要な状態」

### 3.2 例外分類

| 分類ID | 分類名 | 説明 | 例 |
|-------|-------|------|---|
| EX-01 | 入力例外 | 入力不正・不足 | 空入力、500文字超過、必須項目未入力 |
| EX-02 | 判断不能 | AIが結論を出せない | 申請種別の判断不能（交通費/経費の区別がつかない） |
| EX-03 | 業務ルール違反 | 定義済ルール不一致 | 申請期限超過（経費発生日から3ヶ月超）、対象外申請種別 |
| EX-04 | 外部I/F例外 | API障害・タイムアウト | Amazon Bedrock API エラー、レート制限超過 |
| EX-05 | 実行不可 | 権限・状態不整合 | ユーザー未承認での申請書生成試行、キャンセル後の処理継続 |
| EX-06 | データ不整合 | データ形式不正 | 運賃データファイル読み込み失敗、申請書テンプレートファイル不存在、Pydantic バリデーションエラー |
| EX-07 | システム障害 | 基盤障害 | ファイルシステムエラー、ディスク容量不足、セッションデータ破損 |
| EX-08 | 想定外 | 未分類事象 | 予期しない Python 例外（Exception）、LoopLimitError |

## 4. 例外検知レベル

### 4.1 検知レベルの考え方

| レベル | 説明 | 検知主体 |
|-------|------|---------|
| 業務レベル | 業務定義上の異常（申請期限超過・対象外申請・承認未取得等） | エージェント（AG-001〜AG-003） |
| 処理レベル | 処理失敗（ファイル読み込み失敗・バリデーションエラー・申請書生成失敗等） | ツール（TOOL-001, TOOL-002） |
| システムレベル | 基盤・インフラ異常（Bedrock API エラー・LoopLimitError・予期しない例外等） | ErrorHandler |

### 4.2 検知責務

- 検知は **エージェントまたはツール**
- **判断・収束はErrorHandler** が行う

### 4.3 エラー伝播の責務分担

| 責務主体 | 役割 |
|---------|------|
| 各モジュール（エージェント・ツール） | 例外発生箇所でログを出力し、ErrorHandler へ処理を委譲する |
| ErrorHandler | ユーザー向けメッセージを生成して呼び出し元に返す（ログ出力は行わない） |
| 呼び出し元 | 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる |


---

## 5. 例外発生時の基本対応フロー

### 5.1 共通対応フロー

1. 例外検知（エージェント/ツール）
2. 例外分類（ErrorHandler）
3. 継続可否判定（ErrorHandler）
4. 処理分岐（再試行／代替／停止）
5. 状態更新（セッション管理：TERMINATED）
6. ログ・監査記録（呼び出し元モジュール）

## 7. 例外ハンドリング実装

### 7.1 ErrorHandlerの責務方針

ErrorHandlerは以下の責務を担う。

- 例外分類（EX-01〜EX-08）ごとに継続可否を判定し、対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成し、技術的な詳細を隠蔽する
- ErrorHandler 自身はログ出力を行わない。ログ出力は呼び出し元モジュールが行う（ログ出力要件定義に準拠）
- LoopLimitError → Exception の2層構造でキャッチし、各例外クラスに対応する handle_* メソッドに委譲する

具体的なメソッド設計（メソッド名・引数・戻り値）は基本設計（ハンドラー基本設計.md）で定義する。

> ※ 基本設計書への参照: ハンドラー基本設計.md 2.3節


### 7.2 エラーメッセージ設計方針

**ユーザーフレンドリー**:
- 技術的な詳細を隠蔽
- 具体的な対処方法を提示
- 丁寧な言葉遣い

**情報の充実**:
- エラーの原因を明確に説明
- 複数の対処方法を提示
- 必要に応じてシステム管理者への連絡を促す

**例**:
```
# 申請期限超過（EX-03）
「申請期限を超過しているため、申請書を作成できません。
経費発生日から3ヶ月以内に申請してください。
ご不明な点は担当者にお問い合わせください。」

# 運賃データ不存在（EX-06）
「指定された経路の運賃データが見つかりませんでした。
交通費を手動で入力してください。」

# LoopLimitError（EX-08）
「処理の上限回数に達したため、申請処理を終了します。
お手数ですが、最初からやり直してください。」

# 予期しないエラー（EX-08）
「予期しないエラーが発生しました。
担当者にお問い合わせください。」
```

## 8. 例外とエスカレーションの関係

### 8.1 エスカレーション対象例外

| 例外分類 | エスカレーション要否 | 行き先 |
|---------|------------------|-------|
| EX-03（業務ルール違反） | 要（対象外申請・申請期限超過） | ユーザーへの担当者連絡案内 |
| EX-04（外部I/F例外） | 要（リトライ失敗時） | ユーザーへの担当者連絡案内 |
| EX-06（データ不整合） | 要 | ユーザーへの担当者連絡案内 |
| EX-07（システム障害） | 要 | ユーザーへの担当者連絡案内 |
| EX-08（想定外） | 要 | ユーザーへの担当者連絡案内 |

### 8.2 エスカレーション判定責務

**判定主体**: ErrorHandler

**判定基準**: 再試行・代替処理で解決できない例外、または業務ルール上継続不可能な例外はエスカレーション対象とする

## 9. セッション状態への反映方針

### 9.1 例外時の状態更新

| 状態 | 説明 | トリガー |
|-----|------|---------|
| ACTIVE | 処理中（エラー発生前） | — |
| WAITING | 人対応待ち | HumanApprovalHook 発火（EX-05 の承認待ち） |
| TERMINATED | 異常終了 | LoopLimitError、予期しないエラー（EX-07, EX-08）、リトライ失敗（EX-04） |

### 9.2 再開可否

**再開可能な例外**:
- EX-01（入力例外）: 再入力後に継続可能
- EX-02（判断不能）: ユーザー選択後に継続可能
- EX-05（実行不可・修正選択）: 情報収集ステップへ戻り継続可能

**再開不可な例外**:
- EX-03（業務ルール違反・申請期限超過）: 業務終了（新規セッションで再申請）
- EX-04（外部I/F例外・リトライ失敗）: 業務終了（新規セッションで再試行）
- EX-06（データ不整合）: 業務終了（担当者対応後に新規セッション）
- EX-07（システム障害）: 業務終了（担当者対応後に新規セッション）
- EX-08（想定外）: 業務終了（新規セッションで再試行）

## 10. UX観点の扱い（概要）

**利用者への通知有無**:
- 全例外でユーザーへの通知を行う。エラーを握り潰さない

**エラーメッセージの抽象度**:
- 技術的な詳細（スタックトレース・例外クラス名等）は表示しない
- 業務的な原因（申請期限超過・ファイル不存在等）は日本語で明示する

**次の行動を必ず提示**:
- 再入力可能な場合: 「再入力してください」
- 担当者対応が必要な場合: 「担当者にお問い合わせください」
- 新規セッションで再試行可能な場合: 「最初からやり直してください」

## 11. ログ・監査方針

### 11.1 例外ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| error_type | エラータイプ（例外クラス名） | logs/error.log（ERROR） |
| error_message | エラーメッセージ | logs/error.log（ERROR） |
| context | エラーコンテキスト（エージェントID・セッションID等） | logs/error.log（ERROR） |
| exc_info | スタックトレース（exc_info=True） | logs/error.log（ERROR） |
| timestamp | タイムスタンプ | logs/error.log（自動付与） |


### 11.2 監査観点

**再現可能性**:
- セッションID・エージェントID・タイムスタンプをログに記録し、例外発生時の状況を再現可能にする

**判断経路の説明可能性**:
- 例外分類・継続可否判定結果・対応方針をログに記録し、ErrorHandler の判断経路を説明可能にする

## 12. 設計上の留意点・リスク

**例外過多によるUX低下**:
- 頻繁なエラー表示によりユーザーが申請を諦める可能性がある
- 対策: EX-01（入力例外）は再入力を促し、EX-02（判断不能）は選択肢提示で継続できるよう設計する

**例外の握り潰し**:
- ツール関数内で例外を握り潰し、エラーが上位に伝播しない可能性がある
- 対策: ツール関数は辞書形式でエラーを返し（例外を再送出しない）、エージェントが戻り値を確認して ErrorHandler に委譲する

**想定外例外の扱い**:
- 未分類の例外（EX-08）が発生した場合、適切な対応ができない可能性がある
- 対策: LoopLimitError → Exception の2層構造でキャッチし、EX-08 として一律に担当者エスカレーションを行う
