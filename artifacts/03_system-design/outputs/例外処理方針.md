## 1. 目的・適用範囲

### 1.1 目的

システム全体として **「例外とは何か」「どう分類し」「どう収束させるか」** の共通方針を定義する

個々のエージェント設計に **一貫した判断軸** を提供する

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001〜AG-003、TOOL-001〜TOOL-002、HD-001〜HD-003）
- 業務処理、外部連携、UI/UX、運用を含む

### 1.3 本方針で定義しないこと

- エージェント内部の例外検知ロジック
- SDK／ライブラリ依存の例外クラス設計
- 具体的な再試行アルゴリズム

---

## 2. 例外処理の基本原則（システム共通）

- **例外は必ず収束させる（無限リトライ禁止）**：LoopControlHookで最大10回のループ制限。ModelRetryStrategyで最大6回のリトライ後に停止
- **失敗は状態として管理する**：例外発生時はセッション状態をTERMINATEDに遷移させ、ログに記録する
- **例外＝即エラー表示ではない**：リトライ・代替処理を試みた後にユーザーへ通知する
- **ユーザーに内部失敗を見せない**：技術的な詳細（スタックトレース・例外クラス名）はログにのみ記録し、ユーザーには日本語の抽象化されたメッセージを提示する
- **後続処理に影響を残さない**：例外発生時は処理を中断し、セッション状態を確定させる


---

## 3. 例外の定義と分類（システム観点）

### 3.1 例外の定義

「通常の業務フローから逸脱し、継続判断が必要な状態」

### 3.2 例外分類

| 分類ID | 分類名 | 説明 | 例 |
|-------|-------|------|---|
| EX-01 | 入力例外 | 入力不正・不足 | ユーザー入力が500文字超過、必須項目の未入力、対応外交通手段の入力 |
| EX-02 | 判断不能 | AIが結論を出せない | 申請種別の判断不能（JD-03）、経費区分の判断不能 |
| EX-03 | 業務ルール違反 | 定義済ルール不一致 | 申請期限超過（JD-06/JD-08）、対応外経費区分（BRL-12）、上長承認要否閾値超過（JD-07/JD-09） |
| EX-04 | 外部I/F例外 | API障害・タイムアウト | Amazon Bedrock APIエラー・タイムアウト・レート制限 |
| EX-05 | 実行不可 | 権限・状態不整合 | ユーザー未承認での申請書生成試行（APR-001違反） |
| EX-06 | データ不整合 | データ形式不正 | 経路テーブル（train_fares.json）の読み込みエラー、Excelテンプレート不存在、Pydanticバリデーションエラー |
| EX-07 | システム障害 | 基盤障害 | ファイルシステムエラー（output/への書き込み失敗）、LoopLimitError（ループ上限到達） |
| EX-08 | 想定外 | 未分類事象 | 上記に分類できない予期しない例外（Exception） |

## 4. 例外検知レベル

### 4.1 検知レベルの考え方

| レベル | 説明 | 検知主体 |
|-------|------|---------|
| 業務レベル | 業務定義上の異常（申請期限超過・判断不能等） | エージェント（AG-001〜AG-003） |
| 処理レベル | 処理失敗（ツール実行エラー・バリデーションエラー等） | ツール（TOOL-001〜TOOL-002） |
| システムレベル | 基盤・インフラ異常（LoopLimitError・予期しない例外等） | ErrorHandler（HD-001） |

### 4.2 検知責務

- 検知は **エージェントまたはツール**
- **判断・収束はErrorHandler（HD-001）** が行う


---

## 5. 例外発生時の基本対応フロー

### 5.1 共通対応フロー

1. 例外検知（エージェント/ツール）
2. 例外分類（ErrorHandler）
3. 継続可否判定（ErrorHandler）
4. 処理分岐（再試行／代替／停止）
5. 状態更新（セッション管理：ACTIVE/WAITING/TERMINATED）
6. ログ・監査記録（ErrorHandler）

## 7. 例外ハンドリング実装

### 7.1 ErrorHandlerの責務方針

ErrorHandlerは以下の責務を担う。

- 例外分類（EX-01〜EX-08）ごとに継続可否を判定し、対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成し、技術的な詳細を隠蔽する
- 例外発生時のログ出力を行い、スタックトレースはログにのみ記録する（ユーザーには表示しない）
- LoopLimitError → Exception の順でキャッチし、それぞれ対応するメッセージを生成する

具体的なメソッド設計（メソッド名・引数・戻り値）は基本設計（ハンドラー基本設計.md）で定義する。

> ※ 基本設計書への参照: ハンドラー基本設計.md 2.3節


### 7.2 エラーメッセージ設計方針

**ユーザーフレンドリー**:
- 技術的な詳細を隠蔽
- 具体的な対処方法を提示
- 丁寧な言葉遣い

**情報の充実**:
- エラーの原因を明確に説明
- 複数の対処方法を提示
- 必要に応じてシステム管理者への連絡を促す

**例**:
```
【ループ上限到達時】
申し訳ありません。処理の上限回数に達したため、申請処理を中断しました。
お手数ですが、最初からやり直してください。問題が続く場合はシステム管理者にご連絡ください。

【Amazon Bedrock APIエラー時】
申し訳ありません。AIサービスとの通信に失敗しました。
しばらく待ってから再度お試しください。問題が続く場合はシステム管理者にご連絡ください。

【申請書生成失敗時】
申し訳ありません。申請書の生成に失敗しました。
テンプレートファイルが存在するか確認してください。問題が続く場合はシステム管理者にご連絡ください。

【データファイル読み込みエラー時】
申し訳ありません。データの読み込みに失敗しました。
システム管理者にご連絡ください。
```

## 8. 例外とエスカレーションの関係

### 8.1 エスカレーション対象例外

| 例外分類 | エスカレーション要否 | 行き先 |
|---------|------------------|-------|
| EX-04（外部I/F例外） | 要（リトライ失敗時） | ユーザーへ通知（システム管理者への連絡を促す） |
| EX-05（実行不可） | 不要（HumanApprovalHookが制御） | ユーザーへ承認要求 |
| EX-06（データ不整合） | 要 | ユーザーへ通知（システム管理者への連絡を促す） |
| EX-07（システム障害） | 要 | ユーザーへ通知（システム管理者への連絡を促す） |
| EX-08（想定外） | 要 | ユーザーへ通知（システム管理者への連絡を促す） |

### 8.2 エスカレーション判定責務

**判定主体**: ErrorHandler（HD-001）

**判定基準**: 例外分類（EX-04〜EX-08）に該当し、再試行・代替処理で回復不可能な場合にエスカレーションと判定する

## 9. セッション状態への反映方針

### 9.1 例外時の状態更新

| 状態 | 説明 | トリガー |
|-----|------|---------|
| ACTIVE | 処理中（エラー発生前） | — |
| WAITING | 人対応待ち | EX-01（再入力待ち）、EX-02（選択肢提示後）、EX-03（警告提示後の継続） |
| TERMINATED | 異常終了 | EX-04（リトライ失敗）、EX-06、EX-07、EX-08 |

### 9.2 再開可否

**再開可能な例外**:
- EX-01（入力例外）：再入力後に処理継続可能
- EX-02（判断不能）：ユーザー選択後に処理継続可能
- EX-03（業務ルール違反）：警告提示後に処理継続可能（申請期限超過・上長承認要否通知）

**再開不可な例外**:
- EX-04（外部I/F例外）：リトライ失敗時はTERMINATED。新規セッションとして再開
- EX-05（実行不可）：HumanApprovalHookがキャンセル。ユーザーが再度OKを選択すれば継続可能
- EX-06（データ不整合）：TERMINATED。データ修正後に新規セッションとして再開
- EX-07（システム障害）：TERMINATED。新規セッションとして再開
- EX-08（想定外）：TERMINATED。新規セッションとして再開

## 10. UX観点の扱い（概要）

**利用者への通知有無**:
- EX-01〜EX-03：ユーザーへ通知し、次のアクション（再入力・選択・継続）を促す
- EX-04〜EX-08：ユーザーへエラーメッセージを通知し、処理を中断する

**エラーメッセージの抽象度**:
- 技術的な詳細（例外クラス名・スタックトレース・ファイルパス）はユーザーに表示しない
- 原因と対処方法を日本語で簡潔に提示する

**次の行動を必ず提示**:
- 再入力・再試行・システム管理者への連絡など、ユーザーが取るべき次のアクションを必ず提示する

## 11. ログ・監査方針

### 11.1 例外ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| error_type | エラータイプ（例外クラス名） | logs/error.log |
| error_message | エラーメッセージ | logs/error.log |
| context | セッションID・エージェントID・発生箇所 | logs/error.log |
| exc_info | スタックトレース（exc_info=True） | logs/error.log |
| timestamp | タイムスタンプ | logs/error.log |


### 11.2 監査観点

**再現可能性**:
- セッションIDを相関IDとして全ログに含めることで、例外発生時の会話履歴・ツール実行履歴を追跡可能にする

**判断経路の説明可能性**:
- 例外発生前の判断ログ（AUD-003：申請種別判断、AUD-002：承認結果）と組み合わせることで、例外に至った経緯を説明可能にする

## 12. 設計上の留意点・リスク

**例外過多によるUX低下**:
- 過剰なエラーメッセージがユーザーの操作を妨げるリスクがある
- 対策: EX-03（業務ルール違反）は警告提示後に処理継続とし、ユーザーの操作を妨げない

**例外の握り潰し**:
- ツール関数内でエラーを握り潰し、エージェントが誤った情報で処理を継続するリスクがある
- 対策: ツール関数はエラー時に辞書形式でエラーメッセージを返し、エージェントがユーザーへ通知する

**想定外例外の扱い**:
- 未分類の例外（EX-08）が発生した場合、適切な対応ができないリスクがある
- 対策: EX-08はすべてTERMINATEDとして処理中断し、ログにスタックトレースを記録する
