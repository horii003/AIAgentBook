## 1. 目的・適用範囲

### 1.1 目的

システム全体として **「例外とは何か」「どう分類し」「どう収束させるか」** の共通方針を定義する

個々のエージェント設計に **一貫した判断軸** を提供する

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001, AG-002, AG-003）
- 業務処理、外部連携（Amazon Bedrock）、UI/UX、運用を含む

### 1.3 本方針で定義しないこと

- エージェント内部の例外検知ロジック
- SDK／ライブラリ依存の例外クラス設計
- 具体的な再試行アルゴリズム

---

## 2. 例外処理の基本原則（システム共通）

- **例外は必ず収束させる（無限リトライ禁止）**
- **失敗は状態として管理する**
- **例外＝即エラー表示ではない**
- **ユーザーに内部失敗を見せない**
- **後続処理に影響を残さない**


---

## 3. 例外の定義と分類（システム観点）

### 3.1 例外の定義

「通常の業務フローから逸脱し、継続判断が必要な状態」

### 3.2 例外分類

| 分類ID | 分類名 | 説明 | 例 |
|-------|-------|------|---|
| EX-01 | 入力例外 | 入力不正・不足 | 空入力、500文字超過、領収書画像の読み取り失敗 |
| EX-02 | 判断不能 | AIが結論を出せない | 申請種別の判断不能、経費区分の判断不能 |
| EX-03 | 業務ルール違反 | 定義済ルール不一致 | 申請期限超過（経費発生日から3ヶ月超）、申請書提出をAIが試みた場合 |
| EX-04 | 外部I/F例外 | API障害・タイムアウト | Amazon Bedrock呼び出し失敗、接続タイムアウト |
| EX-05 | 実行不可 | 権限・状態不整合 | 申請書テンプレートファイルが見つからない、運賃データが見つからない |
| EX-06 | データ不整合 | データ形式不正 | 運賃データのJSONパース失敗、申請書テンプレートの読み込み失敗 |
| EX-07 | システム障害 | 基盤障害 | ファイルシステムへのアクセス失敗、セッションデータの読み書き失敗 |
| EX-08 | 想定外 | 未分類事象 | 予期しない例外（Exception）、LoopLimitError |

## 4. 例外検知レベル

### 4.1 検知レベルの考え方

| レベル | 説明 | 検知主体 |
|-------|------|---------|
| 業務レベル | 業務定義上の異常（申請期限超過・申請種別判断不能等） | エージェント（AG-001, AG-002, AG-003） |
| 処理レベル | 処理失敗（ツール実行失敗・データ参照失敗等） | ツール（TOOL-001, TOOL-002） |
| システムレベル | 基盤・インフラ異常（Amazon Bedrock障害・ファイルシステム障害等） | ErrorHandler |

### 4.2 検知責務

- 検知は **エージェントまたはツール**
- **判断・収束はErrorHandler** が行う


---

## 5. 例外発生時の基本対応フロー

### 5.1 共通対応フロー

1. 例外検知（エージェント/ツール）
2. 例外分類（ErrorHandler）
3. 継続可否判定（ErrorHandler）
4. 処理分岐（再試行／代替／停止）
5. 状態更新（セッション管理）
6. ログ・監査記録（ErrorHandler）

## 7. 例外ハンドリング実装

### 7.1 ErrorHandlerの責務方針

ErrorHandlerは以下の責務を担う。

- 例外分類（EX-01〜EX-08）ごとに継続可否を判定し、対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成し、技術的な詳細を隠蔽する
- **ログ出力は行わない**（ログ出力は呼び出し元の各モジュールの責務）
- LoopLimitError → Exception の順でキャッチし、例外種別に応じた対応メソッドに委譲する

具体的なメソッド設計（メソッド名・引数・戻り値）は基本設計（ハンドラー基本設計.md）で定義する。

> ※ 基本設計書への参照: ハンドラー基本設計.md 2.3節

### 7.2 エラー伝播の責務分担

| 主体 | 責務 |
|-----|------|
| 各モジュール（エージェント・ツール） | 例外発生箇所でログを出力し（`_logger.error(exc_info=True)`等）、ErrorHandlerへ処理を委譲する |
| ErrorHandler | ユーザー向けメッセージを生成して呼び出し元に返す（ログ出力は行わない） |
| 呼び出し元 | 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる |


### 7.2 エラーメッセージ設計方針

**ユーザーフレンドリー**:
- 技術的な詳細を隠蔽し、ユーザーが理解できる日本語で記述する
- 具体的な対処方法を提示する
- 丁寧な言葉遣いを使用する

**情報の充実**:
- エラーの原因を明確に説明する
- 次に取るべき行動を提示する
- 必要に応じて申請管理部門への問い合わせを促す

**例**:
```
[申請期限超過時]
「申請期限（経費発生日から3ヶ月以内）を超過しているため、申請を受け付けることができません。
申請管理部門にお問い合わせください。」

[Amazon Bedrock接続失敗時]
「システムとの通信に失敗しました。しばらく時間をおいて再度お試しください。
問題が解決しない場合は申請管理部門にお問い合わせください。」

[テンプレートファイル未発見時]
「申請書テンプレートが見つかりませんでした。
申請管理部門にお問い合わせください。」
```

## 8. 例外とエスカレーションの関係

### 8.1 エスカレーション対象例外

| 例外分類 | エスカレーション要否 | 行き先 |
|---------|------------------|-------|
| EX-04（外部I/F例外） | 要（リトライ失敗時） | ユーザーへの通知（申請管理部門への問い合わせ案内） |
| EX-05（実行不可） | 要 | ユーザーへの通知（申請管理部門への問い合わせ案内） |
| EX-06（データ不整合） | 要 | ユーザーへの通知（申請管理部門への問い合わせ案内） |
| EX-07（システム障害） | 要 | ユーザーへの通知（申請管理部門への問い合わせ案内） |
| EX-08（想定外） | 要 | ユーザーへの通知（申請管理部門への問い合わせ案内） |

### 8.2 エスカレーション判定責務

**判定主体**: ErrorHandler

**判定基準**: 再試行・代替処理で解決できない例外（EX-04〜EX-08）はエスカレーション対象とする

## 9. セッション状態への反映方針

### 9.1 例外時の状態更新

| 状態 | 説明 | トリガー |
|-----|------|---------|
| ACTIVE | 処理中（エラー発生前） | — |
| WAITING | 人対応待ち | EX-01（再入力待ち）、EX-02（ユーザー確認待ち） |
| TERMINATED | 異常終了 | EX-03（業務ルール違反）、EX-04（リトライ失敗）、EX-05〜EX-08（中断） |

### 9.2 再開可否

**再開可能な例外**:
- EX-01（入力例外）：再入力後に処理を継続できる
- EX-02（判断不能）：ユーザーが選択肢を選択後に処理を継続できる

**再開不可な例外**:
- EX-03（業務ルール違反）：申請期限超過は新規セッションで再申請が必要
- EX-04（外部I/F例外）：リトライ失敗時は新規セッションで再試行が必要
- EX-05〜EX-08：新規セッションで再試行が必要

## 10. UX観点の扱い（概要）

**利用者への通知有無**:
- 全例外でユーザーへの通知を行う（エラーメッセージを表示する）
- 技術的な詳細（スタックトレース・エラーコード等）は表示しない

**エラーメッセージの抽象度**:
- 業務レベルの例外（EX-01〜EX-03）：具体的な原因と対処方法を提示する
- システムレベルの例外（EX-04〜EX-08）：「システムエラーが発生しました」等の抽象的なメッセージを表示し、申請管理部門への問い合わせを促す

**次の行動を必ず提示**:
- 再入力が必要な場合：再入力を促すメッセージを表示する
- 申請管理部門への問い合わせが必要な場合：問い合わせ先を案内する
- 新規セッションで再試行が必要な場合：再起動を案内する

## 11. ログ・監査方針

### 11.1 例外ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| error_type | エラータイプ（EX-01〜EX-08） | logs/app.log（INFO以上）、logs/error.log（ERROR以上） |
| error_message | ユーザー向けエラーメッセージ | logs/app.log |
| context | エラーコンテキスト（session_id・エージェントID等） | logs/app.log |
| exc_info | スタックトレース（予期しない例外のみ） | logs/error.log（exc_info=True） |
| timestamp | タイムスタンプ | logs/app.log、logs/error.log |

> **ログ出力の責務**: ログ出力は各モジュール（エージェント・ツール）が責任を持って行う。ErrorHandlerはユーザー向けメッセージ生成のみを担当し、ログ出力は行わない。


### 11.2 監査観点

**再現可能性**:
- session_idを全ログに含めることで、例外発生時の会話フローを再現可能にする

**判断経路の説明可能性**:
- 例外分類・適用ルール・判断結果をログに記録することで、例外処理の判断経路を説明可能にする

## 12. 設計上の留意点・リスク

**例外過多によるUX低下**:
- 頻繁なエラーメッセージ表示によりユーザー体験が低下するリスクがある
- 対策: EX-01（入力例外）・EX-02（判断不能）は代替処理（再入力促進・選択肢提示）で対応し、エラーメッセージの表示を最小限にする

**例外の握り潰し**:
- 例外を握り潰して処理を継続することで、後続処理に影響が残るリスクがある
- 対策: 全例外をErrorHandlerに委譲し、継続可否を明示的に判定する

**想定外例外の扱い**:
- 未分類の例外（EX-08）が発生した場合に適切に対処できないリスクがある
- 対策: EX-08はエスカレーション・中断として扱い、スタックトレースをログに記録する
