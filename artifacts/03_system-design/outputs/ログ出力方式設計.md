## 1. 目的・適用範囲

### 1.1 目的

ログ出力要件定義（SR-14）で定義されたログ種別・フォーマット・マスキング要件を具体的な実装方式として定義し、全エージェント・全コンポーネントで一貫したログ出力を実現する。

### 1.2 適用範囲

- 全エージェント（AG-001, AG-002, AG-003）
- 全ハンドラー（ErrorHandler, LoopControlHook, HumanApprovalHook）
- 全ツール（TOOL-001, TOOL-002）
- main.py（エントリーポイント）

### 1.3 本方針で定義しないこと

- ログ分析基盤の構築（ログファイルの参照方法は運用設計に委ねる）
- ログ可視化ダッシュボードの設計

## 2. ログ出力基盤の構成

### 2.1 使用ライブラリ・モジュール

| 区分 | 名称 | 用途 |
|-----|------|------|
| ログフレームワーク | Python 標準 logging | ログ出力の基盤 |
| ファイルハンドラー | RotatingFileHandler | ログファイルのローテーション管理 |
| ロガー定義 | `logging.getLogger(__name__)` | モジュール単位のロガー定義 |

### 2.2 ログレベル設定方針

| 対象 | ログレベル | 理由 |
|-----|----------|------|
| アプリケーション全体（デフォルト） | WARNING | 環境変数 LOG_LEVEL 未設定時のデフォルト値。本番運用時の過剰ログ出力を抑制 |
| アプリケーション全体（環境変数指定時） | LOG_LEVEL の値 | 環境変数（LOG_LEVEL）で制御。DEBUG / INFO / WARNING / ERROR / CRITICAL を指定可能 |
| エラーログファイル | ERROR 以上 | エラー発生の記録のみ |
| Strands イベントループ | WARNING | 過剰なデバッグ出力を抑制（`logging.getLogger("strands").setLevel(logging.WARNING)`） |
| 開発時デバッグ | DEBUG | 開発時のみ LOG_LEVEL=DEBUG を設定して有効化 |

**ログ設定の一元管理方針**:
- ログ設定（レベル・出力先・フォーマット）はエントリーポイント（main.py）で一元管理する
- ログ出力は各モジュールが責任を持って行う
- ログレベルは環境変数（LOG_LEVEL）で制御し、未設定時のデフォルトは WARNING とする
- ユーザー向けメッセージと技術ログは分離する（ユーザー向けメッセージは print/input、技術ログは logging）
- 各モジュールが付加するコンテキスト情報の詳細は基本設計・詳細設計で定義する

### 2.3 出力先・ローテーション設定

| 出力先 | 対象ログレベル | ローテーション方式 | 備考 |
|-------|-------------|----------------|------|
| logs/app.log | INFO 以上 | RotatingFileHandler: 10MB × 5世代 | UTF-8 エンコーディング。LOG_LEVEL=INFO 以下に設定時のみ INFO ログが出力される |
| logs/error.log | ERROR 以上 | RotatingFileHandler: 10MB × 5世代 | UTF-8 エンコーディング |
| コンソール（stderr） | WARNING 以上 | なし | 開発時のみ有効化 |

## 3. 相関ID設計

### 3.1 相関IDの種類と役割

| ID名 | 役割 | 生成タイミング | 生成箇所 |
|-----|------|-------------|---------|
| session_id | セッション単位のトレース識別子 | main.py 起動時 | main.py |

### 3.2 相関IDの形式

| ID名 | 形式 | 例 |
|-----|------|---|
| session_id | `session_{YYYYMMDD_HHMMSS}_{8桁ランダム文字列}` | `session_20260517_162400_a1b2c3d4` |

### 3.3 マルチエージェント間での伝播方式

- session_id は invocation_state 経由で AG-001 から AG-002/AG-003 へ伝播する
- 各エージェントのツール関数内で `tool_context.invocation_state.get("session_id")` により取得する
- session_id は FileSessionManager の初期化にも使用する
- invocation_state は辞書リテラルで渡す（専用の Pydantic モデルは定義しない）

### 3.4 全ログへの付与方式

- 各モジュールのモジュールレベルで `_logger = logging.getLogger(__name__)` を定義する
- ログ出力時に session_id を含むコンテキスト情報を付与する
- フォーマット文字列 `%(asctime)s [%(levelname)s] %(name)s - %(message)s` により自動的にモジュール名が付与される

## 4. ログ種別ごとの実装方式

### 4.1 監査ログ（Audit Log）

| ログID | 記録タイミング | 実装箇所 | 実装方式 |
|-------|-------------|---------|---------|
| AUD-001 | 申請書生成完了時（TOOL-002 実行後） | tools/output_generator.py | `_logger.info("申請書生成完了: file_path=%s, session_id=%s", file_path, session_id)` |
| AUD-002 | HumanApprovalHook 承認結果記録時 | handlers/human_approval_hook.py | `_logger.info("承認結果: result=%s, session_id=%s", result, session_id)` |
| AUD-003 | エージェント処理開始・完了時 | agents/{agent}.py | `_logger.info("処理開始/完了: agent_id=%s, session_id=%s", agent_id, session_id)` |

### 4.2 操作ログ（Operation Log）

| ログID | 記録タイミング | 実装箇所 | 実装方式 |
|-------|-------------|---------|---------|
| OPE-001 | 申請種別判断完了時 | agents/orchestrator_agent.py | `_logger.info("申請種別判断: result=%s, rules=%s", result, applied_rules)` |
| OPE-002 | ツール実行開始・完了時 | tools/{domain}_tools.py | `_logger.info("ツール実行開始/完了: tool_id=%s, params=%s", tool_id, params_summary)` |
| OPE-003 | ループ制限到達時 | handlers/loop_control_hook.py | `_logger.warning("ループ制限到達: agent_id=%s, iterations=%d", agent_id, iterations)` |
| OPE-004 | セッション開始・終了時 | main.py | `_logger.info("セッション開始/終了: session_id=%s", session_id)` |

### 4.3 トレースログ（Trace Log）

| ログID | 記録タイミング | 実装箇所 | 実装方式 |
|-------|-------------|---------|---------|
| TRC-001 | エージェント呼び出し前後 | agents/{agent}.py | `_logger.debug("エージェント呼び出し: agent_id=%s, input_summary=%s", agent_id, summary)` |
| TRC-002 | ツール実行前後 | tools/{domain}_tools.py | `_logger.debug("ツール実行: tool_id=%s, input=%s, output=%s", tool_id, input_summary, output_summary)` |
| TRC-003 | invocation_state 受け渡し時 | agents/orchestrator_agent.py | `_logger.debug("invocation_state 設定: session_id=%s", session_id)` |
| TRC-004 | 委譲発生時（AG-001 → AG-002/AG-003） | agents/orchestrator_agent.py | `_logger.info("委譲: from=AG-001, to=%s, session_id=%s", target_agent, session_id)` |

#### エージェントSDK/フレームワークフックとの連携方式

Strands SDK は HookProvider インターフェースを提供する。LoopControlHook・HumanApprovalHook は HookProvider を継承して実装し、BeforeToolCallEvent・AfterToolCallEvent をフックしてログを出力する。

```python
from strands.hooks import BeforeToolCallEvent, AfterToolCallEvent

def _handle_before_tool_call(self, event: BeforeToolCallEvent) -> None:
    tool_name = event.tool_use["name"] if event.tool_use else "unknown"
    _logger.info("ツール実行前: tool_name=%s", tool_name)

def _handle_after_tool_call(self, event: AfterToolCallEvent) -> None:
    tool_name = event.tool_use["name"] if event.tool_use else "unknown"
    _logger.info("ツール実行後: tool_name=%s, exception=%s", tool_name, event.exception)
```

#### 推論トレース（内部推論）の扱い

| 観点 | 方針 |
|-----|------|
| ユーザーへの表示 | **禁止**。callback_handler=None（全エージェント共通）により LLM の内部推論はユーザーに表示しない。ガードレール要件（GR-004）との整合 |
| ログファイルへの記録 | **記録しない**。LLM の Chain of Thought 等の内部推論はログファイルにも記録しない |
| 記録する場合の形式 | 該当なし（記録しない） |

### 4.4 エラーログ（Error Log）

| ログID | 記録タイミング | 実装箇所 | 実装方式 |
|-------|-------------|---------|---------|
| ERR-001 | バリデーションエラー発生時 | tools/{domain}_tools.py | `_logger.error("バリデーションエラー: %s", e, exc_info=True)` |
| ERR-002 | ファイル読み込み失敗時 | tools/{domain}_tools.py | `_logger.error("ファイル読み込み失敗: path=%s, %s", path, e, exc_info=True)` |
| ERR-003 | LoopLimitError 発生時 | agents/{agent}.py | `_logger.error("LoopLimitError: agent_id=%s, %s", agent_id, e, exc_info=True)` |
| ERR-004 | 予期しないエラー発生時 | agents/{agent}.py | `_logger.error("予期しないエラー: agent_id=%s, %s", agent_id, e, exc_info=True)` |
| ERR-005 | Amazon Bedrock API エラー時 | agents/{agent}.py | `_logger.error("Bedrock API エラー: %s", e, exc_info=True)` |
| ERR-006 | 申請書生成失敗時 | tools/output_generator.py | `_logger.error("申請書生成失敗: %s", e, exc_info=True)` |

#### ErrorHandlerとの連携方式

- ErrorHandler はユーザー向けメッセージ生成のみを担当し、ログ出力は行わない
- ログ出力は呼び出し元モジュール（エージェント・ツール）が行う
- エラー発生時のフロー: 例外検知（エージェント/ツール）→ `_logger.error(..., exc_info=True)` → `ErrorHandler.handle_xxx()` でメッセージ生成 → ユーザーへ表示

### 4.5 デバッグログ（Debug Log）

- 開発時のデバッグ情報は `_logger.debug()` で出力する
- デフォルト（LOG_LEVEL 未設定時）は WARNING 以上のログレベルとなるため、DEBUG ログは出力されない
- 開発時は LOG_LEVEL=DEBUG を設定して有効化する
- デバッグ対象: 変数値・処理経過・invocation_state の内容等

---

### 4.6 メトリクスログ（Metrics Log）

LLMOps の継続的改善サイクルで活用するためのコスト・パフォーマンスメトリクスを記録する。

| 計測対象 | 記録タイミング | 実装箇所 | 実装方式 |
|---------|-------------|---------|---------|
| 実行コスト（LLM推論コスト） | 要件上未定義 | 要件上未定義 | Strands SDK のメトリクス機能を調査して決定 |
| 応答時間（会話ターンごと） | エージェント呼び出し前後 | agents/{agent}.py | `time.time()` で計測し `_logger.info("応答時間: %.2f秒", elapsed)` で記録 |
| 処理時間（ツール実行ごと） | ツール実行前後 | tools/{domain}_tools.py | `time.time()` で計測し `_logger.info("ツール処理時間: tool_id=%s, %.2f秒", tool_id, elapsed)` で記録 |

## 5. ログフォーマット定義

### 5.1 共通フォーマット

| 項目 | 内容 |
|-----|------|
| フォーマット文字列 | `%(asctime)s [%(levelname)s] %(name)s - %(message)s` |
| エンコーディング | UTF-8 |
| タイムスタンプ形式 | `%(asctime)s`（Python logging デフォルト: `YYYY-MM-DD HH:MM:SS,mmm`） |

### 5.2 構造化ログの形式

本システムでは構造化ログ（JSON 形式）は採用しない。Python 標準 logging のテキスト形式を使用する。ログメッセージ内に `key=value` 形式でコンテキスト情報を含める。

```
2026-05-17 16:24:00,123 [INFO] agents.orchestrator_agent - 申請種別判断: result=交通費精算申請, session_id=session_20260517_162400_a1b2c3d4
2026-05-17 16:24:01,456 [ERROR] tools.transport_tools - バリデーションエラー: session_id=session_20260517_162400_a1b2c3d4
Traceback (most recent call last):
  ...
```

## 7. 例外処理方針・実行制御方針との整合

### 7.1 例外ログの出力フロー

1. 例外発生（エージェント/ツール）
2. 呼び出し元モジュールが `_logger.error(..., exc_info=True)` でエラーログを出力
3. `ErrorHandler.handle_xxx()` でユーザー向けメッセージを生成
4. ユーザーへメッセージを表示
5. セッション状態を TERMINATED に更新

### 7.2 実行制御ログの出力フロー

- LoopControlHook: ループ回数カウント時に `_logger.debug()`、制限到達時に `_logger.warning()` を出力
- HumanApprovalHook: 承認結果（OK/修正/キャンセル）を `_logger.info()` で出力
- ModelRetryStrategy: リトライ発生時のログは Strands SDK 内部で出力（WARNING レベル）

## 8. 設計上の留意点・リスク

**個人情報のログ混入リスク**:
- 申請者名・領収書画像等の個人情報がログに混入する可能性がある
- 対策: ログ出力時に個人情報フィールドを除外し、session_id のみを記録する

**ログファイルの肥大化リスク**:
- 長時間の運用によりログファイルが肥大化する可能性がある
- 対策: RotatingFileHandler（10MB × 5世代）でローテーションを管理する

## 9. ログの活用方針（LLMOps連携）

### 9.1 評価・改善サイクルへの接続

収集したログを評価テスト方式設計（SD-08）で定義する評価データセットの構築に活用する。

| ログ種別 | 活用目的 | 活用方法 |
|---------|---------|---------|
| 操作ログ（OPE-001） | 申請種別判断精度の評価 | 判断結果と適用ルールを評価データセットに取り込む |
| トレースログ（TRC-001〜TRC-004） | エージェント動作の品質評価 | ツール実行パターン・委譲フローを分析する |
| メトリクスログ | コスト管理・パフォーマンス最適化 | 応答時間・処理時間の傾向を分析する |
| エラーログ（ERR-001〜ERR-006） | 障害対応・品質改善 | エラー発生パターンを分析し、プロンプト・バリデーションを改善する |

### 9.2 品質劣化の検知方針

- エラーログ（ERR-001〜ERR-006）の発生頻度を監視し、閾値超過時に担当者へ通知する
- 応答時間の増加傾向を検知し、パフォーマンス劣化の早期発見に活用する
- 具体的な閾値・通知方式は運用設計フェーズで決定する
