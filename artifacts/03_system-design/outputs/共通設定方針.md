## 2. 会話管理の方針

### 2.1 基本原則

- **会話履歴の保持**: SlidingWindowConversationManagerを使用し、直近のメッセージのみをLLMに渡す
- **メモリ効率**: ウィンドウサイズを超えたメッセージは切り詰め（should_truncate_results=True）、トークン消費を抑制する
- **コンテキストの継続性**: per_turn=Falseでメッセージ単位でカウントし、会話の文脈を維持する

### 2.2 ウィンドウサイズの基準

| エージェント種別 | ウィンドウサイズ | 理由 |
|----------------|----------------|------|
| オーケストレーター（AG-001） | 30 | 複数の専門エージェントとのやり取りを保持する必要があるため |
| 交通費精算申請エージェント（AG-002） | 20 | 複数ステップの情報収集・処理を行うため |
| 経費精算申請エージェント（AG-003） | 15 | 特定タスクに集中した短い対話のため |
| ツール | — | ツールは会話管理を持たない |

> **【注意】ウィンドウサイズと対話回数上限・ReActループ最大回数は独立した概念**
> - ウィンドウサイズ（30/20/15）：LLMに渡す会話履歴の保持件数
> - ReActループ最大回数（10回）：1回のエージェント呼び出し内でLLM推論→ツール実行を繰り返す上限
> - 対話回数上限：実行制御方針.mdで定義済み。本セクションでは再定義しない

### 2.3 会話管理の実装

**使用するクラス**: `SlidingWindowConversationManager`（Strands SDK提供）

**設定方法**:
```python
from strands.agent.conversation_manager import SlidingWindowConversationManager

conversation_manager = SlidingWindowConversationManager(
    window_size=30,                # AG-001: 30、AG-002: 20、AG-003: 15
    should_truncate_results=True,  # 全エージェント共通
    per_turn=False                 # 全エージェント共通
)
```

**設定パラメータ**:
- `window_size`: 保持するメッセージ数（AG-001: 30、AG-002: 20、AG-003: 15）
- `should_truncate_results`: ウィンドウ超過時に結果を切り詰める（全エージェント共通: True）
- `per_turn`: メッセージ単位でカウント（全エージェント共通: False）

## 3. リトライ戦略の方針

### 3.1 基本原則

- **一時的なエラーへの対応**: Amazon Bedrock APIの一時的なエラー（レート制限・タイムアウト）に対して自動リトライを行う
- **指数バックオフ**: リトライ間隔を指数的に増加させ、APIへの過負荷を防止する
- **最大試行回数の制限**: 最大6回のリトライ後に失敗として処理し、無限リトライを防止する

### 3.2 リトライ対象

| 対象 | リトライ可否 | 理由 |
|-----|----------|------|
| Amazon Bedrock API呼び出し | ○ | 一時的なエラー（レート制限・タイムアウト）が発生しうるため |
| TOOL-001（交通費計算） | × | データファイル読み込みエラーは再試行しても解決しないため |
| TOOL-002（申請書生成） | × | ファイル書き込みエラーは再試行しても解決しないため |
| ユーザー入力 | ○ | 再入力を促すことで回復可能なため（LoopControlHookのループ制限内） |

### 3.3 リトライ戦略の実装

**使用するクラス**: `ModelRetryStrategy`（Strands SDK提供）

**設定方法**:
```python
from strands import ModelRetryStrategy

retry_strategy = ModelRetryStrategy(
    max_attempts=6,    # 全エージェント共通
    initial_delay=4,   # 全エージェント共通（秒）
    max_delay=240      # 全エージェント共通（秒）
)
```

**設定パラメータ**:
- `max_attempts`: モデル呼び出しの最大リトライ回数（全エージェント共通: 6）
- `initial_delay`: 指数バックオフの初期遅延秒数（全エージェント共通: 4）
- `max_delay`: 指数バックオフの最大遅延秒数（全エージェント共通: 240）

## 4. ループ制御の方針

### 4.1 基本原則

- **無限ループの防止**: LoopControlHookがReActループ回数をカウントし、上限到達時に処理を中断する
- **処理の可視化**: LoopControlHookがループ回数をログに記録し、処理状況を可視化する
- **ユーザーへの通知**: LoopLimitError発生時にErrorHandlerがユーザーフレンドリーなメッセージを生成する

### 4.2 最大ループ回数の基準

| エージェント種別 | 最大ループ回数 | 理由 |
|----------------|--------------|------|
| 全エージェント（AG-001〜AG-003） | 10回 | 申請情報収集・計算・申請書生成の一連の処理を完結させるために十分な回数 |

### 4.3 ループ制御の実装

**使用するクラス**: `LoopControlHook`（カスタム実装：handlers/loop_control_hook.py）

**設定方法**:
```python
from handlers.loop_control_hook import LoopControlHook

loop_control_hook = LoopControlHook(max_iterations=10)
```

**設定パラメータ**:
- `max_iterations`: ReActループの最大回数（全エージェント共通: 10）

> ※ LoopControlHookの詳細設計はハンドラー基本設計.md 3.2節を参照

## 5. セッション管理の方針

### 5.1 基本原則

- **会話履歴の永続化**: FileSessionManagerを使用し、セッションデータをローカルファイルシステムに永続化する
- **セッションIDの一意性**: タイムスタンプ＋ランダム文字列でセッションIDを生成し、一意性を保証する
- **エージェント間の共有**: session_idをinvocation_state経由で専門エージェントへ伝播し、各エージェントが独立したセッションを管理する

### 5.2 セッションIDの形式

**形式**: `{YYYYMMDD_HHMMSS}_{8桁ランダム英数字}`

**例**: `20260510_114355_a1b2c3d4`

### 5.3 セッション管理の実装

**使用するクラス**: `FileSessionManager`（Strands SDK提供）

**設定方法**:
```python
from strands.session import FileSessionManager

session_manager = FileSessionManager(
    session_id=session_id,
    storage_dir="storage/sessions"
)
```

**設定パラメータ**:
- `session_id`: セッション識別子（形式: `{YYYYMMDD_HHMMSS}_{8桁ランダム英数字}`）
- `storage_dir`: セッションデータの保存先（全エージェント共通: `storage/sessions`）

**詳細**: セッション管理方針（artifacts/03_system-design/outputs/セッション管理方針.md）を参照

## 6. 承認フックの方針

### 6.1 基本原則

- **重要な操作の承認**: 申請書生成（TOOL-002）前にユーザーの明示的な承認を必須化する
- **ユーザーの選択肢**: OK（承認）・修正要望・キャンセルの3択をユーザーに提示する
- **誤操作の防止**: HumanApprovalHookがTOOL-002の呼び出しをBeforeToolCallEventでフックし、未承認の実行を防止する

### 6.2 承認対象ツールの基準

| ツール種別 | 承認要否 | 理由 |
|----------|---------|------|
| TOOL-002（申請書生成） | 要 | ファイル生成という副作用があり、誤った申請書の生成を防止するため（APR-001） |
| TOOL-001（交通費計算） | 不要 | 参照のみで副作用がないため |
| その他の参照系ツール | 不要 | 副作用がないため |

### 6.3 承認フックの実装

**使用するクラス**: `HumanApprovalHook`（カスタム実装：handlers/human_approval_hook.py）

**設定方法**:
```python
from handlers.human_approval_hook import HumanApprovalHook

human_approval_hook = HumanApprovalHook(
    target_tools=["output_generator"]  # TOOL-002のツール関数名
)
```

**設定パラメータ**:
- `target_tools`: 承認を必要とするツール関数名のリスト（TOOL-002: `output_generator`）

> ※ HumanApprovalHookの詳細設計はハンドラー基本設計.md 3.3節を参照

## 7. 設定の適用方法

### 7.1 エージェント作成時の設定

**基本パターン**:
```python
from strands import Agent, ModelRetryStrategy
from strands.agent.conversation_manager import SlidingWindowConversationManager
from strands.session import FileSessionManager
from handlers.loop_control_hook import LoopControlHook
from handlers.human_approval_hook import HumanApprovalHook
from config.model_config import ModelConfig

# オーケストレーター（AG-001）の設定例
agent = Agent(
    model=ModelConfig.get_model(),
    system_prompt=SYSTEM_PROMPT,
    tools=[specialist_a_agent, specialist_b_agent],
    agent_id="orchestrator_agent",
    name="申請受付窓口エージェント",
    description="申請種別を判断し専門エージェントへ委譲するオーケストレーター",
    conversation_manager=SlidingWindowConversationManager(
        window_size=30,  # AG-001: 30、AG-002: 20、AG-003: 15
        should_truncate_results=True,
        per_turn=False
    ),
    callback_handler=None,
    retry_strategy=ModelRetryStrategy(
        max_attempts=6,
        initial_delay=4,
        max_delay=240
    ),
    session_manager=FileSessionManager(
        session_id=session_id,
        storage_dir="storage/sessions"
    ),
    hooks=[LoopControlHook(max_iterations=10)]
)
```

### 7.2 設定のカスタマイズ

**カスタマイズが必要な場合**:
- ウィンドウサイズ：エージェント種別（AG-001: 30、AG-002: 20、AG-003: 15）に応じて設定する
- フック構成：出力生成ありの専門エージェントはHumanApprovalHookを追加する

**カスタマイズ方法**: 各エージェントの基本設計書（artifacts/04_basic-design/outputs/）を参照

## 8. ログ設計方針

### 8.1 基本原則

- **一元管理**: ログ設定（レベル・出力先・フォーマット）はエントリーポイント（main.py）で一元管理する
- **モジュール責任**: ログ出力は各モジュールが責任を持って行う（`_logger = logging.getLogger(__name__)`）
- **環境変数制御**: ログレベルは環境変数（LOG_LEVEL）で制御し、未設定時のデフォルトはWARNINGとする
- **分離**: ユーザー向けメッセージ（print/input）と技術ログ（logging）は分離する

### 8.2 ログ設定

| 項目 | 内容 |
|-----|------|
| フォーマット | `%(asctime)s [%(levelname)s] %(name)s - %(message)s` |
| エンコーディング | UTF-8 |
| INFO以上 | logs/app.log（RotatingFileHandler: 10MB × 5世代） |
| ERROR以上 | logs/error.log（RotatingFileHandler: 10MB × 5世代） |
| デフォルトレベル | WARNING（LOG_LEVEL未設定時） |
| Strandsイベントループ | WARNINGレベルに設定（過剰なデバッグ出力を抑制） |

### 8.3 コンテキスト情報

各モジュールが付加するコンテキスト情報の詳細は基本設計・詳細設計で定義する。

## 9. エラー伝播の責務分担

### 9.1 基本原則

| 層 | 責務 |
|---|---|
| 各モジュール（エージェント・ツール） | 例外発生箇所でログを出力し、ErrorHandlerへ処理を委譲する |
| ErrorHandler | ユーザー向けメッセージを生成して呼び出し元に返す |
| 呼び出し元 | 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる |

### 9.2 エラー伝播フロー

```
例外発生（ツール/エージェント）
  ↓ ログ出力（exc_info=True）
  ↓ ErrorHandler.handle_xxx() に委譲
  ↓ ユーザー向けメッセージを返す
  ↓ 呼び出し元が上位レイヤーへ伝播
  ↓ ユーザーへ表示
```

## 10. 設計上の留意点・リスク

### 8.1 会話管理

**リスク**: ウィンドウサイズを超えた会話履歴が切り詰められ、重要な文脈が失われる可能性がある

**対策**: ウィンドウサイズをエージェント種別に応じて適切に設定する（AG-001: 30、AG-002: 20、AG-003: 15）

### 8.2 ループ制御

**リスク**: LoopControlHookが正常に動作しない場合、ReActループが無限に継続するリスクがある

**対策**: LoopControlHookの単体テストを実施し、ループ制限が正常に機能することを確認する

### 8.3 セッション管理

**リスク**: セッションファイルが肥大化し、ファイルシステムの容量を圧迫する可能性がある

**対策**: SlidingWindowConversationManagerでウィンドウサイズを制限し、セッションファイルの肥大化を抑制する

### 8.4 承認フック

**リスク**: HumanApprovalHookが全ツール呼び出しで発火するため、対象ツールのフィルタリングが必要

**対策**: HumanApprovalHookのBeforeToolCallEventハンドラー内でtarget_toolsによるフィルタリングを実施する
