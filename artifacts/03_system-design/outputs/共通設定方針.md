## 2. 会話管理の方針

### 2.1 基本原則

- **会話履歴の保持**: SlidingWindowConversationManagerにより直近のメッセージをウィンドウサイズ分保持する
- **メモリ効率**: ウィンドウ超過時は古いメッセージを切り詰める（should_truncate_results=True）
- **コンテキストの継続性**: per_turn=Falseによりメッセージ単位でカウントし、会話の継続性を確保する

### 2.2 ウィンドウサイズの基準

| エージェント種別 | ウィンドウサイズ | 理由 |
|----------------|----------------|------|
| オーケストレーター（AG-001） | 30（大） | 複数の専門エージェントとのやり取りを保持する必要があるため |
| 専門エージェント（複雑な処理）（AG-002） | 20（中） | 複数ステップの情報収集・処理を行うため |
| 専門エージェント（単純な処理）（AG-003） | 15（小） | 特定タスクに集中した短い対話のため |

> **【注意】ウィンドウサイズと対話回数上限・ReActループ最大回数は独立した概念**
> - ウィンドウサイズ: LLMに渡す会話履歴の保持件数（会話管理クラスで設定）
> - ReActループ最大回数: 1回のエージェント呼び出し内でLLM推論→ツール実行を繰り返す上限（LoopControlHookで設定）
> - 対話回数上限: セッション全体でユーザーとエージェントが会話できるターン数の上限（実行制御方針.mdで定義済み）

### 2.3 会話管理の実装

**使用するクラス**: `SlidingWindowConversationManager`（AWS Strands SDK提供）

**設定方法**:
```python
from strands.agent.conversation_manager import SlidingWindowConversationManager

conversation_manager = SlidingWindowConversationManager(
    window_size=20,                # エージェント種別により異なる（上記基準参照）
    should_truncate_results=True,  # 全エージェント共通
    per_turn=False                 # 全エージェント共通
)
```

**設定パラメータ**:
- `window_size`: 保持するメッセージ数（エージェント種別により異なる）
- `should_truncate_results`: ウィンドウ超過時に結果を切り詰める（全エージェント共通: True）
- `per_turn`: メッセージ単位でカウント（全エージェント共通: False）

## 3. リトライ戦略の方針

### 3.1 基本原則

- **一時的なエラーへの対応**: Amazon Bedrock呼び出し失敗（スロットリング・一時的な障害）に対してModelRetryStrategyで自動リトライする
- **指数バックオフ**: 初期遅延4秒から始まり、最大240秒まで指数的に増加する
- **最大試行回数の制限**: 最大6回のリトライで無限リトライを防止する

### 3.2 リトライ対象

| 対象 | リトライ可否 | 理由 |
|-----|----------|------|
| Amazon Bedrock呼び出し | ○ | スロットリング・一時的な障害は再試行で解決する可能性がある |
| TOOL-001（交通費計算） | × | 経路データ未発見は再試行しても解決しない |
| TOOL-002（申請書生成） | × | テンプレートファイル未発見は再試行しても解決しない |
| ファイルシステムアクセス | × | ファイルシステム障害は再試行しても解決しない |

### 3.3 リトライ戦略の実装

**使用するクラス**: `ModelRetryStrategy`（AWS Strands SDK提供）

**設定方法**:
```python
from strands import ModelRetryStrategy

retry_strategy = ModelRetryStrategy(
    max_attempts=6,    # 全エージェント共通
    initial_delay=4,   # 全エージェント共通（秒）
    max_delay=240      # 全エージェント共通（秒）
)
```

**設定パラメータ**:
- `max_attempts`: モデル呼び出しの最大リトライ回数（全エージェント共通: 6）
- `initial_delay`: 指数バックオフの初期遅延秒数（全エージェント共通: 4）
- `max_delay`: 指数バックオフの最大遅延秒数（全エージェント共通: 240）

## 4. ループ制御の方針

### 4.1 基本原則

- **無限ループの防止**: LoopControlHookによりReActループの上限を設定し、無限ループを防止する（GRD-007）
- **処理の可視化**: LoopControlHookがループ回数をカウントし、ログに記録する
- **ユーザーへの通知**: ループ制限到達時はLoopLimitErrorを発生させ、ErrorHandlerがユーザーへのメッセージを生成する

### 4.2 最大ループ回数の基準

| エージェント種別 | 最大ループ回数 | 理由 |
|----------------|--------------|------|
| 全エージェント | 10回 | 申請情報収集・申請書生成に必要な最大ステップ数を考慮した上限値 |

### 4.3 ループ制御の実装

**使用するクラス**: `LoopControlHook`（カスタム実装: handlers/loop_control_hook.py）

**設定方法**:
```python
from handlers.loop_control_hook import LoopControlHook

loop_control_hook = LoopControlHook(max_iterations=10)  # 全エージェント共通
```

**設定パラメータ**:
- `max_iterations`: ReActループの最大回数（全エージェント共通: 10）

> ※ LoopControlHookの詳細設計はハンドラー基本設計.md 3.2節を参照

## 5. セッション管理の方針

### 5.1 基本原則

- **会話履歴の永続化**: FileSessionManagerによりセッションデータをローカルファイルシステム（storage/sessions/）に保存する
- **セッションIDの一意性**: session_idはmain.py起動時に生成し、セッション期間中は変更しない
- **エージェント間の共有**: session_idはinvocation_stateを通じてAG-001からAG-002/AG-003へ伝播する

### 5.2 セッションIDの形式

**形式**: `session_{YYYYMMDD_HHMMSS}_{8桁ランダム文字列}`

**例**: `session_20260521_143022_a1b2c3d4`

### 5.3 セッション管理の実装

**使用するクラス**: `FileSessionManager`（AWS Strands SDK提供）

**設定方法**:
```python
from strands.session import FileSessionManager

session_manager = FileSessionManager(
    session_id=session_id,
    storage_dir="storage/sessions"
)
```

**設定パラメータ**:
- `session_id`: セッション識別子（`session_{YYYYMMDD_HHMMSS}_{8桁ランダム文字列}`形式）
- `storage_dir`: セッションデータの保存先ディレクトリ（全エージェント共通: `storage/sessions`）

**詳細**: セッション管理方針（artifacts/03_system-design/outputs/セッション管理方針.md）を参照

## 6. 承認フックの方針

### 6.1 基本原則

- **重要な操作の承認**: 申請書生成（TOOL-002）の実行前に人の確認を求める（GRD-008）
- **ユーザーの選択肢**: 承認/修正要望/拒否の3択をユーザーに提示する
- **誤操作の防止**: 申請書の誤生成を防止し、申請書提出はAIが行わず必ず人が実行することを保証する（BRL-06）

### 6.2 承認対象ツールの基準

| ツール種別 | 承認要否 | 理由 |
|----------|---------|------|
| TOOL-002（申請書生成） | 要 | 申請書ファイルの生成は副作用があり、人の確認が必要（GRD-008） |
| TOOL-001（交通費計算） | 不要 | 参照のみで副作用なし（自律度Lv1） |
| その他参照系ツール | 不要 | 参照のみで副作用なし |

### 6.3 承認フックの実装

**使用するクラス**: `HumanApprovalHook`（カスタム実装: handlers/human_approval_hook.py）

**設定方法**:
```python
from handlers.human_approval_hook import HumanApprovalHook

human_approval_hook = HumanApprovalHook(
    target_tools=["output_generator"]  # TOOL-002に対応するツール関数名
)
```

**設定パラメータ**:
- `target_tools`: 承認対象のツール関数名リスト（TOOL-002に対応するツール関数名）

> ※ HumanApprovalHookの詳細設計はハンドラー基本設計.md 3.3節を参照

## 7. 設定の適用方法

### 7.1 エージェント作成時の設定

**基本パターン**:
```python
from strands import Agent, ModelRetryStrategy
from strands.agent.conversation_manager import SlidingWindowConversationManager
from handlers.loop_control_hook import LoopControlHook
from handlers.human_approval_hook import HumanApprovalHook
from session.session_manager import create_session_manager

# セッションマネージャの作成
session_manager = create_session_manager(session_id=session_id)

# 共通設定
agent = Agent(
    model=ModelConfig.get_model(),
    system_prompt=SYSTEM_PROMPT,
    tools=[tool_a, tool_b],
    agent_id="agent_id_snake_case",
    name="エージェント表示名",
    description="エージェントの役割説明",
    conversation_manager=SlidingWindowConversationManager(
        window_size=20,                # エージェント種別により異なる
        should_truncate_results=True,
        per_turn=False
    ),
    callback_handler=None,             # 全エージェント共通
    retry_strategy=ModelRetryStrategy(
        max_attempts=6,
        initial_delay=4,
        max_delay=240
    ),
    session_manager=session_manager,
    hooks=[loop_control_hook]          # 出力生成ありの場合は[human_approval_hook, loop_control_hook]
)
```

### 7.2 フック構成の基準

| エージェント種別 | hooks | 理由 |
|----------------|-------|------|
| オーケストレーター（AG-001） | `[LoopControlHook]` | ループ制御のみ（出力生成は専門エージェントが担当） |
| 専門エージェント（出力生成あり）（AG-002, AG-003） | `[HumanApprovalHook, LoopControlHook]` | 出力前の人間承認 + ループ制御 |
| 専門エージェント（出力生成なし） | `[LoopControlHook]` | ループ制御のみ |

### 7.3 設定のカスタマイズ

**カスタマイズが必要な場合**:
- ウィンドウサイズ（エージェント種別により異なる）
- システムプロンプト（エージェントごとに異なる）
- ツールリスト（エージェントごとに異なる）

**カスタマイズ方法**: 各エージェントの基本設計書を参照

## 8. 設計上の留意点・リスク

### 8.1 会話管理

**リスク**: ウィンドウサイズが小さすぎると重要な会話履歴が失われ、申請情報の収集が不完全になる可能性がある

**対策**: エージェント種別に応じたウィンドウサイズを設定し、should_truncate_results=Trueで古いメッセージを切り詰める

### 8.2 ループ制御

**リスク**: 最大ループ回数が小さすぎると申請情報の収集が完了する前にループが終了する可能性がある

**対策**: 最大ループ回数を10回に設定し、申請情報収集・申請書生成に必要な最大ステップ数を考慮する

### 8.3 セッション管理

**リスク**: セッションデータが肥大化してファイルシステムの容量を圧迫する可能性がある

**対策**: セッションデータの保持期間を運用ルールで定め、定期的に削除する

### 8.4 承認フック

**リスク**: HumanApprovalHookが全ツール呼び出しで発火するため、対象ツール以外でも処理が実行される可能性がある

**対策**: HumanApprovalHookのハンドラー内でtarget_toolsによるフィルタリングを必ず実施する

## 9. ログ設計方針

### 9.1 基本原則

- **一元管理**: ログ設定（レベル・出力先・フォーマット）はエントリーポイント（main.py）で一元管理する
- **モジュール責任**: ログ出力は各モジュールが責任を持って行う（`_logger = logging.getLogger(__name__)`）
- **ユーザー向けメッセージと技術ログの分離**: ユーザーへの表示メッセージ（print等）と技術ログ（logging）は分離する
- **ErrorHandlerの責務**: ErrorHandlerはユーザー向けメッセージ文字列の生成のみを担当し、ログ出力は行わない

### 9.2 ログレベル制御

| 設定 | 内容 |
|-----|------|
| 制御方法 | 環境変数（LOG_LEVEL）で制御する |
| デフォルト | 未設定時は WARNING |
| 設定例 | `LOG_LEVEL=INFO`（本番）、`LOG_LEVEL=DEBUG`（開発時） |

### 9.3 コンテキスト情報

各モジュールが付加するコンテキスト情報の詳細は基本設計・詳細設計で定義する。

## 10. エラー伝播の責務分担

### 10.1 責務分担の原則

| 主体 | 責務 |
|-----|------|
| 各モジュール（エージェント・ツール） | 例外発生箇所でログを出力し、ErrorHandlerへ処理を委譲する |
| ErrorHandler | ユーザー向けメッセージを生成して呼び出し元に返す |
| 呼び出し元 | 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる |

### 10.2 実装パターン

```python
# 各モジュール（エージェント・ツール）での実装例
try:
    # 処理
    ...
except ValidationError as e:
    _logger.error("バリデーションエラーが発生しました", exc_info=True)
    error_message = ErrorHandler.handle_validation_error(e)
    return {"success": False, "error_message": error_message}
except Exception as e:
    _logger.error("予期しないエラーが発生しました", exc_info=True)
    error_message = ErrorHandler.handle_unexpected_error(e)
    return {"success": False, "error_message": error_message}
```
