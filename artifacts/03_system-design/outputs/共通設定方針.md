## 2. 会話管理の方針

### 2.1 基本原則

- **会話履歴の保持**: SlidingWindowConversationManager により直近 window_size 件のメッセージを保持する
- **メモリ効率**: ウィンドウ超過時は should_truncate_results=True で結果を切り詰め、トークン消費を抑制する
- **コンテキストの継続性**: per_turn=False でメッセージ単位でカウントし、ターン単位の切り捨てを防ぐ

### 2.2 ウィンドウサイズの基準

| エージェント種別 | エージェント名 | ウィンドウサイズ | 理由 |
|----------------|--------------|----------------|------|
| オーケストレーター | AG-001: 申請受付窓口エージェント | 30 | 複数の専門エージェントとのやり取りを保持する必要があるため |
| 専門エージェント（複雑な処理） | AG-003: 交通費精算申請エージェント | 20 | 複数ステップの情報収集・運賃計算を行うため |
| 専門エージェント（単純な処理） | AG-002: 経費精算申請エージェント | 15 | 領収書情報抽出に集中した短い対話のため |
| ツール | — | — | ツールは会話管理を持たない |

> **【注意】ウィンドウサイズと対話回数上限・ReActループ最大回数は独立した概念**
> - ウィンドウサイズ: LLMに渡す会話履歴の保持件数（会話管理クラスで設定）
> - ReActループ最大回数: 1回のエージェント呼び出し内でLLM推論→ツール実行を繰り返す上限（LoopControlHookで設定）
> - 対話回数上限: セッション全体でユーザーとエージェントが会話できるターン数の上限（実行制御方針.mdで定義済み）

### 2.3 会話管理の実装

**使用するクラス**: `SlidingWindowConversationManager`（strands-agents 提供）

**設定方法**:
```python
from strands.agent.conversation_manager import SlidingWindowConversationManager

conversation_manager = SlidingWindowConversationManager(
    window_size=20,               # エージェント種別に応じて設定
    should_truncate_results=True, # 全エージェント共通: True
    per_turn=False                # 全エージェント共通: False
)
```

**設定パラメータ**:
- `window_size`: 保持するメッセージ数（エージェント種別に応じて設定）
- `should_truncate_results`: ウィンドウ超過時に結果を切り詰める（全エージェント共通: True）
- `per_turn`: メッセージ単位でカウント（全エージェント共通: False）

## 3. リトライ戦略の方針

### 3.1 基本原則

- **一時的なエラーへの対応**: Amazon Bedrock API の一時的なエラー（レート制限・タイムアウト等）に対して自動リトライを行う
- **指数バックオフ**: リトライ間隔を指数的に増加させ、API への過負荷を防ぐ
- **最大試行回数の制限**: 無限リトライを防止するため、最大試行回数を設定する

### 3.2 リトライ対象

| 対象 | リトライ可否 | 理由 |
|-----|----------|------|
| Amazon Bedrock API 呼び出し | ○ | 一時的なエラー（レート制限・タイムアウト等）は再試行で解決可能 |
| TOOL-001（交通費計算） | × | 運賃データ不存在は再試行しても解決しない |
| TOOL-002（申請書生成） | × | 承認後1回のみ実行。失敗時は担当者エスカレーション |
| ユーザー入力（EX-01） | ○ | 再入力を促すことで解決可能（対話回数上限30回まで） |

### 3.3 リトライ戦略の実装

**使用するクラス**: `ModelRetryStrategy`（strands-agents 提供）

**設定方法**:
```python
from strands import ModelRetryStrategy

retry_strategy = ModelRetryStrategy(
    max_attempts=6,   # 全エージェント共通: 6回
    initial_delay=4,  # 全エージェント共通: 4秒
    max_delay=240     # 全エージェント共通: 240秒
)
```

**設定パラメータ**:
- `max_attempts`: モデル呼び出しの最大リトライ回数（全エージェント共通: 6）
- `initial_delay`: 指数バックオフの初期遅延秒数（全エージェント共通: 4）
- `max_delay`: 指数バックオフの最大遅延秒数（全エージェント共通: 240）

## 4. ループ制御の方針

### 4.1 基本原則

- **無限ループの防止**: ReAct ループの最大回数を設定し、無限ループを防止する
- **処理の可視化**: ループ回数をカウントし、WARNING ログに記録する
- **ユーザーへの通知**: ループ制限到達時にユーザーフレンドリーなメッセージを表示する

### 4.2 最大ループ回数の基準

| エージェント種別 | 最大ループ回数 | 理由 |
|----------------|--------------|------|
| 全エージェント（AG-001〜AG-003） | 10回 | 業務処理に必要な最大ステップ数を考慮した上限値 |

### 4.3 ループ制御の実装

**使用するクラス**: `LoopControlHook`（カスタム実装: handlers/loop_control_hook.py、HookProvider として実装）

**設定方法**:
```python
from handlers.loop_control_hook import LoopControlHook

loop_control_hook = LoopControlHook(max_iterations=10)
```

**設定パラメータ**:
- `max_iterations`: ReAct ループの最大回数（全エージェント共通: 10）

**登録イベント**:
- BeforeInvocationEvent: エージェント呼び出し開始時にループカウンタを0にリセット
- AfterModelCallEvent: LLM呼び出し後にループカウンタをインクリメントし、上限到達を監視
- AfterInvocationEvent: エージェント呼び出し完了後にループカウンタを0にリセット

**フック構成（エージェント種別ごと）**:
| エージェント種別 | hooks |
|----------------|-------|
| オーケストレーター（AG-001） | `[LoopControlHook]` |
| 専門エージェント（AG-002, AG-003） | `[HumanApprovalHook, LoopControlHook]` |

## 5. セッション管理の方針

### 5.1 基本原則

- **会話履歴の永続化**: FileSessionManager によりセッションデータをファイルに永続化し、中断後の再開（Resume）に対応する
- **セッションIDの一意性**: session_id は main.py 起動時に生成し、全エージェントで共有する
- **エージェント間の共有**: session_id は invocation_state 経由で専門エージェントに渡し、FileSessionManager の初期化に使用する

### 5.2 セッションIDの形式

**形式**: `session_{YYYYMMDD_HHMMSS}_{8桁ランダム文字列}`

**例**: `session_20260517_162400_a1b2c3d4`

### 5.3 セッション管理の実装

**使用するクラス**: `FileSessionManager`（strands-agents 提供）

**設定方法**:
```python
from strands.session import FileSessionManager

session_manager = FileSessionManager(
    session_id=session_id,
    storage_dir="storage/sessions"
)
```

**設定パラメータ**:
- `session_id`: セッション識別子（main.py で生成）
- `storage_dir`: セッションデータの保存先（全エージェント共通: `storage/sessions`）

**詳細**: セッション管理方針（artifacts/03_system-design/outputs/セッション管理方針.md）を参照

## 6. 承認フックの方針

### 6.1 基本原則

- **重要な操作の承認**: 申請書生成（TOOL-002）の実行前にユーザーの承認を得る
- **ユーザーの選択肢**: OK/修正/キャンセルの3択を提示し、ユーザーが最終判断を行う
- **誤操作の防止**: ユーザーが OK を選択した場合のみ申請書ファイルを生成する

### 6.2 承認対象ツールの基準

| ツール種別 | 承認要否 | 理由 |
|----------|---------|------|
| TOOL-002（申請書生成） | 要 | ファイル生成という副作用のある操作。自律度 Lv4（承認後実行） |
| TOOL-001（交通費計算） | 不要 | 参照のみ。副作用なし。自律度 Lv3（自動実行） |
| Amazon Bedrock API | 不要 | LLM 推論基盤。フレームワーク内部で自動実行 |

### 6.3 承認フックの実装

**使用するクラス**: `HumanApprovalHook`（カスタム実装: handlers/human_approval_hook.py）

**設定方法**:
```python
from handlers.human_approval_hook import HumanApprovalHook

human_approval_hook = HumanApprovalHook(
    target_tools=["output_generator"]  # 承認対象ツール名
)
```

**設定パラメータ**:
- `target_tools`: 承認対象のツール名リスト（TOOL-002 に対応するツール関数名）

**承認選択肢**:
- OK: 申請書生成を実行する
- 修正: 情報収集ステップへ戻る（event.cancel_tool にメッセージを設定）
- キャンセル: 業務を終了する（event.cancel_tool にメッセージを設定）

## 7. 設定の適用方法

### 7.1 エージェント作成時の設定

**基本パターン（全エージェント共通）**:
```python
from strands import Agent, ModelRetryStrategy
from strands.agent.conversation_manager import SlidingWindowConversationManager
from strands.session import FileSessionManager
from handlers.loop_control_hook import LoopControlHook
from handlers.human_approval_hook import HumanApprovalHook
from config.model_config import ModelConfig

agent = Agent(
    # ---- 必須パラメータ ----
    model=ModelConfig.get_model(),
    system_prompt=SYSTEM_PROMPT,
    tools=[tool_a, tool_b],

    # ---- エージェント識別パラメータ ----
    agent_id="agent_id_snake_case",
    name="エージェント表示名",
    description="エージェントの役割説明",

    # ---- 会話管理パラメータ ----
    conversation_manager=SlidingWindowConversationManager(
        window_size=20,               # エージェント種別に応じて設定
        should_truncate_results=True, # 全エージェント共通: True
        per_turn=False                # 全エージェント共通: False
    ),

    # ---- ストリーミング制御 ----
    callback_handler=None,            # 全エージェント共通: None

    # ---- リトライ戦略 ----
    retry_strategy=ModelRetryStrategy(
        max_attempts=6,               # 全エージェント共通: 6
        initial_delay=4,              # 全エージェント共通: 4
        max_delay=240                 # 全エージェント共通: 240
    ),

    # ---- セッション管理 ----
    session_manager=session_manager,  # FileSessionManager インスタンス

    # ---- フック ----
    hooks=[loop_control_hook]         # エージェント種別に応じて設定
)
```

**モデル設定（config/model_config.py）**:
```python
from strands.models import BedrockModel
from functools import lru_cache

class ModelConfig:
    DEFAULT_MODEL_ID = "jp.anthropic.claude-sonnet-4-5-20250929-v1:0"
    GUARDRAIL_ID = ""      # 環境変数 GUARDRAIL_ID から取得
    GUARDRAIL_VERSION = "DRAFT"

    @classmethod
    @lru_cache(maxsize=1)
    def get_model(cls) -> BedrockModel:
        return BedrockModel(
            model_id=cls.DEFAULT_MODEL_ID,
            guardrail_id=cls.GUARDRAIL_ID,
            guardrail_version=cls.GUARDRAIL_VERSION,
            guardrail_trace="enabled"  # 全エージェント共通: "enabled"
        )
```

### 7.2 設定のカスタマイズ

**カスタマイズが必要な場合**:
- window_size: エージェント種別（オーケストレーター/専門エージェント）に応じて変更する
- hooks: 出力生成ありの専門エージェントは `[HumanApprovalHook, LoopControlHook]` を設定する

**カスタマイズ方法**: 各エージェントの基本設計書（artifacts/04_basic-design/outputs/）を参照

## 8. 設計上の留意点・リスク

### 8.1 会話管理

**リスク**: window_size が小さすぎると重要なコンテキストが失われ、エージェントが誤った判断をする可能性がある

**対策**: エージェント種別に応じた適切な window_size を設定する（AG-001: 30、AG-003: 20、AG-002: 15）

### 8.2 ループ制御

**リスク**: max_iterations が小さすぎると正常な業務処理が途中で打ち切られる可能性がある

**対策**: 業務処理に必要な最大ステップ数を考慮した上で max_iterations=10 を設定する

### 8.3 セッション管理

**リスク**: storage/sessions/ ディレクトリが存在しない場合、FileSessionManager の初期化に失敗する可能性がある

**対策**: main.py 起動時に storage/sessions/ ディレクトリの存在確認・自動作成を行う

### 8.4 承認フック

**リスク**: HumanApprovalHook が全ツール呼び出しで発火するため、対象ツール以外でも処理が実行される可能性がある

**対策**: BeforeToolCallEvent のハンドラー内で target_tools によるフィルタリングを必ず実施する
