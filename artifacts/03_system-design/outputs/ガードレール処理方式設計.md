## 1. 目的・適用範囲

### 1.1 目的

ガードレール要件定義（SR-13）で定義された全ガードレール要件を具体的な実装方式として定義し、全エージェントで一貫したガードレール処理を実現する。

### 1.2 適用範囲

- 全エージェント（AG-001, AG-002, AG-003）
- Amazon Bedrock Guardrails（ガードレールツール層）
- HumanApprovalHook・LoopControlHook（アプリケーション層）

### 1.3 本方針で定義しないこと

- Amazon Bedrock Guardrails の具体的なポリシー設定値（詳細設計フェーズで定義）
- CloudFormation テンプレートの詳細（guardrails/guardrails_cloudformation.yaml で管理）

## 2. ガードレール全体構成

### 2.1 ガードレール層の構成

```text
┌─────────────────────────────────────────────────────────────┐
│                    ユーザー入力                               │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│          ガードレールツール層（Amazon Bedrock Guardrails）      │
│  ・入力コンテンツフィルタリング（有害コンテンツ・プロンプトインジェクション）│
│  ・出力コンテンツフィルタリング（業務スコープ外トピック拒否）       │
│  ・ガードレールトレース（enabled）                              │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│              エージェント処理（AG-001〜AG-003）                 │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│          アプリケーション層（HumanApprovalHook）                │
│  ・申請書生成前の承認ゲート（OK/修正/キャンセル）                  │
│  ・LoopControlHook（最大ループ回数制御）                         │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│              ツール実行（TOOL-001, TOOL-002）                   │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 実現方式の分担

| ガードレール層 | 実現方式 | 担当コンポーネント |
|-------------|---------|----------------|
| 入力ガード | Amazon Bedrock Guardrails による入力コンテンツフィルタリング | BedrockModel（guardrail_id 設定） |
| 出力ガード | Amazon Bedrock Guardrails による出力コンテンツフィルタリング | BedrockModel（guardrail_id 設定） |
| 実行ガード | HumanApprovalHook による承認ゲート、LoopControlHook によるループ制御 | handlers/human_approval_hook.py, handlers/loop_control_hook.py |

### 2.3 エージェントごとのガードレール適用方式

| エージェント | 適用ガードレール | 適用方式 |
|-----------|--------------|---------|
| AG-001 | Amazon Bedrock Guardrails + LoopControlHook | BedrockModel に guardrail_id を設定、hooks=[LoopControlHook] |
| AG-002 | Amazon Bedrock Guardrails + HumanApprovalHook + LoopControlHook | BedrockModel に guardrail_id を設定、hooks=[HumanApprovalHook, LoopControlHook] |
| AG-003 | Amazon Bedrock Guardrails + HumanApprovalHook + LoopControlHook | BedrockModel に guardrail_id を設定、hooks=[HumanApprovalHook, LoopControlHook] |

## 3. ガードレールツール設定方式

### 3.1 設定パラメータ管理方式

| 設定項目 | 管理方式 | 備考 |
|--------|---------|------|
| ガードレールID | 環境変数（GUARDRAIL_ID）→ config/model_config.py で参照 | .env ファイルで管理 |
| バージョン | 環境変数（GUARDRAIL_VERSION）→ config/model_config.py で参照 | デフォルト: DRAFT |
| トレース | コード固定値（guardrail_trace="enabled"）| 全エージェント共通 |

### 3.2 エージェントへの適用方式

全エージェントは `ModelConfig.get_model()` が返す `BedrockModel` インスタンスを使用する。`BedrockModel` に `guardrail_id`・`guardrail_version`・`guardrail_trace` を設定することで、全 LLM 呼び出しに自動的にガードレールが適用される。

```python
BedrockModel(
    model_id=cls.DEFAULT_MODEL_ID,
    guardrail_id=cls.GUARDRAIL_ID,       # 環境変数から取得
    guardrail_version=cls.GUARDRAIL_VERSION,  # 環境変数から取得
    guardrail_trace="enabled"             # 全エージェント共通
)
```

### 3.3 設定値の管理方式

- ガードレールID・バージョンは `.env` ファイルで管理し、`python-dotenv` で読み込む
- `.env` ファイルは `.gitignore` に登録し、リポジトリにコミットしない
- `.env.template` に環境変数名のみを記載し、値は空欄とする

|-------------|-----------|---------|---------|-----------|
| GR-001 | トピック制御（業務スコープ外拒否） | Amazon Bedrock Guardrails のトピック拒否設定 | BedrockModel（guardrail_id） | ガードレールが自動的に拒否メッセージを返す。AG-001 がエスカレーション案内を表示 |

## 5. アプリケーションレベルのガードレール

### 5.1 承認ゲートの実装方式

HumanApprovalHook は Strands SDK の HookProvider を継承して実装する。BeforeToolCallEvent をフックし、対象ツール（TOOL-002 に対応するツール関数）の実行前にユーザーの承認を求める。

```python
from strands.hooks import HookProvider, HookRegistry, BeforeToolCallEvent

class HumanApprovalHook(HookProvider):
    def __init__(self, target_tools: list[str]):
        self.target_tools = target_tools

    def register_hooks(self, registry: HookRegistry, **kwargs) -> None:
        registry.add_callback(BeforeToolCallEvent, self._handle_before_tool_call)

    def _handle_before_tool_call(self, event: BeforeToolCallEvent) -> None:
        tool_name = event.tool_use["name"] if event.tool_use else "unknown"
        # 対象ツール以外はスキップ
        if tool_name not in self.target_tools:
            return
        # ユーザーに承認を求める
        result = input("申請書を作成しますか？ [OK/修正/キャンセル]: ")
        if result.strip() != "OK":
            event.cancel_tool = f"{result}を選択しました。"
```

### 5.2 ループ制御との関係

- LoopControlHook は ReAct ループの回数を制御し、無限ループを防止する（実行制御方針参照）
- HumanApprovalHook は申請書生成前の承認ゲートを提供する
- 両フックは独立して動作し、hooks リストに両方を設定することで並列適用される
- 適用順序: `hooks=[HumanApprovalHook, LoopControlHook]`（HumanApprovalHook が先に登録される）

## 6. 自律度レベルとガードレール適用の対応

| 自律度Lv | 適用ガードレール | 実装方式 |
|---------|--------------|---------|
| Lv0（禁止） | 全ガードレール適用 | Amazon Bedrock Guardrails + アプリケーション層で完全ブロック |
| Lv1（参照） | Amazon Bedrock Guardrails（入力・出力フィルタリング） | BedrockModel に guardrail_id を設定 |
| Lv2（提案） | Amazon Bedrock Guardrails（入力・出力フィルタリング） | BedrockModel に guardrail_id を設定 |
| Lv3（自動実行） | Amazon Bedrock Guardrails（入力・出力フィルタリング）+ LoopControlHook | BedrockModel + hooks=[LoopControlHook] |
| Lv4（承認後実行） | Amazon Bedrock Guardrails + HumanApprovalHook + LoopControlHook | BedrockModel + hooks=[HumanApprovalHook, LoopControlHook] |

## 7. ガードレール検知時の処理フロー

### 7.1 ガードレールツール検知時

1. Amazon Bedrock Guardrails が入力/出力コンテンツを検知
2. Guardrails が自動的に拒否メッセージを返す（エージェントの処理は中断）
3. エージェントが拒否メッセージを受け取り、ユーザーへ案内メッセージを表示
4. ガードレールトレース（guardrail_trace="enabled"）により検知結果がログに記録される

### 7.2 アプリケーションレベル検知時

**HumanApprovalHook（承認ゲート）**:
1. BeforeToolCallEvent が発火（TOOL-002 実行前）
2. HumanApprovalHook がユーザーに OK/修正/キャンセルを提示
3. OK 以外の場合: event.cancel_tool にメッセージを設定してツール実行をキャンセル
4. キャンセル: 業務終了メッセージを表示してセッションを CLOSED に更新
5. 修正: 情報収集ステップへ戻る

**LoopControlHook（ループ制御）**:
1. LoopControlHook がループ回数をカウント
2. 最大回数（10回）到達時: LoopLimitError を発生させる
3. エージェントが LoopLimitError をキャッチ
4. ErrorHandler がユーザー向けメッセージを生成
5. セッション状態を TERMINATED に更新して業務終了

### 7.3 例外処理方針との連携

- Amazon Bedrock Guardrails の検知: EX-03（業務ルール違反）または EX-05（実行不可）として処理
- HumanApprovalHook のキャンセル: EX-05（実行不可）として処理
- LoopControlHook の LoopLimitError: EX-08（想定外）として処理（LoopLimitError → Exception の2層構造でキャッチ）

## 8. 設計上の留意点・リスク

**ガードレールIDが未設定の場合のリスク**:
- GUARDRAIL_ID 環境変数が未設定の場合、ガードレールが適用されない
- 対策: main.py 起動時に GUARDRAIL_ID の存在確認を行い、未設定時は警告ログを出力する

**HumanApprovalHook の全ツール発火リスク**:
- BeforeToolCallEvent は全ツール呼び出しで発火するため、対象ツール以外でも処理が実行される可能性がある
- 対策: ハンドラー内で target_tools によるフィルタリングを必ず実施する
