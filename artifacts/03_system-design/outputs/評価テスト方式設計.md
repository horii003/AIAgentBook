## 1. 目的・適用範囲

### 1.1 目的

AIエージェントシステム（経費・交通費精算申請支援システム）の品質を保証するための評価テスト方式を定義する。本書は詳細設計フェーズの評価テスト詳細設計および実装フェーズの評価テストコード生成へのインプットとなる。

> **評価指標・合否基準・実施方法・回帰評価方針は `評価指標一覧.md` を正とする。本書はそれらを参照し、評価の「仕組み・アーキテクチャ・環境」を定義する。**

### 1.2 適用範囲

- AIエージェント評価テスト：LLM-as-Judge方式によるエージェント品質の自動評価（E2E評価テストのみを対象とする）

### 1.3 本方針で定義しないこと

- 個別テストケースの入力データと期待結果（詳細設計で定義）
- 評価テストコードの実装詳細（実装フェーズで定義）
- 評価フレームワーク固有のAPI仕様（詳細設計で定義）
- 評価指標の定義・閾値・実施方法・回帰評価方針（評価指標一覧.mdで定義）

## 2. 評価戦略・基本方針

### 2.1 AIエージェント固有のリスクと評価戦略

| リスク | 対応戦略 |
|--------|---------|
| LLM応答の非決定性 | 各テストケースは1回のみ実行する。複数回実行による集計は行わない |
| プロンプトの意図逸脱 | LLM-as-Judge（段階評価）で申請種別判断・経費区分判断の正確性を評価し、プロンプト変更時は回帰評価を必須とする |
| エージェント間の情報伝達誤り | invocation_state（applicant_name・application_date・session_id）の受け渡しをセッションレベルで検証する |
| ガードレール漏れ | 対象外申請・申請期限超過・上長承認チェックの各ケースを異常系テストケースとして網羅する |
| ユーザーシミュレーションの限界 | ゴールベースのユーザーシミュレーターを使用し、最大ターン数（30回）内での会話完了を検証する |
| LLM-as-Judgeの信頼性 | 判定モデルを評価対象モデルと分離し、判定根拠を必ず出力してレポートに含める |
| フレームワーク間の差異 | strands-agents-evalsフレームワークに依存した抽象概念を本書では定義せず、詳細設計で具体化する |

## 3. 評価テストの共通抽象概念

### 3.1 コア抽象概念の定義

| 抽象概念 | 定義 |
|---------|------|
| テストケース | 評価の単位。入力・期待出力・期待軌跡・メタデータを持つ |
| 評価方式 | テストケースの実行結果に対してスコアリング（LLM-as-Judge・軌跡マッチ等）を適用し、スコア・合否・判定理由を返す評価の方法 |
| 評価結果 | スコア (float 0-1) + 合否 (bool) + 判定理由 (str) |
| 実験 | テストケース群 × 評価方式群のバッチ実行単位 |
| レポート | 全体スコア・個別スコア・判定理由の集約 |

### 3.2 テストケースの構成要素

| 要素 | 型 | 必須/任意 | 説明 |
|------|---|---------|------|
| 識別名 | string | 必須 | テストケースの一意識別子 |
| 入力 | string / messages | 必須 | ユーザー発話またはメッセージ配列 |
| 期待出力 | string | 任意 | 期待する最終応答 |
| 期待軌跡 | list[string] | 任意 | 期待するツール呼び出し順序 |
| メタデータ | dict | 任意 | 参照する追加情報（ゴール・ルーブリック等） |
| セッションID | string | 自動生成 | テレメトリ分離用 |

### 3.3 評価方式の種類

| 評価方式種別 | スコアリング方式 | 適用場面 |
|-----------|---------------|---------|
| **LLM-as-Judge（二値）** | 達成/未達成 → 1.0/0.0 | 全評価観点（MET-001・MET-002・安全性等） |

### 3.4 軌跡マッチモード

| マッチモード | 判定基準 | 適用場面 |
|-----------|---------|---------|
| strict | 完全一致（順序含む） | 厳密なワークフロー検証（CF-002/CF-003の申請書生成フロー） |
| unordered | 順不同で全要素一致 | 順序不問のツール呼び出し検証 |
| subset | 期待が実際の部分集合 | 最低限のツール呼び出し検証（TOOL-002の呼び出し確認） |
| superset | 実際が期待の部分集合 | 不要なツール呼び出しの検出 |

## 4. 評価レベルの定義

### 4.1 評価レベル体系

| 評価レベル | 評価対象 | 評価方式 | 判定粒度 | 対応指標 |
|-----------|---------|---------|---------|---------|
| セッションレベル | 会話全体のゴール達成（申請書ファイル生成完了） | LLM-as-Judge（二値）がセッション全体の軌跡を評価 | セッション単位 | MET-002 |
| ツールレベル | 個別のツール選択判断（TOOL-001/TOOL-002の呼び出し妥当性） | LLM-as-Judge（二値）が各ツール呼び出しの妥当性を評価 | ツール呼び出し単位 | MET-001 |
| 出力レベル | 最終応答の品質（申請種別判断結果・経費区分判断結果） | LLM-as-Judge（二値：達成/未達成） | 応答単位 | MET-001 |
| 軌跡レベル | ツール呼び出し列全体（申請書生成フローの網羅性） | LLM-as-Judge（二値） | 軌跡単位 | MET-001, MET-002 |

### 4.2 評価レベルの選択基準

| 確認したいこと | 選択すべきレベル | 実行方式 |
|-------------|---------------|---------|
| 申請書ファイルが正常生成されるか（MET-002） | セッションレベル | マルチターン |
| 申請種別・経費区分が正しく判断されるか（MET-001） | 出力レベル | マルチターン |
| TOOL-001/TOOL-002が正しく呼び出されるか | ツールレベル | シングルターン |
| 申請書生成フロー全体の順序・網羅性 | 軌跡レベル | マルチターン |

## 5. 評価方式の共通アーキテクチャ

### 5.1 実行フロー

```
┌─────────────────────────────────────────────────────────┐
│                    評価実行基盤                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [1. テストケース定義]                                    │
│    - 入力（ユーザー発話：申請内容テキスト・領収書画像等）    │
│    - メタデータ（ゴール / 期待ツール / ルーブリック）         │
│                                                         │
│  [2. タスク実行]                                         │
│    - エージェントにテストケースの入力を送信                  │
│    - テレメトリによる実行軌跡（スパン）の自動記録            │
│    - マルチターン（ユーザーシミュレーター）                  │
│                                                         │
│  [3. 評価判定]                                           │
│    - 評価方式が実行結果（出力 + 軌跡）をスコアリング          │
│    - LLM-as-Judge（二値）                                │
│                                                         │
│  [4. 結果レポート]                                        │
│    - 全体スコア（overall_score）                          │
│    - 個別スコア・合否・判定理由                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 テストケース設計方針

| 項目 | 方針 |
|------|------|
| テストケースの構成要素 | 識別名・入力テキスト（申請内容/移動情報/領収書情報）・タスク説明・判定基準 |
| テストデータの独立性 | 各テストケースは他のケースに依存しない |
| テレメトリの分離 | ケース実行前にスパンをクリアし、ケース間のデータ混入を防止 |
| 評価データセットの拡充 | 運用開始後も実際の利用データをもとに継続的にケースを追加 |
| 最低件数 | MET-001：申請種別（交通費/経費）×正常/異常の組み合わせを網羅する件数、MET-002：CF-002/CF-003の各フローを網羅する件数 |

### 5.3 ユーザーシミュレーター方式

| 項目 | 方針 |
|------|------|
| シミュレーターの役割 | ゴールに基づき動的なユーザー発話を生成（申請内容入力・不足情報回答・承認選択等） |
| 最大ターン数 | 30ターン（FR-017の対話回数上限に準拠） |
| 会話終了条件 | ゴール達成判定（申請書ファイル生成完了）または最大ターン到達 |
| 入力の起点 | テストケースの入力フィールド（申請内容テキスト） |

### 5.4 LLM-as-Judge設計方針

| 項目 | 方針 |
|------|------|
| 判定モデル | 評価対象（jp.anthropic.claude-sonnet-4-5-20250929-v1:0）とは別モデルを推奨 |
| 判定根拠の出力 | 判定理由を必ず出力しレポートに含める |
| スコアリング尺度 | 全評価観点で二値（達成/未達成） |
| Few-shot例 | 申請種別判断・経費区分判断の複雑なケースにはfew-shot exampleを活用 |

## 6. AIエージェント評価方針

### 6.1 評価観点マッピング

> 各評価観点の確認内容・対応指標IDは `評価指標一覧.md` の「1. 評価観点一覧（マスター）」を参照。

| 評価観点 | 評価方式 |
|---------|---------|
| 妥当性 | LLM-as-Judge（二値：達成/未達成）で申請種別判断・経費区分判断・交通費計算結果の正確性を評価（MET-001） |
| 有用性 | LLM-as-Judge（二値：達成/未達成）で申請書作成ゴール達成を評価（MET-002） |
| 安全性 | LLM-as-Judge（二値）で申請期限超過・対象外申請・上長承認チェックの業務ルール遵守を評価 |
| 一貫性 | LLM-as-Judge（二値）で同一入力に対する応答の安定性を評価 |
| ツール利用の適切性 | LLM-as-Judge（二値）でTOOL-001/TOOL-002の呼び出し有無・順序を評価 |
| 制御機構の有効性 | LLM-as-Judge（二値）でHumanApprovalHook（承認ゲート）・LoopControlHook（ループ制御）の動作を評価 |
| 運用品質 | LLM-as-Judge（二値）で申請書ファイル生成の成功率・ログ出力の正常性を評価 |

## 7. 評価環境設計

### 7.1 実施環境

| 項目 | 値 |
|------|-----|
| Pythonバージョン | 3.10以上 |
| モデルID | jp.anthropic.claude-sonnet-4-5-20250929-v1:0 |
| AWS リージョン | ap-northeast-1 |
| 評価ログ出力先 | `evals/logs/` |
| 評価フレームワーク | strands-agents-evals >= 0.1.0 |

### 7.2 評価実行基盤の要件

| 項目 | 要件 |
|------|------|
| テレメトリ | インメモリスパンエクスポーターで実行軌跡を記録し、ツール呼び出し列（TOOL-001/TOOL-002）を取得する |
| 承認フック無効化 | HumanApprovalHook（HD-003）のブロッキング動作を自動承認（OK選択）に差し替える |
| エージェント生成 | 本番と同一構成（モデルID・システムプロンプト・ツール・ハンドラー）で評価用エージェントを生成する |
| ログ設定 | UTF-8対応・ファイル出力（evals/logs/）・SDKログ抑制 |
| LLM呼び出し | モックなし（本番同等環境でAmazon Bedrock実呼び出し） |
| ブロッキングフック | HumanApprovalHookを自動承認パッチで差し替え（TOOL-002の実行を許可） |
| セッション管理 | テストケースごとに独立したsession_idを生成し、テレメトリを分離する |

### 7.3 テストデータ管理方針

| 項目 | 方針 |
|------|------|
| テストケースの配置 | `evals/`内のPython定数として定義（eval_reimbursement.py） |
| 本番データの利用 | 本番データ（実際の申請内容・領収書画像）は使用しない。匿名化・架空データを使用する |
| バージョン管理 | テストケースはGitで管理し、変更履歴を追跡する |
| 評価データセットの更新 | プロンプト変更・ナレッジ変更・モデルバージョン変更時に回帰評価を実施し、必要に応じてケースを追加する |

## 8. 評価実施フロー

### 8.1 実施工程

```
詳細設計完了
  ↓
評価テスト実施（python evals/eval_reimbursement.py）
  ↓
評価指標測定・合否判定
  ↓
品質チェック通過
```

### 8.2 ゲート条件（合格基準）

> 各MET-xxxの閾値は `評価指標一覧.md` の各指標定義書「5. 合格基準・閾値」を参照。

| ゲート | 移行基準 |
|--------|---------|
| 評価テスト完了 | MET-001（申請種別・経費区分判断正確率）≥ 0.95、MET-002（申請書作成ゴール達成率）≥ 0.90 を達成すること |
| 品質チェック完了 | 評価指標全項目（MET-001〜MET-002）が合否基準を満たすこと |

### 8.3 回帰評価方針

> 回帰評価のトリガー・実施内容・合否基準の詳細は `評価指標一覧.md` の「3.2 回帰評価」を参照。

| トリガー | 実施内容 |
|---------|---------|
| プロンプト変更時 | 全テストケースを再実行し、変更前と同等以上のスコアを確認する |
| ナレッジ変更時（申請ルール・運賃データ更新） | 全テストケースを再実行し、各MET-xxxの閾値を満たすことを確認する |
| モデルバージョン変更時 | 全テストケースを再実行し、各MET-xxxの閾値を満たすことを確認する |

## 9. 設計上の留意点・リスク

**LLM応答の非決定性による不安定な評価**:
- リスク: 同一テストケースでも実行ごとにスコアが変動し、合否判定が安定しない
- 対策: 各テストケースは1回のみ実行する。LLM-as-Judgeの判定根拠をレポートに記録して変動要因を分析する

**LLM-as-Judge の信頼性**:
- リスク: 判定モデルが評価対象モデルと同一の場合、自己評価バイアスが生じる
- 対策: 判定モデルを評価対象モデルと分離し、段階評価にはルーブリックとfew-shot exampleを明示する

**フレームワーク間の差異**:
- リスク: strands-agents-evalsのバージョンアップにより評価APIが変更され、テストコードが動作しなくなる
- 対策: 本書ではフレームワーク非依存の抽象概念のみを定義し、フレームワーク固有のAPI仕様は詳細設計で具体化する
