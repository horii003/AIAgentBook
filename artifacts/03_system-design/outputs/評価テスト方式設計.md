## 1. 目的・適用範囲

### 1.1 目的

AIエージェントシステムの品質を保証するための評価テスト方式を定義する。本書は詳細設計フェーズの評価テスト詳細設計および実装フェーズの評価テストコード生成へのインプットとなる。

> **評価指標・合否基準・実施方法・回帰評価方針は `評価指標一覧.md` を正とする。本書はそれらを参照し、評価の「仕組み・アーキテクチャ・環境」を定義する。**

### 1.2 適用範囲

- AIエージェント評価テスト：LLM-as-Judge方式によるエージェント品質の自動評価
- 対象指標: MET-001（申請種別判断正確率）、MET-002（経費申請ゴール達成率）

### 1.3 本方針で定義しないこと

- 個別テストケースの入力データと期待結果（詳細設計で定義）
- 評価テストコードの実装詳細（実装フェーズで定義）
- 評価フレームワーク固有のAPI仕様（詳細設計で定義）
- 評価指標の定義・閾値・実施方法・回帰評価方針（評価指標一覧.mdで定義）

## 2. 評価戦略・基本方針

### 2.1 AIエージェント固有のリスクと評価戦略

| リスク | 対応戦略 |
|--------|---------|
| LLM応答の非決定性 | 同一テストケースでの試行回数は1回とする。各テストケースは1度のみ実行するため、各テストケースごとの集計は行わない |
| プロンプトの意図逸脱 | LLM-as-Judgeで応答の妥当性を判定し、逸脱時はシステムプロンプトを見直す |
| エージェント間の情報伝達誤り | マルチターン評価（MET-002）でエンドツーエンドの情報伝達を検証する |
| ガードレール漏れ | 異常系テストケース（有害入力・プロンプトインジェクション）を含めてガードレールの動作を検証する |
| ユーザーシミュレーションの限界 | ActorSimulatorを使用してゴールに基づく動的なユーザー発話を生成し、実際の利用シナリオに近い評価を実施する |
| LLM-as-Judgeの信頼性 | 判定理由を必ず出力しレポートに含める。判定モデルは評価対象と同一モデル（jp.anthropic.claude-sonnet-4-5-20250929-v1:0）を使用する |
| フレームワーク間の差異 | strands-agents-evalsフレームワークを使用し、評価テストコードを標準化する |

## 3. 評価テストの共通抽象概念

### 3.1 コア抽象概念の定義

| 抽象概念 | 定義 |
|---------|------|
| テストケース | 評価の単位。入力・期待出力・期待軌跡・メタデータを持つ |
| 評価方式 | テストケースの実行結果に対してスコアリング（LLM-as-Judge・軌跡マッチ等）を適用し、スコア・合否・判定理由を返す評価の方法 |
| 評価結果 | スコア (float 0-1) + 合否 (bool) + 判定理由 (str) |
| 実験 | テストケース群 × 評価方式群のバッチ実行単位 |
| レポート | 全体スコア・個別スコア・判定理由の集約 |

### 3.2 テストケースの構成要素

| 要素 | 型 | 必須/任意 | 説明 |
|------|---|---------|------|
| 識別名 | string | 必須 | テストケースの一意識別子 |
| 入力 | string | 必須 | ユーザー発話テキスト |
| 期待出力 | string | 任意 | 期待する最終応答（GoalSuccessRateEvaluatorで使用） |
| 期待軌跡 | list[string] | 任意 | 期待するツール呼び出し順序 |
| メタデータ | dict | 任意 | タスク説明・ゴール・期待ツール名等 |
| セッションID | string | 自動生成 | テレメトリ分離用（未指定時は自動生成UUID） |

### 3.3 評価方式の種類

| 評価方式種別 | スコアリング方式 | 適用場面 |
|-----------|---------------|---------|
| **LLM-as-Judge（二値）** | 達成/未達成 → 1.0/0.0 | ゴール達成率・ツール選択妥当性 |
| **LLM-as-Judge（段階）** | N段階評価 → 0.0〜1.0 | 応答品質・有用性・一貫性 |
| **軌跡マッチ（決定的）** | ツール呼び出し列の一致判定 | ツール選択の正確性 |
| **完全一致/文字列マッチ** | 出力文字列の一致判定 | 分類タスク・固定回答 |
| **類似度評価** | コサイン類似度・ROUGE-L | 回答の一貫性・要約品質 |
| **カスタムコード** | Python関数によるスコアリング | 業務固有ルール・メトリクス計測 |

### 3.4 軌跡マッチモード

| マッチモード | 判定基準 | 適用場面 |
|-----------|---------|---------|
| strict | 完全一致（順序含む） | 厳密なワークフロー検証 |
| unordered | 順不同で全要素一致 | 順序不問のツール呼び出し検証 |
| subset | 期待が実際の部分集合 | 最低限のツール呼び出し検証 |
| superset | 実際が期待の部分集合 | 不要なツール呼び出しの検出 |

## 4. 評価レベルの定義

### 4.1 評価レベル体系

| 評価レベル | 評価対象 | 評価方式 | 判定粒度 | 対応指標 |
|-----------|---------|---------|---------|---------|
| セッションレベル | 会話全体のゴール達成 | LLM-as-JudgeがセッションのGoalSuccessRateを評価 | セッション単位 | MET-002 |
| ツールレベル | 個別のツール選択判断 | LLM-as-JudgeがToolSelectionAccuracyを評価 | ツール呼び出し単位 | MET-001 |
| 出力レベル | 最終応答の品質 | LLM-as-Judge / 類似度 / 完全一致 | 応答単位 | 要件上未定義 |
| 軌跡レベル | ツール呼び出し列全体 | 決定的マッチ / LLM-as-Judge | 軌跡単位 | 要件上未定義 |

### 4.2 評価レベルの選択基準

| 確認したいこと | 選択すべきレベル | 実行方式 |
|-------------|---------------|---------|
| ユーザーの業務ゴールが達成されるか（MET-002） | セッションレベル | マルチターン（ActorSimulator使用） |
| 正しいツール（エージェント）が選択されるか（MET-001） | ツールレベル | シングルターン（ToolSelectionAccuracyEvaluator使用） |
| 最終応答の品質 | 出力レベル | シングルターン |
| ツール呼び出しの順序・網羅性 | 軌跡レベル | シングル or マルチターン |

## 5. 評価方式の共通アーキテクチャ

### 5.1 実行フロー

```
┌─────────────────────────────────────────────────────────┐
│                    評価実行基盤                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [1. テストケース定義]                                    │
│    - 入力（ユーザー発話）                                  │
│    - メタデータ（ゴール / 期待ツール / タスク説明）           │
│                                                         │
│  [2. タスク実行]                                         │
│    - エージェントにテストケースの入力を送信                  │
│    - テレメトリによる実行軌跡（スパン）の自動記録            │
│    - シングルターン（MET-001）                            │
│      or マルチターン（MET-002: ActorSimulator使用）        │
│                                                         │
│  [3. 評価判定]                                           │
│    - ToolSelectionAccuracyEvaluator（MET-001）           │
│    - GoalSuccessRateEvaluator（MET-002）                 │
│                                                         │
│  [4. 結果レポート]                                        │
│    - 全体スコア（overall_score）                          │
│    - 個別スコア・合否・判定理由                            │
│    - JSONレポート（evals/logs/reports/）                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 テストケース設計方針

| 項目 | 方針 |
|------|------|
| テストケースの構成要素 | 識別名・入力テキスト・タスク説明・判定基準（ゴール or 期待ツール名） |
| テストデータの独立性 | 各テストケースは他のケースに依存しない（ケース実行前にスパンをクリア） |
| テレメトリの分離 | ケース実行前にインメモリスパンエクスポーターをクリアし、ケース間のデータ混入を防止 |
| 評価データセットの拡充 | 運用開始後も実際の利用データをもとに継続的にケースを追加 |
| 最低件数 | MET-001: 交通費精算申請・経費精算申請・判断不能の各ケースを含む最低3件。MET-002: 交通費精算申請・経費精算申請の各ケースを含む最低2件 |

### 5.3 ユーザーシミュレーター方式

| 項目 | 方針 |
|------|------|
| シミュレーターの役割 | ゴールに基づき動的なユーザー発話を生成（ActorSimulator使用） |
| 最大ターン数 | 10ターン（MET-002のマルチターン評価） |
| 会話終了条件 | ゴール達成判定（申請書ファイル生成完了）or 最大ターン到達 |
| 入力の起点 | テストケースの入力フィールド（最初のユーザー発話） |

### 5.4 LLM-as-Judge設計方針

| 項目 | 方針 |
|------|------|
| 判定モデル | 評価対象と同一モデル（jp.anthropic.claude-sonnet-4-5-20250929-v1:0）を使用 |
| 判定根拠の出力 | 判定理由を必ず出力しレポートに含める |
| スコアリング尺度 | 二値を基本（MET-001: ToolSelectionAccuracyEvaluator、MET-002: GoalSuccessRateEvaluator） |
| Few-shot例 | 複雑な判定にはfew-shot exampleを活用（詳細設計で定義） |

## 6. AIエージェント評価方針

### 6.1 評価観点マッピング

> 各評価観点の確認内容・対応指標IDは `評価指標一覧.md` の「1. 評価観点一覧（マスター）」を参照。

| 評価観点 | 評価方式 |
|---------|---------|
| 妥当性（MET-001） | ToolSelectionAccuracyEvaluator（LLM-as-Judge二値）によるツール選択の妥当性評価 |
| 有用性（MET-002） | GoalSuccessRateEvaluator（LLM-as-Judge二値）によるゴール達成率評価 |
| 安全性 | 要件上未定義（ガードレール要件定義に基づく異常系テストケースで検証） |
| 一貫性 | 要件上未定義 |
| ツール利用の適切性 | ToolSelectionAccuracyEvaluatorによるツール選択の妥当性評価（MET-001に含む） |
| 制御機構の有効性 | 要件上未定義（LoopControlHook・HumanApprovalHookの動作は単体テストで検証） |
| 運用品質 | 要件上未定義 |

## 7. 評価環境設計

### 7.1 実施環境

| 項目 | 値 |
|------|-----|
| Pythonバージョン | 3.10以上 |
| モデルID | jp.anthropic.claude-sonnet-4-5-20250929-v1:0 |
| AWS リージョン | ap-northeast-1 |
| 評価ログ出力先 | `evals/logs/` |
| 評価レポート出力先 | `evals/logs/reports/` |

### 7.2 評価実行基盤の要件

| 項目 | 要件 |
|------|------|
| テレメトリ | StrandsEvalsTelemetryのインメモリスパンエクスポーターで実行軌跡を記録 |
| 承認フック無効化 | HumanApprovalHookを自動承認パッチに差し替え（評価時はブロッキングフックを無効化） |
| エージェント生成 | 本番と同一構成で評価用エージェントを生成（helpers.pyのエージェント生成関数を使用） |
| ログ設定 | UTF-8対応・ファイル出力・SDKログ抑制（Strandsイベントループログレベル: WARNING） |
| LLM呼び出し | モックなし（本番同等環境で実呼び出し） |
| trace_attributes | Agent生成時にsession.idとgen_ai.conversation.idを設定（テレメトリのケース間混入防止） |

### 7.3 テストデータ管理方針

| 項目 | 方針 |
|------|------|
| テストケースの配置 | `evals/eval_{evaluation_name}.py`内のEVAL_CASES定数として定義 |
| 本番データの利用 | 本番データは使用しない（テスト用の合成データを使用） |
| バージョン管理 | Gitで管理する（テストケースの変更はコミット履歴で追跡） |
| 評価データセットの更新 | 運用開始後も実際の利用データをもとに継続的にケースを追加する |

## 8. 評価実施フロー

### 8.1 実施工程

```
詳細設計完了
  ↓
評価テストコード生成（evals/eval_*.py）
  ↓
評価テスト実施（python evals/eval_*.py）
  ↓
評価指標測定・合否判定
  ↓
品質チェック通過（MET-001: 0.9以上、MET-002: 0.8以上）
```

### 8.2 ゲート条件（合格基準）

> 各MET-xxxの閾値は `評価指標一覧.md` の各指標定義書「5. 合格基準・閾値」を参照。

| ゲート | 移行基準 |
|--------|---------|
| 評価テスト完了 | MET-001: 0.9以上、MET-002: 0.8以上（最低許容値） |
| 品質チェック完了 | 評価指標全項目が合否基準を満たすこと |

## 9. 設計上の留意点・リスク

**LLM応答の非決定性による不安定な評価**:
- リスク: 同一テストケースでも実行ごとにスコアが変動し、合否判定が不安定になる可能性がある
- 対策: 同一テストケースでの試行回数は1回とする。各テストケースは1度のみ実行するため、各テストケースごとの集計は行わない

**LLM-as-Judge の信頼性**:
- リスク: 判定モデルが誤った判定を下す可能性がある
- 対策: 判定理由を必ず出力しレポートに含め、人手でサンプリング確認する

**フレームワーク間の差異**:
- リスク: strands-agents-evalsのAPIが変更された場合に評価テストコードが動作しなくなる可能性がある
- 対策: strands-agents-evalsのバージョンをpyproject.tomlで固定する
