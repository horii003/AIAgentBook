## 1. 目的・適用範囲

### 1.1 目的

システム全体の品質を保証するためのテスト戦略・評価方式を定義する。本書は詳細設計フェーズの評価テスト詳細設計（DD-09）および実装フェーズのテストコード生成へのインプットとなる。

### 1.2 適用範囲

- 単体テスト：ツール（TOOL-001/TOOL-002）・データモデル・ハンドラー（HD-001〜HD-003）・セッションマネージャ（SM-001）・ナレッジ
- 統合テスト：エージェント間連携（AG-001→AG-002/AG-003）・ツール呼び出しフロー・セッション永続化・ガードレール連携
- E2Eテスト：会話フロー一覧.mdに定義された全フロー（CF-001〜CF-003）

### 1.3 本方針で定義しないこと

- 個別テストケースの入力データと期待結果（DD-09で定義）
- テストコードの実装詳細（実装フェーズで定義）

## 2. テスト戦略・基本方針

### 2.1 品質目標

| 指標 | 目標値 |
|------|--------|
| 単体テストカバレッジ | 80%以上 |
| 統合テスト成功率 | 100%（全テストケース通過） |
| E2Eテスト成功率 | 100%（全会話フロー通過） |
| 評価指標（MET-001〜MET-012）全項目 | 各指標の目標値を達成すること |

### 2.2 AIエージェント固有のリスクとテスト戦略

| リスク | 対応戦略 |
|--------|---------|
| LLM応答の非決定性 | 複数回実行して平均スコアで評価する。定量指標は閾値を設けて合否判定する |
| プロンプトの意図逸脱 | 会話フロー単位のE2Eテストシナリオで意図通りの応答を検証する |
| エージェント間の情報伝達誤り | invocation_stateのフィールド値を統合テストで検証する |
| ガードレール漏れ | 有害コンテンツ入力テストケースでGRD-001〜GRD-006の動作を検証する |

### 2.3 テスト自動化方針

| テスト層 | 自動化方針 | 手動テストとの役割分担 |
|---------|---------|-------------------|
| 単体テスト | 全自動化（pytest） | 手動テストなし |
| 統合テスト | 全自動化（pytest）。LLM呼び出しは実呼び出しまたはモックを選択可能 | 手動テストなし |
| E2Eテスト | 定量指標は自動化（strands-agents-evals）。定性指標（EV-004/EV-005）は手動評価 | 定性評価は評価担当者が実施 |

## 3. テスト層の定義

### 3.1 単体テスト

| 項目 | 内容 |
|------|------|
| 責務 | 各コンポーネントの単体動作の正確性検証 |
| スコープ | TOOL-001（交通費計算）・TOOL-002（申請書生成）・データモデル（InvocationState等）・HD-001（ErrorHandler）・HD-002（LoopControlHook）・HD-003（HumanApprovalHook）・SM-001（SessionManager）・ナレッジ（transport_policies/expense_policies） |
| 実施タイミング | 各コンポーネント実装完了後 |
| カバレッジ目標 | 80%以上 |
| フレームワーク | `pytest` + `pytest-cov` |
| モック方針 | Amazon Bedrock API呼び出しはモック化。ファイルシステムはtmpdir/tmp_pathを使用 |

### 3.2 統合テスト

| 項目 | 内容 |
|------|------|
| 責務 | コンポーネント間連携の正確性検証 |
| スコープ | AG-001→AG-002/AG-003の委譲フロー・TOOL-001/TOOL-002の呼び出しフロー・FileSessionManagerのセッション永続化・HumanApprovalHookの承認ゲート動作 |
| 実施タイミング | 主要コンポーネント統合後 |
| カバレッジ目標 | 主要連携パスを100%カバー |
| フレームワーク | `pytest` |
| モック方針 | Amazon Bedrock API呼び出しは実呼び出しまたはモックを選択可能（環境変数で切り替え） |

### 3.3 エンドツーエンドテスト

| 項目 | 内容 |
|------|------|
| 責務 | ユースケースに基づく会話シナリオ全体の動作検証 |
| スコープ | CF-001（申請種別判断・案内）・CF-002（交通費精算申請書作成）・CF-003（経費精算申請書作成）の全フロー |
| 実施タイミング | システム全体統合後 |
| フレームワーク | `pytest` + `strands-agents-evals` |
| モック方針 | Amazon Bedrock API呼び出しは実呼び出し。ファイル出力はtmpdir使用 |

## 4. 評価指標の測定方式

| 評価指標ID | 指標名 | 測定方法 | 合否基準（閾値） | 測定ツール/手段 |
|----------|-------|---------|-------------|-------------|
| MET-001 | 申請種別判断正解率 | テストシナリオ（申請内容テキスト・期待申請種別ペア）を入力し、AG-001の判断結果と期待値を比較する | 目標値: 0.95以上 / 最低許容値: 0.90以上 | strands-agents-evals / pytest |
| MET-002 | 交通費計算一致率 | テストデータ（出発地・目的地・交通手段・期待交通費）を入力し、TOOL-001の計算結果と期待値を比較する | 目標値: 1.0 / 最低許容値: 1.0 | pytest（TOOL-001単体テスト） |
| MET-003 | 申請書フィールド一致率 | テストデータで申請書を生成し、各フィールドの値を期待値と比較する | 目標値: 1.0 / 最低許容値: 1.0 | pytest（TOOL-002単体テスト） |
| MET-004 | 情報収集完全性スコア | 申請シナリオを実行し、収集された項目と必須項目リストを照合して5段階で採点する | 目標値: 4.5以上 / 最低許容値: 4.0以上 | 手動評価（評価担当者） |
| MET-005 | 業務ルール遵守スコア | BRL-01〜BRL-17の各ルールに対するテストシナリオを実行し、5段階で採点する | 目標値: 4.5以上 / 最低許容値: 4.0以上 | 手動評価（評価担当者） |
| MET-006 | 承認ゲート違反率 | ユーザー承認なしでTOOL-002が呼び出されるケースを検証する | 目標値: 0.0 / 最低許容値: 0.0 | pytest（HumanApprovalHook統合テスト） |
| MET-007 | 有害コンテンツ検知率 | 有害コンテンツを含む入力テストケースを実行し、ガードレールの検知率を測定する | 目標値: 1.0 / 最低許容値: 0.95以上 | pytest（ガードレール統合テスト） |
| MET-008 | 会話ターン応答時間 | 各会話ターンの応答時間を計測する | 目標値: 要件上未定義 / 最低許容値: 要件上未定義 | pytest（時間計測） |
| MET-009 | 処理成功率 | 同一入力に対して複数回実行し、成功率を測定する | 目標値: 0.99以上 / 最低許容値: 0.95以上 | strands-agents-evals / pytest |
| MET-010 | 経費区分判断正解率 | テストシナリオ（品目・期待経費区分ペア）を入力し、AG-003の判断結果と期待値を比較する | 目標値: 0.90以上 / 最低許容値: 0.85以上 | strands-agents-evals / pytest |
| MET-011 | 文字数制限制御率 | 500文字超の入力テストケースを実行し、再入力促進メッセージが返されるかを検証する | 目標値: 1.0 / 最低許容値: 1.0 | pytest（FR-018単体テスト） |
| MET-012 | 対話回数上限制御率 | 30回を超える対話テストケースを実行し、LoopControlHookが正常に動作するかを検証する | 目標値: 1.0 / 最低許容値: 1.0 | pytest（LoopControlHook単体テスト） |

## 5. AIエージェント評価方針

### 5.1 LLM応答品質評価

| 評価観点 | 評価方法 | 合否判定基準 |
|---------|---------|-----------|
| 応答の正確性 | テストシナリオの期待結果と実際の応答を比較する（MET-001/MET-010） | 正解率が最低許容値以上 |
| 応答の一貫性 | 同一入力に対して複数回実行し、応答の一貫性を確認する（MET-009） | 処理成功率が最低許容値以上 |
| 指示への準拠 | 会話フロー定義（CF-001〜CF-003）に沿った応答が返されるかを手動評価する | 評価スコアが最低許容値以上 |
| 日本語品質 | 応答テキストの日本語品質を手動評価する | 評価スコアが最低許容値以上 |

### 5.2 マルチエージェント連携評価

| 評価観点 | 評価方法 | 合否判定基準 |
|---------|---------|-----------|
| エージェント間情報伝達精度 | invocation_stateのフィールド値（applicant_name/application_date/session_id）が正しく伝播されるかを統合テストで検証する | 全フィールドが正しく伝播されること |
| invocation_state整合性 | InvocationStateモデルのバリデーションが正常に動作するかを単体テストで検証する | バリデーションエラーが発生しないこと |
| ツール呼び出し正確性 | TOOL-001/TOOL-002が正しいパラメータで呼び出されるかを統合テストで検証する（MET-002/MET-003） | 一致率が最低許容値以上 |

### 5.3 ガードレール動作検証方針

| 検証観点 | 検証方法 | 合否判定基準 |
|---------|---------|-----------|
| 入力フィルター発動（GRD-001〜GRD-003） | 有害コンテンツ・業務外トピックを含む入力テストケースを実行し、ガードレールが発動するかを検証する | 検知率が最低許容値以上（MET-007） |
| 出力フィルター発動（GRD-004〜GRD-005） | 有害コンテンツを含む出力が遮断されるかを検証する | 検知率が最低許容値以上 |
| ツール実行ガード発動（HumanApprovalHook） | ユーザー承認なしでTOOL-002が呼び出されないかを統合テストで検証する（MET-006） | 違反率が0.0 |
| フォールバック応答内容 | ガードレール発動時のフォールバック応答が適切な日本語メッセージであるかを手動評価する | 適切なメッセージが返されること |

## 6. テスト環境設計

### 6.1 テスト実施環境

| 項目 | 値 |
|------|-----|
| Pythonバージョン | 3.10以上 |
| モデルID | jp.anthropic.claude-sonnet-4-5-20250929-v1:0 |
| AWS リージョン | ap-northeast-1 |
| ストレージパス（テスト用） | tests/fixtures/sessions/（tmpdir使用） |
| テストデータパス | tests/fixtures/ |

### 6.2 モック・スタブ方針

| 対象 | テスト層 | 方針 | 理由 |
|------|--------|------|------|
| Amazon Bedrock（LLM呼び出し） | 単体テスト | モック（unittest.mock） | テスト実行コストの削減・テスト安定性の確保 |
| Amazon Bedrock（LLM呼び出し） | 統合テスト | 実呼び出し（環境変数で切り替え可能） | エージェント間連携の実際の動作を検証するため |
| Amazon Bedrock（LLM呼び出し） | E2Eテスト | 実呼び出し | 実際のLLM応答品質を評価するため |
| ファイルシステム（出力ファイル） | 単体テスト | tmpdir/tmp_path（pytest fixture） | テスト後のクリーンアップを自動化するため |
| セッションストレージ | 単体テスト | tmpdir/tmp_path（pytest fixture） | テスト後のクリーンアップを自動化するため |
| マスタデータ（JSON） | 単体テスト | テスト用フィクスチャデータ（tests/fixtures/） | 本番データへの依存を排除するため |

### 6.3 テストデータ管理方針

| 項目 | 方針 |
|------|------|
| テストデータの配置場所 | `tests/fixtures/` |
| テストデータの形式 | JSON（マスタデータ・テストシナリオ）・Excel（申請書テンプレート） |
| 本番データの利用 | 禁止（テスト専用データを使用する） |
| テストデータのバージョン管理 | Gitで管理する（.gitignoreに含めない） |

## 7. テスト実施フロー

### 7.1 テスト工程

```
詳細設計完了
  ↓
単体テスト実施（tests/unit/）
  ↓
単体テストゲート通過
  ↓
統合テスト実施（tests/integration/）
  ↓
統合テストゲート通過
  ↓
E2Eテスト実施（evals/）
  ↓
評価指標測定・合否判定（MET-001〜MET-012）
  ↓
品質チェック通過
```

### 7.2 ゲート条件（次工程移行基準）

| ゲート | 移行基準 |
|--------|---------|
| 単体テスト完了 | カバレッジ80%以上・全テストケース通過（失敗件数0） |
| 統合テスト完了 | 全テストケース通過（失敗件数0） |
| E2Eテスト完了 | 全会話フロー（CF-001〜CF-003）が正常に完了すること |
| 品質チェック完了 | MET-001〜MET-012の全指標が最低許容値を達成すること |

### 7.3 不具合管理方針

| 項目 | 方針 |
|------|------|
| 重大度分類 | Critical（システム停止・データ破損）/ Major（機能不全）/ Minor（軽微な不具合） |
| Critical不具合の対応方針 | 即座に修正して再テストを実施する。次工程への移行を停止する |
| 再テスト方針 | 不具合修正後は影響範囲のテストケースを再実行する |

## 8. 設計上の留意点・リスク

**LLM応答の非決定性による不安定なテスト**:
- リスク: 同一入力に対してLLMの応答が毎回異なるため、テスト結果が不安定になる可能性がある
- 対策: 複数回実行して平均スコアで評価する。定量指標は閾値を設けて合否判定する

**テスト環境と本番環境の差異**:
- リスク: テスト用モックと本番のAmazon Bedrock APIの動作が異なる場合、テストが本番の問題を検出できない
- 対策: 統合テスト・E2Eテストでは実際のAmazon Bedrock APIを使用する

**テスト実行コスト（Bedrock API呼び出し費用）**:
- リスク: E2Eテストで実際のAmazon Bedrock APIを呼び出すため、テスト実行コストが発生する
- 対策: 単体テストではモックを使用してコストを削減する。E2Eテストは必要最小限のシナリオに絞る
