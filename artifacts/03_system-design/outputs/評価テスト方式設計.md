## 1. 目的・適用範囲

### 1.1 目的

システム全体の品質を保証するためのテスト戦略・評価方式を定義する。本書は詳細設計フェーズの評価テスト詳細設計（DD-09）および実装フェーズのテストコード生成へのインプットとなる。

### 1.2 適用範囲

- 単体テスト：ツール（TOOL-001, TOOL-002）・データモデル・ハンドラー（ErrorHandler, LoopControlHook, HumanApprovalHook）・セッションマネージャ・ガードレール・ナレッジ
- 統合テスト：エージェント間連携（AG-001 → AG-002/AG-003）・ツール呼び出しフロー・セッション永続化
- E2Eテスト：会話フロー一覧.md に定義された全フロー（CF-001〜CF-003）

### 1.3 本方針で定義しないこと

- 個別テストケースの入力データと期待結果（DD-09で定義）
- テストコードの実装詳細（実装フェーズで定義）

## 2. テスト戦略・基本方針

### 2.1 品質目標

| 指標 | 目標値 |
|------|--------|
| 申請種別判断正確性（EVAL-001） | 判断誤り率 0% |
| 交通費計算正確性（EVAL-003） | 計算誤り率 0% |
| 申請書生成完全性（EVAL-004） | 記入漏れ率 0% |
| 業務ルール遵守率（EVAL-005） | 違反率 0% |
| エスカレーション適切性（EVAL-006） | 誤判定率 0% |
| 単体テストカバレッジ | 80%以上 |

### 2.2 AIエージェント固有のリスクとテスト戦略

| リスク | 対応戦略 |
|--------|---------|
| LLM応答の非決定性 | 複数回実行して応答の一貫性を確認する。strands-agents-evals を使用して評価スクリプトを定義する |
| プロンプトの意図逸脱 | 評価シナリオ（正常系・分岐系・異常系）を網羅し、期待結果との一致率を測定する |
| エージェント間の情報伝達誤り | invocation_state の内容を統合テストで検証する |
| ガードレール漏れ | ガードレール要件（GR-001〜GR-004）ごとに検証テストケースを定義する |

### 2.3 テスト自動化方針

| テスト層 | 自動化方針 | 手動テストとの役割分担 |
|---------|---------|-------------------|
| 単体テスト | pytest で完全自動化 | 手動テストなし |
| 統合テスト | pytest で自動化（LLM 呼び出しはモックまたは実呼び出し） | 複雑なシナリオは手動確認 |
| E2Eテスト | strands-agents-evals で自動化 + 手動確認 | 申請書ファイルの内容確認は手動 |

## 3. テスト層の定義

### 3.1 単体テスト

| 項目 | 内容 |
|------|------|
| 責務 | 各コンポーネントの単体動作の正確性検証 |
| スコープ | ツール（TOOL-001, TOOL-002）・データモデル（Pydantic バリデーション）・ハンドラー（ErrorHandler, LoopControlHook, HumanApprovalHook）・セッションマネージャ・ナレッジ |
| 実施タイミング | 各コンポーネント実装完了後 |
| カバレッジ目標 | 80%以上（pytest-cov で計測） |
| フレームワーク | `pytest` |
| モック方針 | Amazon Bedrock API 呼び出しはモック。ファイルシステムは一時ディレクトリを使用 |

### 3.2 統合テスト

| 項目 | 内容 |
|------|------|
| 責務 | コンポーネント間連携の正確性検証 |
| スコープ | AG-001 → AG-002/AG-003 の委譲フロー・invocation_state の伝播・ツール呼び出しフロー・セッション永続化・ガードレール連携 |
| 実施タイミング | 主要コンポーネント統合後 |
| カバレッジ目標 | 主要連携パス 100% |
| フレームワーク | `pytest` |
| モック方針 | Amazon Bedrock API 呼び出しは実呼び出し（テスト用 AWS 環境）またはモック。ファイルシステムは一時ディレクトリを使用 |

### 3.3 エンドツーエンドテスト

| 項目 | 内容 |
|------|------|
| 責務 | ユースケースに基づく会話シナリオ全体の動作検証 |
| スコープ | CF-001（申請種別判断・案内）、CF-002（経費精算申請書作成）、CF-003（交通費精算申請書作成）の全フロー |
| 実施タイミング | システム全体統合後 |
| フレームワーク | `strands-agents-evals` + `pytest` |
| モック方針 | Amazon Bedrock API は実呼び出し。ファイル出力は一時ディレクトリを使用 |

## 4. 評価指標の測定方式

| 評価指標ID | 指標名 | 測定方法 | 合否基準（閾値） | 測定ツール/手段 |
|----------|-------|---------|-------------|-------------|
| EVAL-001 | 申請種別判断正確性 | 評価シナリオ（正常系・分岐系・対象外）に対する AG-001 の判断結果を期待値と比較 | 判断誤り率 0%（全テストケース正答） | strands-agents-evals + pytest |
| EVAL-002 | 経費区分判断正確性 | 品目ごとの経費区分判断結果を期待値と比較 | 要件上未定義（目標値は詳細設計フェーズで決定） | strands-agents-evals + pytest |
| EVAL-003 | 交通費計算正確性 | 運賃データに基づく計算結果を期待値と比較 | 計算誤り率 0%（全テストケース正答） | pytest（TOOL-001 単体テスト） |
| EVAL-004 | 申請書生成完全性 | 生成された申請書ファイルの全必須項目を検証 | 記入漏れ率 0%（全必須項目が正しく記入されていること） | pytest（openpyxl で生成ファイルを検証） |
| EVAL-005 | 業務ルール遵守率 | 業務ルール（BRL-01〜BRL-17）ごとのテストケースに対する遵守率を測定 | 違反率 0%（全ルールが遵守されていること） | pytest + strands-agents-evals |
| EVAL-006 | エスカレーション適切性 | 対象外申請シナリオに対する AG-001 のエスカレーション判断を期待値と比較 | 誤判定率 0%（全テストケース正答） | strands-agents-evals + pytest |
| EVAL-007 | 会話完了率 | CF-001〜CF-003 の全テストシナリオの完了率を測定 | 要件上未定義（目標値は詳細設計フェーズで決定） | strands-agents-evals |

## 5. AIエージェント評価方針

### 5.1 LLM応答品質評価

| 評価観点 | 評価方法 | 合否判定基準 |
|---------|---------|-----------|
| 応答の正確性 | 評価シナリオに対する応答を期待値と比較（strands-agents-evals） | 全テストケースで期待値と一致すること |
| 応答の一貫性 | 同一入力に対して複数回実行し、応答のばらつきを確認 | 申請種別判断・業務ルール適用の結果が一致すること |
| 指示への準拠 | システムプロンプトの指示（業務スコープ・ルール適用等）への準拠を確認 | 業務スコープ外の応答が発生しないこと |
| 日本語品質 | 応答メッセージの日本語品質を手動確認 | 自然な日本語で記述されていること |

### 5.2 マルチエージェント連携評価

| 評価観点 | 評価方法 | 合否判定基準 |
|---------|---------|-----------|
| エージェント間情報伝達精度 | invocation_state の内容（applicant_name, application_date, session_id）を統合テストで検証 | 全フィールドが正しく伝播されていること |
| invocation_state整合性 | AG-001 が設定した invocation_state を AG-002/AG-003 が正しく参照できることを確認 | 申請者名・申請日が申請書に正しく記入されていること |
| ツール呼び出し正確性 | TOOL-001・TOOL-002 の呼び出しパラメータと戻り値を統合テストで検証 | 期待するパラメータでツールが呼び出されること |

### 5.3 ガードレール動作検証方針

| 検証観点 | 検証方法 | 合否判定基準 |
|---------|---------|-----------|
| 入力フィルター発動（GR-001） | 業務スコープ外の入力（有給申請等）を送信し、拒否されることを確認 | 業務スコープ外の処理が実行されないこと |
| 出力フィルター発動 | 有害コンテンツを含む応答が生成されないことを確認 | 有害コンテンツが出力されないこと |
| ツール実行ガード発動（GR-003） | 承認なしで TOOL-002 が実行されないことを確認 | HumanApprovalHook が発火し、承認前にツールが実行されないこと |
| フォールバック応答内容 | ガードレール検知時の応答メッセージを確認 | ユーザーフレンドリーな日本語メッセージが表示されること |

## 6. テスト環境設計

### 6.1 テスト実施環境

| 項目 | 値 |
|------|-----|
| Pythonバージョン | 3.10以上 |
| モデルID | `jp.anthropic.claude-sonnet-4-5-20250929-v1:0` |
| AWS リージョン | `ap-northeast-1` |
| ストレージパス（テスト用） | `tests/fixtures/sessions/`（一時ディレクトリ） |
| 出力ファイルパス（テスト用） | `tests/fixtures/output/`（一時ディレクトリ） |

### 6.2 モック・スタブ方針

| 対象 | テスト層 | 方針 | 理由 |
|------|--------|------|------|
| Amazon Bedrock（LLM呼び出し） | 単体テスト | モック（pytest-mock） | テスト実行コストの削減・応答の安定化 |
| Amazon Bedrock（LLM呼び出し） | 統合テスト | 実呼び出し（テスト用 AWS 環境） | エージェント間連携の実際の動作を検証するため |
| Amazon Bedrock（LLM呼び出し） | E2Eテスト | 実呼び出し | 本番に近い環境での動作検証のため |
| ファイルシステム（出力ファイル） | 単体テスト | 一時ディレクトリ（tmp_path） | テスト後のクリーンアップを容易にするため |
| セッションストレージ | 単体テスト | 一時ディレクトリ（tmp_path） | テスト後のクリーンアップを容易にするため |
| 運賃データ（JSON） | 単体テスト | テスト用フィクスチャデータ | 本番データへの依存を排除するため |

### 6.3 テストデータ管理方針

| 項目 | 方針 |
|------|------|
| テストデータの配置場所 | `tests/fixtures/`（JSON・Excel テンプレート等） |
| テストデータの形式 | JSON（運賃データ）、Excel（申請書テンプレート）、Python（評価スクリプト内定義） |
| 本番データの利用 | 禁止。テスト専用のフィクスチャデータを使用する |
| テストデータのバージョン管理 | Git で管理する |

## 7. テスト実施フロー

### 7.1 テスト工程

```
詳細設計完了
  ↓
単体テスト実施（tests/unit/）
  ↓
単体テストゲート通過（カバレッジ 80%以上・失敗件数 0）
  ↓
統合テスト実施（tests/integration/）
  ↓
統合テストゲート通過（主要連携パス 100%・失敗件数 0）
  ↓
E2Eテスト実施（evals/eval_reimbursement.py）
  ↓
評価指標測定・合否判定（EVAL-001〜EVAL-007）
  ↓
品質チェック通過
```

### 7.2 ゲート条件（次工程移行基準）

| ゲート | 移行基準 |
|--------|---------|
| 単体テスト完了 | カバレッジ 80%以上、失敗件数 0 |
| 統合テスト完了 | 主要連携パス 100%、失敗件数 0 |
| E2Eテスト完了 | 評価シナリオ全件完了、失敗件数 0 |
| 品質チェック完了 | EVAL-001〜EVAL-006 の合否基準を全て満たすこと |

### 7.3 不具合管理方針

| 項目 | 方針 |
|------|------|
| 重大度分類 | Critical（業務ルール違反・申請書生成失敗）/ Major（機能不全）/ Minor（軽微な表示問題） |
| Critical不具合の対応方針 | 即座に修正し、再テストを実施してから次工程へ進む |
| 再テスト方針 | 修正後は影響範囲の単体テスト・統合テストを再実施する |

## 8. 設計上の留意点・リスク

**LLM応答の非決定性による不安定なテスト**:
- リスク: 同一入力に対して LLM の応答が毎回異なる場合、テストが不安定になる
- 対策: 申請種別判断・業務ルール適用等の判断系テストは複数回実行して一貫性を確認する。strands-agents-evals の評価スクリプトで期待値の範囲を定義する

**テスト環境と本番環境の差異**:
- リスク: テスト用 AWS 環境と本番環境でガードレール設定が異なる場合、テスト結果が本番に反映されない
- 対策: テスト環境と本番環境で同一のガードレール設定（GUARDRAIL_ID・GUARDRAIL_VERSION）を使用する

**テスト実行コスト（Bedrock API呼び出し費用）**:
- リスク: 統合テスト・E2Eテストで Amazon Bedrock API を実呼び出しする場合、テスト実行コストが増加する
- 対策: 単体テストは全てモックを使用する。統合テスト・E2Eテストは必要最小限のテストケースに絞る
