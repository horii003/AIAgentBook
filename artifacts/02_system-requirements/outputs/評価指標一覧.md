## 3. 評価の実施方法

### 3.1 開発段階における評価

| フェーズ | 実施方法 | 対象観点 |
|---------|---------|---------|
| 開発初期 | 人手評価（申請種別判断・申請書生成結果を目視確認） | 妥当性・有用性 |
| 開発中期 | 自動評価の段階的導入（MET-001〜MET-002をstrands-agents-evalsで自動実行） | 全観点 |
| リリース前 | 全テストケースの自動実行（MET-001〜MET-002、必須ゲート） | 全観点（必須ゲート） |

### 3.2 回帰評価

| トリガー | 実施内容 | 合否基準 |
|---------|---------|---------|
| プロンプト変更時 | 全テストケースを再実行 | 変更前と同等以上のスコア（各MET-xxxの閾値を満たすこと） |
| ナレッジ変更時（申請ルール・運賃データ更新） | 全テストケースを再実行 | 変更前と同等以上のスコア（各MET-xxxの閾値を満たすこと） |
| モデルバージョン変更時 | 全テストケースを再実行 | 各MET-xxxの閾値を満たすこと |

### 3.3 運用段階における継続的評価

- リリース後も品質は変動し続けるため、評価を運用サイクルに組み込む
- 月次で全テストケースの回帰評価を実施する

## 4. 評価観点と評価レベルの対応

| 評価観点 | 推奨する評価レベル | 実行方式 | 理由 |
|---------|----------------|---------|------|
| 妥当性 | セッションレベル | マルチターン | 申請種別判断・経費区分判断は会話全体の応答品質を評価するため |
| 有用性 | セッションレベル | マルチターン | 申請書作成ゴールの達成はエンドツーエンドで検証するため |

# 評価指標定義書

## 評価指標ID：MET-001
## 評価指標名：申請種別・経費区分判断正確率

### 1. 基本情報

| 項目 | 内容 |
|---|---|
| 評価観点 | 妥当性 |
| 対象エージェント | AG-001（申請種別判断）、AG-002（経費区分判断） |
| 対象会話フロー | CF-001、CF-002 |

### 2. 指標の目的・意味

- 申請内容テキストから申請種別（交通費精算申請/経費精算申請）を正しく判断できているか、および領収書品目から経費区分（事務用品費/宿泊費/資格精算費/その他経費）を正しく判断できているかを評価する
- 業務要件BR-02（申請種別の自動判断）・BR-05（申請書自動作成）の達成度に直結する
- スコアが閾値を下回った場合、プロンプトまたはナレッジの見直しを行う

### 3. 指標定義

| 項目 | 内容 |
|---|---|
| 指標種別 | 定量（LLM-as-Judge段階：1〜5点） |
| 定義式 | 正しく判断されたケース数 ÷ 全テストケース数 |
| 評価単位 | テストケース |
| スコア範囲 | 0.0〜1.0 |
| 高いほど良い | Yes |

### 4. 評価方法・測定手順

| 項目 | 内容 |
|---|---|
| 評価方法 | LLM-as-Judge（段階評価：1〜5点）でエージェントの判断結果と期待値を比較 |
| 実施手順 | 1. 申請内容テキストをエージェントに入力 → 2. 判断結果を取得 → 3. LLMジャッジが期待値と比較し1〜5点で採点 → 4. 4点以上を「正解」としてカウント |
| 評価者 | システム（LLM） |

### 5. 合格基準・閾値

| 項目 | 内容 |
|---|---|
| 目標値 | 0.98以上 |
| 最低許容値 | 0.95以上 |
| 不合格時対応 | プロンプト修正またはナレッジ（申請ルール）の更新を実施し再評価 |

### 6. 評価頻度

| フェーズ | 実施有無 |
|---|---|
| 開発中 | ○ |
| リリース前 | ○（必須ゲート） |
| 運用中 | △ |

## 評価指標ID：MET-002
## 評価指標名：申請書作成ゴール達成率

### 1. 基本情報

| 項目 | 内容 |
|---|---|
| 評価観点 | 有用性 |
| 対象エージェント | AG-002（経費精算申請）、AG-003（交通費精算申請） |
| 対象会話フロー | CF-002、CF-003 |

### 2. 指標の目的・意味

- 社員が申請内容を入力してから申請書ファイル（Excel）が正常に生成・提示されるまでのエンドツーエンドのゴール達成を評価する
- 業務要件BR-05（申請書自動作成）・RQ-03（申請書自動作成）の達成度に直結する
- スコアが閾値を下回った場合、会話フローまたはツール（TOOL-002）の不具合を調査する

### 3. 指標定義

| 項目 | 内容 |
|---|---|
| 指標種別 | 定量（LLM-as-Judge二値：達成/未達成） |
| 定義式 | 申請書ファイルが正常生成されたセッション数 ÷ 全テストセッション数 |
| 評価単位 | セッション |
| スコア範囲 | 0.0〜1.0 |
| 高いほど良い | Yes |

### 4. 評価方法・測定手順

| 項目 | 内容 |
|---|---|
| 評価方法 | LLM-as-Judge（二値）でセッション終了時に申請書ファイルが生成されたかを判定 |
| 実施手順 | 1. テストシナリオに従い会話を実行 → 2. セッション終了時に申請書ファイルの存在を確認 → 3. ファイルが存在し内容が正常であれば「達成」と判定 |
| 評価者 | システム（LLM）＋システム（コード：ファイル存在確認） |

### 5. 合格基準・閾値

| 項目 | 内容 |
|---|---|
| 目標値 | 0.95以上 |
| 最低許容値 | 0.90以上 |
| 不合格時対応 | 会話フロー定義またはTOOL-002（申請書生成ツール）の不具合を調査し修正 |

### 6. 評価頻度

| フェーズ | 実施有無 |
|---|---|
| 開発中 | ○ |
| リリース前 | ○（必須ゲート） |
| 運用中 | △ |
