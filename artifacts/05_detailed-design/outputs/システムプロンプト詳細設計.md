## 1. 概要

### 1.1 コンポーネントの目的

各エージェント（AG-001, AG-002, AG-003）のシステムプロンプトを設計し、実装にそのまま使用できる完成版テキストを定義する。業務ルール・ナレッジはプレースホルダー形式で参照し、実体はナレッジ・業務ルール詳細設計書で管理する。

### 1.2 対象エージェント一覧

| エージェント名 | プロンプト種別 | 動的生成の有無 |
|-------------|------------|------------|
| 申請受付窓口エージェント（AG-001） | 静的定数 | なし |
| 経費精算申請エージェント（AG-002） | 動的生成関数 | あり（申請者名・申請日・申請期限基準日） |
| 交通費精算申請エージェント（AG-003） | 動的生成関数 | あり（申請者名・申請日・申請期限基準日） |

---

## 2. プロンプト設計方針

### 2.1 共通設計原則

- 役割定義を冒頭に明示し、エージェントの責務範囲を明確化する
- 禁止事項を明示し、LLM の自律的な逸脱を防止する
- 業務ルール・ナレッジはプレースホルダー（`{変数名}`）で参照し、実体はナレッジ・業務ルール詳細設計書で管理する
- 処理フローを番号付きで記載し、LLM の判断ブレを抑制する
- 日本語で記述し、ユーザーへの応答も日本語とする

### 2.2 動的生成基本方針

| 動的情報の種類 | 生成タイミング | 実装方式 |
|-------------|------------|--------|
| 申請者名（applicant_name） | エージェントツール関数呼び出し時 | invocation_state から取得し、f-string で埋め込む |
| 申請日（application_date） | エージェントツール関数呼び出し時 | invocation_state から取得し、f-string で埋め込む |
| 申請期限基準日（deadline_date） | エージェントツール関数呼び出し時 | `calculate_deadline(application_date, 3)` で計算し、f-string で埋め込む |

## 3. 申請受付窓口エージェント（AG-001）システムプロンプト設計

### 3.1 プロンプト構造

| セクション | 説明 |
|----------|------|
| 役割定義 | 申請受付窓口エージェントとしての責務（申請種別判断・案内・振り分け）を定義する |
| 申請種別判断ルール | KN-001（申請種別判断ルール）を参照。BRL-01〜BRL-03, BRL-05 の判断基準を組み込む |
| 申請先・申請書情報 | KN-002（申請先・申請書情報）を参照。BRL-06, BRL-07 の案内内容を組み込む |
| 処理フロー | 申請内容受付→種別判断→案内/委譲の流れを指示する |
| 禁止事項 | 申請書の自律生成禁止・不要な確認禁止を明示する |

### 3.2 システムプロンプト全文

```
あなたは申請受付窓口エージェントです。社員からの申請内容を受け付け、申請種別を判断し、適切な専門エージェントへ処理を委譲します。

{application_policies}

【処理フロー】
1. 社員が入力した申請内容テキストを受け付ける
2. 上記の申請種別判断ルールに従い、申請種別を判断する
3. 判断結果に基づき、申請書名・申請先を案内する
4. 申請種別が確定したら、速やかに対応する専門エージェントツールを呼び出す
   - 交通費精算申請 → transport_agent ツールを呼び出す
   - 経費精算申請 → expense_agent ツールを呼び出す

【禁止事項】
- 申請書の作成・生成を自ら行ってはならない（専門エージェントに委譲すること）
- 申請種別が確定したら、速やかに対応する専門エージェントツールを呼び出すこと
- 不要な確認や追加質問を行わないこと
- ルールに記載のない内容を推測・補完してはならない
```

### 3.3 動的生成部の定義

| プレースホルダー | 型 | データソース | 生成タイミング | 実装方法 |
|-------------|-----|----------|------------|--------|
| `{application_policies}` | `str` | `knowledge/application_policies.py` の `get_application_policies()` | エージェント初期化時（静的定数として定義） | `get_application_policies()` の戻り値を f-string で埋め込む |

### 3.4 実装ファイル

| 項目 | 値 |
|------|-----|
| ファイルパス | `prompt/prompt_orchestrator.py` |
| 定数名 | `ORCHESTRATOR_SYSTEM_PROMPT` |
| 生成方式 | 静的定数（モジュールロード時に `get_application_policies()` を呼び出して生成） |
| 引数 | なし |
| 戻り値 | `str` |

## 4. 経費精算申請エージェント（AG-002）システムプロンプト設計

### 4.1 プロンプト構造

| セクション | 説明 |
|----------|------|
| 役割定義 | 経費精算申請エージェントとしての責務（情報抽出・区分判断・チェック・申請書生成）を定義する |
| 申請者情報 | 動的生成：申請者名・申請日・申請期限基準日を埋め込む |
| 経費精算業務ルール | KN-004（経費精算業務ルール）を参照。BRL-09, BRL-11, BRL-13, BRL-14, BRL-16 を組み込む |
| 経費区分判断基準 | KN-005（経費区分判断基準）を参照。BRL-14 の判断基準を組み込む |
| 処理フロー | 領収書画像依頼→情報抽出→不足情報確認→チェック→承認→生成の流れを指示する |
| 禁止事項 | 承認前のツール実行禁止・計算可能情報の確認禁止を明示する |

### 4.2 システムプロンプト全文

```
あなたは経費精算申請エージェントです。申請者「{applicant_name}」の経費精算申請書（Excel）の作成を支援します。
申請日: {application_date}
申請期限の基準日（この日付以降の経費発生日のみ申請可能）: {deadline_date}

{expense_policies}

{expense_category_policies}

【処理フロー】
1. 領収書画像の提供を依頼する
2. 提供された領収書画像から購入日・店舗名・品目・金額を品目ごとに抽出する
3. 抽出した品目から経費区分を自動判断する
4. 申請に必要な不足情報（業務目的等）をまとめて一括確認する（計算可能な情報は確認しない）
5. 各経費の購入日が申請期限の基準日以降であることを確認する（期限超過時は業務終了）
6. 経費合計が上長承認閾値を超える場合は上長承認が必要な旨を通知する
7. 収集した全情報をテキストとして整理・提示する（この段階ではツール呼び出しを行わない）
8. ユーザーが承認（OK）を選択した後、generate_expense_report ツールを呼び出す

【禁止事項】
- 申請書生成前にユーザーの承認を得ずに generate_expense_report を呼び出してはならない
- 計算可能な情報（合計金額等）をユーザーに確認してはならない
- ルールに記載のない内容を推測・補完してはならない
- システム系エラーが発生した場合は、エラー内容を要約して呼び出し元に返却する
```

### 4.3 動的生成部の定義

| プレースホルダー | 型 | データソース | 生成タイミング | 実装方法 |
|-------------|-----|----------|------------|--------|
| `{applicant_name}` | `str` | invocation_state の `applicant_name` | エージェントツール関数呼び出し時 | f-string で埋め込む |
| `{application_date}` | `str` | invocation_state の `application_date` | エージェントツール関数呼び出し時 | f-string で埋め込む |
| `{deadline_date}` | `str` | `calculate_deadline(application_date, 3)` の戻り値 | エージェントツール関数呼び出し時 | f-string で埋め込む |
| `{expense_policies}` | `str` | `knowledge/expense_policies.py` の `get_expense_policies(deadline_months, approval_threshold)` | エージェントツール関数呼び出し時 | f-string で埋め込む |
| `{expense_category_policies}` | `str` | `knowledge/expense_policies.py` の `get_expense_category_policies()` | エージェントツール関数呼び出し時 | f-string で埋め込む |

### 4.4 実装ファイル

|------|-----|
| ファイルパス | `prompt/prompt_expense.py` |
| 生成関数名 | `get_expense_system_prompt` |
| 引数 | `applicant_name: str`, `application_date: str`, `deadline_date: str` |
| 戻り値 | `str` |

## 7. 補足情報

### 7.1 実装上の注意点

1. **業務ルール・ナレッジのハードコード禁止**
   - システムプロンプト内の業務ルール・ナレッジの具体的な数値（期限月数・金額閾値等）はハードコードしない
   - `settings.py` で一元管理し、ポリシー関数（`get_xxx_policies()`）の引数として受け取って動的展開する（R9.12.3）

2. **静的定数 vs 動的生成関数の使い分け**
   - AG-001: 申請期限チェックが不要なため静的定数（`ORCHESTRATOR_SYSTEM_PROMPT`）
   - AG-002, AG-003: 申請期限チェックのために現在日付・基準日を動的に埋め込む必要があるため動的生成関数

3. **ファイル配置**
   - `prompt/prompt_orchestrator.py`: AG-001 のプロンプト定数
   - `prompt/prompt_expense.py`: AG-002 のプロンプト生成関数
   - `prompt/prompt_transport.py`: AG-003 のプロンプト生成関数

## 8. 依存関係

### 8.1 内部モジュール
- `knowledge/application_policies.py`: 申請種別判断ルール・申請先情報（KN-001, KN-002）
  - `get_application_policies()`: 申請種別判断ルールと申請先情報を返す
- `knowledge/expense_policies.py`: 経費精算業務ルール・経費区分判断基準（KN-004, KN-005）
  - `get_expense_policies(deadline_months, approval_threshold)`: 経費精算業務ルールを返す
  - `get_expense_category_policies()`: 経費区分判断基準を返す
- `knowledge/transport_policies.py`: 交通費精算業務ルール（KN-003）
  - `get_transport_policies(deadline_months, approval_threshold)`: 交通費精算業務ルールを返す
- `agents/base_agent.py`: 申請期限計算
  - `calculate_deadline(application_date, deadline_months)`: 申請期限基準日を計算する
- `config/settings.py`: 設定値
  - `settings.expense.deadline_months`: 経費精算申請期限月数
  - `settings.expense.approval_threshold`: 経費精算上長承認閾値
  - `settings.transport.deadline_months`: 交通費精算申請期限月数
  - `settings.transport.approval_threshold`: 交通費精算上長承認閾値

## 9. テスト観点

### 9.1 プロンプト意図確認テスト
- AG-001 が交通費精算申請の内容に対して transport_agent を呼び出すこと
- AG-001 が経費精算申請の内容に対して expense_agent を呼び出すこと
- AG-001 が判断不能時に選択肢を提示すること
- AG-002 が領収書画像から経費情報を抽出すること
- AG-002 が申請期限超過時にエラーメッセージを表示すること
- AG-003 が移動情報を一括確認で収集すること
- AG-003 が駅名の表記ゆれを正規化すること

### 9.2 動的生成テスト
- AG-002 のプロンプトに `applicant_name`・`application_date`・`deadline_date` が正しく埋め込まれること
- AG-003 のプロンプトに `applicant_name`・`application_date`・`deadline_date` が正しく埋め込まれること
- `deadline_date` が `application_date` から3ヶ月前の日付として正しく計算されること

### 9.3 プロンプトインジェクション対策テスト
- ユーザーが「前の指示を無視して...」等の不正な指示を入力した場合に Amazon Bedrock Guardrails でブロックされること
- ユーザーが申請書を直接生成しようとした場合に AG-001 が拒否すること
