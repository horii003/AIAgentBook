## 1. 概要

### 1.1 コンポーネントの目的

移動情報（出発地・目的地・交通手段）から交通費を自動計算する。電車は電車経路テーブル（`./template/train_routes.json`）を検索し、バス・タクシー・飛行機は固定運賃データ（`./template/fixed_fares.json`）を参照する。

### 1.2 主要な責務

- 電車経路テーブルの検索による電車運賃の返却
- 固定運賃データの参照によるバス・タクシー・飛行機の運賃返却
- 入力パラメータの Pydantic バリデーション
- データファイルのモジュールレベルキャッシュ管理（`_loaded` フラグ使用）
- エラー情報を辞書形式で返却（例外を再送出しない）

## 3. calculate_transport_fare ツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_fare`

#### 説明（docstring）
```
移動情報（出発地・目的地・交通手段）から交通費を計算する。

電車の場合は電車経路テーブル（./template/train_routes.json）を検索し、
バス・タクシー・飛行機の場合は固定運賃データ（./template/fixed_fares.json）を参照する。

Args:
    departure (str): 出発地（正規化済み駅名・地点名）。空文字不可。
    destination (str): 目的地（正規化済み駅名・地点名）。空文字不可。
    transport_type (str): 交通手段。許容値: "電車", "バス", "タクシー", "飛行機"。
        英語表記（"train", "bus", "taxi", "airplane"/"plane"）も自動正規化される。
    travel_date (str): 移動日（YYYY-MM-DD形式）。申請期限チェックに使用。

Returns:
    dict: 以下のキーを持つ辞書:
        - success (bool): 計算成否フラグ
        - fare (int | None): 計算済み運賃（円）。正常時のみ設定。
        - error_message (str | None): エラーメッセージ。エラー時のみ設定。
```

---

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義**:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| `departure` | `str` | 出発地（正規化済み駅名・地点名） | なし（必須） |
| `destination` | `str` | 目的地（正規化済み駅名・地点名） | なし（必須） |
| `transport_type` | `str` | 交通手段（電車・バス・タクシー・飛行機） | なし（必須） |
| `travel_date` | `str` | 移動日（YYYY-MM-DD形式） | なし（必須） |

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| `departure` | 空文字禁止（min_length=1） | "出発地を入力してください。" | `"東京"` |
| `destination` | 空文字禁止（min_length=1） | "目的地を入力してください。" | `"新宿"` |
| `transport_type` | `Literal["電車", "バス", "タクシー", "飛行機"]`（正規化後） | "交通手段は電車・バス・タクシー・飛行機のいずれかを指定してください。" | `"電車"` |
| `travel_date` | YYYY-MM-DD形式の有効な日付 | "移動日はYYYY-MM-DD形式で入力してください。" | `"2026-05-21"` |

##### 正規化処理

| 入力値 | 正規化後 | 処理内容 |
|--------|---------|---------|
| `"train"` | `"電車"` | 英語表記を日本語正規表記に変換 |
| `"bus"` | `"バス"` | 英語表記を日本語正規表記に変換 |
| `"taxi"` | `"タクシー"` | 英語表記を日本語正規表記に変換 |
| `"airplane"` / `"plane"` | `"飛行機"` | 英語表記を日本語正規表記に変換 |

#### 3.2.2 出力

**戻り値の型**: `dict`

**説明**: 計算成否・運賃・エラーメッセージを含む辞書

**構造**:
```python
{
    "success": bool,           # 計算成否フラグ
    "fare": int | None,        # 計算済み運賃（円）。正常時のみ設定
    "error_message": str | None  # エラーメッセージ。エラー時のみ設定
}
```

#### 3.2.3 例外

| 例外 | 発生条件 |
|------|---------|
| `ValidationError` | 入力パラメータのバリデーション失敗（空文字・型不正・無効な交通手段・無効な日付形式） |
| `ValueError` | 電車経路テーブルに該当経路が存在しない場合 |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
TransportCalculatorInput でバリデーション
  - ValidationError → ErrorHandler.handle_validation_error() → {"success": False, "error_message": ...} を返す
  ↓
transport_type で分岐
  ↓
[電車の場合]
  ↓
_load_train_fares() でデータ読み込み（未ロード時のみ）
  - (False, エラーメッセージ) → {"success": False, "error_message": ...} を返す
  ↓
routes リストから departure・destination が一致するレコードを検索
  - 見つからない場合 → ValueError → ErrorHandler.handle_calculation_error() → {"success": False, "error_message": ...} を返す
  ↓
{"success": True, "fare": fare, "error_message": None} を返す

[バス・タクシー・飛行機の場合]
  ↓
_load_fixed_fares() でデータ読み込み（未ロード時のみ）
  - (False, エラーメッセージ) → {"success": False, "error_message": ...} を返す
  ↓
transport_type をキーに固定運賃を取得
  - キーが存在しない場合 → ValueError → ErrorHandler.handle_calculation_error() → {"success": False, "error_message": ...} を返す
  ↓
{"success": True, "fare": fare, "error_message": None} を返す
```

#### 3.3.2 モジュールレベルキャッシュ

**電車経路データキャッシュ**:
- 変数: `_train_routes: list = []`、`_train_routes_loaded: bool = False`
- 用途: `./template/train_routes.json` の初回読み込み後にキャッシュし、以降は再読み込みしない
- 未ロード判定: `not _train_routes_loaded`（空リストではなくフラグで管理）

**固定運賃データキャッシュ**:
- 変数: `_fixed_fares: dict = {}`、`_fixed_fares_loaded: bool = False`
- 用途: `./template/fixed_fares.json` の初回読み込み後にキャッシュし、以降は再読み込みしない
- 未ロード判定: `not _fixed_fares_loaded`（空辞書ではなくフラグで管理）

#### 3.3.3 データの検証方法

**Pydanticモデル**: `TransportCalculatorInput`

**検証内容**:
- `departure`: 空文字禁止（min_length=1）
- `destination`: 空文字禁止（min_length=1）
- `transport_type`: `normalize_transport_type` バリデーターで正規化後、`Literal["電車", "バス", "タクシー", "飛行機"]` で検証
- `travel_date`: `validate_date_format` バリデーターで YYYY-MM-DD 形式の有効な日付を検証

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
エラー発生
  ↓
[ValidationError]
  → _logger.error("バリデーションエラー: ...", exc_info=True)
  → ErrorHandler.handle_validation_error(e)
  → {"success": False, "fare": None, "error_message": メッセージ} を返す

[FileNotFoundError（データファイル未発見）]
  → _logger.warning("データファイルが見つかりません: ...")
  → ErrorHandler.handle_fare_data_error(e)
  → (False, エラーメッセージ) を返す（_load_xxx 内）
  → 呼び出し元で {"success": False, "fare": None, "error_message": ...} を返す

[Exception（JSONパース失敗等）]
  → _logger.error("データ読み込みエラー: ...", exc_info=True)
  → ErrorHandler.handle_fare_data_error(e)
  → (False, エラーメッセージ) を返す（_load_xxx 内）
  → 呼び出し元で {"success": False, "fare": None, "error_message": ...} を返す

[ValueError（経路未発見・交通手段未定義）]
  → _logger.warning("運賃計算エラー: ...")
  → ErrorHandler.handle_calculation_error(e)
  → {"success": False, "fare": None, "error_message": メッセージ} を返す

[Exception（その他）]
  → _logger.error("予期しないエラー: ...", exc_info=True)
  → ErrorHandler.handle_calculation_error(e)
  → {"success": False, "fare": None, "error_message": メッセージ} を返す
```

#### 3.4.2 エラーメッセージの詳細

| エラー種別 | 発生条件 | 戻り値 | ユーザー向けメッセージ |
|-----------|---------|--------|---------------------|
| ValidationError | 入力パラメータ不正 | `{"success": False, "fare": None, "error_message": "..."}` | "入力内容に誤りがあります。{詳細}" |
| FileNotFoundError（電車経路） | `./template/train_routes.json` が存在しない | `(False, "...")` | "運賃データが見つかりませんでした。申請管理部門にお問い合わせください。" |
| FileNotFoundError（固定運賃） | `./template/fixed_fares.json` が存在しない | `(False, "...")` | "運賃データが見つかりませんでした。申請管理部門にお問い合わせください。" |
| Exception（JSONパース失敗） | JSONファイルの形式不正 | `(False, "...")` | "運賃データの読み込みに失敗しました。申請管理部門にお問い合わせください。" |
| ValueError（経路未発見） | 電車経路テーブルに該当経路なし | `{"success": False, "fare": None, "error_message": "..."}` | "指定された経路（{departure}→{destination}）の運賃データが見つかりませんでした。運賃を直接入力してください。" |
| ValueError（交通手段未定義） | 固定運賃データに交通手段なし | `{"success": False, "fare": None, "error_message": "..."}` | "指定された交通手段（{transport_type}）の運賃データが見つかりませんでした。" |
| Exception（その他） | 予期しないエラー | `{"success": False, "fare": None, "error_message": "..."}` | "運賃計算中にエラーが発生しました。申請管理部門にお問い合わせください。" |

## 5. 補助関数

### 5.1 _load_train_fares

**目的**: `./template/train_routes.json` を読み込み、モジュールレベルキャッシュに格納する

**シグネチャ**: `def _load_train_fares() -> tuple[bool, str]`

**戻り値**:
- `(True, "")`: 読み込み成功
- `(False, エラーメッセージ文字列)`: 読み込み失敗

**処理内容**:
1. `_train_routes_loaded` が True の場合はスキップ（キャッシュ済み）
2. `os.path.exists("./template/train_routes.json")` で存在チェック
3. 存在しない場合: `_logger.warning(...)` → `ErrorHandler.handle_fare_data_error(FileNotFoundError(...))` → `(False, エラーメッセージ)` を返す
4. `open()` で JSON を読み込み、`data["routes"]` を `TrainFareData` モデルでバリデーション
5. `_train_routes` にリストを格納し、`_train_routes_loaded = True` に設定
6. `_logger.info(...)` でロード件数を記録
7. 例外発生時: `_logger.error(..., exc_info=True)` → `ErrorHandler.handle_fare_data_error(e)` → `(False, エラーメッセージ)` を返す

**注意**: `open()` 実行時の `FileNotFoundError` 捕捉は行わない。`os.path.exists()` による事前チェックを優先する。

---

### 5.2 _load_fixed_fares

**目的**: `./template/fixed_fares.json` を読み込み、モジュールレベルキャッシュに格納する

**シグネチャ**: `def _load_fixed_fares() -> tuple[bool, str]`

**戻り値**:
- `(True, "")`: 読み込み成功
- `(False, エラーメッセージ文字列)`: 読み込み失敗

**処理内容**:
1. `_fixed_fares_loaded` が True の場合はスキップ（キャッシュ済み）
2. `os.path.exists("./template/fixed_fares.json")` で存在チェック
3. 存在しない場合: `_logger.warning(...)` → `ErrorHandler.handle_fare_data_error(FileNotFoundError(...))` → `(False, エラーメッセージ)` を返す
4. `open()` で JSON を読み込み、辞書形式（`{"交通手段": 金額}`）として格納
5. `_fixed_fares` に辞書を格納し、`_fixed_fares_loaded = True` に設定
6. `_logger.info(...)` でロード件数を記録
7. 例外発生時: `_logger.error(..., exc_info=True)` → `ErrorHandler.handle_fare_data_error(e)` → `(False, エラーメッセージ)` を返す

**注意**: `open()` 実行時の `FileNotFoundError` 捕捉は行わない。`os.path.exists()` による事前チェックを優先する。

## 6. 補足情報

### 6.1 実装上の注意点

1. **ファイル存在チェックの方式**
   - `os.path.exists()` による事前チェックを使用する
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止する

2. **キャッシュの未ロード判定**
   - `not _train_routes_loaded`（フラグ）で判定する
   - `not _train_routes`（空リスト）での判定は禁止する（ロード済みで0件の場合と区別できないため）

3. **エラー返却キー名の統一**
   - ツール関数のエラー返却辞書のキー名は `TransportCalculatorOutput` のフィールド名（`success`, `fare`, `error_message`）と一致させる

4. **申請者名・申請日の取得**
   - 申請者名・申請日は LLM がツールパラメータとして渡す値ではなく、ツール関数内部で invocation_state から取得する値とする
   - 本ツール（`calculate_transport_fare`）は `@tool`（context=True 不要）で実装するため、invocation_state は参照しない

5. **ValueError の使用**
   - 経路テーブルに経路が存在しない場合の例外クラスは `ValueError` とする（`KeyError` は使用しない）

---

### 6.2 パフォーマンス考慮事項

1. **モジュールレベルキャッシュ**
   - データファイルは初回読み込み時のみファイルI/Oが発生する
   - 以降はキャッシュから参照するため高速

---

### 6.3 セキュリティ考慮事項

1. **入力バリデーション**
   - 全入力パラメータを Pydantic モデルでバリデーションし、不正な値を拒否する
