## 1. 概要

### 1.1 コンポーネントの目的

出発地・目的地・交通手段・移動日をもとに、経路テーブル（`template/train_routes.json`）または固定運賃データ（`template/fixed_fares.json`）を参照して交通費を計算する。計算根拠（参照データ・経路情報）をエージェントに返し、申請書ドラフト提示に利用させる。経路未登録時はその旨をエージェントに通知し、ユーザーへの手動入力要求を促す。

### 1.2 主要な責務

- **経路テーブル参照**: `template/train_routes.json` から出発地・目的地に一致するレコードを検索し運賃を取得する（電車の場合）
- **固定運賃参照**: バス・タクシー・飛行機の場合は `template/fixed_fares.json` から固定運賃を取得する
- **交通費計算**: 取得した運賃を返す（往復・複数区間の合算はエージェントが担当）
- **計算根拠の提供**: 参照したデータID・経路情報をエージェントに返す
- **モジュールレベルキャッシュ**: データファイルを初回のみ読み込み、2回目以降はキャッシュを使用する

## 3. calculate_transport_fare ツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_fare`

#### 説明（docstring）
```
出発地・目的地・交通手段・移動日をもとに交通費を計算する。

電車の場合は経路テーブル（template/train_routes.json）を参照し、
バス・タクシー・飛行機の場合は固定運賃データ（template/fixed_fares.json）を参照する。
経路が登録されていない場合は not_found=True を返す。

Args:
    departure (str): 出発地（駅名・バス停名等。「駅」「バス停」「空港」等の接尾語は自動除去される）
    destination (str): 目的地（駅名・バス停名等。「駅」「バス停」「空港」等の接尾語は自動除去される）
    transport_type (str): 交通手段。以下のいずれかを指定する（表記ゆれは自動正規化される）:
        - 「電車」「train」「電鉄」「鉄道」→ 電車
        - 「バス」「bus」→ バス
        - 「タクシー」「taxi」「cab」→ タクシー
        - 「飛行機」「airplane」「plane」「flight」「航空」→ 飛行機
    travel_date (str): 移動日（YYYY-MM-DD形式）

Returns:
    dict: 以下のキーを持つ辞書:
        - success (bool): 処理成功フラグ
        - fare (float): 交通費（円）。success=True かつ not_found=False の場合のみ有効
        - calculation_basis (str): 計算根拠（参照データ・経路情報）。success=True かつ not_found=False の場合のみ有効
        - not_found (bool): 経路未登録フラグ。True の場合はユーザーへ手動入力を求める
        - error_message (str): エラーメッセージ。success=False の場合のみ有効
```

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
[ステップ1] 入力バリデーション
  - TransportCalculatorInput(departure, destination, transport_type, travel_date) でバリデーション
  - 駅名正規化（validate_station_name）・交通手段正規化（validate_transport_type）・日付検証（validate_date_format）を実施
  - ValidationError → エラーメッセージを返す（EX-01）
  ↓
[ステップ2] データファイル読み込み（キャッシュ未ロード時のみ）
  - _train_fares_loaded が False の場合: _load_train_fares() を呼び出す
  - _fixed_fares_loaded が False の場合: _load_fixed_fares() を呼び出す
  - 読み込みエラー → エラーメッセージを返す（EX-06）
    戻り値: {"success": False, "fare": 0.0, "calculation_basis": "", "not_found": False, "error_message": message}
  ↓
[ステップ3] 交通手段による分岐
  ↓
  transport_type == "電車"
    ↓
    [ステップ3a] 経路テーブル検索
      - _train_fares から departure・destination が一致するレコードを検索
      - 見つかった場合: fare・calculation_basis を設定して返す
      - 見つからない場合: not_found=True で返す
  ↓
  transport_type in ["バス", "タクシー", "飛行機"]
    ↓
    [ステップ3b] 固定運賃検索
      - _fixed_fares から transport_type が一致するレコードを検索
      - 見つかった場合: fare・calculation_basis を設定して返す
      - 見つからない場合: not_found=True で返す
  ↓
[ステップ4] 結果を返す
  - 正常: {"success": True, "fare": fare, "calculation_basis": basis, "not_found": False, "error_message": ""}
  - 未登録: {"success": True, "fare": 0.0, "calculation_basis": "", "not_found": True, "error_message": ""}
  - エラー: {"success": False, "fare": 0.0, "calculation_basis": "", "not_found": False, "error_message": message}
```

#### 3.3.2 モジュールレベルキャッシュ

**用途**: データファイルの繰り返し読み込みを防止し、パフォーマンスを向上させる

**実装**:
```python
_train_fares: list[dict] = []
_train_fares_loaded: bool = False
_fixed_fares: list[dict] = []
_fixed_fares_loaded: bool = False
```

- 未ロード判定は `_train_fares_loaded` / `_fixed_fares_loaded` フラグで管理する（空リストによる判定禁止: R9.12.4）
- 初回呼び出し時のみファイルを読み込み、以降はキャッシュを使用する

#### 3.3.3 データの検証方法

**Pydanticモデル**: `TransportCalculatorInput`

**検証内容**:
- `departure`: min_length=1、validate_station_name で接尾語除去
- `destination`: min_length=1、validate_station_name で接尾語除去
- `transport_type`: validate_transport_type で正規化後、Literal["電車", "バス", "タクシー", "飛行機"] でチェック
- `travel_date`: validate_date_format で YYYY-MM-DD 形式を検証

#### 3.4.1 エラー処理フロー

```
エラー発生
  ↓
[ValidationError（EX-01）]
```

## 6. 補足情報

### 6.1 実装上の注意点

1. **モジュールレベルキャッシュの未ロード判定**
   - `_train_fares_loaded` / `_fixed_fares_loaded` フラグで管理する
   - 空リスト（`not _train_fares`）による判定は禁止（R9.12.4）

2. **デコレータ**
   - `@tool`（invocation_stateへのアクセス不要）を使用する
   - `@tool(context=True)` は使用しない

3. **エラー返却キー名**
   - 戻り値辞書のキー名は `TransportCalculatorOutput` のフィールド名と一致させること
   - `error_message` キーを使用する（`message` は不可）

4. **往復・複数区間の合算**
   - 本ツールは1区間の運賃のみを返す
   - 往復・複数区間の合算はエージェント（AG-002）が担当する

5. **経路未登録時の例外クラス**
   - 経路テーブルに経路が存在しない場合は `ValueError` を使用する（`KeyError` は使用しない）

6. **ファイル読み込み実装方針**
   - ファイル存在チェックは `os.path.exists()` による事前チェックで実装する
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する
   - FileNotFoundError のログレベルは WARNING とする

7. **データ読み込み系メソッドのエラー時戻り値**
   - `_load_train_fares()` / `_load_fixed_fares()` のエラー時戻り値は `(False, エラーメッセージ文字列)` の `tuple[bool, str]` 形式とする
   - 例外を raise せず、必ず戻り値としてエラー情報を返す

8. **計算系メソッドのエラー時戻り値**
   - `calculate_transport_fare()` のエラー時戻り値は `{"success": False, "error_message": エラーメッセージ文字列, ...}` の dict 形式とする
   - 例外を raise せず、必ず戻り値としてエラー情報を返す

9. **ErrorHandlerメソッドの使い分け**
   - データ読み込みエラー: `ErrorHandler.handle_fare_data_error(e)` を使用する
   - 計算エラー: `ErrorHandler.handle_calculation_error(e)` を使用する

### 6.2 パフォーマンス考慮事項

1. **キャッシュによるファイルI/O削減**
   - データファイルは初回のみ読み込み、以降はモジュールレベルキャッシュを使用する
   - 2回目以降の呼び出しでファイルI/Oが発生しない

### 6.3 セキュリティ考慮事項

1. **ファイルパスの固定**
   - データファイルパスはモジュール定数（`_TRAIN_FARES_PATH`・`_FIXED_FARES_PATH`）で管理する
   - LLMからのパラメータとしてファイルパスを受け取らない

## 8. テスト観点

### 8.1 機能テスト
- 登録済み経路（電車）で正しい交通費・計算根拠が返ること
- バスの固定運賃が正しく返ること
- タクシーの固定運賃が正しく返ること
- 飛行機の固定運賃が正しく返ること
- 経路未登録時に `not_found=True` で返ること
- 2回目以降の呼び出しでキャッシュが使用されること（ファイルI/Oが発生しないこと）

### 8.2 異常系テスト
- 対応外交通手段（「自転車」等）で `success=False` かつ `error_message` が返ること（EX-01）
- 出発地が空文字で `success=False` かつ `error_message` が返ること（EX-01）
- 目的地が空文字で `success=False` かつ `error_message` が返ること（EX-01）
- 移動日が不正形式（「2026/05/10」等）で `success=False` かつ `error_message` が返ること（EX-01）
- `template/train_routes.json` が存在しない場合に `_load_train_fares()` が `(False, エラーメッセージ)` を返すこと（EX-06）

- データファイル不存在時に `calculate_transport_fare()` が `success=False` かつ `error_message` を返すこと（EX-06）

### 8.3 性能テスト
- 2回目以降の呼び出しがキャッシュにより初回より高速であること

### 8.4 境界値テスト
- 運賃が0円のレコードで正しく返ること
- 移動日が有効な日付の境界値（例: 2026-12-31）で正しく処理されること
