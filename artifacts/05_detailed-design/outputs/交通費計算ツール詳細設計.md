## 1. 概要

### 1.1 コンポーネントの目的

運賃データ（電車経路テーブル・固定運賃）を検索し、出発地・目的地・交通手段・移動日に基づいて交通費を計算する。エージェント（AG-003）が対話で収集した移動情報に対して正確な運賃を提供し、交通費精算申請書の明細に記載する費用を自動算出する。

### 1.2 主要な責務

1. **運賃データ読み込み**: `./template/train_routes.json` および `./template/fixed_fares.json` からマスタデータを読み込み、モジュールレベルキャッシュに保持する
2. **経路検索**: 電車の場合、出発駅・到着駅の組み合わせで経路テーブルを検索する
3. **固定運賃取得**: バス・タクシー・飛行機の場合、交通手段に対応する固定運賃を取得する
4. **入力バリデーション**: `TransportCalculatorInput` Pydantic モデルで入力パラメータを検証・正規化する
5. **計算結果返却**: 計算成否・金額・エラーメッセージを辞書形式で返却する

## 3. calculate_transport_fare ツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_fare`

#### 説明（docstring）
```
出発地・目的地・交通手段・移動日から交通費を計算する。

電車の場合は運賃データ（train_routes.json）から経路を検索し運賃を返す。
バス・タクシー・飛行機の場合は固定運賃データ（fixed_fares.json）から運賃を返す。
該当経路が存在しない場合はエラーメッセージを返す。

申請者名・申請日は invocation_state から取得する。

Args:
    departure (str): 出発地（正規化済み駅名。「渋谷駅」ではなく「渋谷」の形式）
    destination (str): 目的地（正規化済み駅名。「渋谷駅」ではなく「渋谷」の形式）
    transport_type (str): 交通手段。以下のいずれかを指定:
        - "電車"（または "train"）
        - "バス"（または "bus"）
        - "タクシー"（または "taxi"）
        - "飛行機"（または "airplane", "plane"）
    travel_date (str): 移動日（YYYY-MM-DD 形式。例: "2026-05-17"）
    tool_context: ツールコンテキスト（invocation_stateを含む）

Returns:
    dict: 以下のキーを持つ辞書:
        - success (bool): 計算成否
        - fare (int | None): 交通費金額（円）。成功時のみ設定
        - message (str | None): エラーメッセージ。失敗時のみ設定
```

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義**:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| `departure` | `str` | 出発地（正規化済み駅名） | なし（必須） |
| `destination` | `str` | 目的地（正規化済み駅名） | なし（必須） |
| `transport_type` | `str` | 交通手段（電車/バス/タクシー/飛行機） | なし（必須） |
| `travel_date` | `str` | 移動日（YYYY-MM-DD 形式） | なし（必須） |
| `tool_context` | `ToolContext` | ツールコンテキスト（invocation_stateを含む） | なし（必須） |

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| `departure` | 空文字禁止（min_length=1）、末尾「駅」を除去して正規化 | "出発地のエラー: 駅名を入力してください。" | `"渋谷駅"` → `"渋谷"` |
| `destination` | 空文字禁止（min_length=1）、末尾「駅」を除去して正規化 | "目的地のエラー: 駅名を入力してください。" | `"新宿駅"` → `"新宿"` |
| `transport_type` | `Literal["電車","バス","タクシー","飛行機"]`、英語表記を正規化 | "交通手段のエラー: 電車・バス・タクシー・飛行機のいずれかを指定してください。" | `"train"` → `"電車"` |
| `travel_date` | YYYY-MM-DD 形式、python-dateutil でパース後に正規化 | "移動日のエラー: 日付の形式が正しくありません（YYYY-MM-DD 形式で入力してください）。" | `"2026/5/17"` → `"2026-05-17"` |

##### 正規化処理

| 入力値 | 正規化後 | 処理内容 |
|--------|---------|---------|
| `"渋谷駅"` | `"渋谷"` | `str.removesuffix("駅").strip()` |
| `"train"` | `"電車"` | マッピング辞書による変換 |
| `"bus"` | `"バス"` | マッピング辞書による変換 |
| `"taxi"` | `"タクシー"` | マッピング辞書による変換 |
| `"airplane"` / `"plane"` | `"飛行機"` | マッピング辞書による変換 |
| `"2026/5/17"` | `"2026-05-17"` | dateutil.parser.parse → strftime |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
[ステップ0] invocation_state から申請者名・申請日を取得
  - applicant_name = tool_context.invocation_state.get("applicant_name")
  - application_date = tool_context.invocation_state.get("application_date")
  ↓
[ステップ1] 入力バリデーション
  - TransportCalculatorInput(**kwargs) でバリデーション・正規化
  - ValidationError → {"success": False, "message": msg} を返して終了
  ↓
[ステップ2] 運賃データ読み込み（キャッシュ確認）
  - _train_routes_loaded が False → _load_train_routes() を呼び出す
  - _fixed_fares_loaded が False → _load_fixed_fares() を呼び出す
  - 読み込み失敗 → {"success": False, "message": msg} を返して終了
  ↓
[ステップ3] 交通手段による分岐
  ↓
transport_type == "電車"
  → [ステップ4a] 経路検索
       - _train_routes から departure + destination が一致するレコードを検索
       - 見つかった → fare を取得
       - 見つからない → ValueError を発生させ、{"success": False, "message": msg} を返して終了
transport_type in ["バス", "タクシー", "飛行機"]
  → [ステップ4b] 固定運賃取得
       - _fixed_fares から transport_type をキーとして運賃を取得
       - キーが存在する → fare を取得
       - キーが存在しない → ValueError を発生させ、{"success": False, "message": msg} を返して終了
  ↓
[ステップ5] 結果返却
  - {"success": True, "fare": fare, "message": None} を返す
```

#### 3.3.2 キャッシュ機構

**モジュールレベルキャッシュ**:

**用途**: 運賃データ JSON ファイルの繰り返し読み込みを防止し、パフォーマンスを向上させる

**実装**:
```python
# モジュールレベルキャッシュ変数
_train_routes: list[dict] = []
_train_routes_loaded: bool = False
_fixed_fares: dict[str, int] = {}
_fixed_fares_loaded: bool = False
```

- `_xxx_loaded` フラグが `False` の場合のみファイルを読み込む
- 読み込み成功後に `_xxx_loaded = True` に更新する
- 空リスト（`not _train_routes`）による判定は禁止（R9.12.4 参照）

#### 3.3.3 データの検証方法

**Pydanticモデル**: `TransportCalculatorInput`

**検証内容**:
- `departure`: 空文字禁止、末尾「駅」を除去して正規化
- `destination`: 空文字禁止、末尾「駅」を除去して正規化
- `transport_type`: 許容値チェック、英語表記を日本語に正規化
- `travel_date`: YYYY-MM-DD 形式チェック、python-dateutil でパース後に正規化

**Pydanticモデル**: `TrainRouteRecord`（マスタデータ読み込み時）

**検証内容**:
- `departure`: 空文字禁止
- `destination`: 空文字禁止
- `fare`: 正の整数（gt=0）

**固定運賃データ**: 辞書形式（`dict[str, int]`）のため、Pydanticモデルによる個別レコードバリデーションは不要。キーの存在確認で運賃を取得する。

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
エラー発生
  ↓
[エラー種別の判定]
  ↓
ValidationError（入力バリデーション失敗）
  → ErrorHandler.handle_validation_error(e) でメッセージ生成
  → _logger.error("バリデーションエラー: %s", e, exc_info=True)
  → {"success": False, "message": msg} を返す

ValueError（経路不存在・固定運賃不存在）
  → エラーメッセージを直接生成
  → _logger.warning("経路が見つかりません: ...")
  → {"success": False, "message": msg} を返す

Exception（データ読み込み失敗・その他）
  → ErrorHandler.handle_unexpected_error(e) でメッセージ生成
  → _logger.error("交通費計算エラー: %s", e, exc_info=True)
  → {"success": False, "message": msg} を返す
```

#### 3.4.2 エラーメッセージの詳細

| エラー種別 | 発生条件 | 対応 | ユーザー向けメッセージ |
|-----------|---------|------|---------------------|
| バリデーションエラー（EX-01） | 入力パラメータが不正 | ErrorHandler.handle_validation_error() | "入力内容にエラーがあります。{フィールド名}のエラー: {詳細}" |
| 経路不存在（EX-06） | 電車の経路が train_routes.json に存在しない | ValueError を発生させ、エラーメッセージ直接生成 | "指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。" |
| 固定運賃不存在（EX-06） | 交通手段が fixed_fares.json に存在しない | ValueError を発生させ、エラーメッセージ直接生成 | "指定された交通手段の運賃データが見つかりませんでした。交通費を手動で入力してください。" |
| データ読み込み失敗（EX-06） | JSON ファイルが存在しない・形式不正 | ErrorHandler.handle_unexpected_error()、ログレベル WARNING（FileNotFoundError時） | "運賃データの読み込みに失敗しました。担当者にお問い合わせください。" |
| 想定外エラー（EX-08） | 予期しない Python 例外 | ErrorHandler.handle_unexpected_error() | "予期しないエラーが発生しました。担当者にお問い合わせください。" |

## 6. 補足情報

### 6.1 実装上の注意点

1. **キャッシュフラグの使用**
   - `not _train_routes` による空リスト判定は禁止。`_train_routes_loaded` フラグで管理する（R9.12.4）

2. **デコレータ**
   - `invocation_state` から申請者名・申請日を取得するため `@tool(context=True)` を使用する
   - 関数の最後の引数に `tool_context: ToolContext` を追加する

3. **invocation_state からの取得項目**
   - `applicant_name`: 申請者名
   - `application_date`: 申請日

4. **エラー返却形式**
   - 失敗時: `{"success": False, "message": エラーメッセージ文字列}`（fareキーは含めない）
   - 成功時: `{"success": True, "fare": 金額, "message": None}`
   - 例外を再送出せず、必ず辞書形式でエラーを返す

5. **経路不存在時の例外クラス**
   - 経路テーブルに経路が存在しない場合は `ValueError` を使用する（`KeyError` は使用しない）

6. **ファイル読み込み実装方針**
   - ファイル存在チェックは `os.path.exists()` による事前チェックで実装する
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する
   - FileNotFoundError 相当（ファイル不存在）のログレベルは WARNING とする

### 6.2 パフォーマンス考慮事項

1. **モジュールレベルキャッシュ**
   - 運賃データは初回呼び出し時のみ読み込み、以降はキャッシュを使用する
   - プロセス再起動時にキャッシュはリセットされる

### 6.3 セキュリティ考慮事項

1. **ファイルパスの固定**
   - 運賃データファイルパスは定数（`_TRAIN_ROUTES_PATH`, `_FIXED_FARES_PATH`）で管理し、外部から変更不可とする

## 8. テスト観点

### 8.1 機能テスト
- 電車（経路テーブル検索）で正しい運賃が返ること
- バス・タクシー・飛行機（固定運賃）で正しい運賃が返ること
- 2回目の呼び出しでキャッシュが使用されること（ファイル読み込みが1回のみ）
- 移動日が正しく受け取られること（travel_date フィールドが辞書に含まれること）

### 8.2 異常系テスト
- 存在しない電車経路を指定した場合に `success=False` が返ること（EX-06）
- `train_routes.json` が存在しない場合に `success=False` が返ること（EX-06）
- `fixed_fares.json` が存在しない場合に `success=False` が返ること（EX-06）
- 出発地に空文字を渡した場合に `success=False` が返ること（EX-01）
- 交通手段に不正値（"自転車"）を渡した場合に `success=False` が返ること（EX-01）
- 移動日に不正な文字列を渡した場合に `success=False` が返ること（EX-01）

### 8.3 性能テスト
- 運賃データ読み込みが初回のみ実行されること（キャッシュ動作確認）
- 100件の経路データに対して検索が1秒以内に完了すること

### 8.4 境界値テスト
- 出発地と目的地が同一の場合の動作（経路不存在として扱う）
- 運賃が0円のレコードが存在する場合の動作（TrainRouteRecord の gt=0 制約でバリデーションエラー）
- 交通手段の英語表記（"train", "bus", "taxi", "airplane", "plane"）が正規化されること
- 駅名末尾の「駅」が除去されること（"渋谷駅" → "渋谷"）
