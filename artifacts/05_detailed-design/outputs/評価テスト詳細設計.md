## 3. テストケース定義

### 3.1 ツール利用の適切性（eval_tool_selection.py）

**評価対象**: AG-001 が申請種別に応じて `transportation_expense_agent` / `general_expense_agent` を正しく選択すること

**件数**: 3件

| ケース識別名 | 入力 | タスク説明 | 期待ツール |
|-----------|------|---------|---------|
| `transport_train_tokyo_shinjuku` | `"申請者は田中太郎です。2026-05-18に東京から新宿まで電車で移動しました。業務目的は顧客訪問です。交通費を申請したいです。"` | 電車移動の交通費精算申請として transportation_expense_agent が選択されること | `transportation_expense_agent` |
| `transport_taxi_business_trip` | `"申請者は鈴木花子です。出張でタクシーを使いました。2026-05-15に品川から渋谷まで移動しました。業務目的は取引先訪問です。交通費の精算をお願いします。"` | タクシー移動の交通費精算申請として transportation_expense_agent が選択されること | `transportation_expense_agent` |
| `expense_office_supplies` | `"申請者は山田次郎です。2026-05-10に東京文具店でボールペンとノートを3,000円で購入しました。業務目的は業務用消耗品の補充です。経費申請をしたいです。"` | 事務用品購入の経費精算申請として general_expense_agent が選択されること | `general_expense_agent` |

**申請種別の判断基準（JD-01/JD-02）**:
- `transportation_expense_agent`: 交通費・移動・電車・バス・タクシー・飛行機・出張・通勤 等
- `general_expense_agent`: 経費・購入・備品・宿泊・資格・文具・事務用品 等

## 4. 評価器設計

### 4.1 ToolSelectionAccuracyEvaluator（MET-001）

| 項目 | 内容 |
|------|------|
| 評価器種別 | LLM-as-Judge（二値） |
| 判定モデル | `get_model()`（評価対象と同一モデル） |
| 判定観点 | 申請内容に対して正しいエージェントツール（transportation_expense_agent / general_expense_agent）が選択されたか |
| スコア尺度 | 1.0（正しい選択）/ 0.0（誤った選択） |
| 判定根拠の出力 | あり |

**注意**: `ToolSelectionAccuracyEvaluator` は `strands_evals` のマッパーがツール名のみを抽出し description を伝播しない。ツール名（`transportation_expense_agent` / `general_expense_agent`）は役割が名前だけで推測できるよう命名されている。

---

### 4.2 GoalSuccessRateEvaluator（MET-002）

| 項目 | 内容 |
|------|------|
| 評価器種別 | LLM-as-Judge（二値） |
| 判定モデル | `get_model()`（評価対象と同一モデル） |
| 判定観点 | セッション全体の trajectory を参照して申請書生成ゴールが達成されたか |
| スコア尺度 | 1.0（ゴール達成）/ 0.0（ゴール未達成） |
| 判定根拠の出力 | あり |

## 5. タスク実行設計

### 5.1 シングルターン実行フロー（eval_tool_selection.py）

```
run_eval_task(case) 開始
  ↓
memory_exporter.clear()  # 前ケースのスパン混入防止
  ↓
エージェント生成（create_orchestrator_agent()）
  - trace_attributes={"session.id": case.session_id, "gen_ai.conversation.id": case.session_id}
  - HumanApprovalHook は patch_human_approval_hook() で自動承認に差し替え済み
  ↓
agent(case.input, invocation_state=create_invocation_state()) を呼び出す
  ↓
spans = list(memory_exporter.get_finished_spans())
  ↓
session = StrandsInMemorySessionMapper().map_to_session(spans, session_id=case.session_id)
  ↓
return {"output": str(response), "trajectory": session}
```

### 5.2 マルチターン実行フロー（eval_goal_success_rate.py）

```
run_eval_task(case) 開始
  ↓
memory_exporter.clear()  # 前ケースのスパン混入防止（ケース開始時に1回のみ）
  ↓
エージェント生成（create_orchestrator_agent()）
  - trace_attributes={"session.id": case.session_id, "gen_ai.conversation.id": case.session_id}
  ↓
user_sim = ActorSimulator.from_case_for_user_simulator(case=case, max_turns=10)
  ↓
user_message = case.input
  ↓
[会話ループ]
  while user_sim.has_next():
    agent_response = agent(user_message, invocation_state=create_invocation_state())
    # memory_exporter.clear() は呼ばない（全ターンのスパンを蓄積）
    user_result = user_sim.act(str(agent_response))
    user_message = str(user_result.structured_output.message)
  ↓
spans = list(memory_exporter.get_finished_spans())
  ↓
session = StrandsInMemorySessionMapper().map_to_session(spans, session_id=case.session_id)
  ↓
return {"output": str(agent_response), "trajectory": session}
```

**重要**: マルチターン評価では会話ループ内で `memory_exporter.clear()` を呼ばない。全ターンのスパンを蓄積してからまとめて Session を構築する。

## 6. 実験実行設計

### 6.1 eval_tool_selection.py の実験構成

```python
experiment = Experiment(
    cases=EVAL_CASES,
    evaluators=[ToolSelectionAccuracyEvaluator(model=get_model())]
)
reports = experiment.run_evaluations(run_eval_task)
```

### 6.2 eval_goal_success_rate.py の実験構成

```python
experiment = Experiment(
    cases=EVAL_CASES,
    evaluators=[GoalSuccessRateEvaluator(model=get_model())]
)
reports = experiment.run_evaluations(run_eval_task)
```

### 6.3 実行順序

1. 環境初期化（UTF-8 設定・sys.path 設定・load_dotenv）
2. helpers import → patch_human_approval_hook() 呼び出し（エージェントモジュールより前）
3. ログ設定・warnings 抑制・SDK ログ抑制
4. 評価器初期化
5. 実験構成（Experiment）
6. 評価実行（experiment.run_evaluations）
7. レポート保存（`ensure_ascii=False, indent=2` で JSON 保存）
8. レポート表示（report.run_display()）

## 7. 非決定性への対処

| 項目 | 内容 |
|------|------|
| 実行回数 | 各テストケースを1回実行する |
| スコア集約方式 | 1回実行のため集約なし |
| 合否判定 | 各テストケースのスコアが閾値（MET-001: 0.9以上、MET-002: 0.8以上）を満たすかで判定 |
