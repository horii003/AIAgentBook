## 1. 概要

### 1.1 テスト設計の目的

詳細設計書に記載された各コンポーネントのテスト観点を集約・体系化し、評価指標一覧（EV-001〜EV-012）の全指標を満たすテストケースを設計する。単体・統合・E2Eの3層構造でテストを整理し、pytestによる自動化可能なテストの実装指針を提供する。

### 1.2 テスト対象範囲

| テスト層 | 対象コンポーネント | 実施方式 |
|---------|--------------|--------|
| 単体テスト | ツール（TL-001/TL-002）・データモデル・ハンドラー・セッションマネージャ・ナレッジ | pytest（自動） |
| 統合テスト | エージェント間連携・ツール呼び出しフロー・セッション永続化 | pytest（自動） |
| E2Eテスト | ユースケース（UC-01〜UC-03）に基づくシナリオ | pytest/手動 |

### 1.3 テスト環境

| 項目 | 値 |
|------|-----|
| Pythonバージョン | `3.10以上` |
| モデルID | `jp.anthropic.claude-sonnet-4-5-20250929-v1:0` |
| テストフレームワーク | `pytest >= 7.4.0` |
| カバレッジ計測 | `pytest-cov >= 4.1.0` |

## 2. テスト全体構成

### 2.1 テスト層とカバレッジ方針

| テスト層 | カバレッジ目標 | 重点観点 |
|---------|------------|--------|
| 単体テスト | 80%以上 | バリデーション・エラーハンドリング・境界値 |
| 統合テスト | 主要フロー網羅 | エージェント間連携・ツール呼び出し・セッション |
| E2Eテスト | 全ユースケース | 業務シナリオ・評価指標検証 |

### 2.2 テストデータ・モック方針

| コンポーネント | モック方針 | 理由 |
|------------|---------|------|
| Amazon Bedrock API | モック（単体・統合テスト） | コスト・速度・再現性 |
| `template/train_routes.json` | テスト用一時ファイル | ファイルI/O分離 |
| `template/fixed_fares.json` | テスト用一時ファイル | ファイルI/O分離 |
| Excelテンプレート | テスト用一時ファイル | ファイルI/O分離 |
| `output/` ディレクトリ | テスト用一時ディレクトリ | ファイルI/O分離 |
| `storage/sessions/` | テスト用一時ディレクトリ | セッションファイル分離 |
| FileSessionManager | モック（単体テスト） | セッション管理分離 |
