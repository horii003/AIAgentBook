## 1. 概要

### 1.1 コンポーネントの目的

領収書画像から経費情報を自動抽出し、不足情報を一括確認で対話収集した後、申請期限・上長承認チェックを実施し、ユーザー承認後に経費精算申請書（Excel）を自動生成する。

### 1.2 主要な責務

1. **領収書情報自動抽出**: LLM の画像解析機能で購入日・店舗名・品目・金額を品目ごとに抽出する
2. **経費区分自動判断**: 抽出した品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断する（BRL-14）
3. **不足情報の一括確認**: 業務目的等の不足情報をまとめて質問し、一括収集する（BRL-04, BRL-16）
4. **申請期限チェック**: 経費発生日から3ヶ月以内かを LLM が判断する（BRL-09）
5. **上長承認チェック**: 経費合計が5,000円超の場合に通知する（BRL-11）
6. **申請書自動生成**: HumanApprovalHook 承認後に generate_expense_report を呼び出す

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（経費発生日から3ヶ月以内）を LLM に判断させるため、現在日付と3ヶ月前の日付をプロンプトに埋め込む必要がある |

##### 動的に埋め込む情報
- 申請者名（applicant_name）
- 申請日（application_date: YYYY-MM-DD）
- 申請期限の基準日（application_date から3ヶ月前: YYYY-MM-DD）

##### プロンプト全文

```
あなたは経費精算申請エージェントです。申請者「{applicant_name}」の経費精算申請書（Excel）の作成を支援します。
申請日: {application_date}
申請期限の基準日（この日付以降の経費発生日のみ申請可能）: {deadline_date}

【役割】
1. 領収書画像からの経費情報自動抽出
   - 提供された領収書画像から購入日、店舗名、品目、金額を品目ごとに抽出する（BRL-13）
   - 複数の品目がある場合は全て抽出する

2. 経費区分の自動判断（BRL-14）
   - 事務用品費: 文房具・コピー用紙・印刷物等の事務用品
   - 宿泊費: 出張時のホテル・旅館等の宿泊費用
   - 資格精算費: 業務に関連する資格取得・試験費用
   - その他経費: 上記に該当しない業務上の経費

3. 不足情報の一括確認（BRL-04, BRL-16）
   - 申請に必要な不足情報（業務目的等）をまとめて質問する
   - 計算可能な情報（合計金額等）は確認しない

4. 申請期限チェック（BRL-09）
   - 各経費の購入日が申請期限の基準日（{deadline_date}）以降であることを確認する
   - 基準日より前の購入日がある場合は申請期限超過エラーを通知し業務終了する

5. 上長承認チェック（BRL-11）
   - 経費合計が5,000円を超える場合は上長の事前承認が必要な旨を通知する
   - 通知後も申請書作成は続行する

6. 申請書ドラフト提示
   - 収集した全情報をテキストとして整理・提示する（この段階ではツール呼び出しを行わない）
   - 申請者名・申請日・経費明細（購入日・店舗名・品目・経費区分・金額・業務目的）・合計金額を提示する

7. 申請書生成
   - ユーザーが承認（OK）を選択した後、generate_expense_report ツールを呼び出す
   - 承認前にツールを呼び出してはならない

【禁止事項】
- 申請書生成前にユーザーの承認を得ずに generate_expense_report を呼び出してはならない
- 計算可能な情報（合計金額等）をユーザーに確認してはならない
- システム系エラーが発生した場合は、エラー内容を要約して呼び出し元に返却する
```

##### 妥当性チェックルールの組み込み方式

| 項目 | 内容 |
|------|------|
| 組み込み方式 | エージェント初期化時に外部ファイルを参照しプロンプトに組み込む |
| ルール参照先 | `agent_knowledge/receipt_policies.py` |
| 組み込みタイミング | `get_expense_system_prompt()` 呼び出し時に `get_expense_policies()` の戻り値をプロンプトに埋め込む |

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的日付埋め込み | 申請期限チェックを LLM に判断させるため、基準日を動的に埋め込む |
| 一括確認パターン | 不足情報を1つずつ聞かず、まとめて質問することで対話回数を削減する |
| ドラフト提示の分離 | 申請書生成前にテキストで内容を提示し、HumanApprovalHook の承認後にのみツールを実行する |
| 外部ルール参照 | 妥当性チェックルールを `agent_knowledge/receipt_policies.py` から取得し、プロンプトに動的に組み込む |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-09 | 購入日が申請期限基準日より前 | 申請期限超過エラーを通知し業務終了 |
| BRL-11 | 経費合計 > 5,000円 | 上長承認が必要な旨を通知し申請書作成を続行 |
| BRL-13 | 領収書画像が提供された | LLM の画像解析で購入日・店舗名・品目・金額を抽出 |
| BRL-14 | 品目が抽出された | 経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断 |
| BRL-16 | 業務目的が未入力 | 業務目的を必須項目として収集する |

#### 2.3.3 処理フロー

```
開始（AG-001 から expense_agent ツール経由で呼び出される）
  ↓
[ステップ1] invocation_state から情報取得
  - applicant_name = tool_context.invocation_state.get("applicant_name")
  - application_date = tool_context.invocation_state.get("application_date")
  - session_id = tool_context.invocation_state.get("session_id")
  - いずれかが None → エラーメッセージを返して終了
  ↓
[ステップ2] エージェント構築
  - deadline_date = calculate_deadline(application_date, deadline_months=3)
  - system_prompt = get_expense_system_prompt(applicant_name, application_date, deadline_date)
  - agent = create_specialist_agent(session_id, system_prompt, tools=[generate_expense_report], ...)
  ↓
[ステップ3] エージェント呼び出し
  - child_invocation_state = {"applicant_name": applicant_name, "application_date": application_date}
  - response = agent(query, invocation_state=child_invocation_state)
  ↓
[ステップ4] 応答返却
  - str(response) を返す
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools 設計

**ツール名**: `expense_agent`

**ツール説明**:
```
経費精算申請書の作成を支援する専門エージェント。
領収書画像から経費情報を抽出し、申請期限・上長承認チェックを実施した後、
ユーザー承認を得て経費精算申請書（Excel）を生成する。
```

**ToolContext の使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から applicant_name・application_date・session_id を取得
- session_id は `create_specialist_agent()` のファクトリ関数で消費し、子エージェントへは渡さない

#### 2.5.2 invocation_state の受け渡し

```python
# AG-001 から受け取る
state = tool_context.invocation_state
session_id = state.get("session_id")
applicant_name = state.get("applicant_name")
application_date = state.get("application_date")

# 子エージェントには session_id を除いて渡す
child_invocation_state = {
    "applicant_name": applicant_name,
    "application_date": application_date
}
response = agent(query, invocation_state=child_invocation_state)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 expense_agent 関数（Agent as Tools ツール関数）

##### 説明
経費精算申請エージェントを構築して呼び出す Agent as Tools ツール関数。

##### 引数
- `query` (str): AG-001 からの指示テキスト（必須）
- `tool_context` (ToolContext): ツールコンテキスト（フレームワーク注入）

##### 戻り値
- `str`: エージェントからの応答テキスト

##### 処理内容
1. `invoke_specialist_agent()` を呼び出す（共通ラッパー）
2. invocation_state から session_id・applicant_name・application_date を取得
3. `_build_expense_agent()` でエージェントを構築
4. 子エージェントに applicant_name・application_date を渡して呼び出す

#### 3.1.2 _build_expense_agent 関数

##### 説明
経費精算申請エージェントのインスタンスを構築するファクトリ関数。

##### 引数
- `session_id` (str): セッションID（必須）
- `applicant_name` (str): 申請者名（必須）
- `application_date` (str): 申請日（必須）
- `deadline` (str): 申請期限（必須）

##### 戻り値
- `Agent`: 構築済みエージェントインスタンス

##### 処理内容
1. `get_expense_system_prompt(applicant_name, application_date, deadline)` でプロンプト生成
2. `create_specialist_agent(session_id, system_prompt, tools=[generate_expense_report], ...)` でエージェント構築

### 3.2 エラーハンドリング

| エラー種別 | 条件 | ログレベル | 戻り値 | メッセージ |
|-----------|------|-----------|--------|----------|
| LoopLimitError | ReActループ上限到達 | WARNING | str | ErrorHandler.handle_loop_limit_error() |
| ContextWindowOverflowException | コンテキストウィンドウ超過 | WARNING | str | ErrorHandler.handle_context_window_error() |
| MaxTokensReachedException | 最大トークン数到達 | WARNING | str | ErrorHandler.handle_max_tokens_error() |

- ログにはクエリの先頭50文字を付加情報として含める

## 5. 補足情報

### 5.1 実装上の注意点

1. **ドラフト提示とツール実行の分離**
   - 申請書生成前にテキストで内容を提示し、HumanApprovalHook の承認後にのみ generate_expense_report を実行する

2. **ファイル配置**
   - `agents/expense_agent.py` に配置する

3. **共通ファクトリ・ラッパーの使用**
   - `agents/base_agent.py` の `create_specialist_agent()` と `invoke_specialist_agent()` を使用する

### 5.2 パフォーマンス考慮事項

1. **SlidingWindow(15)**
   - 領収書情報抽出に集中した短い対話のため、15件で十分

### 5.3 セキュリティ考慮事項

1. **invocation_state の機密情報**
   - 申請者名・申請日は LLM のプロンプトに含まれず、invocation_state 経由でのみ渡す

## 7. テスト観点

### 7.1 機能テスト
- 領収書画像から購入日・店舗名・品目・金額が正しく抽出されること
- 経費区分が品目に基づいて正しく判断されること
- 不足情報（業務目的）が一括確認で収集されること
- 申請期限内の経費で申請書が正しく生成されること
- HumanApprovalHook 承認（OK）後に generate_expense_report が呼び出されること
- 承認前に generate_expense_report が呼び出されないこと

### 7.2 異常系テスト
- 申請期限超過時に適切なエラーメッセージが表示されること
- 経費合計5,000円超時に上長承認通知が表示されること
- テンプレートファイル不存在時にエラーメッセージが返されること
- LoopLimitError 発生時に WARNING ログが出力され、適切なメッセージ（str）が返されること
- ContextWindowOverflowException 発生時に WARNING ログが出力され、適切なメッセージ（str）が返されること
