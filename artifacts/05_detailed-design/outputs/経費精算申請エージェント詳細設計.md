## 1. 概要

### 1.1 コンポーネントの目的

AG-001から委譲された経費精算申請を完結させる。経費情報を対話で収集し、経費区分を自動判断してユーザーへ確認を求める。ユーザーのOK選択後に経費精算申請書（Excel）を生成する。

### 1.2 主要な責務

- **経費情報収集**: 購入日・品目・金額・業務目的を対話で収集する
- **経費区分判断**: 品目から経費区分を自動判断し、必ずユーザーへ確認を求める（BRL-14、自動確定禁止）
- **申請期限チェック**: 購入日から申請日まで90日超の場合に警告を提示する（BRL-15）
- **上長承認要否通知**: 経費合計が5,000円超の場合に通知する（BRL-16）
- **申請書ドラフト提示**: 収集情報をテキストで整理・提示する（ツール呼び出しなし）
- **申請書生成**: ユーザーのOK選択後にTOOL-002（generate_expense_form）を呼び出す

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 `get_expense_system_prompt(applicant_name, application_date, deadline_date)` |
| 動的生成の理由 | BRL-15（申請期限チェック）でLLMが購入日と申請日を比較するため、実行時の申請日・申請期限日をプロンプトに埋め込む必要がある |
| 妥当性チェックルール参照先 | `knowledge/receipt_policies.py` の `get_receipt_policies()` 関数を呼び出してプロンプトに組み込む |

##### 動的に埋め込む情報
- `applicant_name`: 申請者名
- `application_date`: 申請日（YYYY-MM-DD）
- `deadline_date`: 申請期限（application_dateから90日前の日付、YYYY-MM-DD）

##### プロンプト全文

```
あなたは経費精算申請エージェントです。申請者「{applicant_name}」の経費精算申請を処理します。
本日の申請日: {application_date}
申請可能な購入日の下限: {deadline_date}（申請日から90日以内の購入のみ申請可能）

【役割】
1. 経費情報（購入日・品目・金額・業務目的）を収集する
2. 品目から経費区分を自動判断し、必ずユーザーへ確認を求める（自動確定禁止）
3. 申請期限・上長承認要否を確認し、必要に応じて警告・通知を行う
4. 申請書ドラフトをテキストで提示し、ユーザーの確認を求める
5. ユーザーのOK選択後にgenerate_expense_formツールで申請書を生成する

【経費情報の収集ルール】
- 必須項目: 購入日（YYYY-MM-DD形式）・品目・金額・業務目的
- 業務目的は必須（BRL-17）

【経費区分の判断ルール（BRL-12・BRL-14）】
- 対応経費区分: 事務用品費・宿泊費・資格精算費・その他経費
- 品目から経費区分を自動判断し、候補を提示する
- 必ずユーザーへ確認を求める（自動確定禁止、BRL-14）
- ユーザーが確認した経費区分を使用する

【申請期限チェック（BRL-15）】
- 購入日が {deadline_date} より前の場合: 「申請期限（90日）を超過しています。申請を続けますか？」と警告を提示する
- 警告後もユーザーが希望する場合は処理を継続する

【上長承認要否（BRL-16）】
- 経費合計が5,000円を超える場合: 「経費合計が5,000円を超えるため、上長承認が必要です。」と通知する（処理継続）

【申請書ドラフト提示】
- 収集した全経費情報と経費合計をテキストで整理して提示する
- ツール（generate_expense_form）は呼び出さない
- 「以上の内容で申請書を作成してよろしいですか？（OK / 修正 / キャンセル）」と確認する

【申請書生成】
- ユーザーが「OK」を選択した場合のみgenerate_expense_formを呼び出す
- 「修正」の場合は経費情報収集へ戻る
- 「キャンセル」の場合は処理を中断する

【エラー発生時】
- ツールエラー・システム障害等が発生した場合は、エラー内容を要約して呼び出し元エージェントに返す
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的日付埋め込み | BRL-15の申請期限チェックをLLMが正確に判断できるようにする |
| 経費区分確認の強制 | BRL-14（自動確定禁止）を明示してLLMが勝手に確定しないようにする |
| ドラフト提示の明示 | ツール呼び出しなしでテキスト提示することを明示する |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-12 | 経費区分 | 事務用品費・宿泊費・資格精算費・その他経費のいずれかに確定する |
| BRL-14 | 経費区分の自動確定禁止 | 必ずユーザーへ確認を求める |
| BRL-15 | 申請期限超過（90日超） | 警告提示後に処理継続 |
| BRL-16 | 経費合計5,000円超 | 上長承認要否を通知（処理継続） |
| BRL-17 | 業務目的必須 | バリデーションエラーとして再入力を促す |

#### 2.3.3 処理フロー

```
開始（AG-001からの委譲受付）
  ↓
[ステップ1] invocation_stateから申請者名・申請日・session_idを取得
  ↓
[ステップ2] FileSessionManagerを生成（session_idを使用）
  ↓
[ステップ3] 動的システムプロンプト生成
  - get_expense_system_prompt(applicant_name, application_date, deadline_date)
  ↓
[ステップ4] エージェントインスタンス生成
  - _build_expense_agent(session_id, applicant_name, application_date, deadline_date)
  ↓
[ステップ5] エージェント呼び出し
  - agent(query, invocation_state={"applicant_name": ..., "application_date": ...})
  ↓
[エージェント内部処理]
  - 経費情報収集 → 経費区分判断・確認 → 申請期限チェック → ドラフト提示 → generate_expense_form呼び出し
  ↓
[ステップ6] 応答をAG-001へ返す
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | SlidingWindowConversationManager(window_size=20, should_truncate_results=True, per_turn=False) | 複数ステップの情報収集・処理を行うため |
| リトライ戦略 | ModelRetryStrategy(max_attempts=6, initial_delay=4, max_delay=240) | 全エージェント共通 |

| セッション管理 | FileSessionManager（session_idはinvocation_stateから取得） | 会話履歴の永続化 |
| callback_handler | None | ストリーミング無効化 |

#### 2.4.2 利用ツール

| ツールID | ツール名 | 利用目的 | 呼び出し条件 |
|----------|----------|----------|--------------|
| TL-002 | generate_expense_form | 経費精算申請書を生成する | ユーザーのOK選択後（HumanApprovalHook通過後） |

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `expense_agent`

**ツール説明**:
```
経費精算申請を処理する。経費情報を収集し、経費区分を判断して申請書を生成する。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から `applicant_name`・`application_date`・`session_id` を取得

#### 2.5.2 invocation_stateの受け渡し

```python
# AG-001から受け取る
state = tool_context.invocation_state
session_id = state["session_id"]
applicant_name = state["applicant_name"]
application_date = state["application_date"]

# 子エージェントへはsession_idを除いて渡す
child_invocation_state = {
    "applicant_name": applicant_name,
    "application_date": application_date
}
response = agent(query, invocation_state=child_invocation_state)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 expense_agent 関数

##### 説明
AG-001からAgent as Toolsとして呼び出されるツール関数。invocation_stateから申請者情報を取得し、経費精算申請エージェントを生成して処理を実行する。

##### 引数
- `query` (str): AG-001からの指示テキスト（必須）
- `tool_context` (ToolContext): ツールコンテキスト（フレームワーク注入）

##### 戻り値

##### 処理内容
1. `invoke_specialist_agent(query, tool_context, agent_id="AG-003", deadline_months=3, build_agent=_build_expense_agent)` を呼び出す

#### 3.1.2 _build_expense_agent 関数

##### 説明
経費精算申請エージェントのインスタンスを生成する。

##### 引数
- `session_id` (str): セッションID（必須）
- `applicant_name` (str): 申請者名（必須）
- `application_date` (str): 申請日（必須）
- `deadline` (str): 申請期限（必須）

##### 戻り値

1. `get_expense_system_prompt(applicant_name, application_date, deadline)` でシステムプロンプトを生成

| エラー種別 | 条件 | ログレベル | 対応 | 戻り値 |

| LoopLimitError | LoopControlHookがループ上限超過を検知 | WARNING | ErrorHandler.handle_loop_limit_error()に委譲してstr返却 | str（エラーメッセージ） |
| ContextWindowOverflowException | コンテキストウィンドウ超過 | WARNING | ErrorHandler.handle_context_window_error()に委譲してstr返却 | str（エラーメッセージ） |
| MaxTokensReachedException | 最大トークン数到達 | WARNING | ErrorHandler.handle_max_tokens_error()に委譲してstr返却 | str（エラーメッセージ） |

| Exception | 予期しないエラー | ERROR（exc_info=True） | ErrorHandler.handle_unexpected_error()に委譲してstr返却 | str（エラーメッセージ） |

## 5. 補足情報

### 5.1 実装上の注意点

1. **base_agent.py の共通ユーティリティを使用**
   - `create_specialist_agent()` でエージェントを生成する
   - `invoke_specialist_agent()` で呼び出しラッパーを使用する

2. **HumanApprovalHook の登録**
   - `target_tools=["generate_expense_form"]` を設定する

3. **経費区分の自動確定禁止**
   - システムプロンプトでBRL-14を明示し、LLMが勝手に経費区分を確定しないようにする

### 5.2 パフォーマンス考慮事項

1. **会話履歴のウィンドウサイズ**
   - window_size=20 で複数ステップの情報収集を保持する

### 5.3 セキュリティ考慮事項

1. **invocation_state の安全な参照**
   - `tool_context.invocation_state.get()` で安全に参照する
