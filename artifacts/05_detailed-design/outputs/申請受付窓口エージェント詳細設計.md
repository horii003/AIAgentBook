## 1. 概要

### 1.1 コンポーネントの目的

社員が自然言語で入力した申請内容から申請種別（交通費精算申請・経費精算申請）を自動判断し、適切な専門エージェント（AG-002 または AG-003）へ処理を委譲する。判断不能時はユーザーに選択肢を提示し、対象外の場合は担当者へのエスカレーションを案内する。

### 1.2 主要な責務

1. **申請者名の保持**: main.py から受け取った申請者名をインスタンス変数として保持する
2. **申請内容の受付**: 社員が入力した自然言語テキストを受け付ける
3. **申請種別の判断**: 申請ルール（ナレッジ）を参照し、申請種別を自動判定する
4. **専門エージェントへの振り分け**: 確定した申請種別に応じて AG-002 または AG-003 を Agent as Tools として呼び出す
5. **invocation_state の構築・受け渡し**: 申請者名・申請日・セッションID を辞書リテラルで専門エージェントに渡す
6. **エラーハンドリング**: LoopLimitError → Exception の順でキャッチし、ErrorHandler に委譲する

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- `applicant_name` (str, 必須): 申請者名。空文字禁止
- `session_id` (str, 必須): セッションID。セッション開始時にAG-001が生成。形式: `{YYYYMMDDHHMMSS}_{UUID8文字}`（例: `20260323153045_a1b2c3d4`）。生成方法: `datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]`

**対話ループでの入力**:
- ユーザー入力（自然言語での申請内容）
- 特殊コマンド:
  - `exit`, `quit`, `終了`: アプリケーションを終了する
  - `reset`, `リセット`, `最初から`: 会話履歴と申請者情報をリセットし、申請者名の入力からやり直す

**パラメータ**:
- `user_input` (str): ユーザーからの入力テキスト

#### 2.2.2 出力

**出力先**: CLI（標準出力）

**ウェルカムメッセージ**（対話ループ開始時に表示）:
```
============================================================
こちらは申請受付窓口エージェントです
社内の様々な申請作業をサポートします

最初に申請者名を入力してください。その後、申請したい内容をお知らせください。キーワードでも構いません

※終了するには 'exit' または 'quit' と入力ください
※最初からやり直すには 'reset' と入力ください
============================================================
```

**出力メッセージ**:
- 申請内容入力促進: `"申請したい内容を入力してください。"`
- 申請種別案内（交通費精算）: `"交通費精算申請ですね。申請書は「交通費精算申請書」、申請先は所属部門の経理担当者です。"`
- 申請種別案内（経費精算）: `"経費精算申請ですね。申請書は「経費精算申請書」、申請先は所属部門の経理担当者です。"`
- 選択肢提示: `"申請種別を判断できませんでした。以下から選択してください：1. 交通費精算申請 2. 経費精算申請"`
- エスカレーション案内: `"申し訳ございませんが、この申請は本システムの対象外です。担当者にお問い合わせください。"`

**戻り値の形式**: `str`（エージェントからの応答テキスト）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（申請期限チェックはこのエージェントのステップに含まれない） |

##### プロンプト全文

```
あなたは申請受付窓口エージェントです。社員からの申請内容を受け付け、申請種別を判断し、適切な専門エージェントへ処理を委譲します。

【役割】
1. 申請内容の受付
   - 社員が入力した自然言語テキストを受け付ける
   - 申請内容が不明確な場合は、追加情報を求める

2. 申請種別の判断
   - 以下のルールに従い、申請種別を判断する
   - BRL-01: 申請内容に業務上の交通費精算（電車・バス・タクシー・飛行機）に関する内容が含まれる場合 → 交通費精算申請
   - BRL-02: 申請内容に業務上の経費精算（物品購入・宿泊・資格取得等）に関する内容が含まれる場合 → 経費精算申請
   - BRL-03: 申請種別を判断できない場合 → 「交通費精算申請」「経費精算申請」の2択を提示
   - BRL-05: 申請内容が交通費精算・経費精算のいずれにも該当しない場合 → 対象外として担当者へのエスカレーションを案内

3. 申請種別・申請書・申請先の案内
   - BRL-06: 交通費精算申請の場合 → 申請書「交通費精算申請書」、申請先「所属部門の経理担当者」を案内
   - BRL-07: 経費精算申請の場合 → 申請書「経費精算申請書」、申請先「所属部門の経理担当者」を案内

4. 専門エージェントへの委譲
   - 交通費精算申請と確定した場合 → transport_agent ツールを呼び出す
   - 経費精算申請と確定した場合 → expense_agent ツールを呼び出す

【判断プロトコル】
申請種別の判断は以下の優先順序で評価する:
1. BRL-01（交通費精算申請）の条件を確認する
2. BRL-02（経費精算申請）の条件を確認する
3. いずれにも該当しない場合は BRL-03（判断不能）として選択肢を提示する
4. 明らかに対象外の場合は BRL-05（対象外）としてエスカレーションを案内する

【禁止事項】
- 申請書の作成・生成を自ら行ってはならない（専門エージェントに委譲すること）
- 申請種別が確定したら、速やかに対応する専門エージェントツールを呼び出すこと
- 不要な確認や追加質問を行わないこと
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 役割定義 | 申請受付窓口としての責務を明確化し、申請書作成を専門エージェントに委譲させる |
| 判断プロトコル | BRL-01→BRL-02→BRL-03→BRL-05 の優先順序を明示し、LLM の判断ブレを抑制する |
| 禁止事項 | 申請書作成の自律実行を防止し、必ず専門エージェントに委譲させる |

#### 2.3.2 申請種別判断ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-01 | 業務上の交通費精算（電車・バス・タクシー・飛行機）に関する内容 | 交通費精算申請を案内し、transport_agent を呼び出す |
| BRL-02 | 業務上の経費精算（物品購入・宿泊・資格取得等）に関する内容 | 経費精算申請を案内し、expense_agent を呼び出す |
| BRL-03 | 申請種別を判断できない | 「交通費精算申請」「経費精算申請」の2択を提示する |
| BRL-04 | 入力が空 | 再入力を促すメッセージを表示する |
| BRL-05 | 交通費精算・経費精算のいずれにも該当しない | 対象外として担当者へのエスカレーションを案内し業務終了 |

#### 2.3.3 処理フロー

```
開始
  ↓
[ステップ1] 初期化
  - applicant_name をインスタンス変数に保持
  - セッションIDを生成: datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]
  - FileSessionManager を初期化
  - Agent() を構築（tools=[expense_agent, transport_agent]）
  ↓
[ステップ2] 対話ループ開始
  - "申請したい内容を入力してください。" を表示
  ↓
[ステップ3] ユーザー入力受付
  - input("\n\n入力内容（終了時はquit）: ") でユーザー入力を受け取る
  - 空入力 → "入力してください。" を表示して再入力
  - 500文字超過 → "入力が長すぎます（500文字以内）。" を表示して再入力
  - "exit"/"quit"/"終了" → ループを抜けてアプリ終了
  - "reset"/"リセット"/"最初から" → 会話履歴をリセットし、申請者名の再入力へ
  ↓
[ステップ4] エージェント呼び出し
  - invocation_state を構築する
    {
      "applicant_name": self._applicant_name,
      "application_date": datetime.now().strftime("%Y-%m-%d"),
      "session_id": self._session_id
    }
  - response = self._agent(user_input, invocation_state=invocation_state)
  ↓
[ステップ5] 応答表示
  - print(response) で応答を表示
  ↓
[ステップ6] ループ継続判断
  - 業務完了（申請書生成完了・エスカレーション案内）→ ループ終了
  - 継続 → ステップ3へ戻る
```

## 5. 補足情報

### 5.1 実装上の注意点

1. **invocation_state は辞書リテラルで渡す**
   - 専用の Pydantic モデル（InvocationState クラス）は定義しない（バリデーション方針 8.2節）

2. **リセットコマンドの処理**
   - `reset`/`リセット`/`最初から` が入力された場合、会話履歴をリセットし申請者名の再入力へ遷移する

3. **ファイル配置**
   - `agents/orchestrator_agent.py` に配置する

### 5.2 パフォーマンス考慮事項

1. **SlidingWindow(30)**
   - オーケストレーターとして複数専門エージェントとのやり取りを保持するため、最大サイズを採用する

### 5.3 セキュリティ考慮事項

1. **invocation_state の機密情報**
   - 申請者名・申請日・セッションID は LLM のプロンプトに含まれず、invocation_state 経由でのみ渡す

## 7. テスト観点

### 7.1 機能テスト
- 交通費精算申請の内容を入力した場合に transport_agent が呼び出されること
- 経費精算申請の内容を入力した場合に expense_agent が呼び出されること
- 判断不能時に選択肢（1. 交通費精算申請 2. 経費精算申請）が提示されること
- 対象外の申請内容に対してエスカレーション案内が表示されること
- invocation_state に applicant_name・application_date・session_id が正しく設定されること

### 7.2 異常系テスト
- 空入力時に再入力を促すメッセージが表示されること
- 500文字超過時にエラーメッセージが表示されること
- KeyboardInterrupt 発生時にループが終了し、INFOログが出力されること
- LoopLimitError 発生時に適切なメッセージが表示されてループが継続すること
- ContextWindowOverflowException 発生時に適切なメッセージが表示されてループが継続すること
- MaxTokensReachedException 発生時に適切なメッセージが表示されてループが継続すること
- RuntimeError 発生時に適切なメッセージが表示されてループが継続すること（exc_info=True でログ出力）
- 予期しない Exception 発生時に適切なメッセージが表示されてループが継続すること（exc_info=True でログ出力）
- `reset` コマンド入力時に会話履歴がリセットされること
- ログにセッションIDが付加情報として含まれること

### 7.3 性能テスト
- ReAct ループ10回以内で申請種別判断が完了すること
- SlidingWindow(30) で会話コンテキストが正しく保持されること
