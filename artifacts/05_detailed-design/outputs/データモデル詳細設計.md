## 1. 概要

### 1.1 コンポーネントの目的

全ツール・全エージェントで使用するデータ構造を Pydantic v2 モデルとして一元定義し、型安全性を保証する。

### 1.2 主要な責務

- ツール入力モデル・出力モデルの定義
- マスタデータモデルの定義
- 共通バリデーター関数の定義

## 2. 共通バリデーター関数

### 2.1 validate_date_format

**目的**: 日付文字列が YYYY-MM-DD 形式の有効な日付であることを検証する

**シグネチャ**: `def validate_date_format(cls, v: str) -> str`

**処理内容**:
1. `parser.parse(v)` で日付をパースする
2. パース失敗時は `ValueError` を raise する

**定義場所**: `models/data_models.py`

---

### 2.2 normalize_transport_type

**目的**: 交通手段の表記を正規化する（英語表記・略称を日本語正規表記に変換）

**シグネチャ**: `def normalize_transport_type(cls, v: str) -> str`

**マッピング**:

| 入力値 | 正規化後 |
|--------|---------|
| `"train"` | `"電車"` |
| `"bus"` | `"バス"` |
| `"taxi"` | `"タクシー"` |
| `"airplane"` / `"plane"` | `"飛行機"` |
| その他 | そのまま返す（Literal バリデーションで検証） |

**定義場所**: `models/data_models.py`

---

### 2.3 normalize_expense_category

**目的**: 経費区分の表記を正規化する（英語表記・略称を日本語正規表記に変換）

**シグネチャ**: `def normalize_expense_category(cls, v: str) -> str`

**マッピング**:

| 入力値 | 正規化後 |
|--------|---------|
| `"office_supplies"` | `"事務用品費"` |
| `"accommodation"` | `"宿泊費"` |
| `"qualification"` | `"資格精算費"` |
| `"other"` / `"その他"` | `"その他経費"` |
| その他 | そのまま返す（Literal バリデーションで検証） |

**定義場所**: `models/data_models.py`

## 3. ツール入力モデル

### 3.1 TransportCalculatorInput

**目的**: `calculate_transport_fare` ツールの入力バリデーション

**定義場所**: `tools/transport_tools.py`

| フィールド名 | 型 | 必須/任意 | 説明 | 制約 |
|--------------|-----|-----------|------|------|
| `departure` | `str` | 必須 | 出発地（正規化済み駅名・地点名） | min_length=1 |
| `destination` | `str` | 必須 | 目的地（正規化済み駅名・地点名） | min_length=1 |
| `transport_type` | `Literal["電車", "バス", "タクシー", "飛行機"]` | 必須 | 交通手段 | normalize_transport_type バリデーター適用 |
| `travel_date` | `str` | 必須 | 移動日（YYYY-MM-DD形式） | validate_date_format バリデーター適用 |

**バリデーター**:
- `_validate_transport_type = field_validator("transport_type", mode="before")(classmethod(normalize_transport_type))`
- `_validate_travel_date = field_validator("travel_date", mode="before")(classmethod(validate_date_format))`

## 6. バリデーションルール

### 6.1 日付フォーマット
- フォーマット: YYYY-MM-DD（例: 2026-05-21）
- 検証方法: `python-dateutil` の `parser.parse()` で有効な日付であることを確認する
- OK: `"2026-05-21"` / NG: `"2026/05/21"`, `"2026-13-01"`

### 6.2 交通手段
- 許可される値: `"電車"`, `"バス"`, `"タクシー"`, `"飛行機"`
- 正規化: `normalize_transport_type` バリデーターで英語表記・略称を変換する

### 6.3 経費区分
- 許可される値: `"事務用品費"`, `"宿泊費"`, `"資格精算費"`, `"その他経費"`
- 正規化: `normalize_expense_category` バリデーターで英語表記・略称を変換する

### 6.4 金額・運賃
- 制約: 0以上の整数（ge=0）

### 6.5 必須文字列
- 制約: 空文字禁止（min_length=1）

## 7. マルチエージェント連携時のデータ受け渡し

### 7.1 invocation_state（辞書リテラル）

invocation_state は辞書リテラルで渡す。専用の Pydantic モデルは定義しない。

**AG-001 → AG-002/AG-003**:
```python
{
    "user_name": str,       # 申請者名
    "request_date": str,    # 申請日（YYYY-MM-DD形式）
    "session_id": str       # セッションID
}
```

**AG-002/AG-003 → ツール関数内**:
```python
{
    "user_name": str,       # 申請者名
    "request_date": str     # 申請日（YYYY-MM-DD形式）
}
# session_id はファクトリ関数で消費済みのため除外
```

## 8. 依存関係

### 8.1 外部ライブラリ
- `pydantic >= 2.0.0`: データモデル定義・バリデーション
  - `BaseModel`, `Field`, `field_validator`, `ValidationError`
  - `Literal`, `Optional`

  - `parser.parse`
- `typing`: 型ヒント

## 9. テスト観点

### 9.1 機能テスト
- 有効な交通手段（"電車"）で TransportCalculatorInput が生成されること
- 英語表記の交通手段（"train"）が "電車" に正規化されること
- 有効な日付（"2026-05-21"）で TransportRecord が生成されること
- 有効な経費区分（"事務用品費"）で ExpenseRecord が生成されること

### 9.2 異常系テスト
- 無効な交通手段（"自転車"）で ValidationError が発生すること
- 無効な日付形式（"2026/05/21"）で ValidationError が発生すること
- 負の運賃（-1）で ValidationError が発生すること
- 空文字の出発地（""）で ValidationError が発生すること
- 無効な経費区分（"交通費"）で ValidationError が発生すること
