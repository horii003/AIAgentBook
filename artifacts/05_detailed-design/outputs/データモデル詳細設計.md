## 1. 概要

### 1.1 コンポーネントの目的

全ツール・エージェントで使用するPydanticデータモデルを `models/data_models.py` に一元定義し、型安全性とバリデーションの一貫性を保証する。業務ルール（BRL-03・BRL-08・BRL-09・BRL-12・BRL-17等）に基づくバリデーションを共通化する。

### 1.2 モデル分類

| 分類 | 説明 |
|------|------|
| ツール入力モデル | TOOL-001・TOOL-002の入力パラメータの型保証・バリデーション |
| ツール出力モデル | TOOL-001・TOOL-002の出力の型保証 |
| マスタデータモデル | train_fares.json・fixed_fares.jsonの読み込み時の型保証 |
| 明細モデル | ツール入力モデルのitemsフィールドの要素型 |

## 5. 共通バリデーター関数

### 5.1 validate_station_name

**目的**: 駅名・バス停名等の接尾語を除去して正規化する（BRL-08）

**シグネチャ**: `def validate_station_name(v: str) -> str`

**処理内容**:
```python
suffixes = ["駅", "バス停", "空港"]
for suffix in suffixes:
    if v.endswith(suffix):
        return v[:-len(suffix)]
return v
```

**例**: `"東京駅"` → `"東京"`, `"羽田空港"` → `"羽田"`

---

### 5.2 validate_transport_type

**目的**: 交通手段の表記ゆれを正規化する（BRL-03）

**シグネチャ**: `def validate_transport_type(v: str) -> str`

**正規化マッピング**:

| 入力値 | 正規化後 |
|--------|---------|
| `"train"`, `"電鉄"`, `"鉄道"` | `"電車"` |
| `"bus"` | `"バス"` |
| `"taxi"`, `"cab"` | `"タクシー"` |
| `"airplane"`, `"plane"`, `"flight"`, `"航空"` | `"飛行機"` |
| その他 | 入力値をそのまま返す（Literal型チェックで失敗） |

**目的**: 経費区分の表記ゆれを正規化する（BRL-12）

**正規化マッピング**:

| `"宿泊"`, `"ホテル"` | `"宿泊費"` |

---

### 5.4 validate_date_format

**目的**: 日付文字列のYYYY-MM-DD形式を検証する

**シグネチャ**: `def validate_date_format(v: str) -> str`

**処理内容**:
```python
from dateutil import parser as date_parser
try:
    date_parser.parse(v)
    return v
except (ValueError, TypeError):
    raise ValueError(f"日付の形式が正しくありません。YYYY-MM-DD形式で入力してください: {v}")
```

## 7. エラーハンドリング

### 7.1 ValidationError処理フロー

```
ValidationError 発生
  ↓
呼び出し元（ツール関数）が except ValidationError でキャッチ
  ↓
_logger.error("バリデーションエラーが発生しました", exc_info=True)
  ↓
ErrorHandler.handle_validation_error(e) に委譲
  ↓
ユーザー向け日本語メッセージを生成
  ↓
{"success": False, "error_message": message} を返す（例外を再送出しない）
```

### 7.2 バリデーションエラーメッセージ

| フィールド | エラー条件 | メッセージ例 |
|-----------|---------|-----------|
| `transport_type` | 対応外交通手段（BRL-03） | `"交通手段のエラー: 対応していない交通手段が入力されました。対処方法: 電車・バス・タクシー・飛行機のいずれかを入力してください。"` |
| `expense_category` | 対応外経費区分（BRL-12） | `"経費区分のエラー: 対応していない経費区分が入力されました。対処方法: 事務用品費・宿泊費・資格精算費・その他経費のいずれかを選択してください。"` |

## 8. 補足情報

### 8.1 実装上の注意点

1. **共通バリデーター関数の適用**
   - `classmethod(validate_xxx)` として直接渡す（ラムダラッパー禁止: R9.3.3）
   - 例: `_validate_departure = field_validator("departure", mode="before")(classmethod(validate_station_name))`

2. **出力モデルのフィールド名**
   - ツール関数の戻り値辞書のキー名は出力モデルのフィールド名と一致させること
   - `error_message` キーを使用する（`message` は不可）

3. **invocation_stateは辞書リテラルで渡す**
   - 専用Pydanticモデル（InvocationState等）は定義しない

### 8.2 パフォーマンス考慮事項

1. **バリデーションのタイミング**
   - ツール関数の入口で即座にバリデーションを実施する（早期検証）

## 10. テスト観点

### 10.1 正常系テスト
- 有効な交通手段（`"電車"`）で `TransportCalculatorInput` が生成できること
- 交通手段の表記ゆれ（`"train"`）が `"電車"` に正規化されること
- 駅名の接尾語（`"東京駅"`）が `"東京"` に正規化されること
- 有効な経費区分（`"事務用品費"`）で `ExpenseItem` が生成できること
- 経費区分の表記ゆれ（`"事務用品"`）が `"事務用品費"` に正規化されること
- 有効な日付（`"2026-05-10"`）でバリデーションが通過すること
- 金額0円（`amount=0`）でバリデーションが通過すること（ge=0）

### 10.2 バリデーション異常系テスト
- 対応外交通手段（`"自転車"`）で `ValidationError` が発生すること（BRL-03）
- 対応外経費区分（`"交際費"`）で `ValidationError` が発生すること（BRL-12）
- 不正な日付形式（`"2026/05/10"`）で `ValidationError` が発生すること
- 業務目的が空文字（`""`）で `ValidationError` が発生すること（BRL-09/BRL-17）
- 移動明細リストが空（`items=[]`）で `ValidationError` が発生すること
- 負の金額（`amount=-100`）で `ValidationError` が発生すること

### 10.3 境界値テスト
- 金額0円（`amount=0`）でバリデーションが通過すること
- 移動明細1件（最小）で `TransportFormInput` が生成できること

- `model_dump()` で正規化済みフィールド値が返ること
