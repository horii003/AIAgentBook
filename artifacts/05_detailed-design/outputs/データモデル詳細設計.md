## 1. 概要

### 1.1 コンポーネントの目的

ツール関数の入出力データに対して型安全性を保証し、外部データ（運賃データ JSON）の読み込み時にデータ品質を保証する。共通バリデーター関数により表記ゆれの正規化を一元管理する。

### 1.2 モデル分類

| 分類 | 説明 |
|------|------|
| ツール入力モデル | ツール関数に渡されるパラメータの型・制約を検証する |
| ツール出力モデル | ツール関数の戻り値の構造を定義する |
| マスタデータモデル | 運賃データ JSON の読み込み時にデータ構造を検証する |
| ネストモデル | ツール入力モデルの明細リスト要素を定義する |

## 3. 各モデルの詳細設計

### 3.1 TransportCalculatorInput モデル

#### 基本情報

| 項目 | 値 |
|------|-----|
| クラス名 | `TransportCalculatorInput` |
| 基底クラス | `BaseModel` |
| 分類 | ツール入力モデル |
| 説明 | 交通費計算ツール（calculate_transport_fare）の入力パラメータを検証する |

#### フィールド定義

| フィールド名 | 型 | 必須/任意 | デフォルト値 | 説明 | バリデーション制約 |
|------------|-----|-----------|------------|------|-----------------|
| `departure` | `str` | 必須 | なし | 出発地（正規化済み駅名） | `min_length=1`、`validate_station_name` 適用 |
| `destination` | `str` | 必須 | なし | 目的地（正規化済み駅名） | `min_length=1`、`validate_station_name` 適用 |
| `transport_type` | `Literal["電車", "バス", "タクシー", "飛行機"]` | 必須 | なし | 交通手段 | `normalize_transport_type` 適用 |
| `travel_date` | `str` | 必須 | なし | 移動日（YYYY-MM-DD） | `validate_date_format` 適用 |

#### バリデーター

| バリデーター名 | 種別 | 対象フィールド | 処理内容 |
|-------------|------|-------------|---------|
| `_validate_departure` | `field_validator("departure", mode="before")` | `departure` | `validate_station_name` を適用（末尾「駅」除去・空文字禁止） |
| `_validate_destination` | `field_validator("destination", mode="before")` | `destination` | `validate_station_name` を適用（末尾「駅」除去・空文字禁止） |
| `_validate_transport_type` | `field_validator("transport_type", mode="before")` | `transport_type` | `normalize_transport_type` を適用（英語表記→日本語変換） |
| `_validate_travel_date` | `field_validator("travel_date", mode="before")` | `travel_date` | `validate_date_format` を適用（YYYY-MM-DD 形式に正規化） |

#### シリアライズ仕様

| メソッド | 用途 | 注意事項 |
|--------|------|---------|
| `model_dump()` | ツール関数内でバリデーション済みデータにアクセスする際に使用 | バリデーション後の正規化済み値が返される |

## 4. 共通バリデーター関数

### 4.1 validate_station_name

| 項目 | 内容 |
|------|------|
| 関数名 | `validate_station_name` |
| 引数 | `v: str` |
| 戻り値 | `str`（正規化済み駅名） |
| 処理内容 | 空文字チェック（ValueError）、末尾「駅」を `str.removesuffix("駅").strip()` で除去 |
| 例外 | `ValueError`: 空文字列の場合 |

### 4.2 normalize_transport_type

| 項目 | 内容 |
|------|------|
| 関数名 | `normalize_transport_type` |
| 引数 | `v: str` |
| 戻り値 | `str`（正規化済み交通手段） |
| 処理内容 | マッピング辞書で英語表記→日本語に変換。マッピング: `{"train": "電車", "電車": "電車", "bus": "バス", "バス": "バス", "taxi": "タクシー", "タクシー": "タクシー", "airplane": "飛行機", "plane": "飛行機", "飛行機": "飛行機"}` |
| 例外 | `ValueError`: マッピングに存在しない値の場合 |

### 4.3 validate_date_format

| 項目 | 内容 |
|------|------|
| 関数名 | `validate_date_format` |
| 引数 | `v: str` |
| 戻り値 | `str`（YYYY-MM-DD 形式） |
| 処理内容 | `dateutil.parser.parse(v)` でパースし、`strftime("%Y-%m-%d")` で正規化 |
| 例外 | `ValueError`: パース不能な文字列の場合 |

## 5. invocation_state 設計

### 5.1 受け渡しデータ定義

invocation_state は辞書リテラルで渡す。専用の Pydantic モデル（InvocationState クラス）は定義しない（バリデーション方針 8.2節）。

| キー名 | 型 | 送信元エージェント | 受信先エージェント | 説明 |
|--------|-----|----------------|----------------|------|
| `applicant_name` | `str` | AG-001 | AG-002, AG-003 | 申請者名 |
| `application_date` | `str` | AG-001 | AG-002, AG-003 | 申請日（YYYY-MM-DD） |
| `session_id` | `str` | AG-001 | AG-002, AG-003 | セッションID（ファクトリ関数で消費） |

### 5.2 ToolContext 経由での参照

```python
# ツール関数内での invocation_state 参照
applicant_name = tool_context.invocation_state.get("applicant_name")
application_date = tool_context.invocation_state.get("application_date")
```

## 7. エラーハンドリング

### 7.1 ValidationError 処理フロー

```
ValidationError 発生（ツール関数の入口）
  ↓
except ValidationError as e:
  ↓
_logger.error("バリデーションエラー: %s", e, exc_info=True)
  ↓
ErrorHandler.handle_validation_error(e) でメッセージ生成
  ↓
{"success": False, "error_message": msg} を返す（例外を再送出しない）
```

### 7.2 バリデーションエラーメッセージ

| フィールド | エラー条件 | メッセージ |
|-----------|---------|-----------|
| `departure` / `destination` | 空文字列 | "出発地/目的地のエラー: 駅名を入力してください。" |
| `transport_type` | 許容値以外 | "交通手段のエラー: 電車・バス・タクシー・飛行機のいずれかを指定してください。" |
| `travel_date` / `purchase_date` | 不正な日付形式 | "移動日/購入日のエラー: 日付の形式が正しくありません（YYYY-MM-DD 形式で入力してください）。" |
| `amount` / `fare` | 負の値 | "金額/費用のエラー: 0以上の整数を入力してください。" |
| `items` | 空リスト | "明細のエラー: 明細が入力されていません。" |

## 8. 補足情報

### 8.1 実装上の注意点

1. **共通バリデーター関数の適用パターン**
   - `_validate_xxx = field_validator("field_name", mode="before")(classmethod(validate_xxx))` の形式で適用する
   - ラムダラッパー（`classmethod(lambda cls, v: validate_xxx(v))`）は禁止

2. **未使用モデルの定義禁止**
   - 定義したモデルは対応するツール関数内で実際に使用すること

3. **ファイル配置**
   - `models/data_models.py` に全モデルを定義する

### 8.2 パフォーマンス考慮事項

1. **モデルのインスタンス化コスト**
   - Pydantic v2 は高速なバリデーションを提供するため、パフォーマンス上の懸念はない

## 10. テスト観点

### 10.1 正常系テスト
- 有効な出発地・目的地・交通手段・移動日で `TransportCalculatorInput` を生成できること
- 交通手段 `"train"` が `"電車"` に正規化されること
- 駅名 `"渋谷駅"` が `"渋谷"` に正規化されること
- 有効な経費明細リストで `ExpenseReportInput` を生成できること
- 有効な移動明細リストで `TransportReportInput` を生成できること

### 10.2 バリデーション異常系テスト
- 出発地に空文字を渡した場合に `ValidationError` が発生すること
- 交通手段に `"自転車"` を渡した場合に `ValidationError` が発生すること
- 金額に負の値を渡した場合に `ValidationError` が発生すること
- 日付に不正な文字列を渡した場合に `ValidationError` が発生すること
- 空の明細リストを渡した場合に `ValidationError` が発生すること

### 10.3 境界値テスト
- 金額 `0` が許容されること（`ge=0`）
- 金額 `-1` が拒否されること
- 運賃 `1` が許容されること（`gt=0`）
- 運賃 `0` が拒否されること

### 10.4 シリアライズ・デシリアライズテスト
- `model_dump()` でバリデーション後の正規化済み値が返されること
- 正規化後の `transport_type` が `"電車"` であること（入力 `"train"` の場合）
