# Part 1: ErrorHandler 詳細設計書

## 1. 概要

### 1.1 コンポーネントの目的

全エージェント・ツールで発生する例外を分類し、ユーザー向けの日本語メッセージを生成する。技術的な詳細（スタックトレース・例外クラス名）をユーザーから隠蔽し、抽象化されたメッセージを提供する。

### 1.2 主要な責務

- 例外分類（EX-01〜EX-08）ごとのユーザー向けメッセージ生成
- 全メソッドを `@staticmethod` として実装（インスタンス化不要）
- **メソッドの責務はユーザー向けメッセージ文字列の生成と返却のみ**

### 1.3 非責務

- **ログ出力**: 呼び出し元モジュール（エージェント・ツール）が担当
- **例外の再送出**: ツール関数はエラー時に辞書形式で返す（例外を再送出しない）
- **セッション状態管理**: FileSessionManager（SM-001）が担当

## 6. 使用例

### 6.1 ツール関数での使用

```python
from pydantic import ValidationError
from handlers.error_handler import ErrorHandler

try:
    validated = TransportCalculatorInput(**data)
except ValidationError as e:
    _logger.error("バリデーションエラーが発生しました", exc_info=True)
    return {"success": False, "error_message": ErrorHandler.handle_validation_error(e)}
except Exception as e:
    _logger.error("予期しないエラーが発生しました", exc_info=True)
    return {"success": False, "error_message": ErrorHandler.handle_unexpected_error(e)}
```

### 6.2 エージェントでの使用

```python
from handlers.error_handler import ErrorHandler
from handlers.loop_control_hook import LoopLimitError

try:
    response = self.agent(user_input, invocation_state=invocation_state)
except LoopLimitError as e:
    _logger.warning("ループ上限に達しました: session_id={session_id}")
    print(ErrorHandler.handle_loop_limit_error(e))
except Exception as e:
    _logger.error("予期しないエラーが発生しました: session_id={session_id}", exc_info=True)
    print(ErrorHandler.handle_unexpected_error(e))
```

## 7. 補足情報

### 7.1 実装上の注意点

1. **全staticmethod**: インスタンス化不要。`ErrorHandler.handle_xxx()` として直接呼び出す
2. **ログ出力の分離**: ErrorHandler はメッセージ生成のみ担当。ログ出力は呼び出し元が `_logger.error(exc_info=True)` で行う
3. **コンストラクタ引数なし**: `SessionManager` 等の依存注入は行わない
4. **セッション状態更新禁止**: 各メソッド内で `SessionManager` を呼び出さない

# Part 2: LoopControlHook 詳細設計書

## 1. 概要

### 1.1 コンポーネントの目的

全エージェントのReActループ回数をカウントし、上限（10回）到達時にLoopLimitErrorを発生させる。エージェントの暴走（過剰なツール呼び出し・無限ループ）を防止する。

### 1.2 主要な責務

- BeforeInvocationEvent でループカウンタを0にリセット
- AfterModelCallEvent でループカウンタをインクリメントし上限チェック（exception が存在する場合はスキップ）
- AfterInvocationEvent で合計ループ回数をログ出力（リセットは行わない）
- BeforeModelCallEvent でループ回数を INFO レベルでログ出力
- BeforeToolCallEvent でツール名を INFO レベルでログ出力
- AfterToolCallEvent でツール名を INFO レベルでログ出力

### 1.3 非責務

- **エラーメッセージ生成**: ErrorHandler（HD-001）が担当
- **セッション状態管理**: FileSessionManager（SM-001）が担当
- **承認ゲート制御**: HumanApprovalHook（HD-003）が担当

---

## 2. 設計詳細

### 2.1 クラス基本情報

#### クラス名
`LoopControlHook`

#### 説明
ReActループ回数を監視し、上限到達時にLoopLimitErrorを発生させるHookProviderサブクラス。

---

### 2.2 LoopLimitError 定義

`LoopLimitError` はカスタム例外クラスとして `handlers/loop_control_hook.py` に定義する。

```python
class LoopLimitError(Exception):
    """ReActループ上限到達時に発生するカスタム例外。

    Attributes:
        current_iteration: 現在のループ回数
        max_iterations: ループ上限回数
        agent_name: エージェント名
    """
    def __init__(self, current_iteration: int, max_iterations: int, agent_name: str):
        self.current_iteration = current_iteration
        self.max_iterations = max_iterations
        self.agent_name = agent_name
        super().__init__(
            f"ループ上限（{max_iterations}回）に達しました: agent={agent_name}, iteration={current_iteration}"
        )
```

**定義場所**: `handlers/loop_control_hook.py`（`LoopControlHook` クラスと同一モジュール）

**使用箇所**:
- `handlers/loop_control_hook.py`: raise する側
- `agents/orchestrator_agent.py`: except で捕捉する側
- `agents/transport_agent.py`: except で捕捉する側
- `agents/expense_agent.py`: except で捕捉する側
- `handlers/error_handler.py`: handle_loop_limit_error の引数型

---

### 2.3 初期化

#### `__init__(self, max_iterations: int = 10)`

**引数**:
- `max_iterations` (`int`): ループ上限回数（デフォルト: 10）

**インスタンス変数**:
- `_max_iterations`: ループ上限回数
- `_iteration_count`: 現在のループカウンタ（初期値: 0）

---

### 2.4 主要メソッド

#### 2.4.1 register_hooks

##### 説明
フックイベントを登録する。

##### 引数
- `registry` (`HookRegistry`): フックレジストリ
- `**kwargs` (`Any`): 追加引数

##### 処理内容
1. `registry.add_callback(BeforeInvocationEvent, self._handle_before_invocation)`
2. `registry.add_callback(AfterModelCallEvent, self._handle_after_model_call)`
3. `registry.add_callback(AfterInvocationEvent, self._handle_after_invocation)`
4. `registry.add_callback(BeforeModelCallEvent, self._handle_before_model_call)`
5. `registry.add_callback(BeforeToolCallEvent, self._handle_before_tool_call)`
6. `registry.add_callback(AfterToolCallEvent, self._handle_after_tool_call)`

---

### 2.5 フック設計

#### 2.5.1 フック登録

##### `register_hooks`
**登録するイベント**:
- `BeforeInvocationEvent`: `_handle_before_invocation`（ループカウンタリセット）
- `AfterModelCallEvent`: `_handle_after_model_call`（ループカウンタインクリメント・上限チェック）
- `AfterInvocationEvent`: `_handle_after_invocation`（合計ループ回数ログ出力）
- `BeforeModelCallEvent`: `_handle_before_model_call`（ループ回数ログ出力）
- `BeforeToolCallEvent`: `_handle_before_tool_call`（ツール名ログ出力）
- `AfterToolCallEvent`: `_handle_after_tool_call`（ツール名ログ出力）

#### 2.5.2 イベントハンドラー

##### _handle_before_invocation

**説明**: エージェント呼び出し開始時にループカウンタを0にリセットする。

**処理内容**:
1. `self._iteration_count = 0`

##### _handle_after_model_call

**説明**: LLM呼び出し後にループカウンタをインクリメントし、上限チェックを行う。`event.exception` が存在する場合はスキップする。

**処理内容**:
1. `event.exception` が存在する場合: 処理をスキップ（return）
2. `self._iteration_count += 1`
3. `self._iteration_count >= self._max_iterations` の場合:
   - `_logger.warning(f"ループ上限に達しました: iteration={self._iteration_count}, max={self._max_iterations}")`
   - `raise LoopLimitError(current_iteration=self._iteration_count, max_iterations=self._max_iterations, agent_name=event.agent.name if hasattr(event, 'agent') else "unknown")`

##### _handle_after_invocation

**説明**: エージェント呼び出し完了後に合計ループ回数をログ出力する。ループカウンタのリセットは行わない。

**処理内容**:
1. `_logger.info(f"エージェント呼び出し完了: 合計ループ回数={self._iteration_count}")`

##### _handle_before_model_call

**説明**: LLM呼び出し前にループ回数をログ出力する。

**処理内容**:
1. `_logger.info(f"LLM呼び出し開始: iteration={self._iteration_count + 1}/{self._max_iterations}")`

##### _handle_before_tool_call

**説明**: ツール実行前にツール名をログ出力する。

**処理内容**:
1. `tool_name = event.tool_use["name"] if event.tool_use else "unknown"`
2. `_logger.info(f"ツール呼び出し開始: tool={tool_name}")`

##### _handle_after_tool_call

**説明**: ツール実行後にツール名をログ出力する。

**処理内容**:
1. `tool_name = event.tool_use["name"] if event.tool_use else "unknown"`
2. `_logger.info(f"ツール呼び出し完了: tool={tool_name}")`

## 3. ビジネスロジック

### 3.1 ループ制御ロジック

#### 処理フロー

```
BeforeInvocationEvent 発火
  ↓
_iteration_count = 0（リセット）
  ↓
[ReActループ開始]
  ↓
BeforeModelCallEvent 発火
  ↓
INFO: "LLM呼び出し開始: iteration={n}/{max}"
  ↓
AfterModelCallEvent 発火
  ↓
event.exception が存在する? → Yes: スキップ（return）
  ↓
_iteration_count += 1
  ↓
_iteration_count >= _max_iterations?
  ↓
Yes → WARNING ログ出力 → raise LoopLimitError(current_iteration, max_iterations, agent_name)
No  → ループ継続
  ↓
BeforeToolCallEvent 発火
  ↓
INFO: "ツール呼び出し開始: tool={tool_name}"
  ↓
AfterToolCallEvent 発火
  ↓
INFO: "ツール呼び出し完了: tool={tool_name}"
  ↓
AfterInvocationEvent 発火
  ↓
INFO: "エージェント呼び出し完了: 合計ループ回数={_iteration_count}"
（リセットは行わない）
```

# Part 3: HumanApprovalHook 詳細設計書

## 1. 概要

### 1.1 コンポーネントの目的

AG-002・AG-003が申請書生成ツール（`generate_transport_form`・`generate_expense_form`）を呼び出す直前にユーザーへ承認を求める。ユーザーのOK選択後のみツール実行を許可し、修正要望・キャンセル時はツール呼び出しをキャンセルする（APR-001）。

### 1.2 主要な責務

- BeforeToolCallEvent で対象ツール（target_tools）の呼び出しを検知
- ユーザーへOK/修正/キャンセルの選択肢を提示
- 選択結果に応じてツール実行を許可またはキャンセル（`event.cancel_tool`）

### 1.3 非責務

- **ループ制御**: LoopControlHook（HD-002）が担当
- **エラーメッセージ生成**: ErrorHandler（HD-001）が担当
- **申請書生成**: TOOL-002（generate_transport_form・generate_expense_form）が担当
- **申請書ドラフト提示**: エージェント（AG-002/AG-003）がテキストで提示する（ツール呼び出しなし）

---

## 2. 設計詳細

### 2.1 クラス基本情報

#### クラス名
`HumanApprovalHook`

#### 説明
申請書生成ツールの呼び出し前にユーザーの明示的な承認を必須化するHookProviderサブクラス。BeforeToolCallEventで対象ツールをフィルタリングし、承認確認を実施する。

---

### 2.2 初期化

#### `__init__(self, target_tools: list[str], approval_callback: Callable[[str, dict], tuple[bool, str]] | None = None)`

**引数**:
- `target_tools` (`list[str]`): 承認確認を行うツール名のリスト
- `approval_callback` (`Callable[[str, dict], tuple[bool, str]] | None`): 承認確認コールバック関数（省略時はデフォルトのCLI入力を使用）

**インスタンス変数**:
- `_target_tools`: 承認確認対象ツール名リスト
- `_approval_callback`: 承認確認コールバック関数

---

### 2.3 承認コールバック仕様

#### シグネチャ
```python
def approval_callback(tool_name: str, tool_params: dict) -> tuple[bool, str]:
    ...
```

**引数**:
- `tool_name` (`str`): 呼び出されるツール名
- `tool_params` (`dict`): ツールへの入力パラメータ

**戻り値**: `tuple[bool, str]`

| 戻り値 | 意味 | 処理 |
|--------|------|------|
| `(True, "")` | OK（承認） | `event.cancel_tool` は False のまま → ツール実行される |
| `(False, "修正内容")` | 修正要望 | `event.cancel_tool` に修正内容メッセージをセット → ツールキャンセル |
| `(False, "CANCEL")` | キャンセル | `event.cancel_tool` にキャンセルメッセージをセット → ツールキャンセル |

---

### 2.4 主要メソッド

#### 2.4.1 register_hooks

##### 説明
フックイベントを登録する。

##### 引数
- `registry` (`HookRegistry`): フックレジストリ
- `**kwargs` (`Any`): 追加引数

##### 処理内容
1. `registry.add_callback(BeforeToolCallEvent, self._handle_before_tool_call)`

---

### 2.5 フック設計

#### 2.5.1 フック登録

##### `register_hooks`
**登録するイベント**:
- `BeforeToolCallEvent`: `_handle_before_tool_call`（承認確認）

#### 2.5.2 イベントハンドラー

##### _handle_before_tool_call

**説明**: ツール実行前に発火。対象ツールをフィルタリングし、承認コールバックを呼び出す。

**処理内容**:
1. `tool_name = event.tool_use["name"]` でツール名を取得
2. `tool_params = event.tool_use["input"] or {}` でツール入力パラメータを取得
3. `tool_name not in self._target_tools` の場合: 処理をスキップ（return）
4. 対象ツールの場合: `approved, message = self._approval_callback(tool_name, tool_params)` を呼び出す
5. `approved == True`: ツール実行を許可（`event.cancel_tool` は False のまま）
6. `approved == False` かつ `message == "CANCEL"`: `event.cancel_tool = "申請がキャンセルされました。"` でキャンセル
7. `approved == False` かつ `message != "CANCEL"`: `event.cancel_tool = message` で修正要望メッセージをセット

**ログ出力**:
- OK選択時: `INFO: "申請書生成が承認されました: tool={tool_name}"`
- 修正要望時: `INFO: "申請書生成が修正要望でキャンセルされました: tool={tool_name}"`
- キャンセル時: `INFO: "申請書生成がキャンセルされました: tool={tool_name}"`

---

## 3. ビジネスロジック

### 3.1 承認確認ロジック

#### 処理フロー

```
BeforeToolCallEvent 発火
  ↓
tool_name = event.tool_use["name"]
tool_params = event.tool_use["input"] or {}
  ↓
tool_name in self._target_tools?
  ↓
No  → スキップ（return）
Yes → approved, message = self._approval_callback(tool_name, tool_params)
  ↓
approved == True（OK）
  → ツール実行を許可（event.cancel_tool = False のまま）
  → INFO ログ出力
  ↓
approved == False かつ message == "CANCEL"（キャンセル）
  → event.cancel_tool = "申請がキャンセルされました。"
  → INFO ログ出力
  ↓
approved == False かつ message != "CANCEL"（修正要望）
  → event.cancel_tool = message（修正内容メッセージ）
  → INFO ログ出力
```

#### デフォルト承認コールバック（CLI入力）

```python
def _default_approval_callback(tool_name: str, tool_params: dict) -> tuple[bool, str]:
    print("申請書を生成してよろしいですか？")
    print("[1] OK（承認）")
    print("[2] 修正要望")
    print("[3] キャンセル")
    choice = input("選択してください（1/2/3）: ").strip()
    if choice == "1":
        return (True, "")
    elif choice == "2":
        revision = input("修正内容を入力してください: ").strip()
        return (False, revision if revision else "修正要望が選択されました。申請内容を修正してください。")
    else:
        return (False, "CANCEL")
```

## 7. 補足情報

### 7.1 実装上の注意点

1. **全ツール呼び出しで発火**: BeforeToolCallEvent は全ツール呼び出しで発火するため、ハンドラー内で `target_tools` によるフィルタリングを必ず実施する
2. **キャンセル方法**: `event.cancel_tool = "メッセージ"` でキャンセルする（`event.cancel()` は存在しない。`event.stop_reason` も使用しない）
3. **インスタンスの独立性**: AG-002・AG-003それぞれに独立したインスタンスを生成する
4. **承認対象ツール名の一致**: `target_tools` に設定するツール名は申請書生成ツール詳細設計書に定義された全ツール関数名（`generate_transport_form`・`generate_expense_form`）と完全一致させること
5. **承認コールバックの戻り値**: `(True, "")` → OK、`(False, "CANCEL")` → キャンセル、`(False, "修正内容")` → 修正要望

### 7.2 セキュリティ考慮事項

1. **承認の強制**: `target_tools` に登録されたツールは必ず承認確認を経由する。バイパス不可
