# Part 1: ErrorHandler 詳細設計書

## 1. 概要

### 1.1 コンポーネントの目的

全エージェント（AG-001, AG-002, AG-003）および全ツール（TOOL-001, TOOL-002）で発生する例外に対して、ユーザー向けの日本語エラーメッセージを生成する共通コンポーネント。

### 1.2 主要な責務

1. **例外分類ごとのメッセージ生成**: EX-01〜EX-08 の各例外分類に対応するメソッドを提供する
2. **技術的詳細の隠蔽**: スタックトレース・例外クラス名等をユーザーに見せず、業務的な原因と次の行動を提示する
3. **次の行動の提示**: 再入力案内・担当者連絡案内・再試行案内を含むメッセージを生成する

### 1.3 非責務
- **ログ出力**（呼び出し元モジュールが `_logger.error(exc_info=True)` で行う）
- **例外の検知**（エージェントまたはツールが検知し、ErrorHandler に委譲する）
- **セッション状態の更新**（FileSessionManager が行う）
- **リトライ制御**（ModelRetryStrategy が行う）

```
  → str を返す
```

## 6. 使用例

### 6.1 ツール関数での使用例

```python
from pydantic import ValidationError
from handlers.error_handler import ErrorHandler
import logging

_logger = logging.getLogger(__name__)

@tool
def calculate_transport_fare(departure: str, destination: str, ...) -> dict:
    try:
        validated = TransportCalculatorInput(departure=departure, ...)
    except ValidationError as e:
        _logger.error("バリデーションエラー: %s", e, exc_info=True)
        return {"success": False, "fare": None,
                "error_message": ErrorHandler.handle_validation_error(e)}
    except Exception as e:
        _logger.error("交通費計算エラー: %s", e, exc_info=True)
        return {"success": False, "fare": None,
                "error_message": ErrorHandler.handle_calculation_error(e)}
```

### 6.2 エージェントツール関数での使用例

```python
from handlers.error_handler import ErrorHandler
from handlers.loop_control_hook import LoopLimitError
import logging

_logger = logging.getLogger(__name__)

@tool(context=True)
def expense_agent(query: str, tool_context: ToolContext) -> str:
    try:
        return invoke_specialist_agent(...)
    except LoopLimitError as e:
        _logger.warning("ループ制限到達: query=%s", query[:50])
        return ErrorHandler.handle_loop_limit_error(e)
    except Exception as e:
        _logger.error("経費精算申請エージェントエラー: %s, query=%s", e, query[:50], exc_info=True)
        return ErrorHandler.handle_unexpected_error(e)
```

## 7. 補足情報

### 7.1 実装上の注意点

1. **全メソッド @staticmethod**
   - インスタンス化不要。`ErrorHandler.handle_xxx()` として直接呼び出す

2. **ログ出力の責務分離**
   - ErrorHandler はメッセージ生成のみ担当。ログ出力は呼び出し元が `_logger.error(exc_info=True)` で行う

3. **技術的詳細の隠蔽**
   - ユーザー向けメッセージにスタックトレース・例外クラス名等を含めない

### 7.2 パフォーマンス考慮事項

1. **軽量な処理**
   - 文字列生成のみのため、パフォーマンス上の懸念はない

### 7.3 セキュリティ考慮事項

1. **内部情報の非公開**
   - ファイルパス・スタックトレース等の内部情報をユーザー向けメッセージに含めない

# Part 2: LoopControlHook 詳細設計書

## 1. 概要

### 1.1 コンポーネントの目的

全エージェント（AG-001, AG-002, AG-003）の ReAct ループを監視し、最大ループ回数（10回）到達時に LoopLimitError を発生させて無限ループを防止する。

### 1.2 主要な責務

1. **ループ回数のカウント**: LLM 呼び出し1回を1カウントとして AfterModelCallEvent でインクリメントする（`event.exception` が存在する場合はスキップ）
2. **上限到達時の LoopLimitError 発生**: max_iterations 到達時に WARNING ログを出力し LoopLimitError を raise する
3. **カウンタのリセット**: BeforeInvocationEvent でカウンタを0にリセットする
4. **ログ出力**: BeforeModelCallEvent・BeforeToolCallEvent・AfterToolCallEvent・AfterInvocationEvent でINFOレベルのログを出力する

### 1.3 非責務
- **エラーメッセージの生成**（ErrorHandler が担当する）
- **セッション状態の更新**（FileSessionManager が担当する）
- **承認制御**（HumanApprovalHook が担当する）

---

## 2. 設計詳細

### 2.1 クラス基本情報

#### クラス名
`LoopControlHook`

#### 説明
HookProvider を継承し、ReAct ループ回数を監視して無限ループを防止するフッククラス。各エージェントが独立したインスタンスを持つ。

---

### 2.2 初期化

#### `__init__(self, max_iterations: int = 10, agent_name: str = "")`

**引数**:
- `max_iterations` (`int`): ReAct ループの最大回数（デフォルト: 10）
- `agent_name` (`str`): エージェント名（ログ・エラーメッセージ用、デフォルト: ""）

**インスタンス変数**:
- `_max_iterations`: 最大ループ回数
- `_agent_name`: エージェント名
- `_iteration_count`: 現在のループカウンタ（初期値: 0）

---

### 2.3 主要メソッド

#### 2.3.1 register_hooks

##### 説明
フックイベントを HookRegistry に登録する。

##### 引数
- `registry` (`HookRegistry`): フックレジストリ
- `**kwargs` (`Any`): 追加引数

##### 戻り値
- `None`

##### 処理内容
1. `registry.add_callback(BeforeInvocationEvent, self._handle_before_invocation)` を登録する
2. `registry.add_callback(BeforeModelCallEvent, self._handle_before_model_call)` を登録する
3. `registry.add_callback(AfterModelCallEvent, self._handle_after_model_call)` を登録する
4. `registry.add_callback(BeforeToolCallEvent, self._handle_before_tool_call)` を登録する
5. `registry.add_callback(AfterToolCallEvent, self._handle_after_tool_call)` を登録する
6. `registry.add_callback(AfterInvocationEvent, self._handle_after_invocation)` を登録する

---

#### 2.3.2 _handle_before_invocation

##### 説明
エージェント呼び出し開始前にループカウンタを0にリセットする。

##### 引数
- `event` (`BeforeInvocationEvent`): エージェント呼び出し開始前イベント

##### 処理内容
1. `self._iteration_count = 0` でカウンタをリセットする

---

#### 2.3.3 _handle_after_model_call

##### 説明
LLM 呼び出し完了後にカウンタをインクリメントし、上限到達を監視する。

##### 引数
- `event` (`AfterModelCallEvent`): LLM 呼び出し完了後イベント

##### 処理内容
1. `event.exception` が存在する場合はカウントせずリターンする（エラー時はスキップ）
2. `self._iteration_count += 1` でカウンタをインクリメントする
3. `self._iteration_count >= self._max_iterations` の場合、WARNING レベルでログ出力し `LoopLimitError(current_iteration=self._iteration_count, max_iterations=self._max_iterations, agent_name=self._agent_name)` を raise する

##### 例外
- `LoopLimitError`: ループ回数が max_iterations に達した場合（current_iteration・max_iterations・agent_name を保持）

---

## 7. 補足情報

### 7.1 実装上の注意点

1. **各エージェントが独立したインスタンスを持つ**
   - AG-001, AG-002, AG-003 それぞれが独立した LoopControlHook インスタンスを持つ。カウンタの競合は発生しない

2. **LoopLimitError の定義**
   - `LoopLimitError` は `handlers/loop_control_hook.py` 内で定義する
   - `current_iteration`・`max_iterations`・`agent_name` の3フィールドを保持する
   - クラス定義:
     ```python
     class LoopLimitError(RuntimeError):
         def __init__(self, current_iteration: int, max_iterations: int, agent_name: str):
             self.current_iteration = current_iteration
             self.max_iterations = max_iterations
             self.agent_name = agent_name
             super().__init__(
                 f"ループ上限到達: {agent_name} ({current_iteration}/{max_iterations})"
             )
     ```

3. **register_hooks で registry.add_callback() を使用**
   - `add_hook()` は存在しない。必ず `registry.add_callback()` を使用する

### 7.2 パフォーマンス考慮事項

1. **軽量な処理**
   - カウンタのインクリメントと比較のみのため、パフォーマンス上の懸念はない

# Part 3: HumanApprovalHook 詳細設計書

## 1. 概要

### 1.1 コンポーネントの目的

専門エージェント（AG-002, AG-003）の申請書生成ツール実行前に、ユーザーの承認（OK/修正/キャンセル）を得る Human-in-the-Loop 機構を提供する。自律度 Lv4（承認後実行）を実現する。

### 1.2 主要な責務

3. **ツール実行の制御**: OK 選択時はツール実行を継続し、修正・キャンセル選択時は `event.cancel_tool` にメッセージを設定してキャンセルする

- **申請内容の表示**（エージェントのシステムプロンプトで制御する）
- **ループ制御**（LoopControlHook が担当する）
- **エラーメッセージの生成**（ErrorHandler が担当する）
- **対象外ツールの処理**（対象外ツールはスキップする）

## 2. 設計詳細

### 2.1 クラス基本情報

- `approval_callback` (`Callable[[str, dict], tuple[bool, str]]`, optional): 承認コールバック関数。未指定時はデフォルトのCLI承認を使用

```python
human_approval_hook = HumanApprovalHook(
    target_tools=["generate_expense_report"],
    approval_callback=cli_approval_callback
)
loop_control_hook = LoopControlHook(max_iterations=10)

agent = Agent(
    ...
    hooks=[human_approval_hook, loop_control_hook]
)
```

## 7. 補足情報

### 7.1 実装上の注意点

1. **全ツール呼び出しで発火するため対象ツールのフィルタリングが必須**
   - `tool_name not in self._target_tools` の場合は必ずリターンする

2. **ツール呼び出しのキャンセルは event.cancel_tool にメッセージを設定する**
   - `event.cancel()` は存在しない。`event.cancel_tool = "メッセージ"` を使用する

3. **承認対象ツール名の一致確認**
   - `target_tools` に設定するツール名は、エージェント詳細設計書の「利用ツール」に記載されたツール名と完全一致させること
   - AG-002: `["generate_expense_report"]`
   - AG-003: `["generate_transport_report"]`

4. **ドラフト提示とツール実行の分離**
   - ドラフト提示はエージェントのプロンプト指示により実行される（ツール呼び出しなし）
   - HumanApprovalHook は BeforeToolCallEvent でのみトリガーし、承認（OK）後にはじめてツールが実行される

### 7.2 パフォーマンス考慮事項

1. **ユーザー入力待機**
   - `input()` でユーザー入力を待機するため、応答時間はユーザーの操作速度に依存する

### 7.3 セキュリティ考慮事項

1. **承認の強制**
   - 対象ツールは必ず承認ゲートを通過する。承認なしでのツール実行は不可能

## 9. テスト観点

### 9.1 機能テスト
- 対象ツール（generate_expense_report）の実行前に承認コールバックが呼び出されること
  - **入力**: tool_name="generate_expense_report"
  - **期待結果**: approval_callback が tool_name と tool_params を引数に呼び出される
- コールバックが `(True, "")` を返した場合にツール実行が継続されること
  - **入力**: approval_callback が `(True, "")` を返す
  - **期待結果**: event.cancel_tool が False のまま
- コールバックが `(False, "修正内容")` を返した場合に event.cancel_tool に修正メッセージが設定されること
  - **入力**: approval_callback が `(False, "金額を修正してください")` を返す
  - **期待結果**: event.cancel_tool == "金額を修正してください"
- コールバックが `(False, "CANCEL")` を返した場合に event.cancel_tool にキャンセルメッセージが設定されること
  - **入力**: approval_callback が `(False, "CANCEL")` を返す
  - **期待結果**: event.cancel_tool == "キャンセルを選択しました。申請処理を終了します。"
- 対象外ツールの場合に承認コールバックが呼び出されないこと
  - **入力**: tool_name="calculate_transport_fare"
  - **期待結果**: approval_callback が呼び出されない

### 9.2 異常系テスト
- approval_callback が未指定の場合にデフォルトのCLI承認が使用されること

### 9.5 統合テスト
- ドラフト提示ステップ（テキスト整理）では HumanApprovalHook がトリガーされないこと
  - **テスト対象**: エージェントのドラフト提示ステップ
  - **期待結果**: BeforeToolCallEvent が発火しない
- 承認（OK）後にはじめてツールが実行され、Human-in-the-Loop 承認が1回のみ発生すること
  - **テスト対象**: エージェントの申請書生成フロー全体
  - **期待結果**: generate_expense_report が1回のみ実行される

## 10. 設定値

### 10.1 承認対象ツール
- AG-002 用: `target_tools=["generate_expense_report"]`
- AG-003 用: `target_tools=["generate_transport_report"]`

### 10.2 承認コールバック仕様
- コールバックシグネチャ: `(tool_name: str, tool_params: dict) -> tuple[bool, str]`
- OK: `(True, "")` → ツール実行継続
- 修正要望: `(False, "修正内容")` → `event.cancel_tool` にメッセージをセット
- キャンセル: `(False, "CANCEL")` → `event.cancel_tool` にキャンセルメッセージをセット
- ツール中止方法: `event.cancel_tool` にメッセージ文字列をセットする（`event.stop_reason` は使用しない）
