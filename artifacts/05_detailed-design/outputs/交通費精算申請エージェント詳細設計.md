## 1. 概要

### 1.1 コンポーネントの目的

移動情報（移動日・出発地・目的地・交通手段・業務目的）を一括確認で対話収集し、運賃データから交通費を自動計算した後、申請期限・上長承認チェックを実施し、ユーザー承認後に交通費精算申請書（Excel）を自動生成する。

### 1.2 主要な責務

1. **移動情報の一括確認収集**: 移動日・出発地・目的地・交通手段・業務目的を一括で収集する（BRL-17）
2. **駅名正規化**: プロンプト指示により LLM が「渋谷駅」→「渋谷」等の正規化を行う（BRL-15）
3. **交通費自動計算**: calculate_transport_fare を呼び出し、各区間の交通費を計算する（BRL-12）
4. **申請期限チェック**: 移動日から3ヶ月以内かを LLM が判断する（BRL-08）
5. **上長承認チェック**: 交通費合計が10,000円超の場合に通知する（BRL-10）
6. **申請書自動生成**: HumanApprovalHook 承認後に generate_transport_report を呼び出す

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（移動日から3ヶ月以内）を LLM に判断させるため、現在日付と3ヶ月前の日付をプロンプトに埋め込む必要がある |

##### 動的に埋め込む情報
- 申請者名（applicant_name）
- 申請日（application_date: YYYY-MM-DD）
- 申請期限の基準日（application_date から3ヶ月前: YYYY-MM-DD）

##### プロンプト全文

```
あなたは交通費精算申請エージェントです。申請者「{applicant_name}」の交通費精算申請書（Excel）の作成を支援します。
申請日: {application_date}
申請期限の基準日（この日付以降の移動日のみ申請可能）: {deadline_date}

【役割】
1. 移動情報の一括確認収集（BRL-17）
   - 以下の情報を一括で確認収集する: 移動日、出発地、目的地、交通手段、業務目的
   - 費用は自動計算するため確認しない（運賃データに該当経路が存在しない場合のみ確認する）
   - 複数区間がある場合は全区間の情報を収集する

2. 駅名正規化（BRL-15）
   - 入力された駅名の表記ゆれを正規化する（「渋谷駅」→「渋谷」）
   - calculate_transport_fare ツールを呼び出す前に必ず正規化を行う

3. 交通費自動計算（BRL-12）
   - 各区間について calculate_transport_fare ツールを呼び出す
   - 電車: 経路テーブルから運賃を検索する
   - バス・タクシー・飛行機: 固定運賃を取得する
   - 該当経路が存在しない場合: ユーザーに手動での金額入力を案内する

4. 申請期限チェック（BRL-08）
   - 各区間の移動日が申請期限の基準日（{deadline_date}）以降であることを確認する
   - 基準日より前の移動日がある場合は申請期限超過エラーを通知し業務終了する

5. 上長承認チェック（BRL-10）
   - 交通費合計が10,000円を超える場合は上長の承認が必要な旨を通知する
   - 通知後も申請書作成は続行する

6. 申請書ドラフト提示
   - 収集した全情報をテキストとして整理・提示する（この段階ではツール呼び出しを行わない）
   - 申請者名・申請日・移動明細（移動日・出発地・目的地・交通手段・費用・業務目的）・合計金額を提示する

7. 申請書生成
   - ユーザーが承認（OK）を選択した後、generate_transport_report ツールを呼び出す
   - 承認前にツールを呼び出してはならない

【禁止事項】
- 申請書生成前にユーザーの承認を得ずに generate_transport_report を呼び出してはならない
- 費用をユーザーに確認してはならない（運賃データに該当経路が存在しない場合を除く）
- システム系エラーが発生した場合は、エラー内容を要約して呼び出し元に返却する
```

##### 妥当性チェックルールの組み込み方式

| 項目 | 内容 |
|------|------|
| 組み込み方式 | エージェント初期化時に外部ファイルを参照しプロンプトに組み込む |
| ルール参照先 | `agent_knowledge/transportation_policies.py` |
| 組み込みタイミング | `get_transport_system_prompt()` 呼び出し時に `get_transport_policies()` の戻り値をプロンプトに埋め込む |

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的日付埋め込み | 申請期限チェックを LLM に判断させるため、基準日を動的に埋め込む |
| 費用確認の省略 | 費用は自動計算するため確認しない方針を明示し、不要な質問を防止する |
| 駅名正規化の指示 | BRL-15 をツールではなくプロンプト指示で実現し、LLM が正規化してから calculate_transport_fare に渡す |
| 外部ルール参照 | 妥当性チェックルールを `agent_knowledge/transportation_policies.py` から取得し、プロンプトに動的に組み込む |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-08 | 移動日が申請期限基準日より前 | 申請期限超過エラーを通知し業務終了 |
| BRL-10 | 交通費合計 > 10,000円 | 上長承認が必要な旨を通知し申請書作成を続行 |
| BRL-12 | 移動情報入力後 | calculate_transport_fare で各区間の交通費を自動計算 |
| BRL-15 | 駅名入力時 | 「〇〇駅」→「〇〇」の表記ゆれをプロンプト指示で LLM が正規化 |
| BRL-17 | 移動情報入力依頼時 | 全区間の移動情報を一括で確認収集する（費用は確認しない） |

#### 2.3.3 処理フロー

```
開始（AG-001 から transport_agent ツール経由で呼び出される）
  ↓
[ステップ1] invocation_state から情報取得
  - applicant_name = tool_context.invocation_state.get("applicant_name")
  - application_date = tool_context.invocation_state.get("application_date")
  - session_id = tool_context.invocation_state.get("session_id")
  - いずれかが None → エラーメッセージを返して終了
  ↓
[ステップ2] エージェント構築
  - deadline_date = calculate_deadline(application_date, deadline_months=3)
  - system_prompt = get_transport_system_prompt(applicant_name, application_date, deadline_date)
  - agent = create_specialist_agent(session_id, system_prompt,
      tools=[calculate_transport_fare, generate_transport_report], ...)
  ↓
[ステップ3] エージェント呼び出し
  - child_invocation_state = {"applicant_name": applicant_name, "application_date": application_date}
  - response = agent(query, invocation_state=child_invocation_state)
  ↓
[ステップ4] 応答返却
  - str(response) を返す
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools 設計

**ツール名**: `transport_agent`

**ツール説明**:
```
交通費精算申請書の作成を支援する専門エージェント。
移動情報を収集し、運賃データから交通費を自動計算した後、
ユーザー承認を得て交通費精算申請書（Excel）を生成する。
```

**ToolContext の使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から applicant_name・application_date・session_id を取得
- session_id は `create_specialist_agent()` のファクトリ関数で消費し、子エージェントへは渡さない

#### 2.5.2 invocation_state の受け渡し

```python
# AG-001 から受け取る
state = tool_context.invocation_state
session_id = state.get("session_id")
applicant_name = state.get("applicant_name")
application_date = state.get("application_date")

# 子エージェントには session_id を除いて渡す
child_invocation_state = {
    "applicant_name": applicant_name,
    "application_date": application_date
}
response = agent(query, invocation_state=child_invocation_state)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 transport_agent 関数（Agent as Tools ツール関数）

##### 説明
交通費精算申請エージェントを構築して呼び出す Agent as Tools ツール関数。

##### 引数
- `query` (str): AG-001 からの指示テキスト（必須）
- `tool_context` (ToolContext): ツールコンテキスト（フレームワーク注入）

##### 戻り値
- `str`: エージェントからの応答テキスト

##### 処理内容
1. `invoke_specialist_agent()` を呼び出す（共通ラッパー）
2. invocation_state から session_id・applicant_name・application_date を取得
3. `_build_transport_agent()` でエージェントを構築
4. 子エージェントに applicant_name・application_date を渡して呼び出す

#### 3.1.2 _build_transport_agent 関数

##### 説明
交通費精算申請エージェントのインスタンスを構築するファクトリ関数。

##### 引数
- `session_id` (str): セッションID（必須）
- `applicant_name` (str): 申請者名（必須）
- `application_date` (str): 申請日（必須）
- `deadline` (str): 申請期限（必須）

##### 戻り値
- `Agent`: 構築済みエージェントインスタンス

##### 処理内容
1. `get_transport_system_prompt(applicant_name, application_date, deadline)` でプロンプト生成
2. `create_specialist_agent(session_id, system_prompt, tools=[calculate_transport_fare, generate_transport_report], ...)` でエージェント構築

## 5. 補足情報

### 5.1 実装上の注意点

1. **ドラフト提示とツール実行の分離**
   - 申請書生成前にテキストで内容を提示し、HumanApprovalHook の承認後にのみ generate_transport_report を実行する

2. **ファイル配置**
   - `agents/transport_agent.py` に配置する

3. **共通ファクトリ・ラッパーの使用**
   - `agents/base_agent.py` の `create_specialist_agent()` と `invoke_specialist_agent()` を使用する

### 5.2 パフォーマンス考慮事項

1. **SlidingWindow(20)**
   - 複数ステップの情報収集・運賃計算を行うため中サイズを採用する

### 5.3 セキュリティ考慮事項

1. **invocation_state の機密情報**
   - 申請者名・申請日は LLM のプロンプトに含まれず、invocation_state 経由でのみ渡す

## 7. テスト観点

### 7.1 機能テスト
- 移動情報が一括確認で正しく収集されること
- 駅名の表記ゆれ（「渋谷駅」→「渋谷」）が正規化されること
- calculate_transport_fare が各区間ごとに正しく呼び出されること
- 申請期限内の移動で申請書が正しく生成されること
- HumanApprovalHook 承認（OK）後に generate_transport_report が呼び出されること
- 承認前に generate_transport_report が呼び出されないこと

### 7.2 異常系テスト
- 申請期限超過時に適切なエラーメッセージが表示されること

- 経路不存在時に手動入力案内が表示されること
