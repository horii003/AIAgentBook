## 1. 概要

### 1.1 コンポーネントの目的

全エージェント（AG-001, AG-002, AG-003）に対して、Amazon Bedrock Guardrails（ガードレールツール層）とアプリケーション層（HumanApprovalHook・LoopControlHook）の2層構造でガードレールを適用する。有害コンテンツ・プロンプトインジェクション・業務スコープ外トピックをブロックし、申請書生成前の承認ゲートと無限ループ防止を実現する。

### 1.2 ガードレール種別と対象エージェント

| 種別 | 対象エージェント | 説明 |
|------|--------------|------|
| 入力フィルター | AG-001, AG-002, AG-003 | Amazon Bedrock Guardrails による有害コンテンツ・プロンプトインジェクション・業務スコープ外トピックのブロック |
| 出力フィルター | AG-001, AG-002, AG-003 | Amazon Bedrock Guardrails による出力コンテンツフィルタリング |
| ツール実行ガード | AG-002, AG-003 | HumanApprovalHook による申請書生成前の承認ゲート |
| ループ制御 | AG-001, AG-002, AG-003 | LoopControlHook による ReAct ループ回数制御 |

## 2. ガードレール一覧

| ガードレールID | 種別 | 対象エージェント | 概要 | 優先順位 |
|-------------|------|--------------|------|---------|
| GR-001 | 入力フィルター・出力フィルター | AG-001, AG-002, AG-003 | Amazon Bedrock Guardrails による業務スコープ外トピック拒否・有害コンテンツブロック・プロンプトインジェクション防止 | 高 |
| GR-002 | アプリケーション層（業務ルール） | AG-002, AG-003 | 申請期限超過禁止（BRL-08, BRL-09）。エージェント内の業務ロジックで実装 | 高 |
| GR-003 | ツール実行ガード | AG-002, AG-003 | HumanApprovalHook による承認なし申請書生成禁止 | 高 |
| GR-004 | 入力フィルター・出力フィルター | AG-001, AG-002, AG-003 | Amazon Bedrock Guardrails によるコンテンツフィルタリング（推測・補完禁止） | 中 |
| GR-LOOP | ループ制御 | AG-001, AG-002, AG-003 | LoopControlHook による最大ループ回数（10回）制御 | 高 |

## 3. 入力フィルター設計

### 3.1 GR-001: Amazon Bedrock Guardrails 入力フィルター

#### 基本情報

| 項目 | 値 |
|------|-----|
| ガードレールID | GR-001 |
| 適用タイミング | LLM 呼び出し前（BedrockModel が自動適用） |
| 実装方式 | Amazon Bedrock Guardrails（BedrockModel の guardrail_id 設定） |

#### ブロック条件

| 条件ID | 条件内容 | 判定ロジック |
|--------|---------|------------|
| GR-001-IN-01 | 業務スコープ外トピック | Amazon Bedrock Guardrails のトピック拒否設定（交通費精算・経費精算以外のトピック） |
| GR-001-IN-02 | 有害コンテンツ | Amazon Bedrock Guardrails のコンテンツフィルタリング（暴力・差別・不適切表現等） |
| GR-001-IN-03 | プロンプトインジェクション | Amazon Bedrock Guardrails のプロンプトインジェクション検知 |

#### フォールバック応答

| 発動条件 | メッセージ文言 |
|---------|------------|
| 業務スコープ外トピック（GR-001-IN-01） | Amazon Bedrock Guardrails が自動生成するブロックメッセージ。AG-001 がエスカレーション案内を表示 |
| 有害コンテンツ（GR-001-IN-02） | Amazon Bedrock Guardrails が自動生成するブロックメッセージ |
| プロンプトインジェクション（GR-001-IN-03） | Amazon Bedrock Guardrails が自動生成するブロックメッセージ |

#### 処理フロー

```
ユーザー入力受信
  ↓
BedrockModel が LLM 呼び出し前に Amazon Bedrock Guardrails を適用
  ↓
[ブロック条件判定（Guardrails 自動処理）]
  ↓
条件一致（ブロック）
  → Guardrails が拒否メッセージを返す
  → エージェントが拒否メッセージを受け取る
  → ガードレールトレース（guardrail_trace="enabled"）でログ記録
  → AG-001 がエスカレーション案内を表示（業務スコープ外の場合）
条件不一致（通過）
  → ガードレール通過
  → 通常の LLM 処理を継続
```

## 4. 出力フィルター設計

### 4.1 GR-001: Amazon Bedrock Guardrails 出力フィルター

#### 基本情報

| 項目 | 値 |
|------|-----|
| ガードレールID | GR-001 |
| 適用タイミング | LLM 応答生成後（BedrockModel が自動適用） |
| 実装方式 | Amazon Bedrock Guardrails（BedrockModel の guardrail_id 設定） |

#### フィルター条件

| 条件ID | 条件内容 | 判定ロジック | フィルター後処理 |
|--------|---------|------------|--------------|
| GR-001-OUT-01 | 業務スコープ外の出力 | Amazon Bedrock Guardrails のトピック拒否設定 | Guardrails が自動的にブロック・置換 |
| GR-001-OUT-02 | 有害コンテンツの出力 | Amazon Bedrock Guardrails のコンテンツフィルタリング | Guardrails が自動的にブロック・置換 |

#### 処理フロー

```
LLM 応答生成
  ↓
BedrockModel が Amazon Bedrock Guardrails を適用
  ↓
[フィルター条件判定（Guardrails 自動処理）]
  ↓
条件一致（フィルター）
  → Guardrails が応答をブロック・置換
  → ガードレールトレース（guardrail_trace="enabled"）でログ記録
  → フィルター済み応答を返す
条件不一致（通過）
  → 元の応答を返す
```

## 5. ツール実行ガード設計

### 5.1 承認要否マトリクス

| エージェント | ツール名 | 承認要否 | 承認者 |
|-----------|--------|---------|--------|
| AG-002 | `generate_expense_report` | 要 | ユーザー（申請者） |
| AG-003 | `generate_transport_report` | 要 | ユーザー（申請者） |
| AG-003 | `calculate_transport_fare` | 不要 | — |

### 5.2 人間承認フックとの連携

| 項目 | 値 |
|------|-----|
| 実装方式 | HumanApprovalHook（handlers/human_approval_hook.py）。ハンドラー詳細設計書 Part 3 参照 |
| 承認待機タイムアウト | 要件上未定義（CLI の input() でユーザー入力を待機） |
| タイムアウト時の動作 | 要件上未定義 |

### 5.3 GR-003: 承認なし申請書生成禁止

#### 処理フロー

```
エージェントが申請書生成ツールを呼び出そうとする
  ↓
BeforeToolCallEvent 発火
  ↓
HumanApprovalHook._handle_before_tool_call() が実行される
  ↓
tool_name in target_tools?
  ↓
No → スキップ（他のツールは承認不要）
  ↓
Yes → ユーザーに承認選択肢を表示
  "申請書を生成してよろしいですか？"
  "1. OK（生成する）"
  "2. 修正（情報を修正する）"
  "3. キャンセル（申請を中止する）"
  ↓
ユーザー選択
  ↓
"1" / "ok" / "OK" → ツール実行継続
"2" / "修正" → event.cancel_tool = "修正を選択しました。情報を修正してください。"
"3" / "キャンセル" → event.cancel_tool = "キャンセルを選択しました。申請処理を終了します。"
その他 → event.cancel_tool = "無効な選択です。申請処理を中止します。"
```

## 6. 判定ロジック詳細

### 6.1 GR-001 判定ロジック（Amazon Bedrock Guardrails）

| 項目 | 値 |
|------|-----|
| 使用データソース | Amazon Bedrock Guardrails のポリシー設定（AWS コンソールまたは CloudFormation で管理） |
| データ形式 | Amazon Bedrock Guardrails の設定（JSON/CloudFormation） |
| 判定アルゴリズム | Amazon Bedrock Guardrails の内部処理（詳細は AWS 管理） |
| 設定ファイル | `guardrails/guardrails_cloudformation.yaml` |

### 6.2 GR-002 判定ロジック（申請期限チェック）

| 項目 | 値 |
|------|-----|
| 使用データソース | invocation_state の application_date、ユーザー入力の経費発生日/移動日 |
| 判定アルゴリズム | LLM がシステムプロンプトの基準日（application_date から3ヶ月前）と経費発生日/移動日を比較 |
| 閾値 | 3ヶ月（BRL-08: 交通費、BRL-09: 経費） |

### 6.3 GR-003 判定ロジック（承認ゲート）

| 項目 | 値 |
|------|-----|
| 使用データソース | HumanApprovalHook の target_tools リスト |
| 判定アルゴリズム | `event.tool_use["name"] in self._target_tools` |
| 対象ツール | AG-002: `["generate_expense_report"]`、AG-003: `["generate_transport_report"]` |

## 7. フォールバック応答設計

| ガードレール種別 | 発動条件 | メッセージ文言 |
|--------------|---------|------------|
| GR-001（業務スコープ外） | 業務スコープ外のトピックが入力された | Amazon Bedrock Guardrails が自動生成するブロックメッセージ |
| GR-002（申請期限超過） | 経費発生日/移動日から3ヶ月超過 | `"申請期限を超過しているため、申請書を作成できません。経費発生日から3ヶ月以内に申請してください。"` |

## 8. エラーハンドリング

### 8.1 エラー処理フロー

```
ガードレール処理中エラー発生
  ↓
[エラー種別判定]
  ↓
Amazon Bedrock Guardrails エラー（EX-04）
  → ModelRetryStrategy（最大6回）で自動リトライ
  → リトライ失敗時: ErrorHandler.handle_unexpected_error() でメッセージ生成
  → 業務終了

HumanApprovalHook エラー（EX-08）
  → ErrorHandler.handle_unexpected_error() でメッセージ生成
  → 業務終了

LoopControlHook LoopLimitError（EX-08）
  → ErrorHandler.handle_loop_limit_error() でメッセージ生成
  → 業務終了
```

### 8.2 エラーメッセージ

| エラー種別 | 発生条件 | 動作 | ユーザー向けメッセージ |
|----------|---------|------|---------------------|
| Amazon Bedrock Guardrails エラー（EX-04） | Guardrails API エラー | ModelRetryStrategy でリトライ。失敗時は業務終了 | "予期しないエラーが発生しました。担当者にお問い合わせください。" |
| LoopLimitError（EX-08） | ループ回数が10回に達した | 業務終了 | "処理の上限回数に達したため、申請処理を終了します。お手数ですが、最初からやり直してください。" |

## 10. 補足情報

### 10.1 実装上の注意点

1. **Amazon Bedrock Guardrails の設定**
   - ガードレールID・バージョンは `.env` ファイルで管理し、`config/model_config.py` で参照する
   - `.env` ファイルは `.gitignore` に登録し、リポジトリにコミットしない
   - `guardrails/guardrails_cloudformation.yaml` で Guardrails リソースを管理する

2. **GUARDRAIL_ID 未設定時の警告**
   - `GUARDRAIL_ID` 環境変数が未設定の場合、main.py 起動時に WARNING ログを出力する

3. **HumanApprovalHook の全ツール発火対策**
   - BeforeToolCallEvent は全ツール呼び出しで発火するため、`target_tools` によるフィルタリングを必ず実施する

## 12. テスト観点

### 12.1 正常系テスト（通過）
- 業務スコープ内の申請内容（交通費精算・経費精算）が Guardrails を通過すること
- 承認（OK）選択後に申請書生成ツールが実行されること
- LoopControlHook が10回未満のループで LoopLimitError を発生させないこと

### 12.2 ガードレール発動テスト
- 業務スコープ外のトピック（例: 「株式投資について教えて」）が Guardrails でブロックされること
- 申請期限超過（経費発生日から3ヶ月超）でエラーメッセージが表示されること
- 修正選択時に `event.cancel_tool` に修正メッセージが設定されること
- キャンセル選択時に `event.cancel_tool` にキャンセルメッセージが設定されること
- LoopControlHook が10回目のループで LoopLimitError を発生させること

### 12.3 境界値テスト
- 申請期限ちょうど3ヶ月前の経費発生日で申請が通過すること
- 申請期限3ヶ月+1日前の経費発生日で申請期限超過エラーが発生すること
- LoopControlHook が9回目のループで LoopLimitError を発生させないこと
- LoopControlHook が10回目のループで LoopLimitError を発生させること

### 12.4 フォールバック応答テスト
- 申請期限超過時のメッセージが "申請期限を超過しているため..." であること
- キャンセル選択時のメッセージが "キャンセルを選択しました..." であること
- ループ制限到達時のメッセージが "処理の上限回数に達したため..." であること
