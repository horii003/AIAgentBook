## 1. 概要

### 1.1 コンポーネントの目的

Amazon Bedrock Guardrailsを使用して、全エージェント（AG-001〜AG-003）のLLM推論の入力・出力段階にガードレールを設ける。有害コンテンツ・個人情報・業務外トピックの入出力を制御し、申請業務の安全な運用を保証する。

ガードレールはAmazon Bedrock Guardrailsで実現する。ループ制御・承認ゲート等のアプリケーションレベルの制御はハンドラー設計（LoopControlHook・HumanApprovalHook）で対応する。

### 1.2 ガードレール種別と対象エージェント

| 種別 | 対象エージェント | 説明 |
|------|--------------|------|
| 入力フィルター（コンテンツポリシー） | AG-001, AG-002, AG-003 | 有害コンテンツの入力を遮断する |
| 入力フィルター（PIIフィルタ） | AG-001, AG-002, AG-003 | 個人情報をマスキングしてLLMに渡す |
| ルール検証（トピックポリシー） | AG-001, AG-002, AG-003 | 業務外トピックへの回答を禁止する |
| 出力フィルター（コンテンツポリシー） | AG-001, AG-002, AG-003 | 有害コンテンツの出力を遮断する |
| 出力フィルター（PIIフィルタ） | AG-001, AG-002, AG-003 | 出力の個人情報をマスキングする |
| 実行ガード（グラウンディング） | AG-001, AG-002, AG-003 | 業務文脈からの逸脱を検証する（オプション） |
| ツール実行ガード（承認ゲート） | AG-002, AG-003 | 申請書生成前のユーザー承認（HumanApprovalHookで実装） |

## 3. 入力フィルター設計

### 3.1 GRD-001: 入力コンテンツポリシーフィルター

#### 基本情報

| 項目 | 値 |
|------|-----|
| ガードレールID | GRD-001 |
| 適用タイミング | LLM推論の入力段階（Amazon Bedrock Guardrailsが自動適用） |
| 実装方式 | Amazon Bedrock Guardrailsのコンテンツポリシーフィルタ。BedrockModelの `guardrail_id` / `guardrail_version` パラメータで設定 |

#### ブロック条件

| 条件ID | 条件内容 | 判定ロジック |
|--------|---------|------------|
| GRD-001-C1 | 暴力・差別・違法行為等の有害コンテンツ | Amazon Bedrock Guardrailsのコンテンツポリシーフィルタが自動判定 |
| GRD-001-C2 | 業務と無関係な有害コンテンツ | Amazon Bedrock Guardrailsのコンテンツポリシーフィルタが自動判定 |

#### フォールバック応答

| 発動条件 | メッセージ文言 |
|---------|------------|
| 有害コンテンツ検知時 | `"申し訳ありません。入力内容が業務の申請処理に適していないため、処理できません。交通費精算申請または経費精算申請に関する内容を入力してください。"` |

#### 処理フロー

```
ユーザー入力受信
  ↓
Amazon Bedrock Guardrailsがコンテンツポリシーフィルタを適用（自動）
  ↓
有害コンテンツ検知
  ↓
Yes → フォールバック応答を返す（処理中断）
      ↓
      WARNING ログ出力
No  → ガードレール通過 → 通常処理継続
```

## 6. 判定ロジック詳細

### 6.1 GRD-001〜GRD-006 判定ロジック

| 項目 | 値 |
|------|-----|
| 使用データソース | Amazon Bedrock Guardrailsの内部モデル（外部ファイル不使用） |
| データ形式 | Amazon Bedrock Guardrailsの設定（GUARDRAIL_ID・GUARDRAIL_VERSION） |
| 判定アルゴリズム | Amazon Bedrock Guardrailsが自動判定（実装不要） |
| 計算量 | Amazon Bedrock Guardrailsに依存 |

### 6.2 Amazon Bedrock Guardrails設定

| 設定項目 | 値 | 取得方法 |
|---------|-----|---------|
| `guardrail_id` | 環境変数 `GUARDRAIL_ID` | `os.getenv("GUARDRAIL_ID")` |
| `guardrail_version` | 環境変数 `GUARDRAIL_VERSION`（デフォルト: `"DRAFT"`） | `os.getenv("GUARDRAIL_VERSION", "DRAFT")` |
| `guardrail_trace` | `"enabled"` | 固定値 |

### 6.3 BedrockModel設定コード例

```python
from strands.models import BedrockModel

model = BedrockModel(
    model_id=ModelConfig.DEFAULT_MODEL_ID,
    guardrail_id=os.getenv("GUARDRAIL_ID"),
    guardrail_version=os.getenv("GUARDRAIL_VERSION", "DRAFT"),
    guardrail_trace="enabled"
)
```

## 7. エラーハンドリング

### 7.1 エラー処理フロー

```
Amazon Bedrock Guardrails処理中エラー発生
  ↓
Strands SDKがエラーをキャッチ
  ↓
エージェントのエラーハンドリング（LoopLimitError → Exception の順）
  ↓
ErrorHandler.handle_unexpected_error(e) に委譲
  ↓
ユーザーへエラーメッセージを表示して処理中断
```

### 7.2 エラーメッセージ

| エラー種別 | 発生条件 | 動作 | ユーザー向けメッセージ |
|----------|---------|------|---------------------|
| Guardrails API エラー | Amazon Bedrock Guardrails APIが応答しない場合 | フェイルクローズ（処理中断） | `"申し訳ありません。予期しないエラーが発生しました。システム管理者にご連絡ください。"` |

## 9. 補足情報

### 9.1 実装上の注意点

1. **ガードレールIDの管理**
   - `GUARDRAIL_ID` は環境変数で管理する（`.env` ファイルに設定）
   - ガードレールが未設定の場合（`GUARDRAIL_ID` が空）は、ガードレールなしで動作する

2. **全エージェント共通設定**
   - `ModelConfig.get_model()` で `BedrockModel` を生成する際に `guardrail_id`・`guardrail_version`・`guardrail_trace` を設定する
   - 全エージェントで同一のガードレール設定を使用する

3. **GRD-006（グラウンディング）はオプション**
   - Amazon Bedrock Guardrailsのグラウンディングチェックはオプション機能
   - 初期実装では無効化し、必要に応じて有効化する

## 11. テスト観点

### 11.1 正常系テスト（通過）
- 通常の申請内容（「電車代を申請したい」等）でガードレールが通過すること
- 業務関連の経費情報でガードレールが通過すること

### 11.2 ガードレール発動テスト
- 有害コンテンツ（暴力・差別等）の入力でGRD-001が発動すること
- 業務外トピック（投資・医療相談等）の入力でGRD-003が発動すること
- 個人情報（マイナンバー等）の入力でGRD-002がマスキングを実施すること

### 11.3 境界値テスト
- ガードレール判定の閾値付近のコンテンツでの動作確認（Amazon Bedrock Guardrailsの設定に依存）

### 11.4 フォールバック応答テスト
- GRD-001発動時にフォールバックメッセージが返ること
- GRD-003発動時にフォールバックメッセージが返ること
