## 1. ファイル概要

### 1.1 目的

評価テスト方式設計で定義された評価戦略に基づき、実行可能な評価テストスクリプトを生成するために必要な詳細仕様を定義する。

### 1.2 評価スクリプト一覧

| 項目 | eval_tool_selection.py | eval_goal_success_rate.py |
|------|----------------------|--------------------------|
| 対象ファイル | evals/eval_tool_selection.py | evals/eval_goal_success_rate.py |
| 評価レベル | 出力レベル | セッションレベル |
| 評価器種別 | ToolSelectionAccuracyEvaluator | GoalSuccessRateEvaluator |
| 実行方式 | シングルターン | マルチターン |
| 対応指標 | MET-001 | MET-002 |
| スコア形式 | 二値（1.0/0.0） | 二値（1.0/0.0） |

## 2. 評価基盤構成

### 2.1 ファイル構成

```
evals/
├── helpers.py
├── eval_tool_selection.py
├── eval_goal_success_rate.py
└── logs/
    ├── eval_tool_selection.log
    ├── eval_goal_success_rate.log
    └── reports/
        ├── eval_tool_selection_report.json
        └── eval_goal_success_rate_report.json
```

### 2.2 共通ヘルパーの機能一覧

| 機能カテゴリ | 目的 | 入力 | 出力 |
|------------|------|------|------|
| テレメトリ管理 | エージェント実行軌跡（スパン）の記録・取得・クリア | - | 軌跡データ（InMemorySpanExporter） |
| エージェント生成 | 評価対象エージェント（AG-001）を本番同一構成で初期化 | session_id: str, applicant_name: str | OrchestratorAgent インスタンス |
| 状態管理 | 評価用の invocation_state 生成 | session_id: str, applicant_name: str | dict（applicant_name, application_date, session_id） |
| フック制御 | HumanApprovalHook のブロッキング動作を自動承認（OK）に差し替え | - | - |
| 会話シミュレーション | マルチターン会話の実行（ユーザーシミュレーター駆動） | agent, case, state | 会話履歴（list[dict]）、最終応答（str） |

### 2.3 評価フレームワークから利用する概念

| 抽象概念 | 用途 | 採用フレームワークでの実装 |
|---------|------|------------------------|
| テストケース | テスト入力・メタデータの定義 | `strands_evals.Case` |
| 評価器（ツール選択精度） | 申請種別が適切に分類されることのスコア算出 | `strands_evals.evaluators.ToolSelectionAccuracyEvaluator` |
| 評価器（目標達成率） | 経費申請のゴールが達成されることのスコア算出 | `strands_evals.evaluators.GoalSuccessRateEvaluator` |
| 実験 | バッチ実行単位 | `strands_evals.Experiment` |
| セッションマッパー | テレメトリ→評価可能構造への変換 | `strands_evals.mappers.StrandsInMemorySessionMapper` |
| ユーザーシミュレーター | マルチターン会話の自動生成 | `strands_evals.ActorSimulator` |
| テレメトリ | 実行軌跡の記録・取得 | `strands_evals.telemetry.StrandsEvalsTelemetry` |

## 3. テストケース定義

### 3.1 テストケースの構成要素

| 要素 | 型 | 必須条件 | 説明 |
|------|---|---------|------|
| 識別名 | string | 必須 | テストケースの一意識別子 |
| 入力 | string | 必須 | ユーザー入力テキスト（シングルターン：1件、マルチターン：1ターン目） |
| タスク説明 | string | 必須 | テストの目的説明 |
| ゴール基準 | string | セッションレベル時に必須 | GoalSuccessRateEvaluator が判定する達成基準 |
| 期待ツール | string | ツールレベル時に必須 | ToolSelectionAccuracyEvaluator が検証する期待ツール名 |
| メタデータ | dict | 任意 | 評価器が参照する追加情報 |

### 3.2 ケース一覧（eval_tool_selection.py）

> 実行方式: シングルターン  
> 評価器: `strands_evals.evaluators.ToolSelectionAccuracyEvaluator`  
> 判定観点: AG-001 が申請内容を受け取り、transport_agent または expense_agent を適切に選択すること

| 識別名 | 入力 | タスク説明 | 期待ツール |
|--------|------|----------|----------|
| `tool_selection_transport_basic` | 昨日、営業訪問のため渋谷から新宿まで電車で移動しました（運賃170円）。交通費を精算したいです。 | 交通費精算申請に対して transport_agent が選択されることを確認する | `transport_agent` |
| `tool_selection_expense_stationery` | 業務用のボールペンとノートを購入しました。領収書があります。経費精算をお願いします。 | 経費精算申請（事務用品費）に対して expense_agent が選択されることを確認する | `expense_agent` |
| `tool_selection_expense_accommodation` | 先週の出張で東京から横浜へ移動し（運賃480円）、ホテルに宿泊しました。宿泊費の経費精算をお願いします。 | 経費精算申請（宿泊費）に対して expense_agent が選択されることを確認する | `expense_agent` |

### 3.3 ケース一覧（eval_goal_success_rate.py）

> 実行方式: マルチターン  
> 評価器: `strands_evals.evaluators.GoalSuccessRateEvaluator`  
> 判定観点: AG-001→AG-003 の連携により Excel 申請書が正しいデータで生成されること

| 識別名 | 入力（1ターン目） | タスク説明 | ゴール基準 |
|--------|----------------|----------|----------|
| `goal_transport_single_route` | 昨日、営業訪問のため渋谷から新宿まで電車で移動しました（運賃170円）。交通費精算をお願いします。 | 交通費精算申請書（Excel）が正しいデータで生成されることを確認する | 交通費精算申請書ファイルが output/ に生成され、申請者名・移動日・出発地（渋谷）・目的地（新宿）・交通手段（電車）・交通費（170円）・業務目的が正しく記載されていること |
| `goal_expense_stationery` | 先週、業務用の文房具（ボールペン3本、ノート2冊）を購入しました。合計1,500円です。経費精算をお願いします。 | 経費精算申請書（Excel）が正しいデータで生成されることを確認する | 経費精算申請書ファイルが output/ に生成され、申請者名・購入日・品目・経費区分（事務用品費）・金額（1,500円）・業務目的・合計金額が正しく記載されていること |
| `goal_transport_multi_routes` | 今週月曜日に渋谷→新宿（電車・170円）、火曜日に新宿→池袋（電車・170円）の2区間を移動しました。交通費精算をお願いします。 | 複数区間の交通費精算申請書が正しいデータで生成されることを確認する | 交通費精算申請書ファイルが output/ に生成され、2区間（渋谷→新宿170円、新宿→池袋170円）の移動明細・合計金額（340円）が正しく記載されていること |

### 3.4 評価観点トレーサビリティ

| 評価観点 | 対応指標 | 評価レベル | 検証対象ケース | 確認内容 |
|---------|---------|-----------|-------------|---------|
| ツール利用の適切性（申請種別判断） | MET-001 | 出力レベル | tool_selection_transport_basic, tool_selection_expense_stationery, tool_selection_expense_accommodation | 申請内容に応じて transport_agent / expense_agent が正しく選択されること |
| 有用性（目標達成） | MET-002 | セッションレベル | goal_transport_single_route, goal_expense_stationery, goal_transport_multi_routes | 申請書ファイルが正常生成されセッションゴールが達成されること |

## 4. 評価器設計

### 4.1 評価器一覧

| 評価スクリプト | 評価器クラス | 判定入力 | スコアリング方式 | 出力形式 |
|-------------|-----------|---------|---------------|---------|
| eval_tool_selection.py | `strands_evals.evaluators.ToolSelectionAccuracyEvaluator` | エージェントが呼び出したツール名 | 期待ツールと一致→1.0、不一致→0.0 | `{score: float, pass: bool, reason: str}` |
| eval_goal_success_rate.py | `strands_evals.evaluators.GoalSuccessRateEvaluator` | セッション全体の会話履歴 + output/ ディレクトリのファイル存在確認結果 | ゴール基準（申請書ファイル生成完了）に対して達成/未達成を判定し、達成→1.0、未達成→0.0 | `{score: float, pass: bool, reason: str}` |

### 4.2 各評価器の設計

#### eval_tool_selection.py 用評価器（ToolSelectionAccuracyEvaluator）

| 項目 | 値 |
|------|------|
| クラス | `strands_evals.evaluators.ToolSelectionAccuracyEvaluator` |
| 判定観点 | AG-001 が申請内容に応じて transport_agent / expense_agent を正しく選択したか |
| スコア尺度 | 二値（一致: 1.0 / 不一致: 0.0） |
| 期待ツールの参照元 | テストケースの `expected_tool` フィールド |
| 判定根拠の出力 | reason 文字列として出力（レポートに必ず含める） |

#### eval_goal_success_rate.py 用評価器（GoalSuccessRateEvaluator）

| 項目 | 値 |
|------|------|
| クラス | `strands_evals.evaluators.GoalSuccessRateEvaluator` |
| 判定モデル | us.amazon.nova-pro-v1:0 |
| 判定観点 | 申請書ファイル（Excel）が output/ ディレクトリに正常生成され、申請者名・申請日・明細・合計金額が正しく記載されているか |
| スコア尺度 | 二値（達成: 1.0 / 未達成: 0.0） |
| 判定基準の参照元 | テストケースの `goal`（ゴール基準）フィールド |
| 判定根拠の出力 | reason 文字列として出力（レポートに必ず含める） |

## 5. タスク実行設計

### 5.1 シングルターン実行フロー（eval_tool_selection.py）

| 順序 | 処理概要 | 入力 | 出力 |
|------|---------|------|------|
| [1] | テレメトリ初期化（前ケースのスパンをクリア） | - | 初期化済み `strands_evals.telemetry.StrandsEvalsTelemetry` |
| [2] | 評価対象エージェント生成（AG-001 を本番同一構成で初期化） | session_id: str, applicant_name: str | OrchestratorAgent インスタンス |
| [3] | 単一ユーザー入力の送信（テストケースの `input` を1回だけ送信） | テストケース（input）, invocation_state | エージェント応答（str） |
| [4] | 呼び出しツール名の取得（テレメトリスパンから選択されたツール名を抽出） | `strands_evals.telemetry.StrandsEvalsTelemetry` | 呼び出しツール名（str） |
| [5] | 結果構築 | エージェント応答, 呼び出しツール名 | `{output: str, tool_used: str}` |

### 5.2 マルチターン実行フロー（eval_goal_success_rate.py）

| 順序 | 処理概要 | 入力 | 出力 |
|------|---------|------|------|
| [1] | テレメトリ初期化（前ケースのスパンをクリア） | - | 初期化済み `strands_evals.telemetry.StrandsEvalsTelemetry` |
| [2] | 評価対象エージェント生成（AG-001 を本番同一構成で初期化） | session_id: str, applicant_name: str | OrchestratorAgent インスタンス |
| [3] | ユーザーシミュレーター生成（ゴール基準を渡す） | テストケース（goal, input）, 最大ターン数: 30 | `strands_evals.ActorSimulator` インスタンス |
| [4] | セッションマッパー初期化 | session_id: str | `strands_evals.mappers.StrandsInMemorySessionMapper` インスタンス |
| [5] | 会話ループ実行（シミュレーターが発話→エージェントが応答を繰り返す） | エージェント, シミュレーター, invocation_state | 会話履歴（list[dict]）, 最終応答（str） |
| [6] | 実行軌跡の取得（セッションマッパーでスパンを評価可能構造に変換） | `strands_evals.telemetry.StrandsEvalsTelemetry` | セッション構造（`strands_evals.mappers.StrandsInMemorySessionMapper` 出力） |
| [7] | 結果構築 | 最終応答, 会話履歴, セッション構造 | `{output: str, conversation: list[dict], session: object}` |

### 5.3 ユーザーシミュレーター設計

| 項目 | 値 |
|------|------|
| クラス | `strands_evals.ActorSimulator` |
| シミュレーターの役割 | ゴール基準に基づき動的なユーザー発話を生成（申請内容入力・不足情報回答・承認選択等） |
| 最大ターン数 | 30ターン（FR-017の対話回数上限に準拠） |
| 会話終了条件 | ゴール達成判定（申請書ファイル生成完了 or 期待応答の確認）または最大ターン到達 |
| エージェント例外時の動作 | 例外内容を記録してループ終了 |
| シミュレーター例外時の動作 | ループ終了 |

### 5.4 エージェント生成方針

| 項目 | 方針 |
|------|------|
| 本番との同一性 | モデルID（jp.anthropic.claude-sonnet-4-5-20250929-v1:0）・システムプロンプト・ツール構成・ハンドラー構成を本番と一致させる |
| テレメトリ属性 | session_id を `strands_evals.telemetry.StrandsEvalsTelemetry` のスパン属性に紐付け、ケース間のデータ混入を防止する |
| ストリーミング | 無効（callback_handler=None で同期実行） |
| ブロッキング処理 | HumanApprovalHook を自動承認パッチ（OK選択）に差し替え、generate_expense_report / generate_transport_report の実行を許可する |

## 6. 実験実行設計

### 6.1 実験構成

| 評価スクリプト | テストケース群 | 評価器 | タスク関数 |
|-------------|-------------|---------|----------|
| eval_tool_selection.py | tool_selection_transport_basic, tool_selection_expense_stationery, tool_selection_expense_accommodation | `strands_evals.evaluators.ToolSelectionAccuracyEvaluator` | `run_tool_selection_task()` |
| eval_goal_success_rate.py | goal_transport_single_route, goal_expense_stationery, goal_transport_multi_routes | `strands_evals.evaluators.GoalSuccessRateEvaluator` | `run_goal_success_task()` |

### 6.2 実行順序

| 順序 | 処理 | 説明 |
|------|------|------|
| [1] | 環境初期化 | 環境変数（AWS_REGION, AWS_DEFAULT_REGION）・ログ設定（UTF-8, evals/logs/）・SDKログ抑制の準備 |
| [2] | テレメトリ初期化 | `strands_evals.telemetry.StrandsEvalsTelemetry` インスタンスの生成 |
| [3] | 評価器初期化 | `ToolSelectionAccuracyEvaluator` または `GoalSuccessRateEvaluator` インスタンスの生成 |
| [4] | 実験構成 | `strands_evals.Experiment` にテストケース群・評価器・タスク関数を組み立て |
| [5] | 評価実行 | 各ケースに対してタスク関数実行（シングルターン or マルチターン）→ 評価器スコアリング |
| [6] | レポート保存 | JSON 形式で evals/logs/reports/ に保存 |
| [7] | レポート表示 | 全体スコア・個別スコア・合否・判定理由をコンソールに出力 |

### 6.3 非決定性への対処

#### 実行レベルの対処（同一ケースの複数回実行）

| 項目 | 値 |
|------|------|
| 実行回数 | 1回（評価テスト方式設計書 2.1 の方針に従い、複数回実行による集計は行わない） |
| スコア集約方式 | 集計なし（1回実行のスコアをそのまま合否判定に使用） |
| 合否判定の適用先 | 1回実行のスコアに対して閾値を適用 |

#### ケースレベルの対処（テストケースの多様性確保）

| 項目 | 値 |
|------|------|
| 各評価観点の最低ケース件数 | MET-001（ツール利用の適切性）：transport_agent / expense_agent の両方を含む3件、MET-002（有用性）：交通費・経費・複数明細の各フローを含む3件 |
| ケースの多様性方針 | 申請種別（交通費/経費）・複数区間・複数明細の各観点を網羅し、実際の社員シナリオに基づく自然な日本語入力を設計する |

## 9. 回帰評価設計

### 9.1 回帰評価トリガーと実行対象

| トリガー | 実行対象スクリプト | 実施内容 |
