## 1. 概要

### 1.1 コンポーネントの目的

各エージェント（AG-001, AG-002, AG-003）が参照する業務ルール・ナレッジを `knowledge/` ディレクトリの Python モジュールとして定義し、システムプロンプトへの組み込み方式を設計する。ナレッジはシステムプロンプトに直接埋め込む方式（RAG 不使用）で提供する。

### 1.2 ナレッジ・ルールの分類

| 分類 | 説明 |
|------|------|
| 静的知識 | 申請先・申請書情報（KN-002）。組織変更時のみ更新 |
| 業務ルール | 申請種別判断ルール（KN-001）・交通費精算業務ルール（KN-003）・経費精算業務ルール（KN-004）。ルール変更時に更新 |
| 判断基準 | 経費区分判断基準（KN-005）。区分変更時に更新 |

### 2.2 KN-001 / KN-002: 申請種別判断ルール・申請先情報

#### 基本情報

| 項目 | 値 |
|------|-----|
| ナレッジID | KN-001, KN-002 |
| ファイルパス | `knowledge/application_policies.py` |
| 関数名 | `get_application_policies()` |
| 戻り値型 | `str` |
| 利用エージェント | AG-001 |

#### ナレッジ内容定義

```python
def get_application_policies() -> str:
    """申請種別判断ルールと申請先情報を返す。

    Returns:
        str: システムプロンプトに組み込む申請ルールテキスト
    """
    return """【申請種別判断ルール】
BRL-01: 申請内容に「業務上の交通費精算（電車・バス・タクシー・飛行機）」に関する内容が含まれる場合
  → 交通費精算申請を案内し、transport_agent ツールを呼び出す

BRL-02: 申請内容に「業務上の経費精算（物品購入・宿泊・資格取得等）」に関する内容が含まれる場合
  → 経費精算申請を案内し、expense_agent ツールを呼び出す

BRL-03: 申請種別を判断できない場合
  → 「交通費精算申請」「経費精算申請」の2択を提示し、ユーザーの選択を受け付ける

BRL-05: 申請内容が交通費精算・経費精算のいずれにも該当しない場合
  → 対象外である旨を説明し、担当者への連絡を案内して業務終了する

【判断優先順序】
1. BRL-01（交通費精算申請）の条件を確認する
2. BRL-02（経費精算申請）の条件を確認する
3. いずれにも該当しない場合は BRL-03（判断不能）として選択肢を提示する
4. 明らかに対象外の場合は BRL-05（対象外）としてエスカレーションを案内する

【申請先・申請書情報】
BRL-06: 交通費精算申請の場合
  → 申請書: 「交通費精算申請書」
  → 申請先: 所属部門の経理担当者

BRL-07: 経費精算申請の場合
  → 申請書: 「経費精算申請書」
  → 申請先: 所属部門の経理担当者"""
```

#### 基本情報

|------|-----|

#### ナレッジ内容定義

```python
def get_transport_policies(deadline_months: int, approval_threshold: int) -> str:
    """交通費精算業務ルールを返す。

    Args:
        deadline_months: 申請期限月数（settings.transport.deadline_months から取得）
        approval_threshold: 上長承認閾値（settings.transport.approval_threshold から取得）

    Returns:
        str: システムプロンプトに組み込む交通費精算業務ルールテキスト
    """
    return f"""【交通費精算業務ルール】
BRL-08: 申請期限
  → 交通費精算申請は経費発生日（移動日）から{deadline_months}ヶ月以内に申請すること
  → 期限超過時はエラーを通知し業務終了する

BRL-10: 上長承認閾値
  → 交通費合計が{approval_threshold:,}円を超える場合は上長の事前承認が必要
  → 通知後も申請書作成は続行する

BRL-12: 交通費自動計算
  → 電車: 運賃データの経路テーブルから出発駅・到着駅で検索して運賃を取得する
  → バス・タクシー・飛行機: 固定運賃データから運賃を取得する
  → 該当経路が存在しない場合: ユーザーに手動での金額入力を案内する

BRL-15: 駅名正規化
  → 入力された駅名の表記ゆれを正規化する（「渋谷駅」→「渋谷」）
  → calculate_transport_fare ツールを呼び出す前に必ず正規化を行う

BRL-16: 業務目的必須
  → すべての移動区間に業務目的の記載が必要

BRL-17: 移動情報の一括確認収集
  → 移動日・出発地・目的地・交通手段・業務目的を一括で確認収集する
  → 費用は自動計算するため確認しない（運賃データに該当経路が存在しない場合のみ確認する）"""
```

| 組み込み対象エージェント | AG-003（交通費精算申請エージェント） |

### 3.2 BRL-01〜BRL-05: 申請種別判断ルール（AG-001）

#### ルール定義

| 項目 | 値 |
|------|-----|
| ルールID | BRL-01〜BRL-05 |
| 名称 | 申請種別判断ルール |
| 対象エージェント | AG-001（申請受付窓口エージェント） |
| 実装場所 | システムプロンプト内テキスト（`knowledge/application_policies.py` の `get_application_policies()` 経由） |
| 優先順位 | BRL-01 → BRL-02 → BRL-03 → BRL-05 の順で評価 |

#### 条件・処理定義

| 条件（IF） | 処理（THEN） |
|-----------|------------|
| 申請内容に業務上の交通費精算（電車・バス・タクシー・飛行機）が含まれる | 交通費精算申請を案内し、transport_agent を呼び出す（BRL-01） |
| 申請内容に業務上の経費精算（物品購入・宿泊・資格取得等）が含まれる | 経費精算申請を案内し、expense_agent を呼び出す（BRL-02） |
| 申請種別を判断できない（精算に関連する） | 「交通費精算申請」「経費精算申請」の2択を提示する（BRL-03） |
| 申請内容が交通費精算・経費精算のいずれにも関連しない | 対象外として担当者へのエスカレーションを案内し業務終了する（BRL-05） |

#### 例外・特殊ケース

| ケース | 対応 |
|--------|------|
| 交通費と経費の両方が含まれる | LLM が主要な申請種別を判断する。判断不能の場合は BRL-03 に従い選択肢を提示する |

### 3.3 BRL-08 / BRL-09: 申請期限チェック（AG-002, AG-003）

#### ルール定義

| 項目 | 値 |
|------|-----|
| ルールID | BRL-08（交通費）, BRL-09（経費） |
| 名称 | 申請期限チェック |
| 対象エージェント | AG-003（BRL-08）, AG-002（BRL-09） |
| 実装場所 | システムプロンプト内テキスト（KN-003/KN-004）+ LLM 判断（プロンプトに基準日を動的埋め込み） |
| 優先順位 | 高（申請書生成前に必ず確認） |

#### 条件・処理定義

| 条件（IF） | 処理（THEN） |
|-----------|------------|
| 経費発生日（移動日/購入日）が申請期限基準日より前 | 申請期限超過エラーを通知し業務終了する |
| 経費発生日（移動日/購入日）が申請期限基準日以降 | 申請処理を続行する |

#### 例外・特殊ケース

| ケース | 対応 |
|--------|------|
| 申請期限基準日の計算失敗 | `calculate_deadline()` が `"要確認"` を返す。LLM がユーザーに確認を促す |

#### 条件・処理定義

| 条件（IF） | 処理（THEN） |
|-----------|------------|

| 経費合計 > 5,000円（BRL-11） | 上長承認が必要な旨を通知し、申請書作成を続行する |
| 閾値以下 | 上長承認不要として申請処理を続行する |

| ルールID | BRL-14 |
| 名称 | 経費区分自動判断 |
| 対象エージェント | AG-002 |
| 実装場所 | システムプロンプト内テキスト（KN-005）+ LLM 判断 |
| 優先順位 | 高 |

#### 条件・処理定義

| 条件（IF） | 処理（THEN） |
|-----------|------------|
| 品目が文房具・コピー用紙・書籍・オフィス用品等 | 経費区分 = 事務用品費 |
| 品目がホテル・旅館等の宿泊費用 | 経費区分 = 宿泊費 |
| 品目が資格試験・受験料・認定費用等 | 経費区分 = 資格精算費 |
| 上記いずれにも該当しない | 経費区分 = その他経費 |

#### 例外・特殊ケース

| ケース | 対応 |
|--------|------|
| 品目が曖昧で区分を判断できない | 「その他経費」として分類する |

## 4. knowledge モジュール設計

### 4.1 ディレクトリ構成

```
knowledge/
├── __init__.py
├── application_policies.py    # KN-001（申請種別判断ルール）・KN-002（申請先情報）
├── transport_policies.py      # KN-003（交通費精算業務ルール）
└── expense_policies.py        # KN-004（経費精算業務ルール）・KN-005（経費区分判断基準）
```

### 4.2 各ファイルのエクスポート定義

| ファイル | エクスポート関数 | 引数 | 戻り値型 | 説明 |
|--------|----------------|------|---------|------|
| `application_policies.py` | `get_application_policies()` | なし | `str` | KN-001, KN-002 のルールテキストを返す |
| `transport_policies.py` | `get_transport_policies(deadline_months, approval_threshold)` | `int, int` | `str` | KN-003 のルールテキストを返す |
| `expense_policies.py` | `get_expense_policies(deadline_months, approval_threshold)` | `int, int` | `str` | KN-004 のルールテキストを返す |
| `expense_policies.py` | `get_expense_category_policies()` | なし | `str` | KN-005 のルールテキストを返す |

### 4.3 モジュール間依存関係

- `knowledge/` 配下のモジュールは他のモジュールに依存しない（純粋なテキスト返却）
- `prompt/` 配下のモジュールが `knowledge/` を参照する（一方向依存）

## 5. メンテナンス設計

### 5.1 ナレッジ更新手順

1. 対象の `knowledge/[domain]_policies.py` ファイルを開く
2. 該当する関数内のルールテキストを修正する
3. 数値（期限月数・金額閾値等）は `settings.py` で管理されているため、`settings.py` を修正する
4. 修正後にシステムプロンプトが正しく生成されることをテストで確認する

### 5.2 ルール更新手順

1. 業務ルール変更の内容を確認する
2. 対象の `knowledge/[domain]_policies.py` の関数内テキストを修正する
3. 数値変更の場合は `config/settings.py` の対応する設定値を修正する
4. システムプロンプト詳細設計書・ナレッジ・業務ルール詳細設計書を更新する

## 6. 補足情報

### 6.1 実装上の注意点

1. **ビジネスルール値のハードコード禁止**
   - 期限月数・金額閾値等の数値は `settings.py` で一元管理し、ポリシー関数の引数として受け取る（R9.12.3）

2. **knowledge/ モジュールは依存なし**
   - `knowledge/` 配下のモジュールは他のモジュールに依存しない純粋なテキスト返却関数のみを定義する

3. **ファイル配置**
   - `knowledge/application_policies.py`: AG-001 用（KN-001, KN-002）
   - `knowledge/transport_policies.py`: AG-003 用（KN-003）
   - `knowledge/expense_policies.py`: AG-002 用（KN-004, KN-005）

## 8. テスト観点

### 8.1 ナレッジ参照テスト
- `get_application_policies()` が申請種別判断ルールと申請先情報を含む文字列を返すこと
- `get_transport_policies(3, 10000)` が交通費精算業務ルールを含む文字列を返すこと
- `get_expense_policies(3, 5000)` が経費精算業務ルールを含む文字列を返すこと
- `get_expense_category_policies()` が経費区分判断基準を含む文字列を返すこと
- 各ポリシー関数の戻り値がシステムプロンプトに正しく組み込まれること

### 8.2 ルール適用テスト
- BRL-01: 交通費精算の申請内容で transport_agent が呼び出されること
- BRL-02: 経費精算の申請内容で expense_agent が呼び出されること
- BRL-03: 判断不能時に選択肢が提示されること
- BRL-08/09: 申請期限超過時にエラーメッセージが表示されること
- BRL-10/11: 上長承認閾値超過時に通知が表示されること
- BRL-14: 品目から経費区分が正しく判断されること

### 8.3 境界値テスト
- BRL-08/09: 申請期限ちょうど3ヶ月前の経費発生日で申請が通過すること

- BRL-10: 交通費合計 10,001円で上長承認通知が表示されること
- BRL-11: 経費合計 5,000円で上長承認通知が表示されないこと（> 5,000円が条件）
- BRL-11: 経費合計 5,001円で上長承認通知が表示されること
