## 1. 概要

### 1.1 コンポーネントの目的

エージェント（AG-001〜AG-003）が申請種別判断・業務ルール適用に使用するナレッジ・業務ルールを `knowledge/` モジュールに一元管理する。RAGは使用せず、全ナレッジはシステムプロンプトに直接埋め込む。業務ルールの数値（申請期限・上長承認閾値等）は `settings.py` で一元管理し、ポリシー関数の引数として動的に展開する。

### 1.2 ナレッジ・ルールの分類

| 分類 | 説明 |
|------|------|
| 業務ルール（申請種別判断） | 申請内容テキストから申請種別を判断するキーワード・条件（KN-001） |
| 業務ルール（交通費精算申請） | 交通費精算申請に関するルール群（KN-002、BRL-01〜BRL-09） |
| 業務ルール（経費精算申請） | 経費精算申請に関するルール群（KN-003、BRL-10〜BRL-17） |
| 業務ルール（経費区分判断） | 品目から経費区分を判断するルール（KN-004、BRL-14） |
| 業務ルール（申請判断基準） | 申請期限チェック・上長承認要否判断の基準（KN-005） |

#### ナレッジ内容定義

```python
def get_orchestrator_policies() -> str:
    """申請種別判断ルールのテキストを返す。"""
    return """
- 交通費精算申請（JD-01）: 申請内容に「交通費」「移動」「電車」「バス」「タクシー」「飛行機」「出張」「通勤」「乗車」「運賃」等のキーワードが含まれる場合

- 判断不能（JD-03）: 上記のいずれにも明確に該当しない場合、または複数に該当する可能性がある場合
"""
```

| ファイルパス | `knowledge/transport_policies.py` |

#### ナレッジ内容定義

```python
def get_transport_policies(deadline_months: int, approval_threshold: int) -> str:
    """交通費精算申請ルールのテキストを返す。

    Args:
        deadline_months: 申請期限（月数）
        approval_threshold: 上長承認が必要な交通費合計の閾値（円）
    """
    return f"""
- BRL-01: 交通費精算申請書を使用する
- BRL-02: 申請先は経理部門
- BRL-03: 対応交通手段は電車・バス・タクシー・飛行機のみ
- BRL-04: 移動情報は1区間ずつ収集する
- BRL-05: 経路テーブルを参照して交通費を自動計算する（未登録時は手動入力）
- BRL-06: 申請期限は経費発生日から{deadline_months}ヶ月以内
- BRL-07: 交通費合計が{approval_threshold:,}円を超える場合は上長承認が必要であることを通知する
- BRL-08: 出発地・目的地の「駅」「バス停」「空港」等の接尾語は自動除去して正規化する
- BRL-09: 業務目的は必須
"""
```

| 組み込み対象エージェント | AG-002（交通費精算申請エージェント） |

### 3.2 BRL-06: 交通費精算の申請期限

#### ルール定義

| 項目 | 値 |
|------|-----|
| ルールID | BRL-06 |
| 名称 | 交通費精算の申請期限 |
| 対象エージェント | AG-002 |
| 実装場所 | システムプロンプト内テキスト（`{deadline_date}` プレースホルダー）+ `calculate_deadline()` |
| 優先順位 | High |

#### 条件・処理定義

| 条件（IF） | 処理（THEN） |
|-----------|------------|
| 移動日 < `deadline_date`（申請日から `deadline_months` ヶ月前） | 「申請期限（90日）を超過しています。申請を続けますか？」と警告を提示する |
| 移動日 >= `deadline_date` | 申請期限内として処理を継続する |

#### 例外・特殊ケース

| ケース | 対応 |
|--------|------|
| ユーザーが期限超過後も申請を希望する場合 | 警告提示後に処理を継続する（EX-03） |

## 4. knowledgeモジュール設計

### 4.1 ディレクトリ構成

```
knowledge/
├── __init__.py
├── orchestrator_policies.py    # 申請種別判断ルール（KN-001）
├── transport_policies.py       # 交通費精算申請ルール（KN-002、KN-005の交通費部分）
└── expense_policies.py         # 経費精算申請ルール（KN-003、KN-004、KN-005の経費部分）
```

### 4.2 各ファイルのエクスポート定義

| ファイル | エクスポート関数 | 引数 | 型 | 説明 |
|--------|----------------|------|-----|------|
| `orchestrator_policies.py` | `get_orchestrator_policies()` | なし | `str` | 申請種別判断ルールテキストを返す |
| `transport_policies.py` | `get_transport_policies(deadline_months, approval_threshold)` | `int, int` | `str` | 交通費精算申請ルールテキストを返す |
| `expense_policies.py` | `get_expense_policies(deadline_months, approval_threshold)` | `int, int` | `str` | 経費精算申請ルールテキストを返す |

### 4.3 モジュール間依存関係

- `knowledge/` モジュールは他の内部モジュールに依存しない（純粋なテキスト返却）
- `prompt/` モジュールが `knowledge/` モジュールをインポートして使用する

## 5. メンテナンス設計

### 5.1 ナレッジ更新手順

1. 対象の `knowledge/[domain]_policies.py` ファイルを開く
2. 対応するポリシー関数のテキストを修正する
3. 数値（申請期限・閾値等）は `settings.py` で管理するため、`knowledge/` ファイルは修正不要
4. 修正後にテストを実行して動作確認する

### 5.2 ルール更新手順

1. 業務ルールの数値変更（申請期限・閾値等）: `config/settings.py` の対応する設定値を変更する
2. 業務ルールのテキスト変更: `knowledge/[domain]_policies.py` の対応するポリシー関数を修正する
3. 新規ルールの追加: 対応するポリシー関数にルールテキストを追加し、必要に応じてPydanticバリデーションも追加する

## 6. 補足情報

### 6.1 実装上の注意点

1. **業務ルール値のハードコード禁止**
   - 申請期限月数・上長承認閾値等の数値は `settings.py` で一元管理する（R9.12.3）
   - ポリシー関数は数値を引数として受け取り、f-stringで動的展開する

2. **RAG不使用**
   - 本プロジェクトではRAGを使用しない
   - 全ナレッジはシステムプロンプトに直接埋め込む

3. **knowledge/ モジュールの依存禁止**
   - `knowledge/` モジュールは他の内部モジュールに依存しない（純粋なテキスト返却）

## 8. テスト観点

- `get_orchestrator_policies()` が申請種別判断ルールテキストを返すこと
